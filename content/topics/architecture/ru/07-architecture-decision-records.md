# Architecture Decision Records (ADRs) — записи архитектурных решений

> **Область применения:** ADR (Architecture Decision Record — запись архитектурного решения) — это практика документирования, а не паттерн кода. Применяется на любом уровне: одного сервиса, платформы, всей организации. Включена в эту серию потому, что архитектурные решения — это только половина работы; фиксирование *причин* их принятия делает кодовую базу понятной следующему инженеру.

## Проблема, которую решают ADR

Спустя шесть месяцев после разработки системы к проекту присоединяется новый инженер и находит кое-что непонятное:

- Сервис использует PostgreSQL и Redis, казалось бы, для перекрывающихся целей
- Аутентификация реализована на собственном JWT вместо Passport.js
- В монорепозитории есть пакет `shared/`, который используют только два из семи сервисов
- Prisma используется для чтения, но для записи — сырой SQL

У инженера два варианта: принять всё как данность и работать вокруг этого, или потратить время на восстановление логики решений. Часто изначальные инженеры уже ушли. Slack-тред, в котором принималось решение, похоронен под 50 000 других сообщений. Описание PR говорит «добавить кэширование Redis», но не «мы выбрали Redis, а не Memcached, потому что...».

Кодовая база фиксирует *что было решено*. Ничто не фиксирует *почему*.

**ADR (Architecture Decision Record)** — короткий структурированный документ, фиксирующий одно архитектурное решение вместе с его контекстом и последствиями. Его цель — ответить на вопрос «почему это так устроено?» для будущих читателей — включая вас самих через шесть месяцев.

## Формат Нигарда — наиболее распространённая структура

Michael Nygard предложил этот формат в 2011 году. Его приняли команды GitHub, Spotify, ThoughtWorks и многих других компаний, потому что он минималистичен и заставляет автора продумать последствия.

```markdown
# ADR-0003: Использовать PostgreSQL как основное хранилище данных

## Статус
Принято

## Контекст
Нам нужно постоянное хранилище для аккаунтов пользователей, заказов и инвентаря.
Данные реляционные (у пользователей есть заказы, у заказов — позиции, позиции ссылаются
на продукты). Команда хорошо знает SQL; у никого нет продакшн-опыта с MongoDB.
Есть строгие требования к консистентности для финансовых транзакций.
Развёртывание на AWS; RDS для PostgreSQL доступен с управляемыми бэкапами и failover.

## Решение
Использовать PostgreSQL как основную реляционную базу данных.
В продакшн-окружении — AWS RDS, локально — Docker-контейнер.
Использовать Prisma как ORM для управления схемой и типизированных запросов.

## Последствия
**Положительные:**
- Реляционная модель хорошо подходит данным; нет impedance mismatch
- ACID-транзакции работают из коробки для связанных таблиц
- Команда уже знает SQL; нет кривой обучения
- Prisma обеспечивает типобезопасность и автогенерацию миграций

**Отрицательные:**
- Горизонтальное масштабирование записи сложнее, чем с NoSQL-базами
  (приемлемо при прогнозируемой нагрузке <10k req/day)
- Миграции схемы требуют аккуратности в продакшн;
  процесс миграций Prisma требует дисциплины в пайплайнах деплоя

**Нейтральные:**
- Мы принимаем, что смена базы данных позже потребует значительных усилий
  на миграцию. Это осознанный trade-off ради консистентности и знакомости командой.
```

Четыре секции:

- **Статус** — одно из: `Предложено` (обсуждается), `Принято` (решено), `Устарело` (больше не применимо), `Заменено ADR-XXXX` (вытеснено более поздним решением). Статус позволяет читателям понять, активно ли это решение.
- **Контекст** — какова была ситуация, когда принималось это решение? Какие ограничения существовали? Какие варианты рассматривались? Эта секция «датирует» ADR: читатель через 2 года может понять, актуален ли контекст.
- **Решение** — само решение. Одно конкретное, утвердительное утверждение. «Мы будем использовать X», а не «X кажется хорошим выбором».
- **Последствия** — что становится проще, сложнее или иным в результате этого решения? Включайте и положительные, и отрицательные. Именно эта секция делает ADR ценнее, чем простая заметка «мы выбрали X».

## Второй пример — решение об архитектурном паттерне

ADR нужны не только для выбора технологий. Они не менее полезны для решений о паттернах:

```markdown
# ADR-0007: Использовать модульный монолит вместо микросервисов на старте

## Статус
Принято

## Контекст
Мы разрабатываем e-commerce платформу к запуску в Q3. Размер команды — 4 инженера.
Мы обсуждали разбивку на микросервисы (Orders, Users, Inventory, Notifications).

Ключевые ограничения:
- 4 инженера не могут одновременно поддерживать 4+ отдельных пайплайна деплоя,
  настройки мониторинга и контракты сервисов при разработке фич
- Бизнес-требования ещё уточняются; границы сервисов нестабильны
- Прогнозируемая нагрузка на старте <500 concurrent users; горизонтального
  масштабирования одного сервиса хватит на 12+ месяцев
- У нас нет Kubernetes и оркестрации контейнеров

Рассмотренные варианты:
1. Микросервисы с самого начала
2. Модульный монолит с явными границами модулей (выбрано)
3. Неструктурированный монолит

## Решение
Мы будем строить модульный монолит. Каждый бизнес-домен (Orders, Users, Inventory,
Notifications) будет отдельным модулем с явным публичным API (экспорты через index.ts).
Межмодульные импорты разрешены только через публичный API. Прямой доступ к базе данных
между модулями запрещён.

Мы пересмотрим это решение при выполнении любого из условий:
- Команда вырастет до 8+ инженеров
- Требования к масштабированию модуля кардинально разойдутся с остальными
- Независимый деплой модуля ощутимо снизит накладные расходы на координацию

## Последствия
**Положительные:**
- Один пайплайн деплоя, одна настройка мониторинга, одна локальная среда
- ACID-транзакции работают между модулями (одна база данных)
- Изменение границ модулей — перемещение файлов, не распределённая миграция
- Новые инженеры онбордятся в одну кодовую базу

**Отрицательные:**
- Нельзя масштабировать модули независимо (приемлемо при прогнозах нагрузки)
- Баг в одном модуле может влиять на другие (снижается изоляцией тестов на уровне модуля)

**Нейтральные:**
- При необходимости вынести модуль в микросервис явная граница в модульном монолите
  упрощает это: публичный API модуля становится API-контрактом сервиса; его слой
  доступа к данным — репозиторием сервиса
```

## Где хранить ADR

Универсальное соглашение: директория `docs/adr/` в репозитории, последовательно пронумерованные файлы.

```txt
docs/
└── adr/
    ├── 0001-use-nodejs-and-typescript.md
    ├── 0002-use-nestjs-as-framework.md
    ├── 0003-use-postgresql-as-primary-data-store.md
    ├── 0004-use-redis-for-session-caching.md
    ├── 0005-use-jest-for-testing.md
    ├── 0006-use-github-actions-for-ci.md
    └── 0007-modular-monolith-instead-of-microservices.md
```

Почему в репозитории (а не в Confluence, не в Notion):

1. **ADR живут рядом с кодом, который они описывают** — читая код, вы можете добраться до ADR через `git log`
2. **ADR версионируются** — видно, когда решение изменилось, и можно связывать коммиты с ADR
3. **ADR проходят ревью в PR** — pull request с «переходом с Passport.js на кастомный JWT» может включать новый ADR как часть изменения

Нумерация намеренно дополняется нулями (`0001`, а не `1`), чтобы файлы корректно сортировались в файловых системах с лексикографической сортировкой.

## Когда писать ADR

Не каждое решение требует ADR. Тест:

> «Будет ли компетентный инженер, вступающий в проект через 6 месяцев, озадачен тем, почему принято это решение, или захочет отменить его, не зная полного контекста?»

Если да — пишите ADR.

**Пишите ADR для:**
- Выбора технологий (база данных, фреймворк, ORM, очередь, кэш)
- Выбора архитектурных паттернов (слоёный vs чистая vs гексагональная, монолит vs микросервисы)
- Решений с явными trade-off'ами, неочевидными из кода
- Решений *не* использовать что-то, что иначе было бы очевидным выбором («мы оценили GraphQL, но выбрали REST, потому что...»)
- Всего, что обсуждалось в команде и было разрешено в определённом направлении

**Не пишите ADR для:**
- Деталей реализации, понятных из чтения кода
- Решений, тривиально обратимых (какую библиотеку логирования использовать для прототипа)
- Соглашений о стиле кода (используйте `.eslintrc` и секцию README)
- Решений, полностью обусловленных «это было по умолчанию», без значимых рассмотренных альтернатив

## Обновление и замена ADR

ADR **неизменны по умолчанию** — старый ADR не редактируется для изменения решения. Пишется новый ADR, который вытесняет старый.

```markdown
# ADR-0011: Заменить Passport.js собственным JWT-middleware

## Статус
Принято — заменяет ADR-0004

## Контекст
ADR-0004 выбрал Passport.js для аутентификации. С тех пор:
- Мы добавили мультитенантность, требующую per-tenant JWT секретов
- Система стратегий Passport.js добавила сложность, не подходящую нашей
  мультитенантной модели
- Аудит безопасности выявил, что сериализация сессий Passport не нужна для
  нашего stateless API (мы не используем сессии)

## Решение
Заменить Passport.js собственным JWT verification middleware, используя библиотеку
`jsonwebtoken` напрямую. Middleware проверяет токен, устанавливает `req.user`,
обрабатывает поиск per-tenant ключей.

## Последствия
...
```

И обновить старый ADR:

```markdown
# ADR-0004: Использовать Passport.js для аутентификации

## Статус
Заменено ADR-0011
```

Полная история сохранена. Читатель может отследить эволюцию: почему изначально выбрали Passport.js? Почему его потом заменили? Оба ответа доступны без чтения `git blame`.

## Лёгкий вариант — Y-statement

Для команд, которым формат Нигарда кажется громоздким, **Y-statement** фиксирует суть в одном предложении:

```
В контексте <ситуация>,
столкнувшись с <проблема>,
мы решили <вариант>,
чтобы достичь <качество>,
принимая <недостаток>.
```

Пример:
```
В контексте необходимости кэширующего слоя для дорогих запросов к БД,
столкнувшись с высокой латентностью чтения на эндпоинте сводки заказов,
мы решили использовать Redis с TTL 60 секунд,
чтобы достичь времени ответа <100ms для дашборда,
принимая, что кэшированные данные могут устаревать на 60 секунд.
```

Y-statement хорошо работает как inline-комментарий в коде или как первый набросок перед полноценным ADR:

```ts
// ADR-0009 (Y-statement): В контексте webhook'ов статуса заказов,
// столкнувшись с ненадёжной доставкой от платёжных провайдеров,
// мы решили использовать idempotency keys в обработчике,
// чтобы достичь exactly-once обработки,
// принимая дополнительный Redis lookup на каждый webhook.
export async function handlePaymentWebhook(req: Request): Promise<void> {
  const idempotencyKey = req.headers['x-idempotency-key'] as string;
  const alreadyProcessed = await redis.set(idempotencyKey, '1', 'NX', 'EX', 86400);
  if (!alreadyProcessed) return; // уже обработан
  // ...
}
```

## Типичные ошибки на интервью

- **«ADR нужны только для больших решений вроде выбора базы данных»** — ADR для любого решения, о котором будущий инженер спросит «почему?». «Почему у нас собственная обёртка для retry вместо axios-retry?» — валидный ADR. «Почему модуль Orders не вызывает сервис Users напрямую, а читает данные пользователя из денормализованной колонки?» — валидный ADR. Область применения — любое неочевидное решение, а не только те, что попадают на архитектурные диаграммы.

- **«Мы документируем решения в Confluence / Notion»** — проблема не в инструменте, а в дистанции от кода. Документация в отдельной системе постепенно расходится с кодом по мере его изменений, если никто не обновляет её. ADR в репозитории читаются при изменении кода (потому что инженер уже в репо) и обновляются в том же PR, что и изменение. Это ключевое свойство, а не формат markdown.

- **«ADR устаревает при смене решения, значит это накладные расходы на поддержку»** — устаревший ADR (со статусом `Заменено ADR-XXXX`) не бесполезен. Это исторический контекст: он говорит, что было решено, когда, и почему потом изменилось. Именно это нельзя восстановить из истории Slack или сообщений коммитов. Стоимость написания «Заменено ADR-XXXX» в поле статуса ничтожна по сравнению с ценностью сохранённого контекста.

- **«ADR замедляют быстро движущиеся команды»** — написание ADR после архитектурного обсуждения занимает 20–30 минут. Восстановление логики решения 12 месяцев спустя (новый инженер, миграция, отладка странного ограничения) может занять часы или дни — а может оказаться невозможным, если люди ушли. Инвестиция крайне асимметрична.

- **«Если код чистый, ADR не нужны»** — чистый код объясняет *что* он делает. Чистая архитектура объясняет *как* она устроена. Ни то ни другое не объясняет *почему* была выбрана конкретная технология или паттерн, а не разумные альтернативы. «Мы используем Redis для кэширования» очевидно из кода. «Мы выбрали Redis, а не Memcached, потому что нам также нужен pub/sub для инвалидации, а Memcached не поддерживает pub/sub» — нигде не видно кроме ADR.
