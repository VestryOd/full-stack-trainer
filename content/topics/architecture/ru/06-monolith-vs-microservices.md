# Монолит vs Микросервисы

> **Область применения:** Эта статья находится на стыке архитектуры приложения (статьи 01–05) и системного дизайна. В отличие от предыдущих статей, она касается решений о том, сколько деплоящихся единиц иметь — это решение системного уровня. Она включена здесь потому, что напрямую вытекает из модульного проектирования приложений: понимание границ модулей помогает решить, когда (и стоит ли вообще) разбивать их на отдельные сервисы.

## Спектр — от одной деплоящейся единицы к многим

«Монолит vs микросервисы» часто подаётся как бинарный выбор. На самом деле это спектр:

```txt
Big Ball of Mud → Слоёный монолит → Модульный монолит → SOA → Микросервисы
                                              ↑
                              Большинству команд стоит здесь остановиться,
                              прежде чем двигаться правее
```

Решение — не просто о разбиении на сервисы. Оно о том, где ваше приложение находится на этом спектре и стоит ли ценой перехода правее на данном этапе жизни проекта.

## Виды монолитов

Не все монолиты одинаковы. Термин «монолит» описывает модель деплоя (одна деплоящаяся единица), а не внутреннее качество.

### Big Ball of Mud — «большой ком грязи»

Никакой намеренной структуры. Бизнес-логика, доступ к данным, HTTP-обработка — всё перемешано. Быстро стартует, мучительно поддерживается, нормально не тестируется. Именно для предотвращения этого существуют паттерны из статей 01–05.

### Слоёный монолит

Одна деплоящаяся единица, организованная по слоям (Presentation → Service → Repository). Это самая распространённая продакшн-архитектура для малых и средних приложений. Именно так выглядит каждый NestJS или Express проект, если применить паттерны из этой серии.

```txt
Одна деплоящаяся единица: my-app.js / dist/
├── controllers/ (HTTP-слой)
├── services/ (бизнес-логика)
├── repositories/ (доступ к данным)
└── domain/ (типы и правила)

Одна база данных, один пайплайн деплоя, один поток логов.
```

**Сильные стороны:** просто разрабатывать, просто деплоить, просто дебажить. Транзакции элементарны — всё работает в одном процессе. Общий код — просто импорт. Никакой сетевой латентности между «сервисами».

**Слабые стороны:** нельзя масштабировать отдельные части независимо. Весь сервис нужно задеплоить, чтобы изменить один роут. Утечка памяти в одном модуле роняет всё приложение.

### Модульный монолит

Одна деплоящаяся единица, в которой внутренние модули явно изолированы — у каждого модуля есть определённый публичный API, и другие модули не могут залезть в его внутренности. Это идеальная точка остановки для большинства приложений перед переходом (если вообще) к микросервисам.

```txt
my-app/
├── modules/
│   ├── orders/
│   │   ├── index.ts          ← публичный API: что другие модули МОГУТ импортировать
│   │   ├── orders.service.ts
│   │   ├── orders.repository.ts
│   │   └── domain/
│   ├── users/
│   │   ├── index.ts          ← публичный API
│   │   ├── users.service.ts
│   │   └── domain/
│   └── notifications/
│       ├── index.ts          ← публичный API
│       └── ...
└── main.ts
```

Ключевая дисциплина: другие модули могут импортировать только из `orders/index.ts`, никогда напрямую из `orders/orders.service.ts`. Это зеркалит границу интерфейса, которую бы обеспечивал микросервис — но без операционных затрат.

```ts
// modules/orders/index.ts — явный публичный API
export type { Order, OrderStatus } from './domain/order';
export { OrdersService } from './orders.service';
export { ORDER_REPOSITORY_TOKEN } from './orders.repository';
// НЕ экспортируется: детали внутренней реализации

// modules/users/users.service.ts
// ✅ Разрешено: импорт из публичного API модуля orders
import type { Order } from '../orders'; // резолвится в orders/index.ts

// ❌ Запрещено: залезать во внутренности модуля
// import { PrismaOrderRepository } from '../orders/orders.repository';
```

Модульный монолит проще тестировать, проще рефакторить и — критически важно — проще впоследствии вынести в микросервис при реальной необходимости. Граница модуля уже есть; остаётся лишь добавить сетевой вызов и отдельный деплой.

## Микросервисы

**Микросервис** — независимо деплоящийся сервис, который владеет одним ограниченным контекстом (bounded context — связным набором бизнес-возможностей) и своей собственной базой данных.

```txt
API Gateway / Load Balancer
       │
       ├──► Orders Service (порт 3001, владеет orders_db)
       ├──► Users Service (порт 3002, владеет users_db)
       ├──► Notifications Service (порт 3003, без персистентной БД)
       └──► Inventory Service (порт 3004, владеет inventory_db)
```

Правило «владеет своей базой данных» — не опциональное в настоящей микросервисной архитектуре. Если два сервиса делят одну базу — они не независимы: миграция схемы в одном ломает другой. На практике сервисы общаются:

1. **Синхронными вызовами** — REST или gRPC. Один сервис вызывает другой и ждёт ответа.
2. **Асинхронными сообщениями** — Kafka, RabbitMQ, SQS. Один сервис публикует событие; другие подписываются.

### Синхронный пример — HTTP между сервисами

```ts
// orders-service: вызывает users service для проверки кредита
// src/services/orders.service.ts (внутри микросервиса Orders)
export class OrdersService {
  constructor(
    private orderRepository: IOrderRepository,
    private usersClient: IUsersServiceClient, // HTTP-клиент к сервису Users
  ) {}

  async placeOrder(userId: string, total: number): Promise<Order> {
    // Сетевой вызов — может упасть, может быть медленным, есть латентность
    const user = await this.usersClient.getUserById(userId);
    if (!user) throw new Error(`User ${userId} not found`);
    if (user.creditLimit < total) throw new InsufficientCreditError(user.creditLimit, total);

    return this.orderRepository.save({ ... });
  }
}
```

```ts
// HTTP-клиент оборачивает сетевой вызов и предоставляет типизированный интерфейс
// src/clients/users-service.client.ts
export class UsersServiceClient implements IUsersServiceClient {
  private baseUrl = process.env.USERS_SERVICE_URL!;

  async getUserById(id: string): Promise<User | null> {
    const res = await fetch(`${this.baseUrl}/users/${id}`);
    if (res.status === 404) return null;
    if (!res.ok) throw new Error(`Ошибка Users service: ${res.status}`);
    return res.json();
  }
}
```

### Асинхронный пример — событийное взаимодействие

```ts
// orders-service: публикует событие после создания заказа
// orders.service.ts (внутри микросервиса Orders)
async placeOrder(userId: string, total: number): Promise<Order> {
  const order = await this.orderRepository.save({ ... });

  // Fire-and-forget: публикуем событие; notifications service подберёт его сам
  await this.eventBus.publish('order.placed', {
    orderId: order.id,
    userId,
    total,
  });

  return order;
}

// notifications-service: независимо подписывается на событие
// handlers/order-placed.handler.ts (внутри микросервиса Notifications)
export class OrderPlacedHandler {
  async handle(event: OrderPlacedEvent): Promise<void> {
    const user = await this.usersClient.getUserById(event.userId);
    await this.emailService.sendConfirmation(user.email, event.orderId);
  }
}
```

## Сравнение trade-off'ов — честно

```txt
┌───────────────────────┬──────────────────────────────┬────────────────────────────────────┐
│                       │      Монолит                 │       Микросервисы                 │
├───────────────────────┼──────────────────────────────┼────────────────────────────────────┤
│ Скорость разработки   │ Быстро — прямые импорты,     │ Медленно — отдельные репо,         │
│                       │ один репо, один тест-сьют    │ версионированные API, инт. тесты   │
├───────────────────────┼──────────────────────────────┼────────────────────────────────────┤
│ Деплой                │ Один пайплайн, один артефакт │ N пайплайнов, N артефактов,        │
│                       │                              │ оркестрация контейнеров (K8s)      │
├───────────────────────┼──────────────────────────────┼────────────────────────────────────┤
│ Операционная          │ Один поток логов,            │ Distributed tracing,               │
│ сложность             │ одна дашборда метрик         │ централизованные логи, N дашбордов │
├───────────────────────┼──────────────────────────────┼────────────────────────────────────┤
│ Масштабирование       │ Масштабируется всё приложение│ Масштабируется только узкое место  │
├───────────────────────┼──────────────────────────────┼────────────────────────────────────┤
│ Транзакции            │ ACID — одна БД, один процесс │ Нет ACID между сервисами;          │
│                       │                              │ Saga или eventual consistency      │
├───────────────────────┼──────────────────────────────┼────────────────────────────────────┤
│ Изоляция отказов      │ Один сбой может уронить всё  │ Сбои изолированы в одном сервисе   │
├───────────────────────┼──────────────────────────────┼────────────────────────────────────┤
│ Выбор технологий      │ Один язык, один рантайм      │ Каждый сервис — свой язык/БД       │
│                       │                              │ (редко стоит того)                 │
├───────────────────────┼──────────────────────────────┼────────────────────────────────────┤
│ Автономия команды     │ Все в одном репо;            │ Команды деплоят независимо;        │
│                       │ нужна координация            │ меньше межкомандной координации    │
└───────────────────────┴──────────────────────────────┴────────────────────────────────────┘
```

Свобода выбора технологий («каждый сервис на своём языке») часто фигурирует как преимущество микросервисов. На практике разнородные технологические стеки умножают операционную нагрузку с минимальной отдачей. Большинство компаний стандартизируются на 1–2 языках вне зависимости от архитектуры.

## Закон Конвея — почему оргструктура определяет архитектуру

**Закон Конвея** (Melvin Conway, 1967):
> «Организации, проектирующие системы, вынуждены создавать дизайны, которые являются копиями коммуникационных структур этих организаций».

Проще говоря: если системой владеют три команды, система будет состоять из трёх крупных компонентов. Если одна команда владеет всем — система будет монолитом. Архитектура следует за границами команды, а не наоборот.

Из этого есть практическое следствие: микросервисы оправданы, когда есть несколько независимых команд, которым нужно деплоить без координации друг с другом. Для одной команды (или небольшого стартапа) микросервисы добавляют накладные расходы на координацию, не давая взамен преимущества автономии — не от кого быть автономными.

**Inverse Conway Maneuver** («обратный манёвр Конвея») — намеренное структурирование команд под желаемую архитектуру — это реальная стратегия: если хотите микросервисы, сначала организуйте команды по границам сервисов, и архитектура последует.

## Паттерн Strangler Fig — миграция с монолита на микросервисы

**Паттерн Strangler Fig** (назван в честь вида инжира, который оплетает дерево-хозяина и в итоге замещает его) — рекомендуемый способ инкрементально мигрировать существующий монолит на микросервисы.

```txt
Фаза 1: монолит обрабатывает всё
        [Клиент] → [Монолит]

Фаза 2: добавлен прокси/gateway, одна возможность вынесена
        [Клиент] → [Gateway] → [Монолит] (большинство трафика)
                             → [Новый сервис] (вынесенная возможность)

Фаза 3: ещё возможности вынесены
        [Клиент] → [Gateway] → [Users Service]
                             → [Orders Service]
                             → [Монолит] (сжимается — оставшиеся возможности)

Фаза 4: монолит исчез (или настолько мал, что стал просто legacy-адаптером)
        [Клиент] → [Gateway] → [Users Service]
                             → [Orders Service]
                             → [Notifications Service]
```

Ключевое: клиент никогда не меняет свой API. Gateway направляет запросы туда, где теперь живёт каждая возможность. Монолит постепенно сжимается; новые сервисы растут.

```ts
// Пример: маршрутизация gateway в процессе перехода
// api-gateway/src/routes.ts
app.use('/users', proxy({ target: process.env.USERS_SERVICE_URL }));  // вынесено
app.use('/orders', proxy({ target: process.env.ORDERS_SERVICE_URL })); // вынесено
app.use('/', proxy({ target: process.env.MONOLITH_URL }));             // всё остальное ещё в монолите
```

## Когда что выбирать

**Начинайте с модульного монолита, когда:**
- Команда меньше ~15 инженеров
- Ограниченные контексты (бизнес-домены) ещё не ясны — требования продолжают уточняться
- Нет операционной инфраструктуры для нескольких деплоящихся единиц (Kubernetes, мониторинг, distributed tracing)
- Приложение новое — цена ошибки в границах сервисов огромна, а изменения в монолите дёшевы

**Рассматривайте микросервисы, когда:**
- Несколько независимых команд должны деплоить, не блокируя друг друга
- Конкретная часть системы имеет кардинально разные требования к масштабированию (например, сервис обработки видео vs сервис профилей пользователей)
- Регуляторные или требования безопасности обязывают изолировать данные на уровне инфраструктуры
- У вас хорошо понятый домен с чёткими, стабильными ограниченными контекстами
- Есть (или вы готовы построить) платформа: оркестрация контейнеров, distributed tracing, централизованное логирование, API gateway, CI/CD для каждого сервиса

**Признаки, что на микросервисы перешли слишком рано:**
- Сервисы постоянно вызывают друг друга синхронно (они должны быть вместе)
- Изменения требуют деплоя 3+ сервисов одновременно (граница проведена неправильно)
- Каждому разработчику нужно запускать 5+ сервисов локально, чтобы что-то протестировать
- Транзакции охватывают несколько сервисов и требуют паттерна Saga для операций, которые раньше были одной транзакцией в БД

## Типичные ошибки на интервью

- **«Микросервисы всегда лучше для масштабирования»** — монолит можно масштабировать горизонтально (несколько инстансов за балансировщиком). Микросервисы позволяют масштабировать *части* независимо — ценно только тогда, когда у частей кардинально разные профили нагрузки. Горизонтальное масштабирование монолита часто достаточно и несравнимо проще.

- **«Разница между модульным монолитом и микросервисами — только деплой»** — разница в деплое реальная, но вторичная. Более глубокое различие — владение данными: модули в монолите делят базу данных (и поэтому могут делать ACID-транзакции между модулями). Микросервисы владеют своими базами данных эксклюзивно — вы теряете ACID-гарантии через границы сервисов. Это меняет подход к консистентности (eventual consistency, саги, паттерн outbox) и является фундаментальным trade-off'ом, а не просто деталью деплоя.

- **«Начните с микросервисов и избежите боли миграции позже»** — это самая дорогостоящая ошибка в системном дизайне. Микросервисы требуют стабильных ограниченных контекстов (точного понимания, чем должен владеть каждый сервис). В начале проекта эти границы неизвестны. Ошибиться с границами в микросервисной архитектуре катастрофично — вы построили распределённый монолит, где изменение одной фичи требует изменения 4 сервисов и 4 схем баз данных. Ошибиться с границами в монолите — значит отрефакторить несколько файлов. Начинайте с хорошо структурированного монолита; выносите сервисы, когда границы станут ясными, а боль от координации — реальной.

- **«Закон Конвея означает, что нужно структурировать код под команду»** — закон Конвея описателен, а не предписателен. Он описывает то, что происходит без намеренных усилий. Inverse Conway Maneuver — это предписательный ответ: если хотите определённую архитектуру, намеренно структурируйте команды под неё. Но для маленьких команд (до ~10 инженеров) нет границы команды, которую можно отразить в архитектуре — закон просто не применим.

- **«Микросервисы обеспечивают лучшую изоляцию отказов»** — верно в теории, сложно на практике. Да, крэш в сервисе уведомлений не уронит сервис заказов. Но: синхронные вызовы между сервисами создают каскадные отказы (Orders вызывает Users синхронно; Users падает; Orders теперь таймаутит на каждый запрос). Для реальной изоляции отказов нужны circuit breaker'ы, таймауты, retry и bulkhead — паттерны из набора инструментов resilience engineering. Монолит с нормальной обработкой ошибок часто имеет более простые характеристики отказов, чем микросервисная архитектура с необработанными синхронными зависимостями.
