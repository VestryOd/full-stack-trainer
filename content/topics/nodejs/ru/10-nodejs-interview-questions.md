# Node.js: вопросы для интервью (Middle → Senior)

## Как использовать эту шпаргалку

Каждый ответ ниже — это КРАТКАЯ версия того, что подробно разобрано в статьях этого раздела. На senior-интервью почти ЛЮБОЙ из этих вопросов — это не финальный вопрос, а ЗАХОД на уточняющий follow-up ("а почему?", "а что если...?", "приведите пример из практики"). Поэтому после каждой группы — секция **"Типичные follow-up'ы"**, которая показывает, куда обычно ведёт интервьюер дальше. ## Группа 1: Runtime, V8, npm

**1. Что такое Node.js?**
Не "язык программирования" и не "фреймворк" — это RUNTIME: среда выполнения, объединяющая движок V8 (выполнение JS), libuv (event loop, thread pool, асинхронный I/O) и набор встроенных Node API (`fs`, `http`, `crypto` и т.д.), недоступных в браузере. См. [Node.js Fundamentals].

**2. Чем Node отличается от браузера, если оба используют V8?**
V8 в обоих случаях выполняет один и тот же JS, но НАБОР ДОСТУПНЫХ API разный: браузер даёт DOM/`window`/`fetch` (через Web APIs движка), Node даёт `fs`/`process`/`net` (через libuv). Это два разных "хоста" вокруг одного движка.

**3. Что делает V8 помимо "выполнения JS"?**
Конвейер: Parser → AST → Ignition (байткод-интерпретатор, быстрый старт) → TurboFan (JIT-оптимизация горячего кода) → возможный Deopt при изменении формы объектов. Плюс Hidden Classes/Inline Caches для ускорения доступа к свойствам и Generational GC. См. [V8 and the Runtime].

**4. Что такое Hidden Class и зачем он нужен?**
V8 присваивает каждой "форме" объекта (набору ключей в определённом порядке) скрытый класс — это позволяет JIT генерировать оптимизированный код доступа к свойствам по фиксированному смещению, как в статически типизированных языках. Добавление/удаление свойств в разных объектах одного "логического типа" в разном порядке создаёт разные Hidden Classes → Inline Cache становится polymorphic/megamorphic → деоптимизация.

## Группа 2: Event Loop, Microtasks, process.nextTick

**5. Что такое Event Loop простыми словами?**
Это не "очередь задач", а МЕХАНИЗМ С ФАЗАМИ: timers → pending callbacks → poll → check → close callbacks, который на каждой итерации выбирает, какие готовые callback'и выполнить, и между КАЖДЫМ callback'ом полностью дренирует очереди `process.nextTick` и микротасков. См. [The Event Loop].

**6. Перечислите фазы Event Loop и что происходит в каждой.**
`timers` (готовые setTimeout/setInterval) → `pending callbacks` (некоторые системные операции) → `poll` (основная фаза — I/O callbacks, блокируется на epoll/kqueue/IOCP, если нечего делать) → `check` (setImmediate) → `close callbacks` (`'close'` события).

**7. Чем microtask queue отличается от macrotask queue, и КОГДА происходит переключение?**
Microtask queue (`Promise.then`, `queueMicrotask`, async/await) дренируется ПОЛНОСТЬЮ после завершения ЛЮБОЙ операции — включая каждый отдельный callback внутри фазы poll, а не один раз "после скрипта". Macrotask (timers/setImmediate/I/O) ждёт следующей фазы цикла. См. [Microtasks, Macrotasks, and process.nextTick].

**8. Почему `Promise.then` выполняется раньше `setTimeout(fn, 0)`?**
Потому что microtask queue имеет приоритет над ЛЮБОЙ фазой Event Loop — Event Loop не переходит к следующей задаче (включая timers), пока microtask queue не опустеет полностью.

**9. Что такое `process.nextTick` и чем он отличается от microtask queue?**
Отдельная очередь, имеющая ЕЩЁ БОЛЕЕ высокий приоритет, чем microtask queue — дренируется первой, и при этом каждое добавление в неё ВО ВРЕМЯ дренирования также будет обработано до перехода к microtasks.

**10. Почему `process.nextTick` может быть опасен?**
Рекурсивный вызов `process.nextTick` внутри себя создаёт starvation — Event Loop НИКОГДА не доходит до I/O/таймеров, процесс перестаёт отвечать на запросы, но не "падает". Реалистичный пример — рекурсивная обработка очереди сообщений через `nextTick` вместо `setImmediate`.

**11. Чем `setImmediate` отличается от `setTimeout(fn, 0)`?**
На top-level порядок НЕ ГАРАНТИРОВАН (зависит от точности таймера ОС). Но ВНУТРИ I/O callback'а `setImmediate` ВСЕГДА выполнится раньше `setTimeout(fn, 0)`, потому что фаза `check` идёт сразу после `poll`, а `timers` — только на следующей итерации цикла.

## Группа 3: libuv, Thread Pool, Worker Threads, Cluster

**12. Node однопоточный или многопоточный?**
И то и другое — смотря про что говорить. JS-код выполняется в ОДНОМ потоке (V8 + Event Loop). Но процесс Node содержит ещё N потоков Thread Pool libuv (по умолчанию 4) и M фоновых GC-потоков V8. См. [libuv and the Thread Pool] и [Memory and Garbage Collection].

**13. Какие операции используют Thread Pool, а какие — нет?**
ИСПОЛЬЗУЮТ: `fs.*`, `crypto.pbkdf2/scrypt/randomBytes` (async), `zlib.*` (async), `dns.lookup` (через `getaddrinfo`). НЕ используют: `net`/`http`/`tcp` (epoll/kqueue/IOCP — нативный async ОС), `dns.resolve*` (через c-ares, отдельный механизм).

**14. Почему `fs.readFile` не блокирует Event Loop?**
Сам системный вызов `read()` блокирующий — но Node выполняет его на ОТДЕЛЬНОМ потоке Thread Pool, а не на главном JS-потоке. Когда вызов завершается, результат передаётся обратно через тот же механизм уведомлений, что и сетевые события.

**15. Когда нужны Worker Threads, и почему НЕ Thread Pool?**
Thread Pool НЕ ВЫПОЛНЯЕТ ваш JS — он только выполняет встроенные blocking-операции ОС. Если у вас CPU-bound JS-код (обработка изображений, кастомный хэшинг, парсинг больших документов) — единственный способ не блокировать главный поток — Worker Thread с собственным V8 instance. См. [Worker Threads and Cluster].

**16. Чем Worker Threads отличаются от Cluster?**
Worker Threads — несколько ПОТОКОВ в ОДНОМ процессе, с собственным V8/Event Loop каждый, могут обмениваться данными через `SharedArrayBuffer`. Cluster — несколько ПОЛНОСТЬЮ независимых ПРОЦЕССОВ, делящих один порт через round-robin/SO_REUSEPORT, без shared memory.

**17. "Cluster устарел из-за Docker" — согласны?**
Не совсем. Если контейнеру выделено 4 vCPU, а внутри запущен ОДИН Node-процесс — JS event loop всё равно использует только 1 ядро. Либо больше реплик контейнера (предпочтительно в k8s), либо Cluster внутри контейнера — оба варианта решают эту проблему, с разными trade-off'ами по наблюдаемости и graceful shutdown.

## Группа 4: Streams, Backpressure, Buffer

**18. Что такое Stream и почему он "лучше" readFile?**
Не просто "обработка по частям" — главное свойство — АВТОМАТИЧЕСКАЯ синхронизация скорости producer и consumer (backpressure). `readFile` грузит весь файл в память; stream держит в буфере не больше `highWaterMark` (по умолчанию 64KB). См. [Streams and Backpressure].

**19. 4 типа streams и где они применяются?**
Readable (источник — `fs.createReadStream`), Writable (получатель — `fs.createWriteStream`), Duplex (и то и то — TCP socket), Transform (Duplex + преобразование данных — gzip, парсеры).

**20. Что РЕАЛЬНО происходит при backpressure?**
`writeStream.write(chunk)` возвращает `false`, когда внутренний буфер превысил `highWaterMark` — данные ВСЁ РАВНО записываются в буфер, но сигнал говорит "приостанови producer". Без обработки этого сигнала буфер растёт неограниченно. `.pipe()` делает `pause()`/`resume()` на основе этого сигнала и события `'drain'` автоматически.

**21. Чем `pipeline()` лучше `.pipe()`?**
`.pipe()` не освобождает ресурсы при ошибке В СЕРЕДИНЕ цепочки — оставшиеся потоки/файловые дескрипторы остаются открытыми. `pipeline()` корректно вызывает `destroy()` на ВСЕХ потоках цепочки при ошибке в любом из них, и поддерживает async/await.

**22. Что такое Buffer и почему он "не считается" в heapUsed?**
`Buffer` — представление бинарных данных, физически размещённое ВНЕ обычного V8 heap (в "external" памяти). Отслеживание утечек только через `heapUsed` пропустит рост из-за удерживаемых Buffer'ов — нужно смотреть `rss`/`external`.

## Группа 5: Память и Garbage Collection

**23. Stack vs Heap — в чём разница?**
Stack — call frames, примитивы и ССЫЛКИ (не управляется GC, освобождается автоматически при выходе из функции). Heap — объекты/массивы/функции/closures, управляется GC. Утечки происходят только в Heap. См. [Memory and Garbage Collection].

**24. Как GC решает, что удалить?**
Reachability от Root Objects (global, активные stack frames, замыкания) — Mark (помечает достижимое) → Sweep (удаляет непомеченное) → Compact (дефрагментирует Old Space).

**25. Что такое Generational GC, и почему он эффективен?**
New Space (молодое поколение) очищается алгоритмом Scavenge — copying GC, стоимость которого пропорциональна КОЛИЧЕСТВУ ЖИВЫХ объектов (не мусора). Большинство объектов умирают быстро → Scavenge почти всегда быстрый. Выжившие после 2 циклов перемещаются в Old Space, который очищается дороже — Mark-Sweep-Compact.

**26. GC всегда останавливает приложение полностью?**
Уже нет полностью — современный V8 использует Incremental/Concurrent/Parallel Marking и Compaction (фоновые потоки), что резко снижает паузы. Но финальная синхронизация и Sweep всё ещё требуют короткой паузы основного потока, заметной на p99 при большом heap.

**27. Назовите 3 классических паттерна memory leak в Node.**
(1) Глобальный массив/кэш без очистки, растущий по `setInterval`/на каждый запрос. (2) Неудалённые EventEmitter listeners — особенно когда listener захватывает в closure объект с коротким жизненным циклом (socket), а emitter живёт долго. (3) Closure, удерживающий большой объект, который нужен был только при инициализации.

**28. Почему `process.memoryUsage().heapUsed` не показывает ВСЮ утечку?**
`heapUsed` — это только V8 heap. `external`/`arrayBuffers` (содержимое `Buffer`/`TypedArray`) — отдельная категория, не входящая в heapUsed, но влияющая на `rss` и риск OOM Kill.

**29. Контейнер падает по OOM, хотя V8 "думает", что памяти достаточно — почему?**
По умолчанию V8 ставит лимит Old Space на основе памяти ВСЕЙ МАШИНЫ, а не cgroup-лимита контейнера. Нужно явно задать `--max-old-space-size` с запасом под `external`-память, которая этим флагом не ограничивается.

## Группа 6: Модули — CommonJS vs ESM

**30. Главное отличие CommonJS от ESM — это не только синтаксис. В чём ещё?**
CommonJS: `require()` — синхронный вызов функции, экспорт — КОПИЯ значения на момент вызова. ESM: загрузка в 3 фазы (Construction → Instantiation → Evaluation), импорт — LIVE BINDING на "ячейку" в исходном модуле. См. [CommonJS vs ES Modules].

**31. Как circular dependency ведёт себя в CommonJS vs ESM?**
В CommonJS — модуль получает ЧАСТИЧНО заполненный `module.exports` (только то, что экспортировано ДО строки `require()`, вызвавшей цикл). В ESM — функции работают лучше благодаря hoisting и live bindings, но `let`/`const` с вычисляемым значением могут попасть в TDZ при доступе во время цикла.

**32. Может ли CommonJS импортировать чистый ESM-пакет напрямую через `require()`?**
Нет — `require()` синхронный, ESM требует асинхронной загрузки. Единственный путь — динамический `import()` (асинхронный). Обратное (ESM импортирует CJS) работает — `module.exports` становится `default`.

**33. Что такое "dual package hazard"?**
Если библиотека поддерживает и CJS, и ESM сборки, а разные части приложения импортируют её через `require()` и `import()` соответственно — Node загружает ДВА отдельных модуля с ДВУМЯ instance'ами внутреннего состояния. Если библиотека использует Singleton — приложение получает два не связанных между собой singleton'а.

## Самый частый "итоговый" senior-вопрос

**Почему Node способен обрабатывать тысячи соединений одновременно с одним JS-потоком?**

Слабый ответ: "Event Loop и неблокирующий I/O".

Сильный ответ — раскладывает это на конкретные механизмы:
- сетевые операции обрабатываются через epoll/kqueue/IOCP — ОДИН поток может отслеживать тысячи файловых дескрипторов без выделения потока на каждое соединение (см. [libuv and the Thread Pool]);
- для операций без async API ОС (файлы, некоторые crypto) — Thread Pool, ограниченный пул фоновых потоков;
- Event Loop с фазами гарантирует, что готовые callback'и обрабатываются по очереди, не блокируя друг друга дольше, чем длится синхронная часть каждого callback'а (run-to-completion, см. [The Event Loop]);
- ограничение модели — если callback ДОЛГО выполняется синхронно (CPU-bound), ВСЕ остальные соединения ждут — поэтому для CPU-bound работы нужны Worker Threads, а для утилизации нескольких ядер под I/O-bound трафик — Cluster/несколько реплик.

## Типичные ошибки на интервью

- **Заучивание ответов без понимания механизма** — любая модификация вопроса ("а что если добавить ещё один await/listener/процесс?") ломает заученный ответ.

- **Игнорирование follow-up'ов** — каждый из 45 вопросов выше — это ВХОДНАЯ ТОЧКА в более глубокий разговор; неготовность к "а почему?" — главный признак поверхностной подготовки.

- **Путать "что использует Thread Pool" с "что асинхронно"** — почти всё в Node асинхронно, но Thread Pool используется лишь для узкого набора операций (fs/crypto/zlib/dns.lookup).

- **Не связывать темы между собой** — например, не видеть связь между Memory/GC и Cluster (несколько процессов с меньшим heap снижают влияние GC-пауз), или между Streams и Event Loop (stream события идут через тот же Event Loop).
