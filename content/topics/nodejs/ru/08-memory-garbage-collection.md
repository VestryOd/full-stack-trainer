# Memory, Heap, Stack и Garbage Collection

## Почему это вторая по частоте причина продакшен-инцидентов (после Event Loop lag)

Утечка памяти в Node редко выглядит как явная ошибка — приложение продолжает отвечать на запросы, но `RSS` процесса медленно растёт часами или днями, пока оркестратор (k8s) не убьёт контейнер по OOM и не перезапустит его. Между перезапусками всё "работает", поэтому проблему часто замечают только когда частота перезапусков становится подозрительной.

## Stack vs Heap — быстро, но с акцентом на то, что путают

```txt
Stack:
  - function call frames, примитивы, ССЫЛКИ на объекты
  - размер фиксирован при старте (--stack-size), переполнение
    → "Maximum call stack size exceeded" (RangeError)
  - НЕ управляется GC — память автоматически освобождается
    при выходе из функции

Heap:
  - объекты, массивы, функции, замыкания (closures)
  - управляется V8 Garbage Collector
  - именно здесь происходят memory leaks
```

Частая путаница: "примитив, объявленный внутри функции, хранится в Heap, потому что переменные — это объекты". На самом деле сам примитив (`number`, `boolean`, короткая `string`) — на Stack; в Heap размещаются объекты/массивы/функции, а переменная на Stack хранит ССЫЛКУ на них.

## Структура V8 Heap — детальнее, чем "Young + Old Generation"

```txt
V8 Heap состоит из нескольких пространств (spaces):

  New Space (Young Generation)
    - маленький (обычно несколько MB), для новых объектов
    - очищается алгоритмом Scavenge (copying GC) —
      БЫСТРО и ЧАСТО (десятки/сотни раз в секунду
      под нагрузкой)

  Old Space (Old Generation)
    - объекты, "выжившие" после 2 циклов Scavenge
      в New Space, ПЕРЕМЕЩАЮТСЯ сюда (promotion)
    - очищается Mark-Sweep-Compact — РЕЖЕ, но ДОРОЖЕ

  Large Object Space
    - объекты больше определённого порога (например,
      крупные Buffer/массивы) — минуют New Space,
      сразу попадают сюда (не перемещаются — слишком дорого)

  Code Space, Map Space
    - скомпилированный код (JIT), Hidden Classes/Shapes
      (см. [V8 and the Runtime])
```

### Scavenge: почему очистка молодого поколения такая быстрая

```txt
Scavenge — copying GC алгоритм (вариант Cheney's algorithm):

  New Space делится на 2 половины: "from-space" и "to-space"

  1. живые объекты копируются из from-space в to-space
  2. ВСЁ from-space целиком считается мусором и просто
     "забывается" (никакого обхода мёртвых объектов!)
  3. half-space меняются местами

Поэтому стоимость Scavenge пропорциональна количеству
ЖИВЫХ объектов, а не общему объёму мусора — большинство
объектов в New Space умирают быстро (req/res, временные
массивы), поэтому "живых" мало → Scavenge почти всегда
очень быстрый (доли миллисекунды).
```

### Mark-Sweep-Compact: почему очистка старого поколения медленнее

```txt
1. Mark: обход от Root Objects (global, активные stack
   frames, замыкания) — помечает достижимые объекты
2. Sweep: непомеченные объекты освобождаются
3. Compact: живые объекты дефрагментируются (перемещаются
   вместе), чтобы избежать "дырявой" памяти

Old Space обычно МНОГО больше New Space — обход графа
объектов занимает значительно больше времени.
```

### Современный V8: почему "Stop The World" — больше не вся правда

```txt
Старый V8 (до ~2018): Major GC = полная пауза JS-выполнения
на десятки-сотни миллисекунд при большом heap.

Современный V8:
  - Incremental Marking — Mark-фаза разбивается на маленькие
    шаги, чередующиеся с выполнением JS (Orinoco project)
  - Concurrent Marking — часть Mark выполняется в ФОНОВЫХ
    потоках (см. [libuv and the Thread Pool] — это ДРУГИЕ
    потоки, не Thread Pool, а собственные GC-потоки V8)
  - Parallel Compaction — Compact-фаза использует несколько
    потоков одновременно

Итог: полные паузы стали короче (единицы миллисекунд для
большинства приложений), но НЕ исчезли полностью — финальная
синхронизация Mark и сама Sweep-фаза всё ещё требуют
короткой паузы основного потока.
```

Для интервью важно не "GC останавливает мир и это плохо", а: современный GC СНИЖАЕТ влияние на latency, но при достаточно большом heap (гигабайты живых объектов) паузы всё равно заметны на p99/p999 latency — это аргумент в пользу горизонтального масштабирования (больше процессов с меньшим heap каждый) вместо одного процесса с огромным heap, см. [Worker Threads and Cluster].

## process.memoryUsage() — что КАЖДОЕ поле реально означает

```ts
console.log(process.memoryUsage());
// {
//   rss: 85000000,
//   heapTotal: 50000000,
//   heapUsed: 35000000,
//   external: 12000000,
//   arrayBuffers: 8000000
// }
```

```txt
rss (Resident Set Size)
  — ВСЯ память процесса в RAM: V8 heap + Buffers + native
    модули + сам исполняемый код Node. Это число сравнивает
    оркестратор (k8s) с memory limit контейнера для OOM Kill.

heapTotal
  — сколько памяти V8 ВЫДЕЛИЛ под heap (может расти не
    плавно, а скачками — V8 резервирует память блоками)

heapUsed
  — сколько из heapTotal РЕАЛЬНО занято живыми объектами.
    Основной индикатор для отслеживания утечек на JS-стороне

external
  — память, управляемая C++, но связанная с JS-объектами —
    в первую очередь содержимое Buffer/TypedArray.
    НЕ входит в heapUsed!

arrayBuffers
  — подмножество external, конкретно ArrayBuffer/Buffer
```

### Senior-нюанс: утечка через `external`, а не через `heapUsed`

```ts
// ❌ heapUsed может оставаться стабильным, пока external растёт
const buffers: Buffer[] = [];

app.post('/upload', (req, res) => {
  buffers.push(req.body); // удерживаем Buffer навсегда
  res.send('ok');
});
```

```txt
Buffer-данные хранятся ВНЕ обычного V8 heap (в "external"
памяти), но ссылка на Buffer-объект — обычный JS-объект
в heap. Если мониторить ТОЛЬКО heapUsed (как часто советуют
"быстрые" гайды) — рост RSS из-за удерживаемых Buffer'ов
останется НЕЗАМЕЧЕННЫМ. RSS и external — обязательные
метрики при диагностике утечек, связанных с файлами/сетевыми
данными.
```

## Memory leaks — три классических паттерна, но с акцентом на WHY GC не может помочь

```ts
// ❌ Паттерн 1: неограниченно растущая коллекция,
// достижимая из глобального scope (Root Object)
const requestLog: RequestInfo[] = [];
app.use((req, res, next) => {
  requestLog.push({ url: req.url, timestamp: Date.now() }); // никогда не очищается
  next();
});
```

```ts
// ❌ Паттерн 2: EventEmitter listeners, добавляемые
// на каждый запрос/соединение, но не удаляемые
function handleConnection(socket: Socket) {
  const onBroadcast = (msg: string) => socket.send(msg);
  broadcaster.on('message', onBroadcast); // подписка...

  socket.on('close', () => {
    // ❌ забыли: broadcaster.off('message', onBroadcast)
    // broadcaster (живёт долго) теперь держит ссылку на
    // onBroadcast → onBroadcast держит socket (через closure)
    // → socket НИКОГДА не будет собран GC, даже после
    // закрытия соединения
  });
}
```

```ts
// ❌ Паттерн 3: closure удерживает большой объект,
// который нужен был только во время инициализации
function createHandler() {
  const hugeConfigCache = loadHugeConfig(); // 200 MB, нужен ОДИН раз

  return (req: Request) => {
    // hugeConfigCache используется не здесь, но closure
    // ВСЁ РАВНО держит на него ссылку, потому что JS не
    // делает "частичный capture" — захватывается вся
    // переменная из охватывающего scope
    return processRequest(req);
  };
}
```

```txt
Во всех трёх случаях GC работает КОРРЕКТНО — он не удаляет
объект, потому что объект ДОСТИЖИМ из Root (global,
EventEmitter с долгим жизненным циклом, closure). "Утечка" —
это не баг GC, а баг ГРАФА ССЫЛОК в вашем коде. На
собеседовании это стоит явно проговорить: V8 GC не "теряет"
память — он честно сохраняет всё, что вы (по ошибке)
продолжаете считать нужным.
```

### WeakMap / WeakRef — когда они РЕАЛЬНО решают проблему

```ts
// ✅ WeakMap — ключ не удерживает объект живым
const metadataCache = new WeakMap<object, Metadata>();

function attachMetadata(obj: object, meta: Metadata) {
  metadataCache.set(obj, meta);
  // когда obj больше НИГДЕ не используется (даже если
  // metadataCache всё ещё существует) — GC может удалить
  // и obj, и связанную запись в WeakMap
}
```

```txt
WeakMap/WeakRef подходят для: кэшей, привязанных к
жизненному циклу объекта (например, метаданные для DOM-нод
в браузере, или для request-объектов в Node — хотя в Node
это редкий случай, поскольку request живёт недолго и обычный
Map с явной очисткой по 'close' события чаще достаточен).

НЕ подходят для: TTL-кэшей (WeakMap не имеет TTL — объект
живёт, пока на него есть ХОТЬ ОДНА сильная ссылка где-то ещё).
```

## Диагностика в production

```ts
// Базовый мониторинг — экспортировать как метрику
setInterval(() => {
  const mem = process.memoryUsage();
  metrics.gauge('nodejs.heap_used', mem.heapUsed);
  metrics.gauge('nodejs.rss', mem.rss);
  metrics.gauge('nodejs.external', mem.external);
}, 10_000);
```

```txt
Признак утечки: heapUsed (или external/rss) РАСТЁТ МОНОТОННО
после каждого цикла полного GC — то есть локальные пилообразные
колебания (allocation → GC → drop) есть, но "дно" пилы
со временем поднимается.

Признак "это просто нормальное поведение": heapUsed растёт
до некоего плато и стабилизируется — V8 просто увеличил
heapTotal под рабочий набор данных (кэши, connection pools)
и больше не растёт.
```

```bash
# Heap snapshot — для поиска КОНКРЕТНОГО объекта-виновника
node --inspect server.js
# Chrome DevTools → Memory → Heap Snapshot,
# сравнить два снапшота с разницей в N минут → "Comparison" view
# показывает объекты, появившиеся МЕЖДУ снапшотами и не
# освобождённые
```

```txt
Осторожность: снятие heap snapshot — это Stop The World
операция и временный РОСТ памяти (V8 должен материализовать
полное описание графа объектов). Снимать snapshot на
LIVE production-инстансе под нагрузкой — риск кратковременного
полного зависания процесса; обычно делается на одной реплике,
временно выведенной из балансировки, или через clinic.js
heapprofiler с меньшим overhead.
```

## `--max-old-space-size` и контейнеры — частый источник "необъяснимых" OOM

```txt
По умолчанию V8 устанавливает лимит Old Space на основе
ОБЩЕЙ памяти СИСТЕМЫ (не cgroup-лимита контейнера!) —
на машине с 64GB RAM, но контейнере с лимитом 512MB,
V8 может попытаться выделить heap, превышающий лимит
контейнера, и оркестратор убьёт процесс по OOM ДО того,
как сам V8 решит, что нужен GC.

✅ Явно ограничить heap V8 НИЖЕ лимита контейнера:
  node --max-old-space-size=400 server.js
  # оставляя ~25% запас под external/Buffers/native-модули,
  # которые НЕ учитываются в --max-old-space-size
```

```txt
Современные версии Node (≥18) ЧАСТИЧНО решают эту проблему
автоматически (V8 умеет читать cgroup limits), но явная
настройка остаётся best practice — особенно учитывая, что
external-память (Buffers) НЕ ограничивается
--max-old-space-size вообще, и контейнер всё равно может
упасть по OOM из-за накопления Buffer'ов даже при
"безопасном" heap.
```

## Типичные ошибки на интервью

- **"GC останавливает приложение полностью на каждом цикле"** — без уточнения про Incremental/Concurrent/Parallel GC в современном V8, которые резко снижают (но не убирают полностью) влияние на latency.

- **Мониторить только `heapUsed`** — пропуская утечки через `external`/`arrayBuffers` (удерживаемые Buffer'ы), которые не отражаются в heapUsed, но напрямую влияют на RSS и риск OOM Kill.

- **"Объект утёк — это баг Garbage Collector"** — не объяснять, что GC работает корректно по определению reachability, а утечка — это ошибка в графе ссылок кода (забытый listener, растущий глобальный массив, closure с лишним захватом).

- **Не знать про `--max-old-space-size` и контейнерные лимиты памяти** — не понимать, почему процесс может быть убит оркестратором по OOM даже когда V8 "думает", что GC ещё не нужен.

- **Предлагать WeakMap как универсальное решение для кэшей** — не различать "кэш, привязанный к жизненному циклу объекта" (где WeakMap подходит) и "TTL-кэш" (где WeakMap не работает, потому что нет понятия времени жизни без внешних сильных ссылок).
