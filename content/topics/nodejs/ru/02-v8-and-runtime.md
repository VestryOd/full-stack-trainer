# V8 and Node.js Runtime

## V8 — это конвейер, а не "один компилятор"

V8 не компилирует код целиком заранее и не интерпретирует его строка за строкой бесконечно — это многоступенчатый конвейер, который **постепенно повышает качество компиляции для горячего кода** и откатывается назад, если предположения об этом коде оказались неверными.

```txt
Source code
    ↓
Parser → AST (Abstract Syntax Tree)
    ↓
Ignition (baseline interpreter) → Bytecode
    ↓ (если функция вызывается часто — "горячая")
TurboFan (optimizing JIT compiler) → оптимизированный машинный код
    ↑ ↓
    └── Deoptimization (если предположения нарушены —
        откат на bytecode, см. ниже)
```

Главный практический вывод из этой схемы: V8 оптимизирует код **по факту наблюдаемого поведения**, а не по статическому анализу типов (как, например, TypeScript на этапе компиляции). Это значит, что один и тот же код может выполняться с разной скоростью в зависимости от того, какие данные через него проходили ранее — и именно поэтому "тёплый старт" (warm-up) — реальное явление, которое важно учитывать при нагрузочном тестировании Node-сервисов.

## Ignition и TurboFan: baseline vs optimized

```txt
Ignition:
  - запускает функцию сразу, без задержки на компиляцию
  - генерирует компактный bytecode
  - собирает feedback (профиль типов аргументов, форм объектов)

TurboFan:
  - срабатывает, когда V8 видит, что функция "горячая"
    (вызывается много раз с одинаковым "профилем")
  - использует feedback от Ignition, чтобы сгенерировать
    машинный код, специализированный для НАБЛЮДАЕМЫХ типов
  - результат — код, который быстрее интерпретируемого
    в разы, но верен ТОЛЬКО для этого профиля типов
```

```ts
function add(a: number, b: number) {
  return a + b;
}

// Если add вызывается тысячи раз ВСЕГДА с числами —
// TurboFan сгенерирует машинный код, который ожидает
// именно числа (например, использует SMI-представление
// без боксинга)
for (let i = 0; i < 100_000; i++) add(i, i + 1);

// Внезапный вызов с другим типом — TurboFan-код больше
// не валиден для этого случая → DEOPTIMIZATION
add('5', '10'); // V8 откатывается на Ignition bytecode
                // для этого call site (минимум — для него)
```

## Deoptimization — обратная сторона JIT, о которой часто молчат

Деоптимизация — это не "ошибка", это штатный механизм V8: когда оптимизированный код встречает ситуацию, для которой он не был специализирован (другой тип аргумента, изменённая форма объекта, удалённое свойство), V8 **откатывается** на bytecode-исполнение для соответствующего участка, теряя все преимущества TurboFan для этого пути.

```ts
// ❌ Полиморфная функция — вызывается с разными "формами" объектов
function getArea(shape) {
  return shape.width * shape.height;  // у Circle нет height!
}

getArea({ width: 10, height: 5 });          // Rectangle-like shape
getArea({ width: 10, radius: 5 });          // другая "форма" → polymorphic call site
```

Senior-нюанс: один-два полиморфных типа на call site (bimorphic) — это нормально, V8 неплохо справляется с небольшим числом форм. Проблема начинается при **megamorphic** call sites (много разных форм объектов проходят через одну и ту же функцию) — V8 перестаёт пытаться специализировать код и падает на самый медленный generic-путь. Для горячих путей (парсинг тысяч записей в цикле, обработчики middleware на каждый запрос) форма объектов, проходящих через функцию, имеет реальное значение для производительности.

## Hidden Classes (Shapes) и почему форма объекта важна

V8 не хранит объекты как hash-map "имя свойства → значение" — это было бы слишком медленно для доступа к свойствам. Вместо этого каждый объект ассоциируется с **Hidden Class** (внутри V8 называется "Map" или "Shape"), которая описывает раскладку свойств в памяти — похоже на то, как компилятор C++ знает offset полей структуры.

```ts
// Оба объекта создаются с ОДИНАКОВЫМ порядком и набором свойств
const user1 = { name: 'Alice', age: 30 };
const user2 = { name: 'Bob', age: 25 };
// → user1 и user2 делят одну и ту же Hidden Class
// → функции, работающие с такими объектами, могут быть
//   мономорфными и быстрыми

// ❌ Изменение формы объекта ПОСЛЕ создания —
// создаёт НОВУЮ Hidden Class для этого конкретного объекта
const user3 = { name: 'Carol', age: 28 };
user3.address = 'London';  // новая Hidden Class, отличная от user1/user2

// ❌ Удаление свойства — тоже создаёт новую (и часто
// "медленную", dictionary-mode) Hidden Class
delete user3.age;
```

```txt
Практическое следствие — конструкторы/фабрики должны
ВСЕГДА инициализировать ВСЕ поля объекта, даже если
значение пока неизвестно:

  ❌ class User {
       constructor(name) {
         this.name = name;
         // address добавляется позже, в другом методе
       }
     }

  ✅ class User {
       constructor(name, address = null) {
         this.name = name;
         this.address = address;  // поле существует с самого начала,
                                   // даже если значение null
       }
     }
```

Это не "микрооптимизация для энтузиастов" — для кода, который создаёт тысячи объектов одной формы в горячем пути (например, маппинг строк из БД в DTO), консистентная форма объектов напрямую влияет на то, останется ли V8 в "быстром" режиме или перейдёт на dictionary-режим хранения свойств.

## Inline Caches — кеш "где искать свойство"

Inline Cache (IC) — механизм, дополняющий Hidden Classes: для каждого места в коде, где происходит доступ к свойству (`obj.name`), V8 запоминает Hidden Class объекта и offset свойства **прямо в этом месте кода**.

```txt
function getName(obj) {
  return obj.name;  // ← здесь V8 создаёт Inline Cache
}

Первый вызов getName(user1):
  - V8 видит Hidden Class объекта user1
  - запоминает: "для этой Hidden Class, name находится по offset 0"
  - IC становится monomorphic (1 запомненная форма)

Повторные вызовы с объектами ТОЙ ЖЕ Hidden Class:
  - V8 сразу читает offset 0, без поиска свойства — очень быстро

Вызов с объектом ДРУГОЙ Hidden Class:
  - IC становится polymorphic (запоминает 2-4 формы)
  - при превышении лимита форм → megamorphic, IC перестаёт
    помогать, используется generic lookup
```

Hidden Classes и Inline Caches — это две стороны одной идеи: V8 пытается превратить динамическую природу JS ("любое свойство можно добавить/удалить когда угодно") в статические, предсказуемые структуры, похожие на структуры в скомпилированных языках — но это работает только пока код действительно ведёт себя предсказуемо.

## Heap: где V8 хранит объекты, и почему это важно для GC

```txt
Stack:
  - примитивы, ссылки, frame'ы вызовов функций
  - управляется автоматически при входе/выходе из функции

Heap — разделён на "поколения" (generational hypothesis:
большинство объектов умирают быстро):

  New Space (Young Generation):
    - новые объекты создаются здесь
    - небольшой размер → сборка мусора здесь (Scavenger) —
      ЧАСТАЯ, но БЫСТРАЯ (большинство объектов уже мертвы)

  Old Space (Old Generation):
    - объекты, "переживающие" несколько циклов Scavenger,
      перемещаются ("promotion") сюда
    - сборка здесь (Mark-Compact/Mark-Sweep) — РЕЖЕ,
      но ДОРОЖЕ (полный обход графа объектов)
```

Полный разбор алгоритмов сборки мусора, практических паттернов утечек памяти и heap snapshot — в [Memory and Garbage Collection]; здесь важно зафиксировать связь: то, ЧТО объект создаётся в New Space и быстро либо умирает, либо "повышается" в Old Space, напрямую завязано на Hidden Classes и форму объектов — переиспользование объектов одной формы в горячих циклах снижает давление на GC, а не только ускоряет доступ к свойствам.

## Runtime = V8 + Node APIs + libuv

```txt
V8 умеет:           V8 НЕ умеет (и это нормально —
  - выполнять JS       не его ответственность):
  - управлять heap      - читать файлы
  - делать JIT/деопт     - открывать сокеты
  - GC                   - работать с таймерами ОС
                         - DNS-резолвинг
```

Node заполняет этот пробел:

```txt
fs, net, http, crypto, timers, dgram, ...
    ↓ (C++ bindings)
libuv — предоставляет event loop и thread pool
    ↓
системные вызовы ОС (epoll/kqueue/IOCP, читать файл, ...)
```

Эта связка разбирается дальше в [The Event Loop] и [libuv and the Thread Pool] — здесь главное понимать, что когда вы пишете `await fs.promises.readFile(...)`, путь выполнения проходит через ВСЕ три слоя: V8 выполняет ваш async-код и управляет Promise (см. [Microtasks, Macrotasks, and process.nextTick]), Node API оборачивает вызов в формат, понятный libuv, а libuv решает, как физически выполнить операцию (через thread pool или нативный async API ОС).

## Практический вывод: когда V8-детали реально имеют значение

```txt
ИМЕЕТ значение:
  - горячие пути с тысячами/миллионами итераций (парсинг,
    маппинг данных, вычисления в цикле)
  - библиотеки/фреймворки, через которые проходит весь трафик
    (ORM-маппинг строк в объекты, валидаторы, сериализаторы)
  - выбор структур данных для часто мутируемых объектов
    с непредсказуемой формой (Map вместо plain object,
    если ключи динамические)

ПОЧТИ НИКОГДА не имеет значения:
  - бизнес-логика уровня "обработать один HTTP-запрос" —
    разница в наносекундах на доступ к свойству объекта
    исчезает на фоне миллисекунд I/O (запрос к БД, сети)
  - микрооптимизации, ухудшающие читаемость кода ради
    "монографичности" call site, который вызывается 10 раз
    за всё время жизни процесса
```

Это сознательная калибровка, которую стоит проговорить на интервью: знание Hidden Classes/IC демонстрирует понимание движка, но применять эти знания imperative-стилем ("всегда инициализируй все поля в конструкторе, даже не используемые") в коде, где это не горячий путь — пример попытки оптимизации там, где она не нужна, и это тоже сигнал (в данном случае — отрицательный) на senior-собеседовании.

## Типичные ошибки на интервью

- **"V8 интерпретирует JS"** — без упоминания Ignition (baseline bytecode) и TurboFan (optimizing JIT), и того, что современный V8 — это многоступенчатый конвейер, а не чистый интерпретатор.

- **Hidden Classes как "просто факт из Википедии"** — без связи с конкретным следствием: непоследовательная инициализация полей объекта или динамическое добавление/удаление свойств создаёт новые Hidden Classes и может приводить к деоптимизации горячих функций.

- **Не знать о деоптимизации** — воспринимать JIT как "один раз скомпилировалось и всегда быстро", не понимая, что V8 может откатиться на bytecode, если предположения о типах нарушены.

- **Путать V8 и Node** — приписывать V8 возможности fs/net/http, которые на самом деле предоставляются Node через libuv и C++ биндинги.

- **Применять микрооптимизации не там, где они нужны** — переусложнять обычный бизнес-код "ради V8", когда реальный bottleneck — I/O, а не доступ к свойствам объекта.
