# Event Loop

## Зачем нужен Event Loop — переформулировка через "контракт"

JS-движок (V8) умеет выполнять синхронный код и управлять стеком вызовов — и это всё, что он умеет. Event Loop — это механизм, который libuv предоставляет НАД V8, чтобы превратить "выполнить функцию и вернуть результат" в "запланировать операцию, вернуть управление немедленно, и вызвать callback, когда результат будет готов".

```txt
Без Event Loop (синхронная модель):
  fs.readFileSync()  → поток блокируется на время чтения
                         с диска (может быть мс, может — секунды
                         при высокой нагрузке на диск)

С Event Loop (асинхронная модель):
  fs.readFile(cb)    → операция передаётся libuv, JS-поток
                         немедленно продолжает выполнять
                         ДРУГОЙ код; когда диск отдаст данные,
                         cb помещается в очередь, и Event Loop
                         выполнит его, когда дойдёт до
                         соответствующей фазы
```

Это не "Node умный и сам разбирается" — это явный контракт: ваш JS-код выполняется **до конца** (run-to-completion) каждый раз, когда Event Loop передаёт ему управление, и Event Loop не может прервать выполняющуюся JS-функцию. Из этого контракта следуют почти все практические правила этой темы.

## Полная схема фаз — и что РЕАЛЬНО происходит в каждой

```txt
   ┌───────────────────────────┐
┌─>│           timers           │  setTimeout / setInterval,
│  └─────────────┬─────────────┘  готовые по времени
│  ┌─────────────┴─────────────┐
│  │     pending callbacks      │  callbacks некоторых системных
│  │     (pending I/O)          │  операций (например, ошибки TCP)
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │       idle, prepare        │  внутренние нужды libuv
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │            poll            │  ← ЦЕНТРАЛЬНАЯ ФАЗА:
│  │                             │    I/O callbacks (fs, network),
│  │                             │    и ОЖИДАНИЕ новых I/O-событий
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │            check            │  setImmediate
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
└──┤      close callbacks       │  socket.on('close'), и т.п.
   └───────────────────────────┘
```

Между КАЖДОЙ парой фаз (и часто внутри одной фазы, после каждого callback) Event Loop полностью опустошает **microtask queue** (Promise callbacks, `queueMicrotask`) и очередь `process.nextTick` — это критически важная деталь, которая разобрана отдельно в [Microtasks, Macrotasks, and process.nextTick], потому что именно она объясняет большинство "неожиданных" порядков выполнения.

### Poll — самая важная фаза, и почему

Poll — это фаза, где Node проводит **большую часть времени** в реальном приложении (если приложение не загружено вычислениями). Здесь происходит две вещи:

```txt
1. Если в poll-очереди есть готовые I/O callbacks
   (запрос к БД вернулся, файл прочитан, HTTP-ответ получен)
   → выполнить их (с микротасками после каждого)

2. Если очередь пуста:
   - если есть setImmediate() в очереди check → перейти
     к check немедленно
   - если есть таймеры, которые скоро сработают →
     ждать I/O, но не дольше, чем до следующего таймера
   - иначе → БЛОКИРОВАТЬСЯ на этой фазе (ждать I/O от ОС),
     пока не появится событие или не истечёт таймер
```

Этот последний пункт — ключевой: Event Loop **не крутится в горячем цикле, потребляя CPU**, когда нечего делать. Он блокируется на системном вызове ОС (`epoll_wait`/`kqueue`/IOCP), и операционная система "разбудит" процесс, когда придёт сетевой пакет, завершится файловая операция или истечёт таймер. Это и есть причина, по которой Node-процесс с тысячами idle-соединений практически не потребляет CPU.

## setTimeout vs setImmediate — порядок зависит от контекста

Это один из самых частых "вопросов с подвохом", и правильный ответ — "зависит от того, откуда вызывается":

```ts
// В МОДУЛЬНОМ (top-level) коде — порядок НЕ ГАРАНТИРОВАН
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));
// Может вывести "timeout, immediate" ИЛИ "immediate, timeout" —
// зависит от того, сколько времени заняла инициализация
// процесса до этой точки (setTimeout(0) на самом деле
// планируется на ~1ms, и если event loop стартовал раньше
// этой отметки — timers ещё не готовы при первом проходе)
```

```ts
// ВНУТРИ I/O callback — порядок ВСЕГДА детерминирован
fs.readFile(__filename, () => {
  setTimeout(() => console.log('timeout'), 0);
  setImmediate(() => console.log('immediate'));
});
// ВСЕГДА: "immediate, timeout"
//
// Почему: мы находимся в poll-фазе (это I/O callback).
// Следующая фаза после poll — check (setImmediate).
// timers будет следующим проходом цикла — ПОСЛЕ check.
```

Запоминать "что выведется в консоли" бесполезно — важно понимать МЕХАНИЗМ: порядок фаз `timers → poll → check` фиксирован, и единственная переменная — **из какой фазы стартует выполнение**. Внутри I/O callback вы точно в poll, поэтому check (`setImmediate`) гарантированно раньше следующего прохода timers.

## "Event Loop ничего не делает, пока стек не пуст" — практические следствия

```ts
// ❌ Эта функция блокирует Event Loop на секунды —
// НИ ОДИН другой запрос к серверу не будет обработан,
// независимо от того, сколько у них I/O-операций
function blockingHash(password: string) {
  return crypto.pbkdf2Sync(password, 'salt', 500_000, 64, 'sha512');
}

app.post('/login', (req, res) => {
  const hash = blockingHash(req.body.password); // ВСЕ остальные
  res.json({ hash });                            // запросы ждут
});
```

```ts
// ✅ Та же задача, разбитая на чанки через setImmediate —
// между чанками Event Loop может обработать другие задачи.
// Это НЕ ускоряет саму операцию (CPU всё равно занят),
// но даёт возможность серверу оставаться responsive
function processBigArrayInChunks(items: Item[], onDone: () => void) {
  let i = 0;
  function chunk() {
    const end = Math.min(i + 1000, items.length);
    for (; i < end; i++) process(items[i]);
    if (i < items.length) setImmediate(chunk);  // отдать управление
    else onDone();
  }
  chunk();
}
```

Senior-нюанс: "разбивка на чанки" — это **компромисс**, не решение проблемы. Суммарное CPU-время не меняется — меняется лишь то, что между чанками Event Loop может обслужить другие запросы. Для действительно тяжёлых вычислений правильное решение — Worker Threads (см. [Worker Threads and Cluster]), которые используют ОТДЕЛЬНЫЙ поток и реально освобождают главный.

## Мониторинг event loop lag в production — то, что отличает теорию от практики

В реальных системах вопрос не "блокируется ли event loop когда-нибудь", а "насколько он задержан **прямо сейчас**, под текущей нагрузкой". Для этого существует метрика **event loop lag** (или delay):

```ts
import { monitorEventLoopDelay } from 'node:perf_hooks';

const histogram = monitorEventLoopDelay({ resolution: 20 });
histogram.enable();

setInterval(() => {
  // p99 задержки event loop за последний интервал
  console.log('p99 lag (ms):', histogram.percentile(99) / 1e6);
  histogram.reset();
}, 5000);
```

```txt
Практический смысл:
  - lag близкий к 0 — event loop отвечает почти мгновенно
  - lag растёт под нагрузкой — где-то выполняется синхронный
    код, который не даёт event loop "продохнуть" между задачами

  Это метрика, которую стоит экспортировать в Prometheus/Datadog
  для production Node-сервисов — рост event loop lag часто
  предшествует росту latency API ещё ДО того, как CPU usage
  выглядит критично (например, если "горячая" функция вызывается
  не постоянно, а пачками — average CPU usage может казаться
  нормальным, а p99 latency запросов будет расти именно
  из-за lag в моменты этих пачек)
```

## Типичные ошибки на интервью

- **"Event Loop — это while(true) с очередью"** — упрощение, которое не объясняет ни порядок фаз, ни то, почему между ними выполняются микротаски, ни то, что poll-фаза может блокироваться на ожидании I/O без потребления CPU.

- **Заучивать вывод `setTimeout(0)` vs `setImmediate()`** без понимания, что порядок зависит от контекста вызова (top-level vs внутри I/O callback) — на интервью с небольшой вариацией примера заученный ответ оказывается неверным.

- **Не различать "разбить на чанки через setImmediate" и "вынести в Worker Thread"** — первое не уменьшает суммарную нагрузку на CPU, только распределяет её во времени; второе реально освобождает главный поток.

- **Не знать про мониторинг event loop lag** — для собеседований на роли с продакшен-ответственностью ("как вы поймёте, что Node-сервис деградирует из-за блокировки event loop") отсутствие ответа про `monitorEventLoopDelay`/APM-метрики выглядит как чисто теоретическое знание темы.

- **Путать "асинхронно" и "параллельно"** — Event Loop даёт **конкурентность** (concurrency: много операций "в процессе" одновременно за счёт ожидания I/O), но не **параллелизм** (parallelism: одновременное выполнение JS-кода) — для параллелизма нужны Worker Threads или Cluster.
