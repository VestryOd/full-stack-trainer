# libuv и Thread Pool

## "Node однопоточный" — где проходит точная граница

Утверждение "Node однопоточный" верно для одной конкретной вещи — выполнения вашего JS-кода в V8. Но процесс Node содержит:

```txt
1 поток — выполняет JS (V8) + Event Loop libuv
N потоков — Thread Pool libuv (по умолчанию 4)
M потоков — фоновые потоки V8 для GC (см. [Memory and GC])
```

Сетевые операции (`http`, `net`, `tcp`) **не используют Thread Pool вообще** — это критически важная деталь, которую путают чаще всего. Они опираются на нативные асинхронные механизмы ОС:

```txt
Linux:   epoll
macOS:   kqueue
Windows: IOCP (I/O Completion Ports)
```

Эти механизмы позволяют ОДНОМУ потоку отслеживать тысячи файловых дескрипторов (сокетов) и получать уведомление, когда конкретный сокет готов к чтению/записи — без необходимости выделять поток на каждое соединение. Поэтому Node может держать 10,000+ открытых WebSocket-соединений, не имея 10,000 потоков — для сети поток вообще не нужен, пока данные не готовы.

## Почему файловая система — другая история

Логичный вопрос: "если для сети есть epoll, почему `fs.readFile` использует Thread Pool, а не такой же механизм?"

```txt
Ответ: на большинстве платформ (особенно Linux) НАДЁЖНОГО
неблокирующего API для операций с файлами НЕ СУЩЕСТВУЕТ
на уровне ОС — в отличие от сокетов.

epoll отлично работает для сокетов, но при попытке
использовать его для regular files на Linux он либо
не поддерживается, либо ведёт себя непредсказуемо
(всегда сообщает "готово").

Решение libuv: выполнить блокирующий системный вызов
(read(), open(), stat()) В ОТДЕЛЬНОМ потоке Thread Pool —
и когда вызов завершится (поток блокируется, но это
не главный поток!), результат передаётся обратно
в Event Loop через тот же механизм уведомлений.
```

Это объясняет, почему **операции с файлами имитируют асинхронность через потоки**, а сетевые операции — **действительно асинхронны на уровне ОС**. Для пользовательского кода разница незаметна (оба используют callback/Promise), но для понимания, что РЕАЛЬНО потребляет ресурс Thread Pool — это ключевое различие.

## Полная карта: что использует Thread Pool

```txt
ИСПОЛЬЗУЮТ Thread Pool:
  fs.*           (кроме fs.FSWatcher — наблюдение за файлами
                  использует platform-specific уведомления)
  crypto.pbkdf2, crypto.scrypt, crypto.randomBytes (async),
  crypto.generateKeyPair (async)
  zlib.* (gzip/gunzip/brotli — async-версии)
  dns.lookup()   — использует getaddrinfo() системы

НЕ используют Thread Pool (нативный async ОС):
  net, http, https, tcp, udp — epoll/kqueue/IOCP
  dns.resolve*() — использует c-ares, СОБСТВЕННУЮ асинхронную
                   реализацию DNS-резолвинга, НЕ системную
                   getaddrinfo() и НЕ Thread Pool
```

### Senior-нюанс: `dns.lookup` vs `dns.resolve` — частый "gotcha"

```ts
// dns.lookup использует getaddrinfo() ОС → Thread Pool
// (тот же пул, что fs/crypto/zlib!)
dns.lookup('example.com', callback);

// dns.resolve4/resolve6/resolveCname использует c-ares →
// НЕ Thread Pool, отдельный механизм
dns.resolve4('example.com', callback);
```

Если приложение делает много DNS-резолвинга через `dns.lookup` (например, неявно — через `http.get` с доменными именами, которые по умолчанию резолвятся через `lookup`) ОДНОВРЕМЕННО с тяжёлыми `crypto`/`fs`-операциями — они **конкурируют за один и тот же Thread Pool**. Это нетривиальная связь, которую мало кто проговаривает: "медленные исходящие HTTP-запросы к внешним API" и "медленное чтение файлов" могут оказаться СВЯЗАННЫМИ симптомами одной причины — насыщенный Thread Pool.

## Конкретный пример "невидимого" bottleneck'а

```ts
// ❌ На каждый запрос — bcrypt (CPU-heavy, через Thread Pool)
// ПЛЮС чтение файла (тоже Thread Pool)
app.post('/register', async (req, res) => {
  const hash = await bcrypt.hash(req.body.password, 12); // Thread Pool
  const template = await fs.promises.readFile('welcome.html'); // Thread Pool
  // ...
});
```

```txt
При UV_THREADPOOL_SIZE=4 (по умолчанию) и 10 одновременных
регистраций:

  - 4 запроса начинают bcrypt.hash сразу → занимают ВСЕ
    4 потока пула
  - fs.readFile ЭТИХ ЖЕ запросов — ждёт своей очереди,
    хотя само чтение файла занимает доли миллисекунды
  - оставшиеся 6 запросов на bcrypt.hash — ждут полностью

  Наблюдаемый симптом: "API стал медленным", "CPU не на
  пределе" (bcrypt быстрый, но пул узкий), latency растёт
  нелинейно с количеством одновременных запросов
```

### Решение и его границы

```bash
# Увеличение пула — снимает остроту, но не решает корневую причину
UV_THREADPOOL_SIZE=16 node app.js
```

```txt
Важно: UV_THREADPOOL_SIZE должен быть установлен ДО того,
как Node поставит в очередь первую задачу для пула —
т.е. ДО любого require/вызова, который инициирует
Thread Pool (часто — до первого fs/crypto-вызова).
Установка значения "по ходу выполнения" через
process.env в коде уже может не сработать, если
к этому моменту пул уже инициализирован.

Но: больше потоков = больше памяти на стеки + больше
переключений контекста на CPU с ограниченным числом ядер.
"Просто увеличить до 128" — не решение, это перенос
проблемы и потенциально новая (CPU contention).

Реальное решение для CPU-heavy операций (bcrypt, pbkdf2
с большим cost factor) — Worker Threads (см.
[Worker Threads and Cluster]), где вычисление получает
СОБСТВЕННЫЙ выделенный поток, а не делит общий пул
с файловыми операциями всего приложения.
```

## Как диагностировать насыщение Thread Pool в production

```txt
Косвенные признаки:
  - CPU usage НЕ на пределе, но p99 latency растёт
  - latency растёт НЕ ЛИНЕЙНО с RPS, а скачком после
    определённого уровня параллелизма
  - операции, которые "по логике" не связаны (DNS, fs,
    bcrypt), деградируют ОДНОВРЕМЕННО

Прямая диагностика:
  - APM-инструменты (Datadog, New Relic) показывают
    "thread pool queue time" отдельно от "execution time"
  - можно явно измерить через обёртку, засекая время
    между вызовом fs.readFile и началом выполнения callback
    относительно времени постановки в очередь
```

## Связь с другими темами

```txt
[The Event Loop]            — Thread Pool — это ТОЛЬКО один
                               из источников событий для poll-фазы;
                               сеть приходит в poll напрямую через
                               epoll/kqueue, без Thread Pool
[Worker Threads and Cluster] — правильное решение для CPU-heavy
                               задач, которое не делит ресурс
                               с fs/crypto/zlib/dns.lookup
[Memory and Garbage
 Collection]                  — ещё один источник фоновых потоков
                               в процессе (потоки V8 для GC)
```

## Типичные ошибки на интервью

- **"Node однопоточный" без уточнений** — не упоминать ни Thread Pool, ни фоновые потоки V8 для GC, ни то, что "однопоточность" относится конкретно к выполнению JS.

- **Считать, что ВСЕ async-операции в Node используют Thread Pool** — не различать сетевые операции (epoll/kqueue/IOCP, без Thread Pool) и файловые/crypto/zlib (Thread Pool).

- **Не знать про разницу `dns.lookup` vs `dns.resolve*`** — упускать, что DNS-резолвинг через `lookup` конкурирует с fs/crypto за один и тот же пул, а `resolve*` — нет.

- **"Просто увеличить UV_THREADPOOL_SIZE" как универсальное решение** — без понимания, что это паллиатив с собственной стоимостью (память, context switching), а не замена Worker Threads для действительно тяжёлых вычислений.

- **Не уметь диагностировать насыщение пула** — воспринимать "CPU в норме, но latency растёт" как нерешаемую загадку, а не как классический симптом Thread Pool contention.
