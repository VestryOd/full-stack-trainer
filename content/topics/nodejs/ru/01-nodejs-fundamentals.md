# Node.js Fundamentals

## Node.js — это runtime, а не язык и не фреймворк

```txt
JavaScript — язык (спецификация ECMAScript: синтаксис, типы,
  Promise, async/await, ...)

Node.js — runtime: программа на C++, которая встраивает движок
  V8 и добавляет к нему API, которых в спецификации языка нет
  (fs, net, process, Buffer, ...)

Express/Nest/Fastify — фреймворки НАД Node.js
```

Путаница "Node = V8" — частая, но важная для исправления: V8 выполняет JS-код и управляет памятью (heap, GC — см. [Memory and Garbage Collection]), но **не знает**, что такое файл или TCP-сокет. Эти возможности предоставляет сам Node через C++ биндинги к libuv и другим библиотекам. Когда вы вызываете `fs.readFile(...)`, JS-код в V8 вызывает C++ функцию, которая делегирует операцию libuv (см. [libuv and the Thread Pool]).

```txt
┌─────────────────────────────────────────┐
│              Node.js Process              │
│                                            │
│  ┌──────────┐   ┌──────────────────────┐ │
│  │    V8    │   │    Node.js APIs       │ │
│  │ (выполняет│←──│  (fs, net, http,      │ │
│  │  JS, heap,│   │   crypto, Buffer...)  │ │
│  │   GC)    │   └──────────┬───────────┘ │
│  └──────────┘              │              │
│                      ┌──────▼──────┐      │
│                      │    libuv    │      │
│                      │ (Event Loop,│      │
│                      │ Thread Pool)│      │
│                      └─────────────┘      │
└─────────────────────────────────────────┘
```

Эта диаграмма — каркас для всех следующих тем раздела: V8 разбирается в [V8 and the Runtime] и [Memory and Garbage Collection], Event Loop — в [The Event Loop] и [Microtasks, Macrotasks, and process.nextTick], Thread Pool — в [libuv and the Thread Pool], а многопроцессность/многопоточность — в [Worker Threads and Cluster].

## Почему "один поток" — это осознанный архитектурный выбор, а не ограничение

Когда Node появился в 2009 году, доминирующая модель серверов была "поток (или процесс) на каждое соединение" (Apache prefork/threaded). Проблема такой модели: поток — это дорогой ресурс (память на стек, переключение контекста), а большинство времени поток, обслуживающий сетевой запрос, **ничего не делает** — он просто ждёт ответа от БД, диска или другого сервиса.

```txt
Традиционная модель (1 поток = 1 запрос):
  10,000 одновременных соединений → 10,000 потоков,
  каждый в основном в состоянии "жду I/O"
  → память и context-switching становятся bottleneck
  ЗАДОЛГО до того, как CPU реально загружен

Node-модель (1 поток выполняет JS, I/O делегируется):
  10,000 соединений → 1 основной поток обрабатывает события
  по мере готовности I/O-операций
  → память на стеки не тратится, переключений контекста нет
```

Ключевой нюанс, который часто упускают: "single-threaded" относится **только к выполнению вашего JS-кода**. Сам процесс Node — многопоточный: внутри работает Thread Pool libuv (по умолчанию 4 потока — см. [libuv and the Thread Pool]) для операций, у которых нет асинхронного системного API (часть файловой системы, DNS, некоторые криптографические функции), плюс потоки V8 для фоновой сборки мусора. "Один поток" — это про модель программирования (нет гонок данных в вашем коде между двумя кусками JS), а не про физическое число OS-потоков в процессе.

## I/O-bound vs CPU-bound — где граница на самом деле проходит

```txt
I/O-bound (Node — отличный выбор):
  - HTTP/REST API, GraphQL
  - WebSocket / realtime
  - Reverse proxy / API Gateway
  - Streaming (видео, файлы — см. [Streams and Backpressure])
  - BFF (Backend for Frontend) — много параллельных запросов
    к другим сервисам, минимум собственных вычислений

CPU-bound (Node работает хуже "из коробки"):
  - Обработка изображений/видео
  - Сложные вычисления (например, шифрование больших объёмов,
    парсинг огромных документов синхронно)
  - Machine Learning inference
```

Но "Node плохо подходит для CPU-bound" — упрощение, которое стоит уточнить на интервью: проблема не в том, что Node "не может" делать вычисления, а в том, что **любая синхронная CPU-операция блокирует единственный поток, выполняющий event loop** — во время её выполнения Node не может обработать НИ ОДНО другое событие, включая входящие HTTP-запросы от других пользователей.

```ts
// ❌ Блокирует event loop на время выполнения —
// ВСЕ остальные запросы к серверу "зависают" на это время
app.get('/hash', (req, res) => {
  const hash = crypto.pbkdf2Sync(req.query.password, 'salt', 100_000, 64, 'sha512');
  res.json({ hash: hash.toString('hex') });
});

// ✅ Выносим CPU-bound работу в Worker Thread —
// event loop основного потока остаётся свободным
// (полный разбор — в [Worker Threads and Cluster])
app.get('/hash', async (req, res) => {
  const hash = await runInWorker('hash', req.query.password);
  res.json({ hash: hash.toString('hex') });
});
```

Это центральная идея, к которой будут возвращаться многие темы раздела: Node "плохо подходит для CPU-bound" не как абсолютная истина, а как следствие архитектуры с одним JS-потоком — и у этого следствия есть конкретные решения (Worker Threads, Cluster, выгрузка в отдельные сервисы).

## EventEmitter — основа, на которой построен весь Node

Большая часть встроенных API Node (HTTP-сервер, потоки, процессы) — это `EventEmitter` под капотом.

```ts
import { EventEmitter } from 'node:events';

const emitter = new EventEmitter();

emitter.on('userCreated', (user) => {
  console.log('Welcome email queued for', user.email);
});

emitter.emit('userCreated', { email: 'alice@example.com' });
```

### Senior-нюанс №1: событие `'error'` — особый случай

```ts
// ❌ Если на EventEmitter (или Stream, Socket, ...) НЕТ обработчика
// 'error', а событие 'error' эмитится — Node выбрасывает исключение
// и (по умолчанию) ЗАВЕРШАЕТ ПРОЦЕСС, даже если try/catch обернёт
// код вокруг emitter
const socket = net.connect(...);
socket.emit('error', new Error('ECONNRESET')); // → process crash

// ✅ ВСЕГДА подписываться на 'error' для EventEmitter'ов, которые
// могут его эмитить (сокеты, потоки, child processes)
socket.on('error', (err) => {
  logger.error('Socket error', err);
});
```

Это не абстрактная рекомендация "на всякий случай" — это поведение, специфичное именно для `'error'` среди всех событий Node, и оно регулярно ловит даже опытных разработчиков при работе с сетевыми сокетами или дочерними процессами.

### Senior-нюанс №2: утечки памяти через накопление listener'ов

```ts
// ❌ На каждый HTTP-запрос подписываемся на shared emitter,
// но никогда не отписываемся — listeners накапливаются,
// Node выведет предупреждение "MaxListenersExceededWarning"
// после 10 listener'ов по умолчанию, и это сигнал реальной
// утечки памяти (каждый listener держит замыкание с req/res)
app.get('/subscribe', (req, res) => {
  eventBus.on('update', (data) => res.write(JSON.stringify(data)));
});

// ✅ Снимаем listener при закрытии соединения
app.get('/subscribe', (req, res) => {
  const handler = (data: unknown) => res.write(JSON.stringify(data));
  eventBus.on('update', handler);
  req.on('close', () => eventBus.off('update', handler));
});
```

`MaxListenersExceededWarning` часто воспринимается как "просто warning, можно игнорировать" — но в реальности это почти всегда индикатор утечки: либо отсутствует отписка, либо emitter создан не там, где должен быть (например, новый `EventEmitter` на каждый запрос вместо одного shared).

## process — интерфейс к окружению, в котором живёт приложение

```ts
process.env.NODE_ENV     // переменные окружения — конфигурация (см. 12-factor app)
process.argv             // аргументы командной строки
process.pid               // PID процесса — для логов/мониторинга
process.exitCode          // код завершения (0 = успех, !=0 = ошибка для CI/orchestrator)
```

### Senior-нюанс: graceful shutdown через сигналы

```ts
// ❌ Процесс убивается мгновенно (docker stop, k8s rolling update) —
// активные запросы и соединения с БД прерываются на середине
process.on('SIGTERM', () => process.exit(0));

// ✅ Graceful shutdown: перестаём принимать новые соединения,
// даём текущим запросам завершиться, закрываем пул соединений к БД
process.on('SIGTERM', async () => {
  server.close(() => {           // перестать принимать новые соединения
    db.pool.end().then(() => {   // закрыть соединения с БД
      process.exit(0);
    });
  });

  // safety net: если что-то зависло — принудительно выйти через 10 сек
  setTimeout(() => process.exit(1), 10_000);
});
```

Это напрямую связано с тем, как orchestrator (Kubernetes, ECS) обновляет деплои: `SIGTERM` посылается **до** того, как контейнер будет убит (`SIGKILL` следует через grace period, обычно 30 сек). Приложение, не обрабатывающее `SIGTERM`, при каждом деплое обрывает активные запросы пользователей — это частая причина "странных ошибок только во время деплоя", которую интервьюеры любят проверять у backend-кандидатов.

## npm-экосистема: масштаб как преимущество и как риск

```txt
Преимущество:
  практически любая задача (парсинг, валидация, работа с БД,
  криптография) уже решена и опубликована в npm

Риск (supply chain):
  типичный проект на Node имеет СОТНИ транзитивных зависимостей —
  вы доверяете коду, который не писали и, скорее всего,
  не читали, от десятков разных авторов
```

Senior-нюансы, которые стоит знать:

```txt
- package-lock.json / yarn.lock / pnpm-lock.yaml — фиксируют
  ТОЧНЫЕ версии всего дерева зависимостей. Без lock-файла
  "npm install" в CI и на машине разработчика может установить
  разные patch-версии транзитивных пакетов — источник
  "у меня работает, а в проде не работает"

- Semver (^1.2.3 vs ~1.2.3 vs 1.2.3) — "^" разрешает minor/patch
  обновления автоматически; для production-кода многие команды
  предпочитают точные версии + Dependabot/Renovate с review,
  а не "молчаливые" автообновления

- npm audit / Snyk и т.п. — сканирование известных вулнерабилити
  в зависимостях; это не "опционально для безопасных команд",
  а часть стандартного CI pipeline для серьёзного backend-проекта

- postinstall-скрипты — пакет может выполнять произвольный код
  при установке; это реальный вектор атаки (известны случаи
  компрометации популярных пакетов с майнингом/кражей токенов
  через postinstall)
```

## Типичные ошибки на интервью

- **"Node = V8"** — не различать движок (выполнение JS, память, GC) и runtime (предоставляет fs/net/process через C++ биндинги и libuv).

- **"Node однопоточный, точка"** — без уточнения, что однопоточность касается выполнения JS-кода, а сам процесс использует Thread Pool и потоки V8 для GC.

- **"Node не подходит для CPU-задач"** без объяснения МЕХАНИЗМА (блокировка единственного event loop потока) и без упоминания митигации (Worker Threads, выделение в отдельный сервис).

- **Не подписываться на `'error'` у EventEmitter/Stream/Socket** — и не знать, что необработанный `'error'` приводит к падению процесса, в отличие от любого другого события.

- **Игнорировать `SIGTERM`** — не понимать связь между сигналами процесса и graceful shutdown при деплоях в контейнеризированной среде.

- **Недооценивать supply chain риски** — рассматривать npm-зависимости как "бесплатный и безопасный" код без упоминания lock-файлов, semver-стратегии и аудита вулнерабилити.
