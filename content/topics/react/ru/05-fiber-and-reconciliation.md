# Fiber и Reconciliation

## Проблема, которую решает Fiber

До React 16 React использовал **stack reconciler** — единую рекурсивную функцию, которая синхронно обходила всё дерево компонентов. Однажды запущенная, она не могла быть прервана.

```txt
STACK RECONCILER (React < 16)

  reconcile(rootComponent)
    → reconcile(Header)
    → reconcile(Nav)
    → reconcile(Main)
        → reconcile(Sidebar)
        → reconcile(Content)
            → reconcile(ArticleList)
                → reconcile(Article) × 50  ← этот цикл нельзя поставить на паузу
    → reconcile(Footer)

  Суммарное время: столько, сколько нужно для обхода всего дерева.
  В это время: главный поток заблокирован.
  Браузер не может: обрабатывать ввод пользователя, запускать анимации, рисовать кадры.
  Результат: подвисание на деревьях с большим числом компонентов или медленным рендером.
```

Фундаментальное ограничение: JavaScript однопоточен. Рекурсивный стек вызовов нельзя поставить на паузу — нужно дать ему завершиться. Если обход большого дерева занимал 150 мс, интерфейс зависал на 150 мс.

Fiber переписывает reconciler из рекурсивного алгоритма в **итеративный с явной очередью задач**, так что React может ставить работу на паузу, возобновлять её и расставлять приоритеты.

## Что такое узел Fiber

Fiber — объект JavaScript, представляющий единицу работы: один экземпляр компонента, один DOM-элемент, один текстовый узел. Всё дерево компонентов зеркально отражается как дерево Fiber-узлов.

```ts
// Упрощено — реальный FiberNode имеет ~25 полей
type Fiber = {
  // Идентичность
  tag: number;             // FunctionComponent=0, ClassComponent=1, HostComponent=5...
  type: Function | string; // функция компонента, или 'div', 'span' и т.д.
  key: string | null;

  // Структура дерева — Fiber использует связную структуру, не массив
  return: Fiber | null;    // родитель
  child: Fiber | null;     // первый дочерний
  sibling: Fiber | null;   // следующий сиблинг

  // Состояние
  memoizedState: any;      // связный список хуков (useState, useEffect, ...)
  memoizedProps: any;      // props из последнего рендера

  // Работа
  pendingProps: any;       // props для предстоящего рендера
  flags: number;           // битовая маска: Placement | Update | Deletion | ...
  lanes: Lanes;            // приоритет ожидающей работы (см. ниже)

  // Двойная буферизация
  alternate: Fiber | null; // «другая» версия этого fiber (current ↔ work-in-progress)
};
```

### Дерево — это на самом деле связная структура

React не обходит массив дерева — он следует по указателям: `child` для углубления, `sibling` для перехода вправо, `return` для подъёма. Это позволяет прервать обход на любом Fiber-узле и возобновить позже без удержания стека вызовов открытым.

```txt
App
├── child → Header
│             ├── sibling → Nav
│             │               └── sibling → Main
│                                             ├── child → Sidebar
│                                             │             └── sibling → Content
│                                             └── ...
│
(каждый узел также имеет указатель return обратно к родителю)
```

## Двойная буферизация — current и work-in-progress деревья

В любой момент React поддерживает два дерева:

```txt
CURRENT TREE                       WORK-IN-PROGRESS TREE
(что показывается на экране)       (вычисляется в render phase)

  App ─────────────── alternate ──── App'
  │                                   │
  Header ──────────── alternate ──── Header'
  │                                   │
  Main ────────────── alternate ──── Main'
```

Во время render phase React строит work-in-progress дерево, клонируя и обновляя Fiber-узлы. Current дерево не трогается — браузер всё ещё показывает его. Когда render phase завершается, React **атомарно переключает** work-in-progress дерево, делая его новым current деревом. Старое current дерево становится новым пулом work-in-progress (его узлы переиспользуются для следующего рендера).

Вот почему render phase безопасно прерывать и перезапускать: current дерево (то, что видит пользователь) никогда не мутируется во время reconciliation. Модифицируется только work-in-progress дерево.

## Lanes — система приоритетов

React 18 ввёл **Lanes** — систему битовых масок, присваивающую приоритет каждой единице работы. Несколько обновлений могут быть запланированы с разными приоритетами и обрабатываться по порядку приоритетности.

```ts
// Упрощённые значения lanes (реальные — битовые маски)
const SyncLane            = 0b0000000000000000000000000000001; // высший: ввод пользователя
const InputContinuousLane = 0b0000000000000000000000000000100; // drag, scroll
const DefaultLane         = 0b0000000000000000000000000010000; // обычный setState
const TransitionLane      = 0b0000000000000000001000000000000; // useTransition
const IdleLane            = 0b0100000000000000000000000000000; // низший: фоновая работа
```

```txt
ПОРЯДОК ПРИОРИТЕТОВ (высший → низший):
  SyncLane              ← onClick, onKeyDown (пользователь должен увидеть результат немедленно)
  InputContinuousLane   ← onMouseMove, onScroll (должен быть отзывчивым)
  DefaultLane           ← обычный setState в обработчиках событий
  TransitionLane        ← startTransition (может быть прерван)
  IdleLane              ← фоновый префетч, невидимая работа
```

Когда два обновления ожидают обработки, React обрабатывает более приоритетное первым. Если выполняется низкоприоритетный рендер (например, transition) и приходит высокоприоритетное обновление (например, нажатие клавиши), React **прерывает** transition-рендер, синхронно обрабатывает нажатие, затем возобновляет transition с того места, где остановился (или перезапускает, если нажатие изменило входные данные).

```tsx
// Без startTransition:
setQuery(input);        // DefaultLane — результаты поиска обновляются в том же батче, что и набор текста
                        // Большой список результатов блокирует поле ввода

// С startTransition:
startTransition(() => {
  setQuery(input);      // TransitionLane — может быть прерван новыми нажатиями клавиш
});
setInputValue(input);   // DefaultLane — поле ввода обновляется немедленно
```

## Алгоритм reconciliation — правила диффа

Когда render phase создаёт новое дерево React-элементов, React сравнивает его с существующим Fiber-деревом, чтобы определить, что изменилось. Этот дифф имеет явные эвристики (не обобщённый алгоритм диффа деревьев, который был бы O(n³)):

### Правило 1: элементы разных типов создают разные деревья

```tsx
// До:
<div><Counter /></div>

// После:
<span><Counter /></span>

// div → span: разный тип → React размонтирует всё поддерево div
// (включая Counter и всё его состояние), затем монтирует свежее поддерево span.
// Состояние Counter ПОТЕРЯНО.
```

Именно поэтому изменение типа элемента-обёртки уничтожает состояние дочерних компонентов — даже если дети выглядят идентично.

### Правило 2: тот же тип → обновление на месте

```tsx
// До:
<input type="text" className="old" value="hello" />

// После:
<input type="text" className="new" value="world" />

// Тот же тип (input) → React обновляет только изменившиеся атрибуты
// на существующем DOM-узле. DOM-узел переиспользуется.
```

Для типов компонентов: та же функция/класс → React ре-рендерит существующий экземпляр компонента, сохраняя его состояние.

### Правило 3: ключи управляют идентичностью в списках

Без ключей React сопоставляет элементы списка по **позиции**:

```tsx
// До: ['Alice', 'Bob', 'Charlie']
<li>Alice</li>   // позиция 0
<li>Bob</li>     // позиция 1
<li>Charlie</li> // позиция 2

// После: ['Dave', 'Alice', 'Bob', 'Charlie'] (Dave добавлен в начало)
<li>Dave</li>    // позиция 0 ← React думает, что это всё ещё узел "Alice" → обновляет текст
<li>Alice</li>   // позиция 1 ← React думает, что это "Bob" → обновляет текст
<li>Bob</li>     // позиция 2 ← всё ещё "Charlie" → обновляет текст
<li>Charlie</li> // позиция 3 ← новый узел → монтирует
```

Результат: 3 DOM-обновления + 1 монтирование, хотя достаточно было бы 1 монтирования. Хуже того: если элементы имеют состояние (чекбоксы, инпуты), это состояние привязано к позиции, а не к данным.

С ключами React сопоставляет по **идентичности**:

```tsx
<li key="alice">Alice</li>
<li key="bob">Bob</li>
<li key="charlie">Charlie</li>

// После: Dave добавлен в начало
<li key="dave">Dave</li>    // новый ключ → монтирует
<li key="alice">Alice</li>  // тот же ключ → переиспользует существующий DOM-узел, обновление не нужно
<li key="bob">Bob</li>      // тот же ключ → переиспользует
<li key="charlie">Charlie</li> // тот же ключ → переиспользует
```

Результат: 1 монтирование, 0 обновлений — оптимально.

## Почему index-as-key ломает вещи

Использование индекса массива как ключа (`key={index}`) эквивалентно отсутствию ключей для алгоритма диффа — позиция *и есть* идентичность. Проблемы возникают при изменении порядка списка (сортировка, фильтрация, добавление в начало):

```tsx
// Элементы: [{ id: 1, text: 'Купить молоко' }, { id: 2, text: 'Купить яйца' }]
<li key={0}><input defaultValue="Купить молоко" /></li>   // индекс 0
<li key={1}><input defaultValue="Купить яйца" /></li>     // индекс 1

// После удаления элемента 1:
// Элементы: [{ id: 2, text: 'Купить яйца' }]
<li key={0}><input defaultValue="Купить яйца" /></li>     // индекс 0

// React видит: key=0 существует до и после → ТОТ ЖЕ компонент → обновляет на месте.
// Но значение DOM инпута — "Купить молоко" (пользователь мог что-то ввести).
// React обновляет только атрибут `defaultValue` (неконтролируемый инпут),
// поэтому инпут всё ещё показывает "Купить молоко", хотя должен показывать "Купить яйца".
// Идентичность ключа была неверной — переиспользован неправильный DOM-узел.
```

**Безопасные случаи использования индекса как ключа:**
- Список статический (никогда не переупорядочивается, не фильтруется, не дополняется в начало)
- Элементы не имеют локального состояния (нет инпутов, чекбоксов, анимаций)
- Список чисто отображается и элементы никогда не удаляются из середины

Если хотя бы одно из этих условий не выполняется — используйте стабильный уникальный ID из данных.

## Prop key как намеренный механизм сброса

Понимание того, что ключи управляют идентичностью компонентов, раскрывает мощный паттерн: используйте `key` для принудительного перемонтирования компонента с нуля.

```tsx
// Проблема: EditForm кеширует состояние черновика. При изменении userId
// нужна свежая форма — не старый черновик.

// ❌ Требует ручного useEffect для сброса:
function EditForm({ userId }: { userId: string }) {
  const [draft, setDraft] = useState('');
  useEffect(() => {
    setDraft(''); // сбрасываем при изменении userId
  }, [userId]);
  return <input value={draft} onChange={e => setDraft(e.target.value)} />;
}

// ✅ key сообщает React, что это другой экземпляр компонента:
function Page({ userId }: { userId: string }) {
  return <EditForm key={userId} userId={userId} />;
  // При изменении userId React размонтирует старый EditForm и монтирует новый.
  // Состояние автоматически свежее — useEffect не нужен.
}
```

Это идиоматический React — использование собственных правил reconciler для явного выражения намерения.

## Time slicing — что это значит на практике

Time slicing — это способность **разбивать render phase на маленькие порции** и возвращать управление браузеру между порциями, чтобы браузер мог обрабатывать события ввода и рисовать кадры.

```txt
БЕЗ TIME SLICING (синхронный рендеринг):
  ─────────────────────────────────────────────────────────────────▶ время
  [     долгий рендер (100мс)     ][paint][input handler][paint]
        ↑ браузер заблокирован на 100мс во время этого рендера

С TIME SLICING (конкурентный рендеринг):
  ─────────────────────────────────────────────────────────────────▶ время
  [5мс рендер][input][5мс рендер][paint][5мс рендер][5мс рендер][paint]
               ↑ браузер получает управление каждые ~5мс
```

React использует планировщик, который проверяет сколько времени прошло после обработки каждого Fiber-узла. Если бюджет кадра (~5 мс) исчерпан, React уступает управление браузеру и возобновляет рендер в следующей задаче планировщика.

Time slicing применяется только к **прерываемым рендерам** — обновлениям, запланированным через `startTransition` или с `TransitionLane`. Синхронные обновления (пользовательский ввод, `flushSync`) никогда не прерываются.

Результат для пользователей: большой медленный список, рендерящийся в фоне, больше не замораживает поле ввода. Поисковая строка остаётся отзывчивой, потому что React прерывает рендер списка при каждом новом нажатии клавиши.

## Commit phase — порядок эффектов

После того как render phase завершает work-in-progress дерево, commit phase выполняется в трёх подфазах, всегда синхронно:

```txt
BEFORE MUTATION
  → getSnapshotBeforeUpdate (только для классовых компонентов)
  → Чтение из DOM до мутаций (например, позиция скролла)

MUTATION
  → Применить все DOM-вставки, обновления, удаления
  → ref.current устанавливается в null для удалённых узлов

LAYOUT
  → cleanup useLayoutEffect (из предыдущего рендера)
  → setup useLayoutEffect (из этого рендера)
  → ref.current устанавливается на новый DOM-узел
  → вызовы setState здесь синхронны (батчатся в этот commit)

[Браузер рисует здесь]

PASSIVE (запланировано асинхронно, после paint)
  → cleanup useEffect (из предыдущего рендера)
  → setup useEffect (из этого рендера)
```

Различие важно: код в `useLayoutEffect` запускается до отрисовки браузером — он может читать layout, и если вызывает `setState`, результирующий ре-рендер сбрасывается синхронно до paint (без мерцания). Код в `useEffect` запускается после paint — не может синхронно предотвратить визуальные обновления, но и не блокирует пользователя от просмотра экрана.

## Типичные ловушки на интервью

**«Что такое virtual DOM и как он связан с Fiber?»**
«Virtual DOM» — неформальный термин для дерева элементов, которое поддерживает reconciler React. Fiber — внутренняя структура данных, которую React использует для реализации reconciler. Каждый Fiber-узел представляет один компонент/элемент. Дерево Fiber — это и есть то, что люди неформально называют «virtual DOM». Термин «virtual DOM» предшествует Fiber и был точен для эпохи stack reconciler — сегодня «Fiber-дерево» точнее.

**«Что делает React когда setState вызывается внутри useLayoutEffect?»**
`useLayoutEffect` запускается синхронно в подфазе layout во время commit. Если внутри него вызывается `setState`, React планирует дополнительный синхронный рендер и сбрасывает его до отрисовки браузером. Именно поэтому перепозиционирование тултипа через `useLayoutEffect` не имеет видимого мерцания — DOM обновляется дважды в одном кадре.

**«Почему удаление ключа вызывает перемонтирование компонентов?»**
Ключи управляют идентичностью Fiber. Если ключ присутствует в одном рендере и отсутствует в следующем (или наоборот), React считает их разными элементами на этой позиции. Старый Fiber (с его связанным DOM-узлом и состоянием компонента) размонтируется, монтируется новый. Ключи должны быть стабильными и уникальными в пределах своего списка сиблингов на протяжении всего жизненного цикла списка.

**«Могут ли ключи дублироваться в списке?»**
Технически React не крашнется, но поведение не определено — React произвольно выберет один из дублей и отбросит другой. Дублирующиеся ключи в одном списке сиблингов — это баг. Правило ESLint `react/jsx-key` ловит отсутствующие ключи, но не дублирующиеся — для этого нужно `react/no-duplicate-key`.

**«Что такое lane в Fiber? Чем это отличается от приоритета?»**
Lane — это бит в 32-битной целочисленной битовой маске. Несколько обновлений могут находиться в разных lanes одновременно, и React может батчить, разделять или перемежать их. До lanes (React < 18) React использовал простое целочисленное значение приоритета и мог отслеживать только один уровень ожидающего приоритета одновременно — если два обновления ожидали с разными приоритетами, более высокое вытесняло более низкое, потенциально оставляя низкоприоритетную работу голодать. Lanes позволяют React отслеживать множество конкурентных обновлений и гарантировать, что низкоприоритетная работа в конечном счёте завершается (предотвращение голодания).
