# React — Вопросы для интервью (Senior)

## Группа 1: JSX, рендеринг и элементы

**Во что компилируется JSX и почему в старых версиях React должен быть в scope?**

JSX `<Button color="blue">Нажми</Button>` компилируется в `React.createElement(Button, { color: 'blue' }, 'Нажми')` при классическом трансформе, или в `_jsx(Button, { color: 'blue', children: 'Нажми' })` при новом JSX-трансформе (React 17+). В старом коде напрямую вызывался `React.createElement`, поэтому `React` должен быть в scope — даже если нигде не писал `React.`. При новом трансформе (`"jsx": "react-jsx"` в tsconfig) runtime-импорт вставляется автоматически.

---

**Что такое React-элемент и чем он отличается от компонента?**

React-элемент — это обычный JavaScript-объект: `{ $$typeof: Symbol(react.element), type: 'button', props: { color: 'blue' }, key: null, ref: null }`. Он описывает что нужно отрендерить. Компонент — это функция (или класс), принимающая пропсы и возвращающая элементы. Элемент — это описание; компонент — фабрика. `$$typeof` — Symbol: его нельзя JSON-сериализовать, что предотвращает XSS-атаки через JSON, рендерящийся как JSX.

---

**Что вызывает ре-рендер в React?**

Четыре вещи: (1) вызывается `setState` / `dispatch`; (2) изменяется Context, потребляемый компонентом; (3) родительский компонент перерисовывается и передаёт новые пропсы; (4) `forceUpdate` (класс-компоненты). Важно: получение *тех же* пропсов не предотвращает ре-рендер если родитель перерисовывается — React всё равно вызывает функцию компонента. Оптимизация через `Object.is` применяется только когда состояние устанавливается в то же значение что и текущее — в этом случае React пропускает ре-рендер.

---

**Что такое батчинг и как React 18 изменил его поведение?**

Батчинг означает что React группирует несколько вызовов `setState` из одного обработчика событий в один ре-рендер. В React 17 и ниже батчинг работал только внутри обработчиков событий React — `setState` внутри `setTimeout`, `Promise.then` или нативного DOM-события вызывал отдельный ре-рендер на каждый вызов. React 18 ввёл **автоматический батчинг**: все обновления состояния группируются вне зависимости от источника. Для отказа от батчинга для конкретного обновления оберни его в `flushSync(() => setState(...))` — это вызывает синхронный рендер до возврата.

---

**Что делает StrictMode и почему он дважды вызывает рендер компонентов в разработке?**

`<React.StrictMode>` намеренно вызывает функции компонентов дважды в разработке (только фаза рендера, не коммит). Цель — обнаружить сайд-эффекты в теле рендера: если рендер действительно чистый, двойной вызов даёт одинаковый результат. Если он мутирует внешнее состояние, логирует или имеет неидемпотентное поведение — двойной вызов делает это видимым. В production двойного вызова нет. StrictMode также обнаруживает устаревшие API и предупреждает об отсутствующей очистке в эффектах.

## Группа 2: Основы хуков

**Почему правила хуков не произвольны — что сломается при условном вызове хука?**

Хуки хранятся как односвязный список на Fiber-узле. React идентифицирует каждый хук по его позиции в списке (первый вызов → первый узел, второй → второй). Если вызов хука пропущен на условной ветке, каждый хук после него сдвигается на одну позицию — React читает неправильный узел для каждого последующего хука. Результат — молчаливая порча состояния: состояние, принадлежавшее хуку N, теперь читается как состояние хука N-1. Правило существует потому что в связанном списке нет имён, только позиции.

---

**Какова разница между первым рендером и обновлением для useState?**

При монтировании React создаёт новый узел хука на Fiber, вызывает инициализатор (или запускает ленивую функцию инициализации) и сохраняет начальное значение. При обновлении React проходит по существующему связанному списку чтобы найти правильный узел и читает его текущее значение. Сеттер, возвращаемый `useState`, стабилен между рендерами — создаётся один раз и никогда не меняется. Ленивый инициализатор (`useState(() => expensiveCalc())`) вызывается только при монтировании; передача выражения напрямую (`useState(expensiveCalc())`) вызывает его при каждом рендере, хотя результат отбрасывается после монтирования.

---

**Что такое устаревшее замыкание (stale closure) и когда оно появляется с useEffect?**

Устаревшее замыкание возникает когда функция захватывает переменную из внешней области видимости, но та переменная была обновлена в следующем рендере — функция всё ещё держит старое значение из момента своего создания. Классический пример: `useEffect(() => { const id = setInterval(() => console.log(count), 1000); return () => clearInterval(id); }, [])` — `count` захвачен при монтировании и никогда не обновляется. Исправление: добавь `count` в массив зависимостей (эффект перезапускается при изменении count), или используй ref для всегда актуального значения без перезапуска эффекта.

---

**Какая разница между useEffect и useLayoutEffect?**

Оба выполняются после рендера. `useEffect` выполняется после того как браузер отрисовал — он асинхронен относительно отрисовки и не блокирует визуальное обновление. `useLayoutEffect` выполняется после того как React применил DOM-мутации, но **до** отрисовки браузером — он блокирует отрисовку, что необходимо когда нужно измерить DOM-лейаут и обновить состояние на основе этих измерений (чтобы избежать мерцания неправильного лейаута). Правило: используй `useEffect` по умолчанию; применяй `useLayoutEffect` только когда видишь визуальное мерцание, которое можно устранить только синхронным чтением DOM-измерений после коммита. На сервере `useLayoutEffect` ничего не делает (нет DOM) и выводит предупреждение.

---

**Когда запускается очистка useEffect и в каком порядке?**

Очистка выполняется в двух случаях: (1) перед повторным запуском эффекта из-за изменения зависимости — сначала выполняется очистка предыдущего эффекта, затем запускается новый; (2) при размонтировании компонента. В StrictMode (разработка) цикл очистка → эффект намеренно выполняется дважды при монтировании. Порядок при изменении зависимости: `[очистка предыдущего эффекта] → [новый эффект]`. Очистка — это функция, возвращаемая из колбэка эффекта; если колбэк не возвращает функцию, очистки нет.

## Группа 3: Продвинутые хуки

**Когда useMemo реально улучшает производительность, а когда вредит?**

`useMemo` помогает когда: вычисление действительно дорогое (> 1ms), выполняется при каждом рендере, и входные данные меняются редко. `useMemo` вредит (или нейтрален но добавляет шум) когда: вычисление дешёвое (array.filter на < 100 элементов), или массив зависимостей всё равно меняется при каждом рендере (объект/функция создаются inline), делая `useMemo` бесполезным — кэш никогда не используется. Замеряй через `console.time` перед добавлением `useMemo`. Стоимость сравнения плюс аллокация замыкания самого `useMemo` не бесплатны.

---

**Каковы два валидных сценария использования useCallback?**

(1) Передача колбэка как пропа в `React.memo`-обёрнутый дочерний компонент — без `useCallback` при каждом рендере родителя создаётся новая ссылка на функцию, разрушая оптимизацию memo. (2) Включение функции в массив зависимостей другого хука (`useEffect`, `useCallback`, `useMemo`) — без стабилизации ссылки хук будет перезапускаться при каждом рендере. `useCallback` — это `useMemo` для функций: `useCallback(fn, deps)` идентичен `useMemo(() => fn, deps)`.

---

**Каковы четыре применения useRef помимо хранения DOM-ссылок?**

(1) **Паттерн последнего значения**: хранить самое актуальное значение пропа или состояния в ref для доступа внутри устаревшего замыкания без перезапуска эффекта. (2) **Предыдущее значение**: захватывать значение из предыдущего рендера, обновляя ref в `useEffect` после рендера. (3) **Переменные экземпляра**: хранить мутируемые значения, которые должны сохраняться между рендерами, но не должны вызывать ре-рендер при изменении (таймеры, подписки, WebSocket-инстансы). (4) **DOM refs**: исходный сценарий использования — прикрепление к элементу для вызова императивных DOM API.

---

**Что делает useImperativeHandle и когда его применять?**

`useImperativeHandle` настраивает значение ref, доступное родителю когда тот прикрепляет ref к компоненту. Без него прикрепление ref к кастомному компоненту открывает весь DOM-элемент или ничего. С ним ты открываешь только контролируемое API: `useImperativeHandle(ref, () => ({ focus, clear }), [])` — родитель может вызывать только `focus` и `clear`, а не произвольные DOM-методы. Используй для примитивов библиотеки компонентов (кастомные инпуты, видеоплееры, управление фокусом модала) где родитель легитимно нуждается в императивном доступе, но ты хочешь ограничить поверхность API. В React 19 `ref` — обычный проп, `forwardRef` больше не нужен.

---

**Почему useId генерирует ID, стабильные между сервером и клиентом?**

`useId` выводит своё значение из позиции компонента в Fiber-дереве (пути в дереве), а не из счётчика, который инкрементируется независимо на клиенте и сервере. Поскольку структура дерева компонентов одинакова на сервере и клиенте, производный ID тоже одинаков — нет hydration mismatch. `Math.random()` и инкрементирующиеся счётчики расходятся потому что сервер и клиент выполняют один и тот же код независимо и дают разные результаты. Используй `useId` для пар `htmlFor`/`id`, ARIA-атрибутов и любых случаев где нужен стабильный, уникальный, SSR-безопасный ID. Никогда не используй для React `key`.

## Группа 4: Контекст и управление состоянием

**Что заставляет всех потребителей контекста перерисовываться и как это предотвратить?**

React сравнивает значения контекста через `Object.is`. Когда Provider перерисовывается, создаётся новый объект для `value={{ user, setUser }}` — ссылка меняется даже если `user` и `setUser` не изменились. Все потребители перерисовываются потому что `Object.is(prev, next)` возвращает false. Исправление: мемоизируй значение контекста через `useMemo(() => ({ user, setUser }), [user, setUser])`. `React.memo` на потребителе **не помогает** — он предотвращает только ре-рендеры от изменения пропсов, но не контекста.

---

**Почему состояние и dispatch стоит помещать в отдельные контексты?**

`dispatch` (из `useReducer`) имеет стабильную ссылку — она никогда не меняется между рендерами. Состояние меняется при каждом обновлении. Если оба помещены в один объект контекста, каждый компонент, потребляющий контекст, перерисовывается при каждом изменении состояния — даже компоненты, которым нужен только dispatch и не важны текущие значения состояния. Разделение на `StateContext` и `DispatchContext` означает что компоненты, которым нужен только dispatch (кнопки отправки, создатели экшнов), никогда не перерисовываются из-за изменений состояния.

---

**Когда использовать Zustand/Redux вместо Context + useReducer?**

Context + useReducer достаточен для: редко обновляемых данных (авторизация пользователя, тема, язык), данных разделяемых между несколькими компонентами, простой структуры состояния. Переходи на внешнее хранилище когда: обновления частые (каждое нажатие клавиши, каждый кадр анимации), много несвязанных компонентов нуждаются в одном и том же слайсе, нужна отладка путешествием во времени, или нужна персистентность/синхронизация состояния с внешними системами. Ключевое различие: обновления Zustand/Redux не вызывают ре-рендеры React пока компонент не подписан на конкретный изменившийся слайс. Context перерисовывает всех потребителей вне зависимости от того какая часть значения изменилась.

## Группа 5: Fiber, согласование и производительность

**Какую проблему решила архитектура Fiber, которую старый стековый согласователь не мог?**

Старый стековый согласователь был рекурсивным синхронным алгоритмом — однажды запустившись, он выполнялся до конца, блокируя главный поток. Для больших деревьев это могло занимать 50–100ms, вызывая пропуск кадров и неотзывчивый ввод. Fiber заменил рекурсию связанным списком (указатели child → sibling → return), превратив обход дерева в итеративный цикл, который можно приостановить после любого узла. Это обеспечивает нарезку времени: React работает кусками ~5ms и передаёт управление браузеру между кусками, сохраняя частоту кадров при рендере больших деревьев.

---

**Что такое техника двойной буферизации в архитектуре Fiber?**

React поддерживает два Fiber-дерева одновременно: **текущее** (то что отображается) и **work-in-progress** (то что строится). Вся работа согласования происходит в work-in-progress дереве, оставляя текущее дерево нетронутым и всегда готовым к рендеру. Когда work-in-progress дерево завершено, React атомарно меняет их местами в одном коммите — браузер всегда видит полный, консистентный UI. Указатель `alternate` на каждом Fiber-узле связывает текущую и work-in-progress версии одного и того же компонента.

---

**Объясни три правила согласования, которые React использует при сравнении деревьев.**

(1) **Разный тип → размонтировать и перемонтировать**: если элемент на позиции меняется с `<div>` на `<span>` или с `<ComponentA>` на `<ComponentB>`, React полностью размонтирует старое поддерево и монтирует новое. (2) **Тот же тип → обновить на месте**: React обновляет пропсы существующего Fiber, сохраняя DOM-узлы и состояние компонента. (3) **Ключи → идентичность**: среди списка дочерних React сопоставляет детей по `key`, а не по позиции. Дочерний элемент с key "a" — это тот же элемент где бы он ни находился в списке — React обновляет его, а не размонтирует и перемонтирует.

---

**Что не так с использованием индекса массива как ключа?**

При переупорядочивании, фильтрации или вставке элементов не в конец, ключи на основе индекса заставляют React сопоставлять неправильные идентичности элементов. Пример: список `[A, B, C]` с ключами `[0, 1, 2]` — удаление A даёт `[B, C]` с ключами `[0, 1]`. React думает что ключ 0 — это всё ещё A (был A раньше), поэтому обновляет Fiber A данными B вместо размонтирования A и переиспользования существующего Fiber B. Для неконтролируемых инпутов это приводит к появлению неправильного содержимого в поле. Используй стабильные, уникальные идентификаторы из данных. Исключение: индексные ключи безопасны когда список статичен и никогда не переупорядочивается.

---

**В чём разница между фазой рендера и фазой коммита?**

**Фаза рендера** (также называемая согласованием) — чистая и прерываемая: React вызывает функции компонентов, сравнивает вывод с предыдущим деревом и строит work-in-progress Fiber-дерево. DOM-мутации не происходят. Может быть приостановлена, перезапущена или отменена. **Фаза коммита** — синхронная и непрерываемая: React применяет накопленные DOM-мутации (подфазы before mutation → mutation → layout), выполняет очистку и колбэки `useLayoutEffect`, затем передаёт управление браузеру для отрисовки. После отрисовки React выполняет очистку и колбэки `useEffect` (фаза пассивных эффектов).

---

**Что означают приоритеты lanes в React и как их использует startTransition?**

Lanes — это битмаск-система где каждый lane представляет уровень приоритета: `SyncLane` (наивысший, для дискретного ввода пользователя), `InputContinuousLane` (перетаскивание, наведение), `DefaultLane` (обычный setState), `TransitionLane` (низкоприоритетная фоновая работа), `IdleLane` (наименьший). React планирует работу по lanes и обрабатывает более приоритетные lanes первыми. `startTransition` помечает обновление состояния `TransitionLane` — наименьшим интерактивным приоритетом. Если более приоритетное обновление (ввод в инпут) приходит пока идёт рендер перехода, React прерывает переход, обрабатывает высокоприоритетное обновление, затем возобновляет или перезапускает переход.

## Группа 6: Паттерны производительности

**Каковы три условия, которые должны выполняться одновременно для пользы от React.memo?**

(1) Компонент рендерится часто — его родитель перерисовывается часто из-за несвязанных изменений состояния. (2) Ре-рендер дорогой — компонент имеет много дочерних или выполняет тяжёлую работу во время рендера. (3) Пропсы референтно стабильны между рендерами — примитивы не меняются, а объекты/массивы/функции мемоизированы через `useMemo`/`useCallback`. Если условие 3 не выполняется, сравнение пропсов всегда возвращает "изменились" потому что при каждом рендере родителя создаются новые ссылки — `React.memo` добавляет накладные расходы на сравнение без всякой пользы.

---

**Что такое паттерн children (подъём контента) и какую проблему производительности он решает?**

Когда компонент владеет быстро меняющимся состоянием (позиция мыши, скролл, таймеры), любой импортируемый им компонент будет перерисовываться при каждом изменении состояния. Исправление: вместо импорта тяжёлого компонента принимай его через `children`. Родительский компонент, рендерящий тяжёлый компонент, теперь перерисовывается только при изменении собственного состояния — он не отслеживает позицию мыши. Fiber тяжёлого компонента уже создан дедом; `MouseTracker` просто получает его как готовое значение и помещает в вывод не вызывая свежего рендера.

---

**Когда применять виртуализацию и когда она не помогает?**

Виртуализация (react-window, @tanstack/react-virtual) решает проблему **количества DOM-узлов**: 10 000 строк списка создают 10 000 DOM-узлов, которые дорого прикреплять, измерять и отрисовывать. Виртуализация рендерит только видимые строки плюс небольшой буфер overscan. Она не решает проблему **частоты ре-рендеров** — если видимые строки перерисовываются при каждом нажатии клавиши, виртуализация не поможет. Применяй когда: список 500+ элементов, каждая строка нетривиальна, и пользователь активно прокручивает. Не применяй когда: список менее ~100 элементов (накладные расходы не оправданы), или узкое место совсем в другом.

---

**Что измеряет React DevTools Profiler и что не измеряет?**

Profiler измеряет **фазу рендера**: время вызова функций компонентов и сравнения вывода. Он не измеряет: длительность фазы коммита (применение DOM-мутаций), время выполнения `useEffect`, и время отрисовки браузером. Компонент может выглядеть быстрым в Profiler, но всё равно вызывать медленное воспринимаемое время если его коммит генерирует много дорогих DOM-мутаций или `useEffect` делает тяжёлую работу. Для полного времени кадра включая коммит и отрисовку используй вкладку Performance в Chrome DevTools. Компонент `<Profiler>` (не DevTools) работает в production-сборках и может отправлять `actualDuration` и `baseDuration` в аналитику.

---

## Группа 7: Конкурентные возможности

**В чём разница между startTransition и useDeferredValue?**

Оба помечают работу как низкоприоритетную, но отличаются местом применения контроля. `startTransition` используется когда ты **владеешь сеттером состояния** — оборачиваешь вызов `setState`: `startTransition(() => setResults(filter(data, query)))`. `useDeferredValue` используется когда ты **получаешь значение извне** (проп, состояние родителя, библиотека) и хочешь отложить его локально: `const deferredQuery = useDeferredValue(query)`. Используй тот, что соответствует источнику значения — если контролируешь обновление, используй `startTransition`; если получаешь его, используй `useDeferredValue`.

---

**Как работает Suspense механически — что фактически означает "суспендировать"?**

Когда компонент "суспендируется", он бросает специальное значение (Promise) во время фазы рендера. React перехватывает брошенное значение на ближайшем `<Suspense>`-boundary и показывает его `fallback` UI. React регистрирует колбэк на брошенном Promise — когда тот резолвится, React повторяет рендер суспендированного компонента. При повторной попытке компонент не должен бросать снова (данные должны быть готовы) — он рендерится нормально и React коммитит результат, заменяя fallback реальным UI. Библиотеки данных (React Query, SWR) реализуют этот протокол кешируя запросы и бросая in-flight Promise при первом вызове.

---

**Почему оборачивание навигации в startTransition предотвращает мерцание Suspense fallback?**

Без `startTransition`: навигация считается срочной. React немедленно переключается на Suspense boundary новой страницы, которая сразу показывает fallback — даже если данные приходят через 50ms, видно мерцание. С `startTransition`: React помечает обновление как переход. Вместо замены текущей страницы fallback'ом, React оставляет текущую страницу видимой (устанавливая `isPending = true`) пока новая страница рендерится в фоне. Когда Suspense новой страницы резолвится и контент готов, React коммитит полный переход за один шаг — старая страница исчезает и новая появляется без промежуточного пустого экрана.

---

**Отличается ли useDeferredValue от дебаунсинга?**

Да, это разные механизмы. Дебаунсинг задерживает само **обновление состояния** — сеттер вызывается только после срабатывания таймера, промежуточные значения отбрасываются. `useDeferredValue` получает уже обновлённое значение и говорит React обрабатывать его с низким приоритетом — текущий рендер использует предыдущее отложенное значение пока новый рендер обрабатывается в фоне. Нет таймера, нет задержки, нет отброшенных обновлений: React в конечном счёте всегда отрендерит с последним значением. Воспринимаемая задержка возникает от депроритизации React отложенного рендера при наличии более приоритетной работы, а не от искусственной задержки.

---

## Группа 8: Server Components и React 19

**Что такое граница сериализации и почему функции не могут её пересекать?**

Когда Server Component рендерит Client Component, вывод сериализуется в RSC payload — JSON-подобный проводной формат. Пересекать могут только сериализуемые значения: примитивы, массивы, простые объекты, React-элементы. Функции нельзя сериализовать потому что сериализация кода — это угроза безопасности (произвольное выполнение кода на клиенте). Поэтому обработчики событий должны жить в Client Components — они не могут быть определены на сервере и переданы как пропсы клиенту. React-элементы (вывод рендера другого Server Component) могут пересекать границу, потому что это простые объекты, а не код.

---

**Означает ли 'use client' что компонент выполняется только на клиенте?**

Нет. `'use client'` маркирует **границу** между деревом серверных компонентов и деревом клиентских. Client Components по-прежнему выполняются на сервере при SSR/SSG для генерации начального HTML — они просто также выполняются на клиенте при гидратации и последующих обновлениях. `'use client'` означает: этот компонент и его поддерево используют клиентские возможности React (состояние, эффекты, browser API) и должны быть включены в JavaScript-бандл, отправляемый браузеру. Это не означает "никогда не выполнять на сервере".

---

**В чём разница между Server Actions и API routes?**

API routes — явные HTTP-эндпоинты: определяешь URL, парсишь тело запроса, обрабатываешь аутентификацию, возвращаешь ответ. Server Actions — функции, помеченные `'use server'`, которые фреймворк автоматически открывает как POST-эндпоинты. Вызываешь их как обычные функции из Client Components; фреймворк обрабатывает сериализацию, транспорт и десериализацию. Server Actions интегрируются с React form model (`<form action={serverAction}>`) и работают без JavaScript (прогрессивное улучшение). Могут вызывать `revalidatePath`/`revalidateTag` для инвалидации кэшей Next.js напрямую. Используй API routes для публичных API, потребляемых третьими сторонами; Server Actions — для внутренних сабмитов форм и мутаций из UI.

---

**Каковы четыре основных причины hydration mismatch?**

(1) **Browser API в рендере**: обращение к `window`, `localStorage` или `document` в функции рендера — они undefined на сервере. (2) **Рендер времени/даты**: `new Date().toLocaleTimeString()` даёт разные значения когда сервер рендерит vs когда клиент гидратирует миллисекунды спустя. (3) **Случайные значения**: `Math.random()` генерирует разные значения на сервере и клиенте. (4) **Условный рендер на основе browser-only данных**: `typeof window !== 'undefined'` на сервере false, на клиенте true — разный вывод. Паттерны исправления: используй `useEffect` для чтения browser-only значений после гидратации; используй стабильные/детерминированные значения (хэш user ID → цвет); используй `suppressHydrationWarning` для известных намеренных несоответствий.

---

**В чём разница между useActionState и useFormStatus в React 19?**

`useActionState` управляет **состоянием, возвращаемым экшном формы** — оборачивает функцию экшна, хранит результат (данные успеха / ошибки) и предоставляет `isPending` для выполняющегося экшна. Используется в компоненте, владеющем формой, и предоставляет проп `action`. `useFormStatus` читает **статус отправки ближайшего родительского `<form>`** — предоставляет поля `pending`, `data`, `method` и `action`. Предназначен для компонентов, вложенных в форму (кнопка отправки, индикатор загрузки), которые не владеют экшном, но должны знать статус формы. Они компонуются: `useActionState` предоставляет `action` форме; `useFormStatus` читает статус этой формы изнутри дочерних компонентов.

---

**Что фактически делает React Compiler и что он не может исправить?**

React Compiler — это build-time Babel/SWC-плагин, который статически анализирует компоненты, определяет какие значения "реактивны" (зависят от пропсов, состояния или других реактивных значений), и автоматически вставляет `useMemo`, `useCallback` и мемоизацию на уровне компонента там где это безопасно. Он устраняет необходимость вручную писать большую часть мемоизации. Что он не может исправить: структурные проблемы дерева компонентов — состояние, живущее слишком высоко (из-за чего большие поддеревья перерисовываются), inline-объекты значений Context, или ре-рендеры вызванные архитектурными решениями. Compiler оптимизирует отдельные компоненты; он не может исправить дизайн где не тот компонент владеет состоянием.

## Группа 9: Паттерны и архитектура

**Что такое паттерн Compound Components и чем он отличается от API на основе пропсов?**

Compound Components разбивают сложный компонент на родителя (владеет состоянием и логикой координации, открывает её через контекст) и несколько дочерних компонентов (потребляют контекст без prop drilling). Потребитель собирает UI из суб-компонентов и полностью контролирует layout. API на основе пропсов (`<Select options={...} renderOption={...} maxHeight={...}>`) вынуждает автора компонента предвидеть каждую потребность кастомизации. Compound Components (`<Select><Select.Option value="a">A</Select.Option></Select>`) делегируют layout и компоновку потребителю — автор компонента должен определить только логику координации.

---

**Что перехватывает Error Boundary и что нет?**

Error Boundary перехватывает ошибки в **фазе рендера** (тело функции компонента), **методах жизненного цикла** (lifecycle класс-компонентов) и **конструкторах** дочерних компонентов. Он **не** перехватывает: ошибки в обработчиках событий (используй try/catch внутри обработчика и сохраняй ошибку в состоянии), ошибки в асинхронном коде (setTimeout, отклонения Promise, async/await — они выполняются вне цикла рендера React), и ошибки в самом Error Boundary. Чтобы async-ошибка попала к boundary, перехвати её вручную и запиши в состояние — React бросит значение состояния при следующем рендере, и boundary его поймает.

---

**Когда использовать Portal и чем он отличается от обычного рендера inline?**

Portal рендерит дочерний элемент в DOM-узел вне корневого элемента React, оставляя дочерний внутри React-дерева (контекст и всплытие событий работают нормально). Используй Portal когда CSS-предок создаёт проблему содержания: `overflow: hidden` обрезает контент, низкий `z-index` скрывает его за соседями, или CSS-трансформ создаёт новый stacking context. Классический пример: модальный диалог внутри скроллируемой карточки с `overflow: hidden`. Без Portal он обрезается; с Portal рендерится в `document.body` и не подвержен ни одному CSS-ограничению предка.

---

**Почему Higher-Order Components вышли из моды по сравнению с кастомными хуками?**

HOC работают, но имеют накапливающиеся проблемы: (1) каждый HOC добавляет компонент-обёртку видимый в DevTools, делая трассировки запутанными; (2) несколько HOC, внедряющих одноимённый проп, молча перезаписывают друг друга; (3) TypeScript-типы требуют механического `Omit<P, 'injectedProp'>` boilerplate; (4) refs требуют явного пробрасывания через `forwardRef`. Кастомные хуки достигают того же переиспользования логики как обычные функции, чьи возвращаемые значения именуются вызывающим кодом — никаких компонентов-обёрток, никаких коллизий пропсов, никакого пробрасывания refs, чистые TypeScript-типы. Единственный оставшийся валидный сценарий для HOC — оборачивание Error Boundary, которые должны быть класс-компонентами.

---

**В чём разница между контролируемым и неконтролируемым компонентом, и почему React Hook Form использует неконтролируемые инпуты по умолчанию?**

Контролируемый инпут хранит значение в React-состоянии — `value={value} onChange={setValue}`. Каждое нажатие клавиши вызывает ре-рендер. Неконтролируемый инпут управляет собственным значением в DOM — `defaultValue` устанавливает начальное значение, а текущее читается через ref или `FormData` при сабмите. React Hook Form использует неконтролируемые инпуты чтобы избежать ре-рендеров при каждом нажатии клавиши — для формы с 20 полями контролируемый подход мог бы вызывать 20 ре-рендеров на цепочку нажатий. React Hook Form регистрирует инпуты через ref и вызывает ре-рендеры только для изменений состояния валидации и явных обновлений на уровне формы, а не для отдельных вводимых символов.
