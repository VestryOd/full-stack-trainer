# Основы хуков

## Зачем существуют хуки — проблема классовых компонентов

До хуков (React < 16.8) стейтовая логика жила в классовых компонентах. Главная проблема была не в синтаксисе — а в том, что **стейтовая логика была неотделима от компонента, в котором она использовалась**. Два компонента, нуждающихся в одной и той же логике подписки, либо дублировали её, либо использовали Higher-Order Components / render props — что порождало глубоко вложенные деревья компонентов из чисто организационных соображений («wrapper hell»). Хуки позволяют извлечь стейтовую логику в функцию и переиспользовать её между компонентами без изменения иерархии.

Вторичные проблемы, которые решают хуки:
- Привязка `this` в методах класса (постоянный источник багов)
- Логика, разбросанная по lifecycle-методам: связанный код размазан между `componentDidMount`, `componentDidUpdate`, `componentWillUnmount`
- Классы хуже минифицируются и делают hot reload менее надёжным

## Связный список — почему Правила Хуков не произвольны

Хуки хранятся как **однонаправленный связный список**, прикреплённый к Fiber-узлу компонента. Нет словаря, нет имён, нет магической рефлексии. React отслеживает хуки по **порядку вызовов** — первый вызов хука это узел 1, второй — узел 2, и так далее.

```txt
FIBER NODE для <MyComponent>
│
└── memoizedState (первый хук)
      │  { state: 0, queue: ..., next: ─────────────────────────────┐ }
      │                                                               │
      └────────────────────── next ──────────────────────────────────▼
                                        { state: 'Alice', queue: ..., next: ──┐ }
                                                                                │
                                                                                ▼
                                                            { effect: ..., next: null }
```

При первом рендере React строит этот список. При каждом последующем рендере React **проходит тот же список в том же порядке** и сопоставляет каждый вызов `useState` / `useEffect` с его существующим узлом.

Именно поэтому:

```ts
// ПРАВИЛО: никогда не вызывайте хуки условно

// ✅ Правильно — одинаковое количество хуков при каждом рендере:
function Component({ show }: { show: boolean }) {
  const [a, setA] = useState(0);  // узел 1
  const [b, setB] = useState(''); // узел 2
  useEffect(() => {}, []);        // узел 3
  return <div>{a}</div>;
}

// ❌ Неправильно — условный хук, разное количество узлов:
function Component({ show }: { show: boolean }) {
  if (show) {
    const [a, setA] = useState(0); // узел 1 при одних рендерах, отсутствует при других
  }
  const [b, setB] = useState('');  // узел 1 или узел 2 в зависимости от `show`
  // React читает значение узла 2 в [b, setB] при рендере где show=false,
  // но этот узел 2 хранил значение `a` из предыдущего рендера.
  // Состояние теперь повреждено.
}
```

Правило линтера `eslint-plugin-react-hooks` проверяет это статически. Сообщение "Rendered more hooks than during the previous render" означает несовпадение длин списков.

**Почему не словарь?** React намеренно так устроен — никаких динамических имён хуков, строковых ключей, рефлексии. Связный список сохраняет реализацию минимальной и гарантирует стабильность идентичности хуков между рендерами, пока порядок вызовов стабилен.

## useState — механика изнутри

```tsx
const [count, setCount] = useState(0);
```

При **первом рендере** (mount) React:
1. Создаёт новый узел хука в связном списке
2. Запускает инициализатор (`0` — или функцию, если передать `() => expensiveCompute()`)
3. Сохраняет начальное состояние в `node.memoizedState`
4. Возвращает `[node.memoizedState, node.queue.dispatch]`

При **последующих рендерах** (update) React:
1. Находит существующий узел, проходя список до той же позиции
2. Обрабатывает все поставленные в очередь обновления (от вызовов `setCount`) по порядку
3. Сохраняет новое состояние в `node.memoizedState`
4. Возвращает обновлённые `[state, dispatch]`

### Ленивый инициализатор — когда его использовать

```tsx
// ❌ Выполняет дорогое вычисление при КАЖДОМ рендере:
const [state, setState] = useState(JSON.parse(localStorage.getItem('data') ?? '{}'));

// ✅ Выполняется только при монтировании (функциональная форма):
const [state, setState] = useState(() => JSON.parse(localStorage.getItem('data') ?? '{}'));
```

Передавайте функцию, когда начальное значение дорого вычислять. React вызовет её только один раз — при монтировании — и игнорирует при ре-рендерах.

### Сеттер состояния стабилен

Функция `setCount`, возвращаемая `useState`, **ссылочно стабильна** — это одна и та же ссылка на функцию во всех рендерах компонента. Поэтому её безопасно опускать из массивов зависимостей `useEffect` / `useCallback`:

```tsx
useEffect(() => {
  const timer = setInterval(() => {
    setCount(c => c + 1); // setCount стабилен — в deps добавлять не нужно
  }, 1000);
  return () => clearInterval(timer);
}, []); // ✅ пустой массив deps здесь правильный
```

## useEffect — полная ментальная модель

`useEffect` — это не «lifecycle-метод с другим именем». Правильная ментальная модель:

```txt
useEffect(setup, deps)

→ После каждого рендера, при котором deps изменились (сравнение Object.is):
    1. Запустить cleanup предыдущего эффекта (если был)
    2. Запустить setup

→ При размонтировании:
    3. Запустить cleanup последнего эффекта
```

Функция setup вызывается всегда **после** отрисовки экрана браузером (асинхронно, после commit phase). Это намеренно — не блокирует paint.

### Три формы массива зависимостей

```tsx
// Форма 1: нет массива — запускается после КАЖДОГО рендера
useEffect(() => {
  document.title = `Count: ${count}`;
});

// Форма 2: пустой массив — запускается ОДИН РАЗ после монтирования, cleanup при размонтировании
useEffect(() => {
  const sub = store.subscribe(handler);
  return () => sub.unsubscribe();
}, []);

// Форма 3: с зависимостями — запускается после монтирования И при каждом изменении зависимости
useEffect(() => {
  const sub = store.subscribe(userId, handler);
  return () => sub.unsubscribe();
}, [userId]); // переподписывается при изменении userId
```

**Типичная ловушка:** «хочу запустить только один раз» → пустой массив. Но если setup эффекта читает props или state, это устаревшие замыкания после первого рендера. Правильно: включайте всё, что эффект читает из области видимости компонента, в массив зависимостей. Правило `exhaustive-deps` ESLint проверяет это.

### Функции cleanup — что это и когда они вызываются

Функция, возвращаемая из `useEffect`, называется *функцией cleanup*. React вызывает её в двух ситуациях:
1. Перед повторным запуском эффекта (когда deps изменились)
2. При размонтировании компонента

```tsx
useEffect(() => {
  // SETUP: подписываемся
  const controller = new AbortController();
  fetch('/api/data', { signal: controller.signal })
    .then(res => res.json())
    .then(setData)
    .catch(err => { if (err.name !== 'AbortError') setError(err); });

  // CLEANUP: отменяем fetch если компонент размонтируется или deps изменились
  // (предотвращает установку state на размонтированный компонент)
  return () => controller.abort();
}, [userId]);
```

Без cleanup: если `userId` меняется до завершения fetch, два запроса гонятся; более медленный может завершиться последним и перезаписать результат более быстрого нового (**race condition**). Cleanup отменяет предыдущий fetch перед запуском нового.

### Почему эффекты запускаются после paint (и почему это важно)

```txt
Рендер → Commit → Отрисовка браузером → useEffect

Если бы useEffect блокировал отрисовку (как useLayoutEffect),
пользователь видел бы пустой экран до завершения эффекта —
плохо для всего асинхронного.
useEffect намеренно отложен.
```

Следствие: если ваш эффект читает измерения layout (размеры элементов, позицию скролла), к моменту его запуска браузер уже отрисовал страницу с потенциально устаревшими размерами. Используйте `useLayoutEffect` для этого случая.

## useLayoutEffect — что он решает

`useLayoutEffect` имеет ту же сигнатуру, что и `useEffect`, но срабатывает **синхронно после DOM-мутаций и до отрисовки браузером**:

```txt
Рендер → Commit (DOM-мутации) → useLayoutEffect → Отрисовка браузером → useEffect
```

Сценарий использования: читать layout из DOM и синхронно скорректировать его до того, как пользователь увидит промежуточное состояние.

```tsx
function Tooltip({ anchor }: { anchor: HTMLElement }) {
  const tooltipRef = useRef<HTMLDivElement>(null);
  const [position, setPosition] = useState({ top: 0, left: 0 });

  useLayoutEffect(() => {
    // Запускается ДО того, как браузер показал тултип в его начальной позиции.
    // Измеряем и перепозиционируем в одном цикле отрисовки.
    const rect = tooltipRef.current!.getBoundingClientRect();
    const anchorRect = anchor.getBoundingClientRect();
    setPosition({ top: anchorRect.bottom, left: anchorRect.left });
  }, [anchor]);

  return <div ref={tooltipRef} style={position}>...</div>;
}
```

С `useEffect`: тултип мигнёт на неправильной позиции один кадр (браузер отрисовал до запуска эффекта). С `useLayoutEffect`: DOM обновляется до отрисовки — никакого видимого мерцания.

**Оговорка для SSR:** `useLayoutEffect` пропускается на сервере (где нет DOM), React предупреждает об этом. Если компонент с `useLayoutEffect` рендерится на сервере, либо используйте `useEffect`, либо рендерите компонент только на клиенте.

## Ловушки массива зависимостей — полный список

### 1. Устаревшие замыкания (stale closures)

```tsx
function Timer() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      console.log(count); // захватывает count из рендера, когда запустился этот эффект
      // count всегда будет 0, если deps = []
    }, 1000);
    return () => clearInterval(id);
  }, []); // ← пропущен count в deps
}
```

Исправление: либо добавить `count` в deps (эффект пересоздаёт таймер при каждом изменении count), либо использовать функциональное обновление, чтобы вообще не читать count: `setCount(c => c + 1)`.

### 2. Объекты и массивы как deps (сравнение по ссылке)

```tsx
function Component({ config }: { config: { timeout: number } }) {
  useEffect(() => {
    const timer = setTimeout(fetch, config.timeout);
    return () => clearTimeout(timer);
  }, [config]); // config — новый объект при каждом рендере родителя → бесконечный цикл
}
```

Исправление: зависьте от примитивных значений вместо объекта:

```tsx
useEffect(() => { ... }, [config.timeout]);
```

Или мемоизируйте объект в родителе с помощью `useMemo` (см. статью о продвинутых хуках).

### 3. Функции как deps

```tsx
function Component({ onData }: { onData: (data: Data) => void }) {
  useEffect(() => {
    fetch('/api').then(onData); // onData может меняться при каждом рендере родителя
  }, [onData]); // → повторный fetch при каждом рендере родителя
}
```

Исправление: оберните `onData` в `useCallback` в родителе, или используйте `useEffectEvent` (React 19+) для захвата последней версии без добавления в deps.

### 4. Ложные негативы exhaustive-deps

Правило линтера `exhaustive-deps` ловит пропущенные deps, но не всегда обнаруживает проблемы с мутируемыми refs. Мутируемые refs (`ref.current`) намеренно исключаются из массивов deps — объект ref стабилен, но его содержимое может меняться незаметно. Это корректное поведение, но оно означает, что чтение `ref.current` внутри эффекта даёт актуальное значение, а не захваченный снимок, что может удивить.

## Ментальная модель «каждый рендер — снимок»

Каждый раз когда компонент рендерится, React вызывает вашу функцию. Этот вызов захватывает текущие значения props и state в замыкании. Setup `useEffect` запускается со значениями из рендера, который его запланировал — не с актуальными значениями на момент запуска эффекта.

```tsx
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    setTimeout(() => {
      // Этот `count` захвачен из рендера, запланировавшего этот эффект.
      // Если count был 0 когда эффект запустился, count будет 0 в этом callback
      // даже если setCount(5) был вызван до срабатывания таймаута.
      console.log(count);
    }, 3000);
  }, [count]);

  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}
```

Это не баг — это определённое поведение. У каждого рендера есть своя версия каждого значения. Если нужно всегда читать актуальное значение (не снимок), используйте `useRef` (разбирается в статье о продвинутых хуках).

## Типичные ловушки на интервью

**«Можно ли вызывать хуки внутри цикла?»**
Нет — количество и порядок вызовов хуков должны быть одинаковыми при каждом рендере. Вызов хуков внутри цикла давал бы разное количество хуков в зависимости от длины массива. Если нужен state для каждого элемента — вынесите каждый элемент в отдельный компонент.

**«В чём разница между `useEffect(() => {}, [])` и `componentDidMount`?»**
В StrictMode (разработка) `useEffect` с `[]` запускается дважды: mount → cleanup → mount. `componentDidMount` запускается один раз. Важнее то, что ментальная модель отличается: `componentDidMount` — «сделай это когда компонент смонтируется». `useEffect` — «синхронизируйся с этими значениями» — пустой массив говорит «нет значений для синхронизации, поэтому запускать только один раз».

**«Почему мой `useEffect` запускается бесконечно?»**
Почти всегда: объект или массив, созданный inline в компоненте, указан как зависимость. Объекты сравниваются по ссылке, поэтому новый `{}` или `[]` при каждом рендере всегда «изменился». Исправление: вынесите объект за пределы компонента, мемоизируйте с `useMemo`, или зависьте от примитивных свойств вместо объекта.

**«Безопасно ли делать `async` функции setup в useEffect?»**
Setup `useEffect` не может возвращать Promise — он должен возвращать либо ничего, либо функцию cleanup. `async`-функция всегда возвращает Promise, поэтому `useEffect(async () => { ... })` вызовет предупреждение и функция cleanup (если была) будет потеряна. Правильный паттерн:

```tsx
useEffect(() => {
  let cancelled = false;
  async function load() {
    const data = await fetch('/api').then(r => r.json());
    if (!cancelled) setData(data);
  }
  load();
  return () => { cancelled = true; };
}, []);
```

**«Запускается ли `useEffect` на сервере?»**
Нет. Ни `useEffect`, ни `useLayoutEffect` не запускаются при серверном рендеринге. Это означает, что любой код, который должен работать только в браузере (доступ к DOM, localStorage, браузерные API), должен быть внутри `useEffect`. Прямой доступ к этим API в теле компонента вызывает ошибки гидратации.
