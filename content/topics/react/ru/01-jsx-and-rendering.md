# JSX и конвейер рендеринга

## JSX — синтаксический сахар, но важно понимать, что именно он скрывает

JSX выглядит как HTML внутри JavaScript. Это точно, но неполно — за этим описанием скрыта механика, объясняющая почти каждое решение по производительности в React.

```tsx
// Что вы пишете
const element = <Button variant="primary" onClick={handleClick}>Сохранить</Button>;

// Что компилирует Babel/SWC
const element = React.createElement(
  Button,
  { variant: 'primary', onClick: handleClick },
  'Сохранить'
);

// React 17+ (новый JSX-трансформ) — импорт React не нужен:
import { jsx as _jsx } from 'react/jsx-runtime';
const element = _jsx(Button, { variant: 'primary', onClick: handleClick, children: 'Сохранить' });
```

`React.createElement` (или `_jsx` нового JSX-трансформа) не создаёт DOM-узел. Он создаёт **обычный JavaScript-объект** — React element:

```ts
{
  $$typeof: Symbol(react.element),  // помечает объект как React-элемент (защита от XSS)
  type: Button,                      // строка для DOM-тегов, функция/класс для компонентов
  key: null,
  ref: null,
  props: { variant: 'primary', onClick: handleClick, children: 'Сохранить' },
}
```

Этот объект **иммутабельный** и **дёшевый в создании** — просто выделение `{}`. Реальная работа с DOM происходит намного позже, в отдельной фазе.

### Почему `$$typeof` — Symbol

`JSON.parse` не может породить Symbol. Если пользовательский JSON случайно попадёт в рендер (`{JSON.parse(userInput)}`), у него никогда не будет `$$typeof: Symbol(react.element)` — React отклонит его. Это защита от класса XSS-уязвимостей через опасный рендеринг API-ответов.

## Две фазы каждого рендера: Render и Commit

Процесс рендеринга в React разбит на две принципиально разных фазы:

```txt
RENDER PHASE (чистая, прерываемая)
────────────────────────────────────
  1. React вызывает функцию вашего компонента с текущими props/state
  2. Компонент возвращает дерево React-элементов (обычных объектов)
  3. React запускает reconciler (алгоритм диффа Fiber)
  4. React вычисляет список нужных изменений DOM
  → DOM НЕ мутируется

COMMIT PHASE (нечистая, синхронная, непрерываемая)
────────────────────────────────────────────────────
  1. React применяет все DOM-мутации из шага 4
  2. Запускает layout effects: cleanup useLayoutEffect → callback useLayoutEffect
  3. Браузер рисует экран
  4. Запускает passive effects: cleanup useEffect → callback useEffect
```

Это разделение не деталь реализации — у него есть наблюдаемые последствия:

- Можно вызывать `setState` во время рендера (осторожно) — render phase не трогает DOM.
- `useLayoutEffect` срабатывает синхронно *после* DOM-мутаций, но *до* отрисовки браузером — блокирует paint, подходит для измерения layout.
- `useEffect` срабатывает *после* того, как браузер уже отрисовал экран — правильное место для подписок и невизуальных побочных эффектов.
- В StrictMode / concurrent mode render phase может быть прервана, отброшена и запущена заново. Если в вашем рендере есть побочные эффекты — они выполнятся дважды. Commit phase никогда не прерывается.

## Что вызывает повторный рендер — полный список

```txt
ИСТОЧНИК                        ЧТО ПРОИСХОДИТ
─────────────────────────────────────────────────────────────
setState / useState setter       планирует повторный рендер компонента и его поддерева
useReducer dispatch              то же самое
Изменение значения Context       все consumers этого Context перерисовываются
Повторный рендер родителя        дети перерисовываются (если не memo'd)
forceUpdate (класс)              обходит shouldComponentUpdate, перерисовывает
```

Ключевое понимание: **повторный рендер — это не обновление DOM**. Повторный рендер означает «React снова вызвал вашу функцию компонента». Будет ли DOM обновлён — зависит целиком от того, найдёт ли reconciler различия.

```tsx
function Counter() {
  const [count, setCount] = useState(0);

  // Повторный рендер инициирован setCount.
  // Если count не изменился, React всё равно вызовет Counter() снова,
  // но после диффа не сделает никаких изменений в DOM.
  return <div>{count}</div>;
}
```

### Оптимизация «то же значение» (Object.is bail-out)

```tsx
const [count, setCount] = useState(0);
setCount(0); // count уже равен 0
```

React использует `Object.is` для сравнения нового значения стейта с текущим. Если они идентичны, React **может** пропустить ре-рендер — но только после первого пропуска (чтобы эффекты успели выполниться хотя бы раз). Поэтому установка state на тот же объект-ссылку не вызывает ре-рендер, а новый объект с той же структурой — вызывает:

```tsx
// Нет ре-рендера (та же ссылка):
setUser(user);

// Есть ре-рендер (новая ссылка, даже если структурно идентично):
setUser({ ...user });
```

## Почему обновления state батчатся — и что изменилось в React 18

Батчинг означает, что React собирает несколько вызовов `setState` из одного события и применяет их в одном проходе ре-рендера, вместо того чтобы перерисовывать после каждого вызова.

**До React 18** — батчинг работал только внутри React event handlers:

```tsx
function handleClick() {
  setA(1); // нет ре-рендера
  setB(2); // нет ре-рендера
}        // → один ре-рендер с обоими обновлениями

// НЕ батчилось (до React 18):
setTimeout(() => {
  setA(1); // ре-рендер здесь
  setB(2); // ещё один ре-рендер
}, 0);
```

**React 18 — автоматический батчинг везде:**

```tsx
// Всё это теперь батчится в React 18:
setTimeout(() => { setA(1); setB(2); });
fetch('/api').then(() => { setA(1); setB(2); });
element.addEventListener('click', () => { setA(1); setB(2); });
```

Если нужно принудительное синхронное небатченное обновление (редко), `flushSync` из `react-dom` выходит из батчинга:

```tsx
import { flushSync } from 'react-dom';

flushSync(() => setA(1)); // DOM обновился здесь
flushSync(() => setB(2)); // DOM обновился ещё раз
```

**Почему батчинг важен:** два вызова `setState` → один вызов функции (вашего компонента) → один проход reconciliation → одна DOM-мутация. Без батчинга: два вызова функции, два прохода, две DOM-мутации. Для компонентов с многими обновлениями state в одном обработчике разница может составлять 2–10x по производительности.

## Функциональные обновления — правильный паттерн, когда новый state зависит от старого

```tsx
// Неправильно — захватывает устаревшее closure-значение count:
setCount(count + 1);
setCount(count + 1); // всё ещё использует то же устаревшее `count`

// Правильно — использует гарантированно актуальное значение:
setCount(c => c + 1);
setCount(c => c + 1); // каждый updater получает результат предыдущего
```

Функциональные обновления необходимы, когда вы вызываете `setState` несколько раз в одном батче и каждый вызов зависит от результата предыдущего. Также это безопасный паттерн внутри `useEffect` с пустым массивом зависимостей, когда эффект должен инкрементировать счётчик.

## Double-rendering в StrictMode: объяснение

`<React.StrictMode>` в режиме разработки вызывает функцию вашего компонента **дважды** за каждый рендер, дважды вызывает инициализаторы состояния, дважды вызывает редюсеры `useReducer`, и дважды вызывает функции setup в `useEffect` (намеренно запуская cleanup и повторный setup).

```txt
ЧТО REACT ДЕЛАЕТ В STRICT MODE (только dev):
  1. Вызывает функцию компонента → сохраняет вывод
  2. Вызывает функцию компонента СНОВА → проверяет, что вывод тот же
  3. Если разный: выводит предупреждение

ЗАЧЕМ: проверяет, что ваш рендер — чистая функция от props + state.
       Рендер с побочными эффектами (мутация глобального, запись в DOM)
       будет вести себя непредсказуемо в concurrent mode, где React
       может отбросить рендер на полпути и перезапустить его.
```

Double-invocation происходит **только в разработке**, не в production. Это исключительно инструмент проверки чистоты.

```tsx
let renderCount = 0;

function MyComponent() {
  renderCount++; // НЕПРАВИЛЬНО: мутирует внешнее состояние
  return <div>Отрендерено {renderCount} раз</div>;
}

// В StrictMode dev: рендерит "Отрендерено 2 раза", потом "4 раза" и т.д.
// В production:     рендерит "Отрендерено 1 раз", потом "2 раза" и т.д.
// Разница обнажает баг — renderCount не идемпотентен.
```

Double-invocation `useEffect` (mount → cleanup → mount) добавили в React 18, чтобы проверить, что эффекты корректно убирают за собой. Классический сценарий, который это ловит:

```tsx
useEffect(() => {
  const sub = eventBus.subscribe(handler);
  // Если вы забыли cleanup:
  // return () => eventBus.unsubscribe(sub);
}, []);
// В StrictMode: subscribe вызывается дважды, unsubscribe — никогда.
// Вы увидите дублирующиеся обработчики событий в dev — баг обнаружен.
```

## Вывод рендера — это не HTML, а описание

Путаница между JSX и HTML — корень нескольких распространённых ошибок:

```tsx
// Это НЕ HTML-атрибуты. React маппит их на DOM-свойства:
<div className="box"        // className → element.className (не class)
     htmlFor="input-id"    // htmlFor → label.htmlFor (не for)
     onClick={handler}     // onClick → addEventListener('click', ...)
     style={{ color: 'red', fontSize: 16 }} // style — ОБЪЕКТ, не строка
/>

// React управляет системой событий (synthetic events):
// события всплывают через React-дерево, а не DOM-дерево.
// Важно для Portals (см. статью о паттернах).
```

Фрагменты (`<>...</>` или `<React.Fragment>`) нужны потому, что JSX-вызовы `React.createElement` принимают один корневой аргумент — нельзя вернуть несколько сиблингов как отдельные корни. Fragment — это React-элемент с `type: Symbol(react.fragment)`, который reconciler знает, как «разворачивать» без создания DOM-узла.

## Типичные ловушки на интервью

**«Вызов setState немедленно перерисовывает компонент?»**
Нет. `setState` планирует ре-рендер. React батчит обновления и обрабатывает их асинхронно (в том же тике, но после завершения текущего синхронного кода). Функция компонента вызывается снова в следующем проходе рендера, а не inline.

**«JSX медленнее прямого вызова `React.createElement`?»**
Нет. JSX компилируется в `React.createElement` (или новый JSX-трансформ) во время сборки — нулевых runtime-накладных расходов. Скомпилированный вывод идентичен.

**«Что React рендерит в React Native?»**
Не DOM. «Рендерер» заменяем: `react-dom` рендерит в браузерный DOM, `react-native` — в нативные UI-элементы мобильной платформы, `react-three-fiber` — в сцену Three.js. Ядро React (reconciler, Fiber) общее.

**«Может ли render phase иметь побочные эффекты?»**
Не должна. Render phase должна быть чистой функцией от props и state. В Concurrent Mode React может вызвать вашу функцию рендера несколько раз для одного commit или прервать и отбросить рендер в процессе. Побочные эффекты в рендере (подписки, мутации, таймеры) небезопасны, потому что могут выполниться ноль, один или несколько раз для одного «логического» рендера.

**«Зачем React использует Virtual DOM?»**
Формулировка немного неточная. React использует не «virtual DOM» для ускорения DOM-манипуляций — реальные DOM-манипуляции вызываются один раз за commit, что уже максимально быстро. Цель reconciler — вычислить *минимальный* набор изменений DOM между рендерами, чтобы незатронутые узлы не трогались вообще. Fiber-дерево — деталь реализации reconciler, а не «копия DOM».
