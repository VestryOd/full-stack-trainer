## Группа 1: MobX — модель реактивности

**1. Как MobX знает, какой компонент ре-рендерить при изменении observable?**

Отслеживание зависимостей во время выполнения. Когда `observer`-компонент рендерится, MobX запускает "контекст отслеживания". Каждое чтение `observable` во время этого рендера фиксируется как зависимость данного компонента. При изменении любого из них MobX запланирует ре-рендер именно этого компонента — и никакого другого. Поэтому гранулярность автоматическая: компонент, читающий только `store.total`, не ре-рендерится при изменении `store.isLoading`, даже если оба поля на одном объекте стора.

---

**2. В чём разница между `makeObservable` и `makeAutoObservable`, и когда каждый из них не работает?**

`makeObservable` требует явно перечислить каждое поле с аннотацией. `makeAutoObservable` выводит по соглашению: поля → `observable`, геттеры → `computed`, методы → `action`. `makeAutoObservable` не работает с наследованием — он не может безопасно пройти по цепочке прототипов для сбора аннотаций, поэтому наследование класса с `makeAutoObservable` бросает ошибку во время выполнения. Для наследования используйте `makeObservable` в каждом классе явно.

---

**3. Почему деструктуризация observable-объекта вне `observer` ломает реактивность?**

MobX регистрирует зависимость только в момент *чтения свойства внутри контекста отслеживания*. Деструктуризация извлекает обычные JavaScript-значения до того, как JSX рендерится — эти чтения происходят вне реактивного контекста рендера, поэтому MobX никогда не фиксирует зависимость. Полученные переменные — снимки, не живые ссылки.

```tsx
// ❌ Чтения происходят до JSX — зависимость не зарегистрирована
const MyComponent = observer(({ store }: { store: CartStore }) => {
  const { total, itemCount } = store; // обычные числа, не реактивные
  return <span>{total}</span>;        // ре-рендера при изменении НЕ будет
});

// ✅ Чтения происходят внутри JSX — в контексте отслеживания
const MyComponent = observer(({ store }: { store: CartStore }) => {
  return <span>{store.total} ({store.itemCount} товаров)</span>;
});
```

---

**4. Что такое `computed` — чем отличается от обычного геттера, и когда он НЕ пересчитывается?**

`computed` — мемоизированная, лениво вычисляемая производная. Отличается от обычного геттера в трёх аспектах: (1) результат кэшируется до изменения зависимостей; (2) если нет наблюдателей (ни компонент, ни reaction не читает его), он "засыпает" и не пересчитывается совсем; (3) если зависимости изменились, но *результат* одинаков (например, количество осталось 3 после добавления и удаления), MobX не уведомляет наблюдателей — в отличие от обычного геттера, который выполняется при каждом вызове.

---

**5. `autorun` vs `reaction` vs `when` — как выбирать?**

| | `autorun` | `reaction` | `when` |
|---|---|---|---|
| Запускается сразу | ✅ да | ❌ нет | ❌ нет |
| Получает предыдущее значение | ❌ нет | ✅ да | ❌ нет |
| Самодиспоузится | ❌ нет | ❌ нет | ✅ да |
| Возвращает Promise | ❌ нет | ❌ нет | ✅ да |

Правило: `autorun` — для побочных эффектов, которые должны сработать сразу и при каждом изменении (синхронизация document.title, localStorage). `reaction` — для аналитики или дебаунс-сохранения, где нужно предыдущее значение и не нужен начальный запуск. `when` — для одноразовых условий: "дождаться загрузки пользователя, затем сделать X".

---

**6. Почему `async/await` нарушает strict mode MobX, и какие два решения существуют?**

`async/await` компилируется в машину состояний, где код после `await` выполняется в новом тике микротаск — вне исходного контекста `action`. В режиме `enforceActions: 'always'` любая мутация observable вне action бросает ошибку. Решения: (1) `runInAction(() => { ... })` — обернуть каждый блок мутаций после `await` вручную; (2) `flow` — использует генератор с `yield` вместо `await`, и MobX автоматически оборачивает каждый шаг после `yield` в action. `flow` — идиоматический подход; он также поддерживает отмену через возвращаемую функцию `cancel()`.

## Типичные follow-up вопросы (Группа 1)

```txt
"Что произойдёт, если прочитать computed вне observer?" →
  Вычислится и вернёт значение — но без кэширования. Работает как
  обычный геттер: нет мемоизации, нет подписки, нет автообновления.
  С computedRequiresReaction: true бросит ошибку, что помогает
  поймать баги на ранней стадии.

"Если два observer-компонента читают один computed, сколько раз
он пересчитается при изменении зависимости?" →
  Один раз. computed глобален для стора, не привязан к компоненту.
  Оба компонента получат новое кэшированное значение. Это ключевое
  преимущество производительности перед useMemo (который
  привязан к экземпляру компонента).

"Какая утечка памяти возникает при забытом dispose()?" →
  MobX держит ссылку из списка зависимостей observable на reaction.
  После unmount компонента, если dispose() не вызван, reaction
  удерживает компонент в памяти — GC не может его собрать.
  В React Strict Mode это проявляется как предупреждения
  "Can't perform a state update on an unmounted component".
```

## Группа 2: Redux / RTK — поток данных

**7. Опишите полный поток данных Redux от клика по кнопке до обновления DOM.**

```txt
Пользователь кликает
  → dispatch(action)          диспатчит простой объект
  → Цепочка Middleware        thunk, logger и т.д. — каждый вызывает next(action)
  → Reducer(state, action)    чистая функция → возвращает новый объект состояния
  → Store.setState(newState)  стор заменяет внутреннюю ссылку на состояние
  → useSelector перезапускается  для каждого подписанного компонента
  → React сравнивает результаты  если результат selector изменился (по ===) → ре-рендер
  → Компонент ре-рендерится   с новыми данными из стора
```

Каждый шаг синхронный (кроме middleware типа thunk, добавляющего асинхронность). Этот детерминированный, отслеживаемый поток — причина, по которой Redux DevTools может воспроизвести любую последовательность actions.

---

**8. Что генерирует `createSlice`, и почему мутация `state` внутри него работает?**

`createSlice` генерирует: (1) строки типов action (`'cart/addItem'`); (2) функции-создатели action; (3) функцию-reducer. Он оборачивает reducer с помощью Immer. Внутри Immer-обёрнутого reducer, `state` — это Proxy: любая мутация (`state.items.push(item)`) перехватывается и производит новый иммутабельный объект. Реальный объект состояния Redux никогда не мутируется. Можно также вернуть новое значение state явно (для сброса) — но нельзя одновременно мутировать и возвращать.

---

**9. `createAsyncThunk` — в чём разница между `rejectWithValue` и `throw`?**

Оба вызывают срабатывание кейса `rejected`, но помещают данные в разные места:

```ts
// throw → action.error.message (SerializedError, ограниченная типизация)
// rejectWithValue(data) → action.payload (ваши типизированные данные)

const fetchUser = createAsyncThunk<User, string, { rejectValue: string }>(
  'user/fetch',
  async (id, { rejectWithValue }) => {
    try {
      return await api.getUser(id);
    } catch (e) {
      // rejectWithValue для контролируемых, типизированных ошибок
      return rejectWithValue((e as Error).message);
    }
  }
);

// В extraReducers:
.addCase(fetchUser.rejected, (state, action) => {
  // action.payload типизирован как string (ваш rejectValue)
  state.error = action.payload ?? action.error.message;
})
```

**10. Почему selector'ы без `createSelector` вызывают лишние ре-рендеры?**

`useSelector` сравнивает возвращаемое значение с предыдущим по строгому равенству (`===`). Если selector создаёт новый объект или массив при каждом вызове (через `.filter()`, `.map()`, литерал объекта), ссылка всегда разная — компонент ре-рендерится при каждом Redux dispatch, даже полностью не связанном с его данными.

```ts
// ❌ Новый массив при каждом вызове → ре-рендер на каждый dispatch
const selectActiveUsers = (state: RootState) =>
  state.users.filter(u => u.isActive);

// ✅ Мемоизировано — пересчитывается только при изменении state.users
export const selectActiveUsers = createSelector(
  [(state: RootState) => state.users],
  (users) => users.filter(u => u.isActive)
);
```

**11. Что делает система инвалидации тегов RTK Query, и почему она лучше ручного управления кэшем?**

Теги — это метки, связывающие запросы (провайдеры данных) с мутациями (инвалидаторы данных). При успешном завершении мутации RTK Query автоматически перезапрашивает все активные запросы, предоставлявшие инвалидированные теги. Это устраняет паттерн ручного диспатча "refresh"-экшенов после мутаций — кэш всегда консистентен без координирующего кода.

```ts
endpoints: (builder) => ({
  getUsers: builder.query<User[], void>({
    providesTags: ['User'],         // кэш этого запроса помечен тегом 'User'
  }),
  deleteUser: builder.mutation<void, string>({
    query: (id) => ({ url: `/users/${id}`, method: 'DELETE' }),
    invalidatesTags: ['User'],      // после удаления — перезапросить все 'User'-запросы
  }),
}),
// Ручной dispatch не нужен — getUsers перезапрашивается автоматически после deleteUser
```

## Типичные follow-up вопросы (Группа 2)

```txt
"Можно ли иметь два Redux-стора в одном приложении?" →
  Технически — да. Но вся экосистема (DevTools, хуки react-redux,
  RTK Query) предполагает один стор. Несколько сторов = потеря
  кросс-сторовых selector'ов, объединённого DevTools и кэша RTK Query.
  Реальный ответ: используйте отдельные slice'ы, не отдельные сторы.

"Какое ограничение у Immer в RTK-reducerах?" →
  Нельзя одновременно мутировать state и возвращать значение.
  Если возвращаете что-то — Immer игнорирует мутации. Если мутируете —
  не возвращайте (или возвращайте undefined). Частая ошибка:
  return state.items.push(item) — push возвращает новую длину, не state,
  и RTK заменяет всё состояние числом.

"Когда бы вы НЕ использовали RTK Query?" →
  При использовании WebSockets/SSE для real-time обновлений (поддержка
  в RTK Query экспериментальная). Когда нужна тонкая логика
  оптимистичных обновлений, которую onQueryStarted не моделирует хорошо.
  Когда в проекте уже используется React Query и стоимость миграции
  превышает выгоду.
```

## Группа 3: Zustand — API и паттерны

**12. Чем модель ре-рендеров Zustand отличается от React Context?**

Context ре-рендерит всех потребителей при изменении любого значения в объекте контекста — встроенных selector'ов нет. Zustand использует модель подписок с selector'ами на уровне компонента: каждый вызов `useStore(selector)` ре-рендерит только при `selector(newState) !== selector(prevState)` (по `===`). Компоненты — независимые подписчики, а не потребители общего объекта контекста.

```tsx
// Context: CartSummary ре-рендерится при изменении ЛЮБОГО поля CartContext
const CartSummary = () => {
  const ctx = useContext(CartContext); // подписывается на весь контекст
  return <span>{ctx.items.length}</span>;
};

// Zustand: CartSummary ре-рендерится ТОЛЬКО при изменении items.length
const CartSummary = () => {
  const count = useCartStore(state => state.items.length);
  return <span>{count}</span>;
};
```

---

**13. Почему функции-action в Zustand — стабильные ссылки, и почему это важно для React?**

Функции, определённые внутри колбэка `create`, создаются один раз при инициализации стора и никогда не пересоздаются при обновлении state. Это значит, что их безопасно извлекать и передавать как пропсы или зависимости без `useCallback`, и они не вызовут лишних ре-рендеров у дочерних компонентов.

```tsx
// ✅ addItem стабильна — не заставит ProductCard ре-рендериться
const addItem = useCartStore(state => state.addItem);
<ProductCard onAdd={addItem} /> // useMemo/useCallback не нужен
```

---

**14. Что такое паттерн slices в Zustand и когда его использовать?**

Паттерн slices разбивает большой стор на функции `StateCreator`, каждая управляет своим доменным срезом, которые затем spread'ятся в единственный вызов `create`. Аналог `combineReducers` в Redux без boilerplate. Использовать когда: (1) стор разрастается до ~5+ action/полей и становится трудноориентируемым; (2) разные члены команды владеют разными доменами; (3) нужен cross-slice доступ (каждый slice получает `get()`, который читает полный стор). Для маленьких приложений — один плоский `create` проще.

---

**15. Что делает middleware `persist`, и для чего нужен `partialize`?**

`persist` оборачивает стор и синхронизирует состояние с хранилищем (localStorage по умолчанию) при каждом обновлении, а при инициализации гидрирует из него. `partialize` — функция, выбирающая, *какие* поля сохранять — предотвращает запись чувствительных данных (токены, пароли) или эфемерного UI-состояния (isLoading, error) в localStorage.

```ts
persist(
  (set) => ({ theme: 'light', token: null, isLoading: false }),
  {
    name: 'app-store',
    partialize: (state) => ({ theme: state.theme }), // сохранять только theme
    // token и isLoading НЕ пишутся в localStorage
  }
)
```

## Типичные follow-up вопросы (Группа 3)

```txt
"Может ли Zustand заменить RTK Query?" →
  Нет. Zustand — для client-state (UI-состояние, локальные данные).
  RTK Query — для server-state (кэш, дедупликация, фоновый рефетч,
  инвалидация). Можно хранить серверные данные в Zustand
  (fetch + set), но тогда придётся вручную реализовывать кэширование,
  дедупликацию и инвалидацию. Правильная комбинация: Zustand + React Query.

"Как тестировать Zustand-стор?" →
  Прямой доступ к API стора: const { getState, setState } = useMyStore.
  Сброс между тестами: setState(initialState, true) (replace: true).
  Тест action: getState().addItem(item); expect(getState().items).toHaveLength(1).
  Без React, без render — стор это обычный объект.

"Что делает useShallow и когда нужен?" →
  Когда из selector возвращается объект: useStore(s => ({ a: s.a, b: s.b })).
  Без useShallow новый объект {} !== {} вызывает ре-рендер при каждом
  обновлении состояния. useShallow делает shallow-сравнение полей объекта,
  поэтому ре-рендер происходит только при реальном изменении a или b.
```

## Группа 4: Сравнение и trade-offs

**16. MobX использует мутабельное состояние. Redux — иммутабельное. Каковы практические последствия?**

**Мутабельное (MobX):** пишете естественные мутации объектов (`this.items.push(item)`); не нужно spread-ить глубоко вложенные структуры; проще для разработчиков с OOP-бэкграундом. Следствие: нужно использовать `action` для батчинга; без strict mode мутации могут просочиться куда угодно; time-travel отладка сложнее, потому что прошлые состояния не сохраняются.

**Иммутабельное (Redux):** каждое изменение производит новый объект состояния; предыдущее состояние всегда сохраняется; time-travel тривиален (повторно применить actions к начальному состоянию). Следствие: обновление глубоко вложенных объектов требует spread-цепочек или Immer; концептуально сложнее для разработчиков с OOP-бэкграундом.

---

**17. Как каждая библиотека обрабатывает производные/вычисляемые данные?**

**MobX:** `computed` — первоклассный гражданин: ленивый, глобально кэшированный, автоматически подписанный. Ручная мемоизация не нужна; если входные данные не меняются — результат не пересчитывается одновременно для всех потребителей.

**Redux:** `createSelector` (Reselect) мемоизирует selector'ы. Мемоизация привязана к экземпляру selector'а, не к компоненту. Нужно создавать явно; если забыть — каждый `useSelector` с новым объектом/массивом вызовет ре-рендер.

**Zustand:** нет встроенной концепции derived/computed. Пишете функции в сторе (`getTotal: () => get().items.reduce(...)`) — они пересчитываются при каждом вызове. Для мемоизации — комбинируйте с `createSelector` из Reselect или вычисляйте inline в компоненте через `useMemo`.

---

**18. Senior-инженер говорит "будем использовать Redux для всего". Как вы реагируете?**

Признать сильные стороны Redux: audit trail, time-travel отладка, явный поток данных, большая экосистема. Затем спросить: "Что конкретно мы пытаемся решить?" Если ответ "нам нужно предсказуемое управление состоянием" — это обосновано. Если "у нас сложные серверные данные" — RTK Query решает это, но React Query может быть проще. Если команда маленькая и домен несложный — overhead boilerplate Redux (даже с RTK) — реальная стоимость без отдачи. Правильный ответ — не "Redux везде" и не "никогда Redux" — а "Redux для проблем, которые Redux решает хорошо".

---

**19. Что такое "server state" vs "client state" и почему это важно при выборе библиотеки?**

**Server state:** данные, живущие на сервере, загружаемые асинхронно, которые могут устаревать и требуют кэширования, дедупликации, фонового обновления. Примеры: список пользователей, каталог товаров, история заказов. Лучше управляются React Query, SWR или RTK Query — библиотеками, созданными именно для этого.

**Client state:** данные, существующие только в браузере и не имеющие серверного представления. Примеры: открыт/закрыт modal, активный tab, черновик формы, корзина (если не персистируется). Лучше управляются Zustand, MobX или даже `useState` для локального состояния.

Большинству приложений нужно и то, и другое. Ошибка — управлять серверными данными в MobX/Zustand без стратегии кэширования: это ручная реализация React Query, но хуже.

**20. React-приложение имеет серьёзные проблемы с производительностью ре-рендеров. Как диагностировать и исправить в зависимости от библиотеки?**

**Диагностика (всегда первый шаг):** React DevTools Profiler — найти, какие компоненты ре-рендерятся, как часто и почему. Библиотека React Scan или `why-did-you-render` для автоматических аннотаций.

**Redux-исправление:** проверить selector'ы — возвращают ли они новые ссылки? Добавить `createSelector`. Проверить `useSelector` — выбирает ли всё состояние целиком? Разбить на field-specific selector'ы. Использовать `shallowEqual` для объектных selector'ов.

**MobX-исправление:** проверить, что `observer` оборачивает нужный компонент (тот, который читает данные, а не родитель). Проверить деструктуризацию вне observer. Перенести `computed` ближе к месту вычисления производных данных.

**Zustand-исправление:** проверить, подписываются ли компоненты на весь стор или на конкретные поля. Добавить гранулярные selector'ы. Использовать `useShallow` для мультипольных selector'ов.

"У какой библиотеки лучшая поддержка TypeScript?" →

## Группа 5: Продвинутые и сценарные вопросы

**21. Корзина должна сохраняться при перезагрузке страницы. Как реализовать в каждой библиотеке?**

**Zustand:** добавить middleware `persist` — одна строка, ноль дополнительного кода. Опционально `partialize` для исключения транзитных полей. Гидрация автоматическая при инициализации стора.

**Redux/RTK:** использовать библиотеку `redux-persist`. Настроить `persistConfig` с хранилищем, whitelist/blacklist. Обернуть стор в `persistStore`, приложение — в `PersistGate`. Больше настройки, но больше контроля (вложенный persist, миграции).

**MobX:** реализовать вручную — `autorun`, который записывает `JSON.stringify(store.serializableState)` в localStorage при каждом изменении; при конструировании стора читать и применять. Или использовать библиотеку `mobx-persist`.

Zustand выигрывает по простоте для этого конкретного сценария.

---

**22. Два async-action'а создают race condition. Как обработать в каждой библиотеке?**

**Redux/RTK:** `createAsyncThunk` со `signal` — передать `signal` в `fetch`, вызвать `promise.abort()` на предыдущем dispatch перед запуском нового. Или в `extraReducers` сравнивать `action.meta.requestId` с `state.currentRequestId` и отбрасывать устаревшие результаты.

**MobX flow:** `flow` возвращает объект с возможностью отмены. Вызвать `cancel()` на предыдущем запуске перед стартом нового:

```ts
class SearchStore {
  private cancelPrev?: () => void;
  *search(q: string) { /* ... */ }

  startSearch(q: string) {
    this.cancelPrev?.();
    const run = this.search(q);
    this.cancelPrev = run.cancel;
  }
}
```

**Zustand:** встроенного механизма нет. Использовать `AbortController` в async-action, отменять при следующем вызове или отслеживать счётчик `requestId` и отбрасывать устаревшие результаты.

**23. Как реализовать undo/redo для состояния текстового редактора в каждой библиотеке?**

Основной механизм: массив истории состояний (или команд) и указатель.

**Redux:** идеальное применение — reducer'ы чистые функции, состояние иммутабельно. Хранить `past: State[]`, `present: State`, `future: State[]`. При каждом action — пушить present в past. Action `UNDO`: pop из past, push present в future. Time-travel в Redux DevTools делает это бесплатно в разработке.

**Zustand с Immer:** использовать middleware `temporal` (библиотека `zundo`) — оборачивает стор и автоматически поддерживает историю undo/redo. `useTemporalStore().undo()` и `.redo()`.

**MobX:** action'ы MobX по умолчанию не сериализуемы — undo/redo требует явных снимков (`getSnapshot`/`applySnapshot` из `mobx-state-tree`) или ручного ведения истории action'ов и повторного применения.

**24. Что такое проблема "stale closure" в Zustand и как `get()` её решает?**

Когда async-action захватывает `state` из колбэка `set`, этот снимок устаревает к моменту завершения async-операции — к тому времени могут произойти другие синхронные обновления:

```ts
// ❌ Stale closure — state захвачен в момент вызова addItem
addItem: (item) => {
  set((state) => {
    fetch('/api/cart').then(() => {
      // state здесь устарел — захвачен до async-вызова
      console.log(state.items); // может не отражать параллельные обновления
    });
    return { items: [...state.items, item] };
  });
},

// ✅ get() читает ТЕКУЩЕЕ состояние в момент вызова
addItem: (item) => {
  set((state) => ({ items: [...state.items, item] }));
  fetch('/api/cart').then(() => {
    console.log(get().items); // всегда текущее состояние
  });
},
```

**25. Когда стоит мигрировать с одной библиотеки на другую, и каков процесс?**

Миграция оправдана когда: (1) ограничения текущей библиотеки вызывают реальную, измеримую боль (не просто "я предпочитаю X"); (2) команда растёт и boilerplate становится бременем координации; (3) реально нужна новая возможность (например, time-travel отладка, кэширование RTK Query).

Процесс: (1) точно определить болевую точку; (2) запустить обе библиотеки параллельно — новые фичи на новой, существующий код без изменений; (3) мигрировать домен за доменом, не переписывать всё сразу; (4) удалить старую библиотеку только когда не осталось ни одной ссылки. Самая опасная ошибка миграции: переписывать работающий код по архитектурным причинам, одновременно поставляя фичи — переключения контекста умножают риски.

## Типичные follow-up вопросы (Группа 5)

```txt
"React Server Components становятся мейнстримом. Как они меняют
управление состоянием?" →
  RSC убирают большую часть получения серверных данных с клиента.
  Данные, которые раньше жили в Redux/Zustand (пользователь, товары),
  теперь приходят как пропсы серверного компонента — клиентский стор
  не нужен. Клиентское состояние становится: только UI-взаимодействия
  (modal, форма, анимация). Zustand/MobX лучше вписываются в RSC —
  нет обёртки Provider, нет проблем hydration mismatch.

"Готов ли Zustand к production в крупном enterprise-приложении?" →
  Да — используется в production крупными компаниями. "Крупный" не
  значит "нужен Redux". Вопрос в том, нужны ли команде audit trail
  и инструментарий уровня Redux. Если нет — Zustand со slices
  масштабируется нормально. Заблуждение — отождествлять размер бандла
  или количество строк с production-готовностью.
```
