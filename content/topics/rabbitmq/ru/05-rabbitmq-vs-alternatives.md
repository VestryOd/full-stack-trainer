# RabbitMQ vs Альтернативы

## Рамка, которая делает сравнения полезными

Сравнивать брокеры по спискам функций ("у Kafka есть log compaction, у RabbitMQ — per-message TTL") — значит промахиваться мимо цели. Полезная рамка — **какую задачу каждая система решала в первую очередь**:

```txt
RabbitMQ   → создавался для маршрутизации и доставки отдельных сообщений
             Ключевой вопрос: "как доставить конкретное сообщение нужному потребителю?"

Kafka      → создавался для надёжных, упорядоченных, воспроизводимых журналов событий в масштабе
             Ключевой вопрос: "как хранить поток событий, который любой потребитель может читать
             по порядку с любого момента времени?"

Redis Pub/Sub → создавался для in-memory вещания в реальном времени подключённым клиентам
             Ключевой вопрос: "как разослать сигнал всем текущим подписчикам с минимальной задержкой?"

AWS SQS    → создавался для развязанных, управляемых, at-least-once очередей задач без накладных расходов на эксплуатацию
             Ключевой вопрос: "как надёжно ставить фоновые задачи в очередь, не управляя инфраструктурой?"
```

Понимание исходного замысла даёт больше, чем любой список функций.

## RabbitMQ vs Kafka

Это самое частое сравнение на senior-собеседованиях, обычно формулируемое как "умный брокер, тупой потребитель" vs "тупой брокер, умный потребитель".

### Архитектурное различие

```txt
RabbitMQ ("умный брокер"):
  - Брокер отслеживает, какие сообщения доставлены каждому потребителю
  - Брокер маршрутизирует по правилам exchange/binding
  - Брокер удаляет сообщения после ack потребителя
  - Потребителю не нужно состояние — он просто обрабатывает приходящее
  - Доставка push-based: брокер толкает к потребителю

Kafka ("умный потребитель"):
  - Брокер хранит сообщения в упорядоченных неизменяемых логах (топики с партициями)
  - Брокер НЕ отслеживает состояние на потребителя — просто дописывает и отдаёт
  - Потребитель сам отслеживает свою позицию (называется "offset") в логе
  - Потребитель может воспроизвести с любого offset: вчера, неделю назад, с самого начала
  - Доставка pull-based: потребитель опрашивает брокер
```

Владение offset — это ключевое различие. В RabbitMQ после ack сообщение исчезает — перечитать невозможно. В Kafka данные хранятся настраиваемый период (дни, недели, вечно), и любой новый потребитель может читать с самого начала.

### Пропускная способность и сценарии использования

```txt
Сильные стороны RabbitMQ:
  ✓ Сложная маршрутизация (4 типа exchange, wildcard-паттерны, заголовки)
  ✓ Per-message TTL, приоритетные очереди, DLQ — из коробки
  ✓ Push-доставка → меньшая задержка для отдельных сообщений
  ✓ Простые конкурирующие потребители (несколько воркеров на одну очередь)
  ✓ Понятная операционная модель, management UI в комплекте
  ✓ ~50k–150k сообщений/сек на ноду (типичная настроенная установка)

Сильные стороны Kafka:
  ✓ Воспроизводимый журнал событий — потребители могут перечитывать историю
  ✓ Exactly-once semantics с Kafka транзакциями
  ✓ Очень высокая пропускная способность: миллионы сообщений/сек на скромном кластере
  ✓ Несколько независимых групп потребителей с собственными offset
  ✓ Потоковая обработка с Kafka Streams / ksqlDB
  ✓ Долгосрочное хранение событий (event sourcing, audit log, инжест в data lake)

Слабые стороны RabbitMQ:
  ✗ Нет replay сообщений — после ack они исчезают
  ✗ Нет групп потребителей с независимыми offset
  ✗ Не подходит для высокого fan-out с потребностью в независимом replay

Слабые стороны Kafka:
  ✗ Нет нативной маршрутизации на уровне сообщения — только по топику/партиции
  ✗ Нет per-message TTL (только политики retention на уровне лога)
  ✗ Высокая операционная сложность (ZooKeeper или KRaft, управление партициями)
  ✗ Pull-модель добавляет задержку на поллинг
  ✗ Нет встроенного DLQ (реализуется в коде приложения)
  ✗ Гарантия порядка — на уровне партиции, не топика
```

### Гарантии порядка

```txt
RabbitMQ:
  - Один потребитель на очередь → гарантированный FIFO-порядок
  - Несколько потребителей на очередь → нет гарантии глобального порядка
    (потребитель A может обработать сообщение 2, пока B ещё обрабатывает сообщение 1)
  - Конкурирующие потребители жертвуют порядком ради пропускной способности

Kafka:
  - Внутри одной партиции → строго упорядочено
  - Между партициями → нет гарантии порядка
  - Для гарантии порядка конкретной сущности (например, все события user_id=42)
    используйте partition key: сообщения с одним ключом всегда идут в одну партицию
```

```ts
// Kafka: принудительный порядок для событий конкретного пользователя
producer.send({
  topic: 'user-events',
  messages: [{
    key: userId.toString(),  // одинаковый ключ → одна партиция → порядок
    value: JSON.stringify(event),
  }],
});
```

### Когда выбирать RabbitMQ вместо Kafka

- Нужна сложная маршрутизация (wildcards, заголовки, фильтрация на потребителя)
- Сообщения — это задачи или команды, а не события: должны быть обработаны один раз, затем удалены
- Нужен per-message TTL или приоритет
- Нужны DLQ и логика повторов с минимальным кодом в приложении
- Малая или средняя система, и вы не хотите операционной сложности Kafka

### Когда выбирать Kafka вместо RabbitMQ

- Нужно воспроизводить события (новый сервис должен догнать с первого дня)
- Несколько независимых групп потребителей с разной скоростью обработки
- Event sourcing или audit log, который никогда не должен терять данные
- Очень высокая пропускная способность (миллионы/сек)
- Уже работаете в экосистеме Kafka (Kafka Connect, Kafka Streams, ksqlDB)

## RabbitMQ vs Redis Pub/Sub

Redis Pub/Sub — самый быстрый способ разослать сигнал подключённым клиентам — и самый хрупкий.

```txt
Redis Pub/Sub:
  - Только in-memory: если подписчик офлайн в момент публикации — сообщение пропущено навсегда
  - Нет персистентности: перезапуск Redis = все pending-сообщения потеряны
  - Нет подтверждений: fire-and-forget на уровне протокола
  - Нет очереди: сообщения не хранятся — они рассылаются и исчезают
  - Очень низкая задержка: суб-миллисекунды для внутреннего Redis

RabbitMQ:
  - Персистентность по умолчанию (durable queue + persistent messages)
  - Сообщения ждут в очереди, пока потребитель не подключится и не ack
  - Полная семантика ack/nack/retry
  - Офлайн-потребители догоняют при переподключении
```

Redis также предлагает **Redis Streams** (добавлено в Redis 5.0) — это ближе к Kafka, чем к Pub/Sub: есть группы потребителей, персистентное хранилище, ID сообщений для отслеживания позиции. Redis Streams — разумная лёгкая альтернатива Kafka для событийного стриминга малого и среднего объёма.

```txt
Redis Pub/Sub: выбирайте когда
  ✓ Вещание в реальном времени подключённым клиентам (чат, live-дашборд, уведомления)
  ✓ Сигналы инвалидации кеша (сообщить серверам приложений "инвалидируй этот ключ")
  ✓ Потеря сигнала допустима (следующий придёт скоро)
  ✓ Redis уже есть и не хочется отдельного брокера

RabbitMQ: выбирайте когда
  ✓ Доставка должна быть гарантирована даже при офлайн-потребителе
  ✓ Нужны durable queues, DLQ, логика повторов
  ✓ Бизнес-критичные операции, которые не могут терять сообщения
```

## RabbitMQ vs AWS SQS

AWS SQS (Simple Queue Service) — полностью управляемый сервис очередей: нет серверов для развёртывания, нет брокеров для сопровождения.

```txt
AWS SQS:
  - Полностью управляемый: нет кластера, автоматическое масштабирование, SLA 99.9%
  - At-least-once доставка гарантирована AWS
  - Standard queues: очень высокая пропускная способность, сообщения могут прийти не по порядку
  - FIFO queues: exactly-once обработка + порядок (меньшая пропускная способность: ~3k TPS)
  - Visibility timeout: сообщение становится "невидимым" во время обработки
    (не настоящая блокировка — другой потребитель может взять его при истечении timeout)
  - Встроенная поддержка DLQ (через redrive policy, без дополнительной топологии)
  - Pull-based: потребители вызывают ReceiveMessage (поддержка long polling)
  - Нет exchange/маршрутизации: одна очередь на логический поток
  - Цена за сообщение (~$0.40 за миллион сообщений на Standard)

RabbitMQ:
  - Self-hosted (или управляемые CloudAMQP/Amazon MQ)
  - Сложная маршрутизация через exchange
  - Push-based доставка (меньшая задержка для отдельных сообщений)
  - Больше контроля над топологией, поведением и настройкой
  - Нет цены за сообщение (фиксированная стоимость инфраструктуры)
```

### Visibility timeout в SQS — концепция, которую проверяют на интервью

В SQS при получении сообщения потребителем оно становится **невидимым** для других потребителей на время visibility timeout (по умолчанию 30 секунд). Если потребитель не удаляет сообщение до истечения timeout — оно становится видимым снова, и другой потребитель может его получить.

```txt
Поток visibility timeout в SQS:
  Потребитель A получает сообщение → оно невидимо 30с
  Потребитель A обработал, вызывает DeleteMessage → сообщение удалено ✓
  
  Потребитель A падает после получения, но до удаления:
  → visibility timeout истекает → сообщение снова видимо
  → Потребитель B получает и обрабатывает его
  (Так SQS обеспечивает at-least-once без явных ack)
```

Это отличается от модели ack в RabbitMQ: в SQS вы "удаляете" вместо "подтверждения", а механизм timeout обеспечивает повторную доставку при сбое — вместо явного nack.

### SQS FIFO queues — когда важен exactly-once в AWS

```txt
Standard SQS:
  - Пропускная способность: практически неограниченная
  - Порядок: best-effort (не гарантирован)
  - Доставка: at-least-once (возможны дубликаты)

FIFO SQS:
  - Пропускная способность: 300 TPS (или 3 000 с batching)
  - Порядок: гарантирован внутри группы сообщений
  - Доставка: exactly-once (deduplication ID предотвращает дубликаты)
  - Применение: финансовые транзакции, последовательности обработки заказов
```

### Когда выбирать SQS вместо RabbitMQ

- Вы уже на AWS и хотите нулевого управления инфраструктурой
- Нужно автоматическое масштабирование без планирования ёмкости
- Требования к маршрутизации простые (point-to-point или fan-out через SNS)
- Предсказуемость стоимости за сообщение важнее задержки
- Хотите управляемый DLQ без конфигурации (redrive policy)

### Когда выбирать RabbitMQ вместо SQS

- Нужна сложная маршрутизация на основе exchange (topic wildcards, атрибуты сообщений)
- Не на AWS или нужно избежать vendor lock-in
- Нужна push-доставка для меньшей задержки на уровне сообщения
- Высокий объём, и цена за сообщение становится заградительной
- Нужен тонкий контроль над поведением retry, prefetch, приоритетом

## Типичные ошибки на интервью

- **"Kafka всегда лучше RabbitMQ, потому что быстрее"** — Kafka быстрее по суммарной пропускной способности и потоковой обработке, но RabbitMQ имеет меньшую задержку для доставки отдельных сообщений (push vs pull). Для очередей задач со сложной маршрутизацией RabbitMQ проще и подходит лучше. "Лучше" полностью зависит от задачи.

- **"Redis Pub/Sub — хорошая замена RabbitMQ"** — только если вы готовы терять сообщения при офлайн-потребителях. Redis Pub/Sub не имеет персистентности, нет очереди, нет гарантий доставки. Это механизм вещания, а не очередь сообщений.

- **"Kafka гарантирует exactly-once по всему топику"** — гарантии порядка и exactly-once в Kafka — на уровне партиции, не топика. Топик с 10 партициями — это 10 независимых упорядоченных логов. События для одной сущности должны маршрутизироваться в одну партицию через partition key для сохранения порядка.

- **"SQS FIFO гарантирует глобальный порядок"** — SQS FIFO гарантирует порядок внутри **группы сообщений** (`MessageGroupId`). Сообщения из разных групп могут обрабатываться параллельно и в произвольном порядке относительно друг друга. Одна FIFO-очередь с одной группой сообщений — фактически однопоточная.

- **"Нужно всегда использовать Kafka, это industry standard"** — операционная сложность Kafka (управление партициями, ребалансировка групп потребителей, управление offset) — это реальная цена. Для команды из 3 инженеров, строящих B2B SaaS с умеренным трафиком, поднимать Kafka-кластер для фоновой отправки email — это оверинжиниринг. Используйте простейший инструмент, соответствующий требованиям.

- **"RabbitMQ подойдёт для event sourcing"** — плохо. RabbitMQ не хранит сообщения после ack, не умеет replay и не имеет log-семантики. Event sourcing требует append-only хранилища, которое можно воспроизвести с нуля. Используйте Kafka, EventStoreDB или БД с CDC (change data capture) для event sourcing.
