# RabbitMQ — Вопросы для интервью

## Группа 1: Основы

**Какую проблему решает очередь сообщений, которую не решает прямой HTTP-вызов?**

HTTP синхронен и связывает доступность: если сервис B медленный или недоступен — запрос от сервиса A падает. Очередь сообщений разрывает эту связность тремя способами: (1) **временная развязка** — продюсер и потребитель не обязаны быть доступны одновременно; (2) **развязка нагрузки** — всплеск трафика наполняет очередь, а не перегружает потребителя; (3) **изоляция сбоев** — краш потребителя не влияет на продюсера; потребитель переподключается и продолжает с того места, где остановился. Компромисс: eventual consistency вместо мгновенного подтверждения и накладные расходы на эксплуатацию брокера.

---

**Что такое AMQP и почему это важно?**

AMQP (Advanced Message Queuing Protocol, Расширенный протокол очередей сообщений) — открытый wire-level протокол для middleware, ориентированного на сообщения. "Wire-level" означает: точные байты по сети стандартизированы — любой AMQP-совместимый клиент на любом языке может работать с RabbitMQ без привязки к вендору. RabbitMQ реализует AMQP 0-9-1 (не несовместимый AMQP 1.0). Ключевое архитектурное ограничение AMQP, которое определяет всё остальное: продюсеры никогда не публикуют напрямую в очереди — они публикуют в exchange, который маршрутизирует в очереди через привязки. Именно это косвенное звено и обеспечивает гибкость маршрутизации RabbitMQ.

---

**В чём разница между durable queue и persistent message?**

Это независимые настройки, которые часто путают. **Durable queue** (`{ durable: true }`) означает, что определение очереди (имя, настройки, привязки) переживает перезапуск брокера — "канал" не теряется. **Persistent message** (`{ persistent: true }`, delivery mode 2) означает, что содержимое сообщения записывается на диск — переживает перезапуск брокера даже при необработанном состоянии. В продакшне нужно и то, и другое: durable-очередь без persistent-сообщений будет пуста после краша; persistent-сообщение в non-durable-очереди не выживет, потому что сама очередь исчезает при перезапуске.

---

**В чём разница между at-most-once, at-least-once и exactly-once доставкой?**

**At-most-once**: брокер доставляет сообщение один раз; если потребитель упал до обработки — сообщение потеряно (`noAck: true` в RabbitMQ). **At-least-once**: брокер повторно доставляет при сбое; потребитель может обработать одно сообщение более одного раза (RabbitMQ по умолчанию с `ack`/`nack`). **Exactly-once**: каждое сообщение обрабатывается ровно один раз — требует координации между брокером, потребителем и хранилищем. RabbitMQ не предоставляет exactly-once нативно. Практический путь: проектировать под at-least-once и делать потребителей идемпотентными. Это даёт безопасность exactly-once без его сложности.

## Группа 2: Ключевые концепции

**Назовите четыре типа exchange в RabbitMQ и когда использовать каждый.**

**Direct**: маршрутизирует в очереди, чей binding key точно совпадает с routing key. Применение: маршрутизация задач, конкретные типы уведомлений, per-tenant маршрутизация. **Topic**: маршрутизирует по wildcard-паттернам (`*` = одно слово, `#` = ноль или более слов). Применение: event-driven микросервисы, где разные сервисы интересуются подмножествами событий (`order.*`, `*.placed`). **Fanout**: игнорирует routing key, доставляет во все привязанные очереди. Применение: вещание событий всем заинтересованным (паттерн Pub/Sub). **Headers**: маршрутизирует по key-value парам заголовков сообщения (`x-match: all/any`). Применение: атрибутная маршрутизация, когда routing key недостаточно выразителен — редко на практике, topic покрывает большинство случаев.

---

**Что такое default exchange и как с ним связан `sendToQueue`?**

Default exchange — предварительно объявленный direct exchange с пустым именем (`""`). Его особое правило: каждая очередь автоматически привязана к нему со своим именем как binding key. `channel.sendToQueue('my-queue', payload)` — это синтаксический сахар для `channel.publish('', 'my-queue', payload)`: публикация в default exchange с именем очереди как routing key. Нельзя вручную добавлять привязки к default exchange (auto-binding неявный). Подходит для простых случаев, но явные именованные exchange дают гибкость маршрутизировать одно сообщение в несколько очередей без изменений продюсера.

---

**В чём разница между `nack` с `requeue: true` и `nack` с `requeue: false`?**

`channel.nack(msg, false, true)` — сообщение возвращается в очередь и немедленно доставляется повторно. Без задержки это создаёт плотный цикл повторов при перманентных ошибках (сломанный payload, отсутствующая зависимость), что может насытить и потребителя, и брокера. `channel.nack(msg, false, false)` — сообщение отбрасывается; если у очереди настроен `x-dead-letter-exchange`, оно маршрутизируется туда. Продакшн-паттерн: `nack` с `requeue: false` при ошибке, с настроенной dead-letter queue, и отдельная retry-топология (delayed queue на TTL) для управляемого backoff перед повторной доставкой.

---

**Что такое Dead Letter Queue (DLQ) и что запускает попадание в неё?**

DLQ — очередь, принимающая сообщения, которые невозможно обработать. Сообщение становится "мёртвым" в трёх случаях: (1) отклонено через `nack`/`reject` с `requeue: false`; (2) истёк его TTL (`x-message-ttl`) до обработки; (3) очередь достигла лимита `x-max-length`. Настройка: установить `x-dead-letter-exchange` на исходной очереди. RabbitMQ добавляет заголовки `x-death` к dead-lettered сообщениям: исходная очередь, причина (`rejected`, `expired`, `maxlen`), счётчик попыток. Потребители DLQ обычно алертят команду, сохраняют сообщения в БД для инспекции или повторно публикуют с исправленными данными после фикса.

## Группа 3: Интеграция с Node.js

**Почему `prefetch` нужно вызывать до `consume` и как выбрать значение?**

Без `prefetch` RabbitMQ пушит все сообщения из очереди потребителю сразу. При 50 000 сообщений в очереди и медленной обработке все 50 000 окажутся в памяти процесса — очередь фактически переместилась из управляемого хранилища RabbitMQ в heap вашего приложения. `prefetch(N)` говорит брокеру: "не доставляй больше N неподтверждённых сообщений одновременно." Вызывать до `consume`, так как настраивает QoS канала до начала доставки. Выбор значения: prefetch=1 для строгой честности и порядка (низкая пропускная способность); prefetch=10–50 для типичных фоновых задач (баланс); prefetch=100+ для быстрых, лёгких по I/O потребителей (высокая пропускная способность, больше памяти).

---

**Что означает `false`, возвращённый `channel.publish()`, и что делать?**

`false` означает, что внутренний буфер записи канала заполнен — сигнал backpressure от RabbitMQ. Если игнорировать и продолжать публиковать, память иссякнет. Правильная реакция: прекратить публикацию и ждать события `drain` на канале. В коде: `const canSend = channel.publish(...); if (!canSend) { await new Promise(resolve => channel.once('drain', resolve)); }`. Особенно важно в циклах пакетной публикации. Игнорирование backpressure — частая причина алертов "RabbitMQ consumer memory alarm" в продакшне.

---

**Что такое `noAck: true` и когда это уместно?**

`noAck: true` означает, что RabbitMQ удаляет сообщение из очереди в момент доставки — до того, как потребитель что-либо с ним сделал. Ack отправлять не нужно. Если потребитель упал между получением и обработкой — сообщение потеряно навсегда. Уместно только там, где потеря сообщений приемлема: fire-and-forget аналитика, метрики, доставка логов, обновления live-дашборда — случаи, где пропускная способность важнее гарантированной доставки. Никогда не использовать `noAck: true` для бизнес-критичных операций (платежи, заказы, запись пользовательских данных).

---

**Как реализовать graceful shutdown для RabbitMQ-потребителя?**

Цель: перестать принимать новые сообщения, позволив завершить in-flight обработку. Шаги: (1) при `SIGTERM`/`SIGINT` вызвать `channel.cancel(consumerTag)` — сообщает брокеру прекратить доставку новых сообщений этому потребителю; (2) дождаться завершения in-flight сообщений (короткий таймаут или счётчик in-flight); (3) `channel.close()`, затем `connection.close()`; (4) `process.exit(0)`. Без отмены `connection.close()` вызывает nack in-flight сообщений и их повторную доставку — приемлемо для идемпотентных потребителей, но может вызвать дублированную обработку для не-идемпотентных.

## Группа 4: Надёжность

**Что такое паттерн Transactional Outbox и зачем он нужен?**

Без него: запись в БД и публикация в RabbitMQ — две отдельные операции. Краш процесса между ними оставляет систему в несогласованном состоянии — либо в БД есть данные без события, либо событие опубликовано для неподтверждённой транзакции. Паттерн Outbox: сообщение записывается в таблицу `outbox` в той же транзакции БД, что и бизнес-данные. Отдельный relay-процесс читает `WHERE published_at IS NULL` через `SELECT FOR UPDATE SKIP LOCKED` (предотвращает обработку одной строки двумя инстансами) и публикует в RabbitMQ с publisher confirms. Только после подтверждения от брокера relay помечает строку как опубликованную. Гарантирует атомарность без распределённых транзакций.

---

**Как реализовать повторные попытки с экспоненциальным backoff в RabbitMQ?**

Нативной задержки повторов в RabbitMQ нет. Стандартный паттерн — TTL + Dead Letter Exchange: (1) основная очередь имеет `x-dead-letter-exchange`, указывающий на retry exchange; (2) при ошибке `nack` с `requeue: false` — сообщение уходит на retry exchange; (3) retry-очередь имеет `x-message-ttl` (например, 30 секунд) и `x-dead-letter-exchange`, указывающий обратно на основной exchange — после истечения TTL сообщение возвращается в основную очередь. Для экспоненциального backoff: несколько retry-очередей с нарастающими TTL (5с, 30с, 5мин), маршрутизация в последовательно более медленные очереди по заголовку `x-death[0].count`. После максимального числа попыток — в постоянный DLQ.

---

**Что такое идемпотентный потребитель и как его реализовать?**

Идемпотентный потребитель даёт одинаковый результат независимо от того, обработал он сообщение один раз или десять — необходимо, потому что RabbitMQ гарантирует at-least-once (сообщения могут быть повторно доставлены после сбоев). Два подхода: (1) **дедупликация через Redis**: `SET processed:{messageId} 1 NX EX 86400` — если `NX` возвращает null, это дубликат, ack и пропустить; (2) **upsert в БД**: `INSERT INTO table (...) ON CONFLICT (message_id) DO NOTHING` — уникальный constraint делает операцию no-op при повторе. `messageId` должен устанавливать продюсер (`{ messageId: crypto.randomUUID() }`). Без стабильного `messageId` дедупликация требует upsert по бизнес-ключу (например, `ON CONFLICT (order_id) DO NOTHING`).

---

**Что такое poison message и как его обнаружить и обработать?**

Poison message (отравленное сообщение) вызывает краш или ошибку потребителя при каждой попытке обработки — сломанный payload, неожиданная версия схемы, баг, проявляющийся на конкретных данных. Без защиты потребитель входит в бесконечный цикл краш-повтор. Обнаружение: заголовки `x-death` накапливаются при каждой повторной доставке; когда `x-death[0].count >= MAX_RETRIES` — сообщение ядовито. Обработка: не повторять — поместить в карантин. Ack сообщение (убрать из очереди), записать в таблицу `quarantined_messages` с полным payload и заголовками, отправить алерт команде. Исправить причину (обычно баг или изменение схемы), затем опционально повторно опубликовать исправленные сообщения из карантинной таблицы.

## Группа 5: Архитектура и сравнения

**В чём архитектурная разница между RabbitMQ и Kafka?**

RabbitMQ — "умный брокер": отслеживает состояние доставки на потребителя, маршрутизирует через exchange, удаляет сообщения после ack. Потребители — stateless-получатели. Kafka — "умный потребитель": брокер хранит сообщения в неизменяемых упорядоченных логах (партициях) бессрочно, потребители сами отслеживают свою позицию (offset) и могут воспроизводить с любого момента. Практическое следствие: в RabbitMQ после ack сообщение исчезает (нет replay). В Kafka новые потребители могут читать полную историю. Выбирайте RabbitMQ для сложной маршрутизации, очередей задач, per-message TTL и умеренной пропускной способности. Выбирайте Kafka для replay событий, независимых групп потребителей, event sourcing и миллионов сообщений в секунду.

---

**Почему RabbitMQ не может гарантировать порядок при нескольких потребителях на одной очереди?**

С одним потребителем на очереди доставка FIFO — гарантирована. С несколькими потребителями сообщение 1 идёт к потребителю A, сообщение 2 — к потребителю B. Потребитель A обрабатывает медленно (или падает и получает повторную доставку), потребитель B завершает первым. С точки зрения бизнеса сообщение 2 "обработано" раньше сообщения 1. Для сохранения порядка при нескольких потребителях: одна очередь на потребителя (паттерн Pub/Sub с fanout exchange, а не конкурирующие потребители), или Kafka с partition key (все события для конкретной сущности идут в одну партицию, обрабатываемую одним потребителем).

---

**Когда вы выбрали бы Redis Pub/Sub вместо RabbitMQ?**

Redis Pub/Sub: когда нужна суб-миллисекундная доставка в реальном времени подключённым клиентам, и потеря сообщения при офлайн-подписчике приемлема. Сценарии: обновления live-дашборда, присутствие в чате, сигналы инвалидации кеша, курсоры совместного редактирования. Redis Pub/Sub не имеет персистентности — подписчик, который был офлайн, пропускает сообщение навсегда. RabbitMQ: когда сообщение должно быть доставлено даже при офлайн-потребителе, нужны durable queues, DLQ, семантика ack или сложная маршрутизация. Redis Streams (Redis 5.0+) — промежуточный вариант: персистентный, группы потребителей, отслеживание offset — ближе к Kafka, стоит рассмотреть для простого событийного стриминга, если Redis уже в стеке.

---

**Когда вы выбрали бы AWS SQS вместо RabbitMQ?**

SQS: когда вы на AWS и хотите нулевого управления инфраструктурой (нет кластера, автомасштабирование, SLA 99.9%), цена за сообщение приемлема, требования к маршрутизации простые (одна очередь на тип), и нужен управляемый DLQ через redrive policy без конфигурации. SQS использует pull-модель с visibility timeout вместо push + ack. RabbitMQ: когда нужна сложная маршрутизация (topic wildcards, headers), push-доставка для меньшей задержки, гранулярный контроль над retry и prefetch, или нужно избежать vendor lock-in в AWS. CloudAMQP и Amazon MQ предоставляют управляемый RabbitMQ, если нужна мощь маршрутизации без операционной нагрузки.

## Группа 6: Системное проектирование

**Как бы вы спроектировали надёжную систему уведомлений о заказах с RabbitMQ?**

Настройка: topic exchange `order-events`. Order Service использует Transactional Outbox (событие пишется в таблицу `outbox` атомарно с заказом, relay публикует с publisher confirms). Email Service привязывает очередь `order-email` к `order.placed` с настроенным DLQ. Потребитель идемпотентен через `ON CONFLICT (message_id) DO NOTHING` на таблице `sent_notifications`. Повторы: `nack` с `requeue: false` при временных ошибках → сообщения ждут в retry-очереди (TTL 30с) → возвращаются в основную → максимум 3 попытки → постоянный DLQ с алертингом. Prefetch = 10 — параллелизм без перегрузки SMTP-сервера. Graceful shutdown: отмена потребителя при SIGTERM, ожидание завершения in-flight.

---

**Как предотвратить создание бесконечного цикла повторов при падающем потребителе?**

Три взаимодополняющих защиты: (1) **Максимальное число попыток**: читать `x-death[0].count` из заголовков; если `>= MAX_RETRIES` — маршрутизировать в постоянный DLQ вместо повтора. (2) **Задержка повтора**: `nack` с `requeue: false` → dead-letter на retry exchange → retry-очередь с `x-message-ttl` → возврат в основную очередь после задержки. Без задержки повторы мгновенные и цикличные. (3) **Обнаружение poison message**: если ошибка структурная (ошибка парсинга JSON, валидация схемы) — пропустить повторы полностью, немедленно поместить в карантин. Комбинация: временные ошибки получают до N отложенных повторов; структурные ошибки — сразу в карантин.

---

**Как масштабировать RabbitMQ-потребителя для обработки в 10 раз большей нагрузки?**

Горизонтальное масштабирование: запустить несколько инстансов потребителя, каждый подключён к одной очереди. RabbitMQ распределяет доставку по всем подключённым потребителям round-robin — никакого кода координации не нужно. Настройка prefetch на инстанс: если каждый может обработать 20 параллельных сообщений — `prefetch(20)`. Для CPU-интенсивной работы: один потребитель на CPU-ядро с `prefetch(1)`. Для I/O-интенсивной работы: высокий prefetch + больше инстансов. RabbitMQ: добавить ноды для кластеризации; использовать Quorum Queues (реплицированы между нодами) вместо Classic Queues для HA. Quorum Queues обеспечивают автоматический failover лидера и являются рекомендуемым выбором по умолчанию для продакшна с RabbitMQ 3.8.

---

**Что такое паттерн саги на основе хореографии и каковы его компромиссы?**

В саге на хореографии каждый сервис реагирует на события и публикует свои — нет центрального координатора. Пример: Order Service публикует `order.placed` → Inventory Service резервирует и публикует `order.inventory-reserved` → Payment Service списывает и публикует `payment.processed` → Order Service обновляет статус. Компромиссы: **плюсы**: нет единой точки отказа, слабая связность, каждый сервис развёртывается независимо. **минусы**: трудно отследить одну бизнес-транзакцию через несколько сервисов и очередей без distributed tracing (correlation ID на каждом сообщении, OpenTelemetry); компенсирующие транзакции должны быть явными (если оплата упала — опубликовать `order.inventory-release`). Альтернатива — оркестрация (координатор саги вызывает каждый сервис по порядку) — проще для понимания, но вводит центральный сервис, который может стать узким местом и единой точкой отказа.
