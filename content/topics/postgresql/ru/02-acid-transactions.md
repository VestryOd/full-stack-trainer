# ACID и Transactions

## Транзакция — единица работы с гарантированными свойствами, а не просто "группа запросов"

Транзакция — это последовательность операций с базой данных, которая переводит БД из одного **консистентного состояния** в другое. Ключевое слово — "консистентное": не просто "атомарная группа", а группа, которая не оставляет данные в частично-изменённом или невалидном состоянии.

```sql
-- Перевод средств: классический пример, где частичное обновление катастрофично
BEGIN;

UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- Если здесь упадёт сервер — без транзакции деньги "испарились" бы

UPDATE accounts SET balance = balance + 100 WHERE id = 2;

COMMIT;   -- только ЗДЕСЬ изменения становятся видимы другим транзакциям
-- либо:
ROLLBACK; -- откатывает ВСЕ изменения с момента BEGIN, как будто их не было
```

```txt
Автокоммит (autocommit): в PostgreSQL каждый SQL-оператор вне
явного BEGIN/COMMIT выполняется в собственной неявной транзакции —
BEGIN + COMMIT "оборачивается" автоматически. Это называется
autocommit-режим и является поведением по умолчанию.

Практическое следствие: INSERT/UPDATE без BEGIN — это уже транзакция
(атомарная), просто из одного оператора. Несколько UPDATE без BEGIN —
каждый в своей транзакции, нет общего отката.
```

## A — Atomicity: "всё или ничего" на уровне реализации

```txt
Атомарность реализуется через Write-Ahead Log (WAL):
  1. Перед любым изменением данных на диске PostgreSQL записывает
     запись в WAL (журнал транзакции)
  2. При COMMIT: запись в WAL помечается как committed → данные
     можно применить к heap
  3. При ROLLBACK (или crash до COMMIT): PostgreSQL при старте
     читает WAL и "отматывает" незавершённые транзакции

WAL — это то, что даёт D (Durability) и A (Atomicity) одновременно.
```

```sql
-- SAVEPOINT — частичный откат внутри транзакции
BEGIN;

INSERT INTO orders (user_id, total) VALUES (1, 500);

SAVEPOINT after_order;

INSERT INTO payments (order_id, amount) VALUES (currval('orders_id_seq'), 500);
-- Предположим, платёж временно отклонён

ROLLBACK TO after_order;  -- откат только до SAVEPOINT, заказ сохранён

-- Можно попробовать другой способ оплаты
INSERT INTO payments (order_id, method, amount)
VALUES (currval('orders_id_seq'), 'alternative', 500);

COMMIT;
```

## C — Consistency: гарантии constraints, а не "логическая корректность бизнес-логики"

```txt
Популярное заблуждение: "Consistency" в ACID гарантирует, что
данные всегда "логически правильные". Это не так.

PostgreSQL гарантирует только то, что CONSTRAINTS (NOT NULL,
CHECK, FOREIGN KEY, UNIQUE) не будут нарушены после COMMIT.
Бизнес-логика ("нельзя перевести больше, чем есть на счёте")
— ответственность ПРИЛОЖЕНИЯ или CHECK-ограничения.

Пример разницы:
  CHECK (balance >= 0)  → СУБД заблокирует перевод при нарушении
  Без CHECK             → СУБД позволит balance стать отрицательным
                          (это "логически некорректно", но ACID не
                          помогает — нет constraints для проверки)
```

```sql
-- Constraints, обеспечивающие C (Consistency)
CREATE TABLE accounts (
    id      BIGSERIAL PRIMARY KEY,
    balance NUMERIC(15, 2) NOT NULL DEFAULT 0
              CHECK (balance >= 0),          -- C: база не позволит
    owner   TEXT NOT NULL                   --    уйти в минус
);

-- Транзакция нарушит CHECK → ROLLBACK автоматически
BEGIN;
UPDATE accounts SET balance = balance - 10000 WHERE id = 1; -- откат если balance < 10000
COMMIT;
```

## I — Isolation: "параллельные транзакции не мешают друг другу" — сложнее, чем кажется

```txt
Полная изоляция (Serializable) = транзакции выполняются так,
как будто последовательно, одна за другой. На практике это дорого
— блокировки снижают параллелизм.

PostgreSQL поддерживает несколько уровней изоляции с разными
компромиссами между изоляцией и производительностью
(подробнее — [Isolation Levels]):

  READ COMMITTED   — по умолчанию; видит данные, закоммиченные
                     ДО начала КАЖДОГО оператора внутри транзакции
  REPEATABLE READ  — видит снапшот на момент НАЧАЛА транзакции
  SERIALIZABLE     — полная изоляция, реализованная через SSI
                     (Serializable Snapshot Isolation)

Каждый уровень по-разному защищает от: dirty read, non-repeatable
read, phantom read, serialization anomaly.
```

## D — Durability: COMMIT = данные на диске (но не так просто)

```txt
После COMMIT приложение получает подтверждение — и данные гарантированно
выживут при сбое питания, краше процесса, перезагрузке ОС.

Реализация:
  WAL fsync — перед подтверждением COMMIT PostgreSQL вызывает
  fsync() (или fdatasync()) для WAL-файла — это БЛОКИРУЮЩИЙ вызов,
  который ждёт физической записи на диск.

Настройки, влияющие на D:
  synchronous_commit = on    — стандарт (D гарантировано)
  synchronous_commit = off   — подтверждение ДО fsync (риск потери
                              ~200мс данных при краше, зато быстрее)
  fsync = off                — ОЧЕНЬ опасно: полная потеря гарантии D
                              (используется только для bulk-загрузки
                              данных с возможностью воспроизведения)
```

## Deadlock — как возникает и как PostgreSQL его обнаруживает

```txt
Deadlock — циклическое ожидание блокировок между двумя и более
транзакциями:

  Транзакция A: держит блокировку строки 1, ждёт строки 2
  Транзакция B: держит блокировку строки 2, ждёт строки 1
  → Ни одна не может продолжить → deadlock
```

```sql
-- Классический сценарий deadlock (запускаются конкурентно)
-- Транзакция A:
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- (пауза, B выполняет свой первый UPDATE)
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

-- Транзакция B:
BEGIN;
UPDATE accounts SET balance = balance - 50  WHERE id = 2;
-- (пауза, A выполняет свой первый UPDATE)
UPDATE accounts SET balance = balance + 50  WHERE id = 1;
COMMIT;
-- → PostgreSQL обнаружит deadlock и откатит одну из транзакций
--   с ошибкой: ERROR: deadlock detected
```

```txt
PostgreSQL обнаруживает deadlock через граф ожидания блокировок
(lock wait graph): deadlock_timeout (по умолчанию 1 секунда) —
через этот интервал PostgreSQL проверяет граф на циклы. При
обнаружении цикла — выбирает жертву (обычно транзакцию с
наименьшим объёмом работы) и откатывает её с ERROR.

Приложение ДОЛЖНО перехватывать SQLSTATE 40P01 (deadlock detected)
и повторять транзакцию.

Профилактика deadlock:
  - всегда обновлять строки в ОДИНАКОВОМ порядке (обе транзакции
    сначала id=1, потом id=2 — deadlock невозможен)
  - держать транзакции короткими (минимальное окно для конкуренции)
  - использовать SELECT ... FOR UPDATE с NOWAIT или SKIP LOCKED
    для явного контроля блокировок
```

## Транзакции в приложении: Prisma, ошибки и длинные транзакции

```ts
// Prisma: интерактивная транзакция (рекомендуется для сложной
// бизнес-логики с условиями между запросами)
await prisma.$transaction(async (tx) => {
  const sender = await tx.account.findUniqueOrThrow({ where: { id: senderId } });

  if (sender.balance < amount) {
    throw new Error('Insufficient funds'); // автоматический ROLLBACK
  }

  await tx.account.update({
    where: { id: senderId },
    data: { balance: { decrement: amount } },
  });

  await tx.account.update({
    where: { id: recipientId },
    data: { balance: { increment: amount } },
  });
  // При выходе из коллбека без ошибки → автоматический COMMIT
});
```

```txt
Критически важно для production:

1. Транзакции должны быть КОРОТКИМИ:
   - Пока транзакция открыта, она держит блокировки
   - Долгая транзакция блокирует VACUUM (см. [MVCC, Locks, and Vacuum])
   - Долгая транзакция удерживает WAL, увеличивая его размер

2. Сетевые вызовы ВНУТРИ транзакции — антипаттерн:
   await prisma.$transaction(async (tx) => {
     const user = await tx.user.findFirst();
     await fetch('https://external-api.com/notify'); // ← ПЛОХО
     // Если fetch завис на 30 секунд — транзакция держит блокировки 30 секунд
   });

3. Правильно: сначала завершить транзакцию, потом делать внешние вызовы.
```

## Типичные ошибки на интервью

- **"C (Consistency) в ACID означает, что данные всегда логически корректны"** — не понимать, что Consistency в контексте ACID означает соблюдение только тех invariants, которые выражены через database constraints. Логическая корректность бизнес-правил — ответственность приложения.

- **"ROLLBACK удаляет данные из таблицы"** — не понимать механизм: PostgreSQL через WAL "отматывает" изменения, которые ещё не были применены к heap (или применяет undo через MVCC версии строк).

- **"deadlock — это когда два запроса ждут дольше обычного"** — не отличать deadlock (циклическое ожидание — принципиальная неразрешимость) от lock contention (высокая конкуренция за одну блокировку — разрешается при снятии блокировки).

- **"В Prisma.$transaction всегда безопасно делать внешние HTTP-вызовы"** — не понимать, что пока транзакция открыта, блокировки удерживаются, и зависание внешнего API блокирует строки в БД на всё время ожидания.

- **"synchronous_commit = off небезопасно — нужно всегда держать on"** — не знать о контексте применения: для некритичных данных (логи событий) `synchronous_commit = off` даёт значительный прирост производительности при приемлемом риске потери нескольких сотен миллисекунд данных.
