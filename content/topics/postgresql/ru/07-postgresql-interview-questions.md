# PostgreSQL: Вопросы для интервью

Вопросы сгруппированы тематически. Внутри каждой группы — полный senior-ответ + типичные follow-up вопросы. Цель: воспроизвести цепочку "базовый вопрос → уточнения → нюансы".

## Группа 1: Основы и архитектура

### Что такое PostgreSQL и чем он отличается от других СУБД?

PostgreSQL — объектно-реляционная СУБД (ORDBMS) с открытым исходным кодом, выпускаемая под PostgreSQL License (схожа с MIT). Ключевые особенности: ACID-транзакции через WAL, MVCC без блокировок чтения, богатая система типов (включая JSONB, Arrays, hstore, range types, пользовательские типы), расширяемость (расширения: PostGIS, pg_stat_statements, pgcrypto).

Отличие от MySQL: PostgreSQL строже следует стандарту SQL, нативно поддерживает сложные запросы (window functions, CTEs, LATERAL JOIN), JSONB с GIN-индексами; MySQL исторически быстрее на простых OLTP-запросах, но PostgreSQL давно сравнялся.

```txt
Типичные follow-up:

Q: "Что значит 'объектно-реляционная'?"
A: Поддержка наследования таблиц (CREATE TABLE child INHERITS parent),
   пользовательских типов (CREATE TYPE), перегрузки операторов,
   массивов как нативного типа данных — выходит за рамки чистой
   реляционной модели Кодда.

Q: "Когда выбрать PostgreSQL вместо MySQL?"
A: Для сложных запросов с JOIN'ами, аналитики, работы с JSONB,
   геоданных (PostGIS), когда важно соответствие SQL-стандарту.
   MySQL предпочтительнее для: простого OLTP с высоким write
   throughput (MySQL Cluster), если уже налажена экосистема MySQL.
```

### Как SQL-запрос проходит через PostgreSQL?

```txt
1. Parser → AST (синтаксическая проверка, ошибка синтаксиса здесь)
2. Analyzer → Query Tree (разрешение имён таблиц, столбцов, типов)
3. Rewriter → правила (VIEW expansion, RLS-политики)
4. Planner/Optimizer → cost-based оптимизация, выбор плана
5. Executor → выполнение плана, возврат строк
6. Buffer Manager → shared_buffers (page cache в RAM), промах → диск
```

```txt
Типичные follow-up:

Q: "Что такое shared_buffers?"
A: Общий RAM-кэш страниц данных (по умолчанию 128MB, обычно
   настраивают на 25% RAM). Страница читается с диска один раз
   и кэшируется; повторные обращения — из RAM.

Q: "Что такое WAL?"
A: Write-Ahead Log — журнал операций. Перед записью данных на диск
   запись попадает в WAL (fsync). Обеспечивает A (Atomicity) и D
   (Durability): при краше — WAL replay восстанавливает состояние.
```

## Группа 2: ACID и транзакции

### Объясните каждый из принципов ACID с примером реализации в PostgreSQL

**A — Atomicity**: реализована через WAL. Либо все операции транзакции применяются к heap (COMMIT), либо WAL использует undo для отмены (ROLLBACK / crash recovery). Примитив: `BEGIN / COMMIT / ROLLBACK`.

**C — Consistency**: PostgreSQL обеспечивает соблюдение CONSTRAINTS (NOT NULL, CHECK, FOREIGN KEY, UNIQUE) на момент COMMIT. Бизнес-логическую "consistency" (нельзя уйти в минус) — только если есть `CHECK (balance >= 0)`.

**I — Isolation**: реализована через MVCC-снапшоты. Три уровня: READ COMMITTED (снапшот per-statement), REPEATABLE READ (снапшот per-transaction), SERIALIZABLE (SSI — отслеживание зависимостей).

**D — Durability**: WAL fsync перед подтверждением COMMIT. `synchronous_commit = on` гарантирует D; `off` — быстрее, но риск потери ~200ms данных при краше.

```txt
Типичные follow-up:

Q: "Что такое SAVEPOINT?"
A: Точка частичного отката внутри транзакции. ROLLBACK TO SAVEPOINT
   отменяет только операции после SAVEPOINT, не всю транзакцию.
   Полезно: попробовать операцию, откатить только её при ошибке,
   продолжить транзакцию другим путём.

Q: "Что случится, если не вызвать COMMIT/ROLLBACK?"
A: При разрыве соединения PostgreSQL автоматически ROLLBACK'ит
   незавершённую транзакцию. Проблема: если соединение из пула
   "возвращается" без COMMIT/ROLLBACK — следующий клиент получит
   транзакцию в "aborted" состоянии, все запросы будут отклоняться
   до явного ROLLBACK.
```

### Как работает Deadlock и как его предотвратить?

Deadlock = циклическое ожидание блокировок: Transaction A держит row 1 и ждёт row 2; Transaction B держит row 2 и ждёт row 1 → ни одна не может продолжить.

PostgreSQL обнаруживает deadlock через граф ожидания блокировок через `deadlock_timeout` (по умолчанию 1 сек) и откатывает "жертву" с `ERROR: deadlock detected` (SQLSTATE 40P01).

Профилактика: всегда обновлять строки в ОДИНАКОВОМ порядке (`WHERE id = 1`, потом `WHERE id = 2` — в обеих транзакциях). Приложение должно перехватывать 40P01 и повторять транзакцию.

```txt
Типичные follow-up:

Q: "Чем deadlock отличается от lock contention?"
A: Lock contention — высокая конкуренция за одну блокировку
   (второй ждёт пока первый освободит). Разрешается само собой
   при COMMIT/ROLLBACK первого. Deadlock — ЦИКЛИЧЕСКОЕ ожидание,
   принципиально неразрешимое без внешнего вмешательства (СУБД
   откатывает одну транзакцию).
```

## Группа 3: Уровни изоляции

### Опишите разницу между READ COMMITTED, REPEATABLE READ и SERIALIZABLE

**READ COMMITTED** (по умолчанию): снапшот создаётся на каждый ОПЕРАТОР. Видит все коммиты до начала каждого SELECT → возможен Non-Repeatable Read (повторный SELECT возвращает другое значение если между ними был COMMIT другой транзакции).

**REPEATABLE READ**: снапшот создаётся ОДИН РАЗ на транзакцию. Все SELECT видят одни и те же данные. PostgreSQL (через MVCC) дополнительно защищает от Phantom Read (стандарт этого не требует). Возможна Write-Skew Anomaly.

**SERIALIZABLE** (SSI): PostgreSQL отслеживает зависимости read/write между транзакциями. Если их конкурентный результат не эквивалентен никакому последовательному порядку — откатывает одну с `ERROR: could not serialize access` (SQLSTATE 40001). Приложение обязано обрабатывать retry.

```txt
Типичные follow-up:

Q: "Что такое Write-Skew Anomaly?"
A: Оба врача читают "в клинике 2 дежурных" и уходят — в итоге
   никого нет. Каждая транзакция отдельно не нарушает инвариант
   (есть кто-то в клинике), но вместе — нарушают. REPEATABLE READ
   не защищает, SERIALIZABLE обнаруживает конфликт через SSI.

Q: "PostgreSQL поддерживает READ UNCOMMITTED?"
A: Технически да (SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED),
   но реализован как READ COMMITTED — dirty reads физически невозможны
   из-за MVCC.
```

## Группа 4: Индексы и внутренности

### Объясните B-Tree индекс, Left Prefix Rule и когда Planner не использует индекс

**B-Tree** — сбалансированное дерево, высота которого O(log N). Листовые узлы отсортированы и связаны (двусвязный список) → эффективны range-запросы. Поддерживает: =, <, <=, >, >=, BETWEEN, ORDER BY.

**Left Prefix Rule** для составного индекса `(a, b, c)`: данные отсортированы сначала по `a`, потом по `b` внутри одного `a`, потом по `c`. Без фиксации `a` нельзя эффективно найти `b` — он разбросан по всему индексу. Правило: индекс работает для `(a)`, `(a, b)`, `(a, b, c)`, но не для `(b)` или `(c)` в одиночку.

**Когда Planner не использует индекс**: selectivity низкая (WHERE country = 'USA' при 90% USA-пользователей → Seq Scan дешевле), таблица маленькая, функция в WHERE (`WHERE LOWER(email) = ...`), implicit type cast, `random_page_cost` слишком высокий для SSD.

```txt
Типичные follow-up:

Q: "Что такое Partial Index?"
A: Индекс с WHERE-условием: CREATE INDEX ... ON orders(created_at)
   WHERE status = 'pending'. Индексирует только подмножество строк
   → меньше, быстрее. Запрос с тем же WHERE автоматически его
   использует.

Q: "Что такое Index-Only Scan?"
A: Все нужные данные SELECT берутся прямо из индекса (key + INCLUDE
   столбцы) без обращения к heap. Требует: все SELECT-поля в индексе
   И Visibility Map = "all-visible" для страниц heap.

Q: "Чем BRIN отличается от B-Tree?"
A: BRIN хранит мин/макс значения по диапазонам страниц (tens of KB
   vs GBs для B-Tree). Эффективен только при ВЫСОКОЙ КОРРЕЛЯЦИИ
   физического порядка строк и значения столбца (временны́е ряды,
   append-only логи). Для случайных данных BRIN бесполезен.
```

## Группа 5: MVCC, VACUUM, Locks

### Объясните MVCC — механизм и зачем PostgreSQL хранит несколько версий строки

MVCC (Multi-Version Concurrency Control): вместо блокировки строки при чтении — хранить несколько версий одной строки одновременно.

При UPDATE: старая строка не перезаписывается. В её `xmax` записывается XID текущей транзакции. В heap вставляется НОВАЯ строка с `xmin` = XID транзакции. Каждая читающая транзакция использует свой снапшот (xmin/xmax/xip) для определения видимости каждой версии строки.

Результат: "читатели не блокируют писателей, писатели не блокируют читателей" — параллелизм без блокировок чтения.

```txt
Типичные follow-up:

Q: "Что такое dead tuple и как он появляется?"
A: Старая версия строки после UPDATE или DELETE с заполненным xmax.
   Транзакциям больше не нужна, но физически ещё в heap. Занимает
   место, замедляет Seq Scan (нужно проверить видимость каждой версии).

Q: "Что делает VACUUM и VACUUM FULL?"
A: VACUUM: помечает space dead tuples как свободный (FSM), удаляет
   dead index entries, обновляет Visibility Map. НЕ возвращает место ОС.
   VACUUM FULL: перестраивает таблицу с нуля (ACCESS EXCLUSIVE LOCK →
   всё блокируется), возвращает место ОС. Для production — pg_repack.

Q: "Зачем VACUUM нужен для XID Wraparound?"
A: PostgreSQL использует 32-битный XID (счётчик транзакций). VACUUM
   "замораживает" старые строки (обнуляет xmin для строк, которые видны
   ВСЕМ транзакциям). Без заморозки через 2^32 транзакций — wraparound:
   PostgreSQL остановит приём запросов во избежание потери данных.

Q: "Когда нужен SELECT ... FOR UPDATE?"
A: При паттерне "check-then-act": прочитал значение → проверил
   бизнес-условие → обновил. Без FOR UPDATE — race condition
   (две транзакции читают одно и то же значение, обе "видят" что
   условие выполнено, обе обновляют → инвариант нарушен).
   Для обычного чтения без последующего UPDATE — не нужен.
```

## Группа 6: Query Planner и производительность

### Как работает EXPLAIN ANALYZE и что в нём искать?

`EXPLAIN` — показывает план выполнения (оценочные cost, rows) без реального выполнения. `EXPLAIN ANALYZE` — реально выполняет запрос и добавляет actual time, actual rows, loops.

Что искать:
- **Seq Scan на большой таблице с WHERE** → нужен индекс
- **estimated rows ≠ actual rows** (rows=100 vs actual=100000) → устаревшая статистика → `ANALYZE`
- **Nested Loop с большим loops** → N+1 проблема в SQL, нужен индекс на inner
- **Sort Method: external merge Disk** → work_mem слишком мал (`SET work_mem = '256MB'` в сессии)
- **shared read высокий** → данные не умещаются в shared_buffers, много disk I/O

```sql
-- Всегда использовать BUFFERS для полной картины I/O
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) SELECT ...;
```

```txt
Типичные follow-up:

Q: "Почему Planner иногда выбирает Seq Scan при наличии индекса?"
A: Planner оценивает cost: при низкой selectivity (мало строк
   фильтруется) random I/O через индекс дороже sequential Seq Scan.
   Также: random_page_cost слишком высокий для SSD (нужно снизить до
   1.1-2.0); функция в WHERE ломает использование индекса.

Q: "Что такое pg_stat_statements?"
A: Расширение, аккумулирующее статистику по ALL выполненным запросам
   (total_exec_time, calls, mean_exec_time). Позволяет найти "дорогие"
   запросы в production без EXPLAIN ANALYZE каждого из них.
   Требует shared_preload_libraries = 'pg_stat_statements'.

Q: "Как найти медленные запросы в production без остановки?"
A: pg_stat_statements — самый частый ответ. Также: pg_stat_activity
   (текущие активные запросы), log_min_duration_statement (логировать
   запросы дольше N мс в PostgreSQL лог), auto_explain (автоматически
   логировать EXPLAIN для медленных запросов).
```
