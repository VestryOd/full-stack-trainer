# Query Planner и EXPLAIN ANALYZE

## Самый недооцененный компонент PostgreSQL

Многие думают:

```txt
SQL -> выполняется
```

---

На самом деле:

```txt
SQL
 ↓
Parser
 ↓
Planner
 ↓
Executor
```

---

Planner — мозг PostgreSQL.

---

# Что делает Planner

Получает запрос:

```sql
SELECT *
FROM users
WHERE email = 'max@test.com';
```

---

И решает:

```txt
как быстрее получить результат
```

---

# Возможные варианты

Например:

```txt
Seq Scan
Index Scan
Bitmap Index Scan
Index Only Scan
```

---

Planner выбирает лучший.

---

# Почему это важно

Один и тот же запрос может выполняться:

```txt
5 ms
```

или

```txt
5 minutes
```

---

В зависимости от выбранного плана.

---

# Seq Scan

Sequential Scan.

---

Самый простой вариант.

---

PostgreSQL читает таблицу полностью.

---

Условно:

```txt
row1
row2
row3
...
row10M
```

---

Сложность:

```txt
O(n)
```

---

# Когда Seq Scan хорошо

Маленькие таблицы.

---

Например:

```txt
50 строк
```

---

Индекс будет даже медленнее.

---

# Index Scan

Используется индекс.

---

Алгоритм:

```txt
найти значение в индексе
получить указатель
прочитать строку
```

---

Обычно:

```txt
O(log n)
```

---

# Bitmap Index Scan

Иногда Planner комбинирует индекс и чтение страниц.

---

Особенно полезно:

```txt
много совпадающих строк
```

---

Например:

```sql
WHERE status='ACTIVE'
```

---

если таких пользователей:

```txt
100 000
```

---

# Index Only Scan

Самый быстрый вариант.

---

Таблица вообще не читается.

---

Данные берутся только из индекса.

---

Пример:

```sql
SELECT email
FROM users
WHERE email='a@test.com';
```

---

Если email уже есть в индексе.

---

# Откуда Planner знает что быстрее

Из статистики.

---

PostgreSQL собирает:

```txt
количество строк
распределение значений
селективность
```

---

# ANALYZE

Обновляет статистику.

---

Запускается автоматически через:

```txt
autovacuum
```

---

Либо вручную:

```sql
ANALYZE users;
```

---

# Selectivity

Очень важная тема.

---

Представим:

```txt
users = 10M строк
```

---

Поле:

```txt
country
```

---

95% пользователей:

```txt
USA
```

---

Запрос:

```sql
WHERE country='USA'
```

---

Индекс бесполезен.

---

Planner выберет:

```txt
Seq Scan
```

---

Потому что всё равно придется читать почти всю таблицу.

---

# Cost Based Optimizer

Planner не знает точное время выполнения.

---

Он считает стоимость.

---

Условно:

```txt
Seq Scan cost = 500
Index Scan cost = 200
```

---

Берется меньшая стоимость.

---

# EXPLAIN

Показывает план выполнения.

---

Пример:

```sql
EXPLAIN
SELECT *
FROM users
WHERE email='test@test.com';
```

---

Результат:

```txt
Index Scan using idx_users_email
```

---

# EXPLAIN ANALYZE

Самая важная команда.

---

Не только показывает план.

---

Но реально выполняет запрос.

---

И показывает:

```txt
actual time
rows
loops
buffers
```

---

Пример:

```sql
EXPLAIN ANALYZE
SELECT *
FROM users
WHERE email='test@test.com';
```

---

# На что смотреть

## Actual Time

Настоящее время выполнения.

---

## Rows

Сколько строк реально прочитано.

---

## Cost

Оценка Planner.

---

## Scan Type

Очень важно.

---

Ищем:

```txt
Seq Scan
Index Scan
Bitmap Scan
Index Only Scan
```

---

# Почему Planner ошибается

Статистика устарела.

---

Например:

```txt
было 100 строк
стало 10 миллионов
```

---

Planner принимает неправильные решения.

---

Помогает:

```sql
ANALYZE;
```

---

# Очень популярный вопрос

Почему PostgreSQL не использует индекс?

---

Ответ:

Потому что Planner посчитал,
что стоимость использования индекса выше,
чем стоимость полного сканирования таблицы.

---

# Еще один популярный вопрос

Почему запрос может внезапно стать медленным?

---

Причины:

```txt
устаревшая статистика
table bloat
неудачный план
изменение распределения данных
отсутствие нужного индекса
```

---

# Senior Interview Answer

Что делает Query Planner?

Planner анализирует SQL-запрос и на основе статистики выбирает наиболее эффективный план выполнения. PostgreSQL использует cost-based optimization и может выбирать между Sequential Scan, Index Scan, Bitmap Scan и другими стратегиями доступа к данным. Для анализа планов выполнения используется EXPLAIN ANALYZE.
