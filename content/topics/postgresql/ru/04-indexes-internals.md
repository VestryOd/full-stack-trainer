# PostgreSQL Indexes Internals

## Что такое индекс

Индекс — это дополнительная структура данных,
которая ускоряет поиск строк.

---

Без индекса:

```sql
SELECT *
FROM users
WHERE email = 'max@test.com';
```

PostgreSQL делает:

```txt
Full Table Scan
```

---

То есть:

```txt
строка 1
строка 2
строка 3
...
строка 10 000 000
```

---

Сложность:

```txt
O(n)
```

---

# Что делает индекс

Создает отдельную структуру данных.

Упрощенно:

```txt
email
 ↓
row location
```

---

Тогда поиск становится:

```txt
O(log n)
```

вместо

```txt
O(n)
```

---

# Почему индекс ускоряет поиск

Представь книгу.

---

Без индекса:

```txt
ищем слово постранично
```

---

С индексом:

```txt
открываем оглавление
```

---

Принцип абсолютно тот же.

---

# Самый важный индекс PostgreSQL

B-Tree

Используется по умолчанию.

---

# Как выглядит B-Tree

Упрощенно:

```txt
            [50]
          /      \
      [20]       [80]
     /   \      /    \
 [10] [30] [70] [90]
```

---

Каждый узел содержит диапазон значений.

---

Поиск:

```txt
50?
 ↓
меньше
 ↓
20?
 ↓
больше
 ↓
30
```

---

Количество шагов растет логарифмически.

---

# Почему O(log n)

Предположим:

```txt
1 000 000 строк
```

---

Full Scan:

```txt
до миллиона проверок
```

---

B-Tree:

```txt
примерно 20 переходов
```

---

Потому что:

```txt
log₂(1 000 000)
≈ 20
```

---

# Где работает B-Tree

Отлично подходит для:

```sql
=
<
>
<=
>=
BETWEEN
ORDER BY
```

---

Пример:

```sql
WHERE created_at > now()
```

---

или

```sql
ORDER BY created_at
```

---

# Composite Index

Можно индексировать несколько полей.

---

Пример:

```sql
CREATE INDEX idx_users_name_age
ON users(name, age);
```

---

Индекс хранится:

```txt
(name, age)
```

---

Очень важно правило:

Left Prefix Rule

---

Индекс:

```txt
(name, age)
```

работает для:

```sql
WHERE name = ...
```

работает для:

```sql
WHERE name = ... AND age = ...
```

---

Но НЕ работает эффективно для:

```sql
WHERE age = ...
```

---

Это один из любимых вопросов интервьюеров.

---

# Почему индекс может не использоваться

Потому что Planner оценивает стоимость.

---

Пример

Таблица:

```txt
100 строк
```

---

Поиск:

```sql
WHERE is_active = true
```

---

Если:

```txt
95% строк имеют true
```

то индекс бесполезен.

---

Быстрее сделать:

```txt
Sequential Scan
```

---

# EXPLAIN ANALYZE

Используется для проверки.

---

Пример:

```sql
EXPLAIN ANALYZE
SELECT *
FROM users
WHERE email = 'test@test.com';
```

---

Можно увидеть:

```txt
Index Scan
```

или

```txt
Seq Scan
```

---

# Недостатки индексов

Индексы не бесплатны.

---

Каждый INSERT:

```sql
INSERT ...
```

обновляет индекс.

---

Каждый UPDATE:

```sql
UPDATE ...
```

может обновлять индекс.

---

Каждый DELETE:

```sql
DELETE ...
```

тоже обновляет индекс.

---

Поэтому:

```txt
много индексов
=
медленнее запись
```

---

# Covering Index

PostgreSQL может вернуть данные
вообще не читая таблицу.

---

Называется:

```txt
Index Only Scan
```

---

Пример:

Индекс содержит:

```txt
email
id
```

---

Запрос:

```sql
SELECT id
FROM users
WHERE email = ...
```

---

Таблица может вообще не читаться.

---

# Частичные индексы

Очень мощная фича PostgreSQL.

---

Пример:

```sql
CREATE INDEX idx_active_users
ON users(email)
WHERE is_active = true;
```

---

Индекс содержит только активных пользователей.

---

Размер меньше.

Работает быстрее.

---

# GIN Index

Используется для:

```txt
JSONB
Arrays
Full Text Search
```

---

Пример:

```sql
CREATE INDEX idx_settings
ON users
USING GIN(settings);
```

---

# Почему GIN нужен для JSONB

Допустим:

```json
{
  "theme": "dark"
}
```

---

GIN индексирует содержимое JSON.

---

И позволяет быстро выполнять:

```sql
WHERE settings @> '{"theme":"dark"}'
```

---

# Hash Index

Оптимизирован под:

```sql
=
```

---

Используется редко.

Обычно B-Tree лучше.

---

# GiST

Используется для:

```txt
геоданные
ranges
PostGIS
```

---

# BRIN

Очень полезен для огромных таблиц.

---

Например:

```txt
логирование
метрики
телеметрия
```

---

Хранит информацию о диапазонах страниц.

---

Размер индекса очень маленький.

---

# Интервью вопрос

Почему индекс ускоряет поиск?

Ответ:

Потому что PostgreSQL использует структуры данных
(обычно B-Tree),
которые позволяют выполнять поиск за O(log n)
вместо полного прохода по таблице O(n).

---

# Интервью вопрос

Почему индекс может ухудшать производительность?

Ответ:

Потому что индекс необходимо поддерживать.

Каждый INSERT, UPDATE и DELETE
требует обновления индексных структур.
Поэтому ускорение чтения всегда оплачивается
замедлением записи.
