# Адаптивный и современный CSS

## Медиазапросы — что они на самом деле измеряют

Медиазапросы реагируют на **вьюпорт** — размеры окна браузера. Это работает для решений на уровне страницы ("переключить с одной колонки на две, когда экран достаточно широкий"), но ломается для компонентов.

Фундаментальная проблема: компонент не знает, какой у него ширины — он знает только ширину вьюпорта. Карточка может быть во всю ширину на мобильном, занимать половину в двухколоночной раскладке, и треть — в трёхколоночной. Ширина вьюпорта не говорит карточке, в каком из этих сценариев она находится.

```css
/* Подход с медиазапросами — ломается, когда один компонент появляется
   в контейнерах разной ширины при одинаковом размере вьюпорта */
@media (min-width: 768px) {
  .card {
    display: grid;
    grid-template-columns: 120px 1fr;
  }
}
/* При вьюпорте 900px:
   - Карточка в полноширинном контейнере → работает, места достаточно
   - Карточка в сайдбаре 300px → тоже переключается → сломано, слишком узко */
```

## Container queries — смена парадигмы

Container queries реагируют на **размер контейнера**, а не вьюпорта. Компонент может адаптироваться к пространству, которое у него реально есть, независимо от причины его размера.

### Настройка контекста ограничения

Container query требует двух вещей: (1) **контекст ограничения**, объявленный на контейнере, и (2) правило `@container` на компоненте.

```css
/* Шаг 1: Объявить контейнер */
.card-wrapper {
  container-type: inline-size;
  /* inline-size: реагирует на inline-размер контейнера (ширину в LTR) */
  /* size: реагирует на оба размера — inline и block */
  container-name: card; /* опционально — для именованных container queries */
}

/* Шаг 2: Написать container query */
@container (min-width: 400px) {
  .card {
    display: grid;
    grid-template-columns: 120px 1fr;
  }
}

/* Именованный container query — нацелен на конкретный контейнер по имени */
@container card (min-width: 400px) {
  .card-title {
    font-size: 1.25rem;
  }
}
```

Теперь `.card` адаптируется к ширине своего контейнера, а не вьюпорта. Поместите в сайдбар 300px — остаётся одноколоночным. Поместите в основную область 600px — переключается на grid. Один компонент, один CSS, корректное поведение в обоих контекстах.

### Сокращение container

```css
/* Эквивалент container-type + container-name */
.wrapper {
  container: card / inline-size;
  /* имя / тип */
}
```

### Единицы container query

Внутри блока `@container` доступны специальные единицы, относительные к размеру контейнера:

```css
@container (min-width: 400px) {
  .card-title {
    font-size: 5cqi;   /* 5% от inline-размера контейнера */
    padding: 3cqb;     /* 3% от block-размера контейнера */
    margin: 2cqw;      /* 2% от ширины контейнера */
    height: 10cqh;     /* 10% от высоты контейнера */
    font-size: min(5cqi, 2rem); /* адаптивный, но ограниченный */
  }
}
```

`cqi` (container query inline) и `cqb` (container query block) наиболее полезны — они учитывают направление (корректно работают с логическими свойствами).

### Что container queries решают, а медиазапросы — нет

```html
<!-- Карточка из дизайн-системы используется в трёх разных раскладках одновременно -->
<div class="sidebar" style="width: 280px;">
  <article class="card">...</article>  <!-- должна быть компактной -->
</div>

<div class="main-grid" style="width: 600px;">
  <article class="card">...</article>  <!-- должна быть средней -->
</div>

<div class="featured" style="width: 900px;">
  <article class="card">...</article>  <!-- должна быть расширенной -->
</div>
```

```css
.card-container {
  container-type: inline-size;
}

/* Компактная: < 300px */
.card { /* одна колонка, мелкий текст */ }

/* Средняя: 300px–600px */
@container (min-width: 300px) {
  .card {
    display: grid;
    grid-template-columns: 100px 1fr;
  }
}

/* Расширенная: > 600px */
@container (min-width: 600px) {
  .card {
    grid-template-columns: 200px 1fr;
    font-size: 1.125rem;
  }
}
```

Три карточки на одной странице, один размер вьюпорта, разные раскладки — невозможно с медиазапросами, тривиально с container queries.

### Стилевые container queries

Помимо размера, container queries могут запрашивать **вычисленные значения стилей**:

```css
/* Родитель устанавливает кастомное свойство */
.theme-dark {
  --theme: dark;
}

/* Потомки реагируют на него */
@container style(--theme: dark) {
  .card {
    background: #1a1a2e;
    color: #e0e0e0;
  }
}
```

Стилевые запросы пока экспериментальны (частичная поддержка по состоянию на 2024), но показывают направление: компоненты, адаптирующиеся к своему семантическому контексту, а не только к физическим размерам.

## Логические свойства — почему они важны для i18n

Физические свойства (`margin-left`, `padding-right`, `border-top`) привязаны к физическим направлениям экрана. Логические свойства привязаны к **режиму письма и направлению текста** — они адаптируются автоматически при изменении направления.

### Таблица соответствий

| Физическое | Логическое | LTR | RTL |
|---|---|---|---|
| `margin-left` | `margin-inline-start` | лево | право |
| `margin-right` | `margin-inline-end` | право | лево |
| `margin-top` | `margin-block-start` | верх | верх |
| `margin-bottom` | `margin-block-end` | низ | низ |
| `padding-left` | `padding-inline-start` | лево | право |
| `width` | `inline-size` | ширина | ширина |
| `height` | `block-size` | высота | высота |
| `top` | `inset-block-start` | верх | верх |
| `left` | `inset-inline-start` | лево | право |

В горизонтальном письме слева направо (`ltr`): **inline** = горизонталь, **block** = вертикаль.
В вертикальном режиме письма (например, японский): **inline** = вертикаль, **block** = горизонталь.

### Практическая проблема, которую решают логические свойства

```css
/* Физический подход — нужно вручную зеркалировать для RTL */
.button-icon {
  margin-left: 8px; /* иконка справа от текста */
}

/* Требуется RTL-переопределение: */
[dir="rtl"] .button-icon {
  margin-left: 0;
  margin-right: 8px;
}

/* Логический подход — работает в обоих направлениях автоматически */
.button-icon {
  margin-inline-start: 8px;
  /* LTR: margin-left: 8px → иконка справа от текста ✓ */
  /* RTL: margin-right: 8px → иконка слева от текста ✓ */
}
```

Переопределение `[dir="rtl"]` не нужно. Браузер сам сопоставляет логическое направление на основе вычисленных значений `direction` и `writing-mode`.

### Реальный пример — раскладка сообщений в чате

```css
/* Физический подход требует дублированных RTL-стилей */
.message {
  margin-left: auto;
  text-align: left;
  border-radius: 12px 12px 12px 0; /* нижний левый угол плоский */
}
[dir="rtl"] .message {
  margin-left: 0;
  margin-right: auto;
  text-align: right;
  border-radius: 12px 12px 0 12px; /* нижний правый угол плоский */
}

/* Логический подход — ноль RTL-переопределений */
.message {
  margin-inline-start: auto;
  text-align: start;
  border-start-start-radius: 12px;
  border-start-end-radius: 12px;
  border-end-end-radius: 12px;
  border-end-start-radius: 0; /* угол у начала сообщения */
}
```

### Логические свойства для border-radius

```css
/* Физическое → Логическое для border-radius */
border-top-left-radius     → border-start-start-radius
border-top-right-radius    → border-start-end-radius
border-bottom-left-radius  → border-end-start-radius
border-bottom-right-radius → border-end-end-radius
```

### Сокращение `inset`

```css
/* Физическое позиционирование */
position: absolute;
top: 0;
right: 0;
bottom: 0;
left: 0;

/* Физическое сокращение */
inset: 0; /* все четыре стороны */
inset: 10px 20px; /* block / inline */

/* Логическое позиционирование */
inset-block: 0;   /* block-start и block-end */
inset-inline: 0;  /* inline-start и inline-end */
inset-inline-start: 20px;
```

### Когда применять логические свойства на практике

- **Всегда**: новые проекты, где вы контролируете кодовую базу; любой проект, которому в будущем может понадобиться RTL
- **Выборочно**: существующие проекты, где полный переход потребует обширного тестирования
- **Избегать**: когда нужно именно физическое направление вне зависимости от режима письма (например, декоративный элемент, прикреплённый к физическому левому краю вьюпорта)

## `clamp()`, `min()`, `max()` — адаптивные значения без медиазапросов

Эти три функции позволяют создавать **флюидный** адаптивный дизайн — значения, масштабирующиеся непрерывно с вьюпортом, а не скачками на брейкпоинтах.

### `min()` и `max()`

```css
/* min(): используется наименьшее из значений */
width: min(500px, 100%);
/* Вьюпорт 400px: 100% (400px) < 500px → 400px */
/* Вьюпорт 800px: 500px < 100% (800px) → 500px */
/* Практически: элемент максимум 500px, но сжимается на маленьких экранах */

/* max(): используется наибольшее из значений */
font-size: max(16px, 1.5vw);
/* Вьюпорт 800px: 1.5vw = 12px < 16px → 16px (никогда не меньше 16px) */
/* Вьюпорт 1200px: 1.5vw = 18px > 16px → 18px */
/* Практически: шрифт масштабируется с вьюпортом, но не меньше 16px */
```

### `clamp()` — функция плавного масштабирования

`clamp(минимум, предпочтительное, максимум)` — значение масштабируется с предпочтительным выражением, но ограничено между минимумом и максимумом:

```css
/* Размер шрифта, масштабирующийся между 1rem и 2rem в зависимости от ширины */
font-size: clamp(1rem, 2.5vw, 2rem);
/* Вьюпорт 400px: 2.5vw = 10px. clamp: 16px (минимум, 10px < 1rem) */
/* Вьюпорт 800px: 2.5vw = 20px. clamp: 20px (предпочтительное) */
/* Вьюпорт 1200px: 2.5vw = 30px. clamp: 32px (максимум, 30px > 2rem) */
```

### Флюидная типографика с `clamp()`

"Предпочтительное" значение — как правило, viewport-зависимое выражение. Цель: типографика, плавно масштабирующаяся от минимального (мобильный) до максимального (десктоп) размера без медиазапросов.

```css
/* Формула: clamp(мин-размер, предпочтительное, макс-размер)
   предпочтительное = линейное выражение через vw */

:root {
  /* Флюидный базовый шрифт: 16px при вьюпорте 320px, 20px при 1200px */
  font-size: clamp(1rem, 0.875rem + 0.625vw, 1.25rem);

  /* Флюидный заголовок: 2rem при 320px, 3.5rem при 1200px */
  --h1-size: clamp(2rem, 1.571rem + 2.143vw, 3.5rem);
}

h1 { font-size: var(--h1-size); }
```

### Флюидные отступы с `clamp()`

```css
:root {
  /* Padding секции: 24px на мобильном, 80px на десктопе */
  --section-padding: clamp(24px, 4vw + 16px, 80px);

  /* Расстояние между колонками: минимум 16px, масштабируется до 32px */
  --column-gap: clamp(16px, 3vw, 32px);
}

.section {
  padding-block: var(--section-padding);
}

.grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(280px, 1fr));
  gap: var(--column-gap);
}
```

### Адаптивная раскладка без медиазапросов

```css
/* Карточка, которая всегда читаема без брейкпоинтов */
.card {
  /* Ширина: минимум 280px, максимум 400px, масштабируется с контейнером */
  width: clamp(280px, 100%, 400px);

  /* Padding масштабируется с шириной карточки */
  padding: clamp(16px, 5%, 32px);

  /* Шрифт масштабируется с вьюпортом */
  font-size: clamp(0.875rem, 0.8rem + 0.375vw, 1.125rem);
}
```

### `min()` / `max()` в сетке и flex

```css
/* Колонка шириной минимум 200px (предотвращает слишком узкие колонки) */
grid-template-columns: repeat(auto-fit, minmax(min(200px, 100%), 1fr));
/* min(200px, 100%): при контейнере 150px — 100% < 200px → используется 100% */
/* Предотвращает переполнение колонки, если контейнер уже минимума minmax */

/* Сайдбар с разумным диапазоном ширины */
.sidebar {
  width: min(300px, 30%);
  /* Занимает 30% контейнера, но не более 300px */
}
```

## Свойство `aspect-ratio`

`aspect-ratio` определяет предпочтительное соотношение сторон элемента. Браузер поддерживает это соотношение автоматически при изменении размера элемента.

```css
/* Контейнер видео 16:9 */
.video-wrapper {
  aspect-ratio: 16 / 9;
  width: 100%;
  /* Высота вычисляется автоматически: ширина × (9/16) */
}

/* Квадратный аватар */
.avatar {
  aspect-ratio: 1;   /* эквивалентно 1 / 1 */
  width: 64px;       /* высота: 64px автоматически */
}

/* Фотографическое соотношение 3:2 */
.photo {
  aspect-ratio: 3 / 2;
  width: 100%;
  overflow: hidden; /* обрезка при несоответствии изображения */
}

.photo img {
  width: 100%;
  height: 100%;
  object-fit: cover; /* заполнить без искажений */
}
```

### До `aspect-ratio` — хак с padding

До появления `aspect-ratio` использовался трюк с процентным padding-top:

```css
/* Старая техника — padding-top в процентах относителен к ШИРИНЕ элемента */
.video-wrapper {
  position: relative;
  padding-top: 56.25%; /* 9/16 = 56.25% */
  height: 0;
}
.video-wrapper iframe {
  position: absolute;
  top: 0; left: 0;
  width: 100%; height: 100%;
}
```

`aspect-ratio` полностью заменяет этот хак и работает с любым типом элемента.

### `aspect-ratio` с контентом

Когда у элемента есть контент, `aspect-ratio` — *предпочтение*, а не ограничение. Если контент выше, чем позволяет соотношение, элемент вырастает:

```css
.card {
  aspect-ratio: 4 / 3;
  /* Если контент выше 75% ширины карточки → карточка растёт вертикально */
  /* aspect-ratio переопределяется контентом */
}

/* Чтобы соблюдать соотношение независимо от контента: */
.card {
  aspect-ratio: 4 / 3;
  overflow: hidden; /* обрезать выходящий за рамки контент */
}
```

### Интринсическое соотношение сторон для изображений

Изображения имеют интринсическое соотношение сторон из исходных размеров. `aspect-ratio` может резервировать место до загрузки изображения (предотвращение сдвига раскладки):

```css
img {
  aspect-ratio: 16 / 9;
  width: 100%;
  object-fit: cover;
}

/* Резервирование места до загрузки — предотвращает CLS */
.image-container {
  aspect-ratio: 16 / 9;
  background: #f0f0f0; /* цвет-заглушка */
}
.image-container img {
  width: 100%;
  height: 100%;
  object-fit: cover;
  display: block;
}
```

## Собираем вместе — современный адаптивный компонент

```css
/* Компонент карточки со всеми современными техниками */

.card-grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(min(280px, 100%), 1fr));
  gap: clamp(16px, 3vw, 32px);
  padding: clamp(16px, 4vw, 48px);
  container-type: inline-size;
}

.card {
  display: flex;
  flex-direction: column;
}

.card-image {
  aspect-ratio: 16 / 9;
  overflow: hidden;
  border-radius: 8px;
}

.card-image img {
  width: 100%;
  height: 100%;
  object-fit: cover;
}

.card-body {
  padding-block: clamp(12px, 2vw, 24px);
  padding-inline: clamp(12px, 2vw, 24px);
  flex: 1;
}

.card-title {
  font-size: clamp(1rem, 1.5cqi, 1.5rem);
  margin-block-end: 0.5em;
}

/* Container query: переключение на горизонтальную раскладку в широких контейнерах */
@container (min-width: 500px) {
  .card {
    flex-direction: row;
  }
  .card-image {
    aspect-ratio: 1;
    flex: 0 0 200px;
  }
}
```

## Типичные ошибки на интервью

**"В чём разница между container queries и медиазапросами?"**

Медиазапросы реагируют на **размер вьюпорта** — полезны для решений на уровне страницы. Container queries реагируют на **размер контейнера** — делают компоненты по-настоящему переиспользуемыми, потому что компонент адаптируется к пространству, которое он реально занимает, а не к глобальному измерению экрана. Карточка, которая появляется в сайдбаре 280px и в основной области 600px при одном и том же размере вьюпорта, может корректно адаптироваться только с container queries.

---

**"Что такое `container-type: inline-size` vs `container-type: size`?"**

`inline-size`: позволяет запрашивать inline-размер контейнера (ширину в горизонтальных режимах письма). Block-размер (высота) не отслеживается — элемент не требует явной высоты для участия. `size`: позволяет запрашивать оба размера — inline и block. Требует, чтобы block-размер контейнера не зависел от его содержимого (иначе — циклическое вычисление). Используйте `inline-size` в большинстве случаев; `size` — только когда нужно запрашивать высоту контейнера.

---

**"В чём разница между `margin-left` и `margin-inline-start`?"**

`margin-left` — физическое свойство, всегда относится к левой стороне элемента независимо от направления. `margin-inline-start` — логическое свойство, относится к началу inline-направления. В LTR оба идентичны. В RTL (`dir="rtl"`) `margin-inline-start` отображается в `margin-right`. В вертикальных режимах письма — в верх или низ. Логические свойства устраняют необходимость в `[dir="rtl"]`-переопределениях.

---

**"Объясните `clamp(1rem, 2.5vw, 2rem)`."**

Значение равно `2.5vw`, ограниченному минимумом `1rem` и максимумом `2rem`. На маленьких вьюпортах, если `2.5vw` опускается ниже `1rem`, используется `1rem`. На больших вьюпортах, если `2.5vw` превышает `2rem`, используется `2rem`. Между этими крайними значениями — плавное линейное масштабирование с шириной вьюпорта. Это заменяет два медиазапроса-брейкпоинта одним непрерывно масштабирующимся значением.

---

**"Какую проблему `aspect-ratio` решает лучше хака с padding-top?"**

Хак с padding-top (`padding-top: 56.25%`) эксплуатирует то, что проценты padding относительны к ширине элемента — создаёт бокс с фиксированным соотношением сторон, делая высоту пропорциональной ширине через padding. Проблемы: требует `height: 0`, требует `position: absolute` у потомков, нарушает поток контента. `aspect-ratio` — декларативен, работает с потоком контента, применяется к любому типу элемента и читаем. Поддерживается во всех основных браузерах с 2021 года.

**"Когда использовать `min()` в определении колонки сетки?"**

`repeat(auto-fit, minmax(min(280px, 100%), 1fr))` — `min(280px, 100%)` предотвращает минимальный размер колонки, превышающий размер контейнера. Без этого при контейнере 200px `minmax(280px, 1fr)` установит минимум 280px для 200px-контейнера — колонка переполнится. С `min(280px, 100%)` при 200px-контейнере `100% < 280px`, минимум становится `100%` (200px) и колонка вписывается.
