# Конвейер рендеринга и производительность

## Конвейер — что происходит между HTML и пикселями

При рендеринге страницы браузер выполняет последовательность чётко разделённых этапов. Понимание того, какие этапы запускает изменение CSS, — разница между "добавлю transition" и "добавлю transition, работающий на 60fps."

```
HTML/CSS → Parse → Style → Layout → Paint → Composite → Экран
```

Каждый этап может инвалидировать последующие — запуск Layout всегда запускает Paint и Composite. Запуск только Composite пропускает Layout и Paint полностью.

### Этап 1: Parse (разбор)

Браузер разбирает HTML в DOM-дерево и CSS в CSSOM-дерево. Они объединяются в **Render Tree** — включаются только элементы, влияющие на визуальный вывод (`display: none` исключается; `visibility: hidden` включается, просто рисуется прозрачным).

Блокирующее поведение:
- CSS **блокирует рендеринг**: браузер не нарисует страницу до построения CSSOM (частичный CSSOM вызвал бы мерцание неоформленного контента)
- JavaScript **блокирует парсер** по умолчанию (если не `async` или `defer`): тег `<script>` приостанавливает разбор HTML до скачивания и выполнения скрипта
- `defer`-скрипты выполняются после завершения разбора, по порядку
- `async`-скрипты выполняются сразу после скачивания, возможно не по порядку

### Этап 2: Style (пересчёт стилей)

Браузер вычисляет **computed style** для каждого элемента в render tree — разрешает наследование, каскад, кастомные свойства и относительные единицы. Здесь `em`, `%`, `var()` превращаются в абсолютные пиксельные значения.

Что запускает полный пересчёт стилей:
- Добавление/удаление класса на любом элементе
- Любая вставка или удаление из DOM
- Изменения CSS-кастомных свойств
- Смена состояния псевдоклассов (`:hover`, `:focus`)

Стоимость пересчёта стилей масштабируется с количеством затронутых элементов. Глубокие цепочки селекторов (`div > section > article > .card > p`) заставляют браузер проходить больше дерева — одна из причин, почему мелкие селекторы и BEM быстрее в больших DOM.

### Этап 3: Layout (Reflow — перестройка разметки)

Layout вычисляет **геометрию** каждого элемента — позицию, размер и взаимосвязи с другими элементами. Это самый дорогостоящий этап, потому что layout **взаимозависим**: изменение размера одного элемента может каскадно изменить позиции сиблингов, родителей и всего документа.

```
Layout вычисляется в порядке документа.
Ширина каждого блочного элемента определяется его контейнером.
Высота каждого блочного элемента определяется его содержимым (если не задана явно).
Изменение высоты элемента сдвигает все сиблинги ниже него.
Изменение размера flex-контейнера перепозиционирует все flex-элементы.
```

**Что запускает Layout:**

Любое свойство, влияющее на геометрию элемента или окружения:
- `width`, `height`, `min-width`, `max-width`, `min-height`, `max-height`
- `padding`, `margin`, `border-width`
- `top`, `right`, `bottom`, `left` (для позиционированных элементов)
- `font-size`, `font-weight`, `line-height`, `letter-spacing`
- `display`, `position`, `float`, `flex`, `grid` и их подсвойства
- `overflow`, `white-space`
- `content` (на псевдоэлементах)

Чтение определённых JavaScript-свойств принудительно запускает **синхронный layout** (называемый "forced reflow") до того, как JavaScript получит точное значение:

```javascript
// Эти свойства запускают синхронный layout при чтении:
el.offsetWidth, el.offsetHeight
el.offsetTop, el.offsetLeft
el.scrollWidth, el.scrollHeight
el.scrollTop, el.scrollLeft
el.clientWidth, el.clientHeight
window.getComputedStyle(el)
el.getBoundingClientRect()
```

### Этап 4: Paint (рисование)

Paint записывает инструкции рисования для каждого слоя: фоны, рамки, текст, тени, контуры. Результат — список команд рисования, а не пиксели. Paint происходит по **слоям** — элементы на одном слое рисуются вместе.

**Что запускает Paint (но не Layout):**

Свойства, меняющие внешний вид без изменения геометрии:
- `color`, `background-color`, `background-image`
- `border-color`, `outline-color`
- `box-shadow`, `text-shadow`
- `visibility` (но не `display`)
- `border-radius` (спорно — некоторые браузеры могут пропускать layout)

### Этап 5: Composite (композитинг)

Браузер берёт все отрисованные слои, применяет трансформации и прозрачность, и объединяет их в финальный кадр — с использованием GPU. Это самый дешёвый этап: происходит полностью на стороне GPU без участия основного потока.

**Что запускает только Composite (пропуская Layout и Paint):**

Только два CSS-свойства композитируются без перерисовки во всех основных браузерах:
- `transform` (включая `translate`, `rotate`, `scale`)
- `opacity`

Это важнейшее правило производительности в CSS: **анимировать `transform` и `opacity`, а не `left`/`top` или `color`.**

```css
/* Запускает Layout + Paint + Composite на каждом кадре — дорого */
@keyframes move-bad {
  from { left: 0; }
  to   { left: 200px; }
}

/* Запускает только Composite — полностью на GPU, плавные 60fps */
@keyframes move-good {
  from { transform: translateX(0); }
  to   { transform: translateX(200px); }
}
```

## GPU-слои — что их создаёт

GPU-compositor работает с дискретными слоями. Определённые CSS-свойства заставляют браузер **повысить** элемент до собственного GPU-слоя (compositing layer):

**Гарантированное повышение до GPU-слоя:**
- `will-change: transform` или `will-change: opacity`
- `transform: translateZ(0)` или `transform: translate3d(0,0,0)` (старый хак)
- Элементы с CSS-анимациями на compositor thread (анимации transform/opacity)
- Элементы с `position: fixed`
- Элементы `<video>`, `<canvas>`, `<iframe>`

**Условное повышение:**
- Элементы, перекрывающие другие повышенные элементы
- Элементы с `filter` или `backdrop-filter`

### Почему повышение до GPU-слоя помогает

Оказавшись на собственном GPU-слое, изменения transform и opacity элемента не требуют CPU для перерисовки — GPU обрабатывает композитинг напрямую. Содержимое слоя загружается в GPU-память один раз и переиспользуется между кадрами.

```
Без повышения до GPU-слоя:
  Кадр 1: animate left → Layout → Paint → Composite (CPU + GPU)
  Кадр 2: animate left → Layout → Paint → Composite (CPU + GPU)
  Кадр N: animate left → Layout → Paint → Composite (CPU + GPU)
  → Дорого на каждом кадре

С повышением (transform):
  Загрузка: отрисовка в GPU-текстуру один раз
  Кадр 1: Composite (только GPU) — применение матрицы трансформации
  Кадр 2: Composite (только GPU) — применение матрицы трансформации
  Кадр N: Composite (только GPU) — применение матрицы трансформации
  → Дёшево на каждом кадре
```

### Цена слишком многих слоёв

GPU-слои потребляют GPU-память. Слишком много повышенных элементов вызывает:
- Высокое потребление GPU-памяти (видно как тормоза в Chrome DevTools → Memory)
- **Взрыв слоёв**: когда повышенный элемент перекрывает множество других, браузер может повысить все перекрывающиеся элементы — создавая десятки непреднамеренных слоёв

## `will-change` — использование и риски

`will-change` подсказывает браузеру, что элемент изменит конкретное свойство, позволяя браузеру заранее подготовить оптимизацию (обычно — повышение до GPU-слоя).

```css
/* Сообщить браузеру: скоро transform этого элемента изменится */
.modal {
  will-change: transform;
  /* Браузер повышает до GPU-слоя до начала анимации */
}
```

### Правила ответственного использования `will-change`

**1. Применять перед изменением, снимать после**

```css
/* Неправильно: всегда активен, постоянно тратит GPU-память */
.card {
  will-change: transform;
}

/* Правильно: добавлять при наведении (даёт браузеру время подготовиться), убирать после */
.card:hover {
  will-change: transform;
}

/* Или через JavaScript для явного контроля */
button.addEventListener('click', () => {
  element.style.willChange = 'transform';
  element.addEventListener('animationend', () => {
    element.style.willChange = 'auto';
  }, { once: true });
});
```

**2. Не использовать как карго-культное исправление производительности**

`will-change` сам по себе ничего не ускоряет — он лишь повышает до GPU-слоя, что экономит время при анимации, но тратит GPU-память. Если элемент не анимируется, `will-change` — чистые накладные расходы.

**3. Не применять повсеместно**

```css
/* Никогда не делайте так */
* { will-change: transform; }
/* Повышает каждый элемент до собственного GPU-слоя — катастрофическое потребление памяти */
```

**4. Использовать редко — только для элементов, которые точно будут анимированы**

Хорошие кандидаты: модалы, drawer-ы, тултипы, лоадеры, параллакс-элементы.

### `will-change` vs старый хак с `translateZ(0)`

```css
/* Старая техника: принудительно создать GPU-слой 3D-трансформацией */
.element { transform: translateZ(0); }

/* Современный эквивалент */
.element { will-change: transform; }
```

`will-change` предпочтительнее: декларативен (выражает намерение, а не реализацию), браузер сам выбирает лучшую стратегию оптимизации, и не добавляет трансформацию, способную влиять на контексты наложения.

## Свойство `contain` — изоляция layout и paint

`contain` сообщает браузеру, что поддерево элемента **независимо** от остального документа в определённых аспектах. Это позволяет браузеру пропускать целые разделы документа при пересчёте layout или paint.

```css
.widget {
  contain: layout;
  /* Изменения layout внутри .widget не влияют на внешние элементы.
     Браузер может перестроить .widget, не перестраивая остальную страницу. */
}
```

### Значения `contain`

```css
contain: layout;
/* Внутренний layout изолирован.
   Изменения внутри не влияют на внешние элементы, и наоборот.
   Также создаёт: stacking context, block formatting context,
   containing block для absolute/fixed позиционирования. */

contain: paint;
/* Элемент действует как вьюпорт для поддерева.
   Потомки не могут выходить за пределы элемента и быть видимыми снаружи.
   Также создаёт: stacking context, block formatting context,
   containing block для absolute/fixed позиционирования.
   Похоже на overflow: hidden, но с дополнительными подсказками производительности. */

contain: style;
/* Счётчики CSS и кавычки ограничены этим поддеревом.
   (Редко нужен, ограниченный выигрыш производительности) */

contain: size;
/* Размер элемента не зависит от содержимого.
   Браузеру не нужно смотреть на потомков для определения размера элемента.
   Требует явных width + height — иначе элемент схлопнется до 0. */

contain: strict;       /* layout + paint + size + style */
contain: content;      /* layout + paint + style (наиболее полезная комбинация без size) */
```

### Реальное использование `contain`

```css
/* Виджет в ленте новостей — контент часто меняется, не должен перестраивать ленту */
.news-item {
  contain: content; /* layout + paint + style */
}

/* Карточка с фиксированным размером в сетке — размер никогда не меняется */
.thumbnail-card {
  contain: strict;
  width: 280px;
  height: 200px;
}

/* Компонент со сложными внутренними анимациями — предотвратить распространение paint-инвалидации */
.chart-container {
  contain: paint;
  /* Внутренние перерисовки canvas не запускают paint для соседних элементов */
}
```

### `contain: layout` и абсолютное позиционирование

Важный побочный эффект: `contain: layout` (и `contain: paint`) делает элемент containing block для абсолютно позиционированных потомков — как `position: relative`. Это намеренно: если внутренний layout изолирован, абсолютно позиционированные потомки должны быть ограничены внутри.

## `content-visibility` — пропуск рендеринга внеэкранного контента

`content-visibility` указывает браузеру полностью пропустить работу по рендерингу для внеэкранного контента:

```css
.article-section {
  content-visibility: auto;
  contain-intrinsic-size: 0 500px; /* расчётный размер пока не рендерится */
}
```

С `content-visibility: auto`:
- Внеэкранные секции не проходят layout, paint или compositing
- Браузер использует `contain-intrinsic-size` как заглушку для сохранения позиции скролла
- При появлении в зоне видимости секция рендерится в обычном режиме

Выигрыш производительности на длинных страницах может быть значительным — список из 10 000 элементов платит стоимость рендеринга только за видимые. В отличие от виртуализации (которая требует JavaScript), `content-visibility` — чистый CSS.

## Избегание layout thrashing

**Layout thrashing** происходит, когда JavaScript чередует чтение и запись layout-свойств в цикле, заставляя браузер выполнять синхронные reflow при каждом чтении:

```javascript
// Layout thrashing — принудительный синхронный reflow на каждой итерации
const boxes = document.querySelectorAll('.box');
boxes.forEach(box => {
  const width = box.offsetWidth; // принудительный reflow для точного значения
  box.style.width = width * 2 + 'px'; // инвалидирует layout
  // Следующая итерация: offsetWidth снова принудительный reflow, т.к. layout инвалидирован
});

// Исправление: сначала пакетное чтение, затем пакетная запись
const widths = Array.from(boxes).map(box => box.offsetWidth); // пакетное чтение
boxes.forEach((box, i) => {
  box.style.width = widths[i] * 2 + 'px'; // пакетная запись
});
// Один reflow для всех чтений, одно обновление layout для всех записей
```

Принцип: **сначала читать, потом писать.** Никогда не чередовать чтение и запись layout-влияющих свойств.

**`requestAnimationFrame`** группирует записи на начало следующего кадра:

```javascript
// Планируем визуальные обновления в правильный момент конвейера
function animate() {
  requestAnimationFrame(() => {
    element.style.transform = `translateX(${pos}px)`;
    pos += 1;
    if (pos < 200) animate();
  });
}
```

## Таблица триггеров CSS

Самая важная таблица для senior-интервью:

| Свойство | Layout | Paint | Composite |
|---|---|---|---|
| `width`, `height`, `margin`, `padding` | ✓ | ✓ | ✓ |
| `top`, `left` (позиционированные) | ✓ | ✓ | ✓ |
| `font-size`, `line-height` | ✓ | ✓ | ✓ |
| `display`, `position` | ✓ | ✓ | ✓ |
| `color`, `background-color` | — | ✓ | ✓ |
| `box-shadow`, `border-radius` | — | ✓ | ✓ |
| `visibility` | — | ✓ | ✓ |
| `outline`, `outline-color` | — | ✓ | ✓ |
| **`transform`** | **—** | **—** | **✓** |
| **`opacity`** | **—** | **—** | **✓** |

`transform` и `opacity` находятся в отдельном классе: затрагивают только этап Composite.

## Типичные ошибки на интервью

**"В чём разница между reflow, repaint и compositing?"**

Reflow (Layout): браузер пересчитывает геометрию — позиции и размеры всех затронутых элементов. Запускается любым свойством, меняющим размеры или позиционирование. Самый дорогой — каскадирует по всему документу. Repaint (Paint): браузер перезаписывает команды рисования для визуального вида слоя. Запускается изменениями цвета/стиля, не влияющими на геометрию. Compositing: GPU объединяет слои в финальный кадр — только transform и opacity можно менять без reflow или repaint. Самый производительный — работает на GPU без участия основного потока.

---

**"Почему анимация `transform: translateX()` дешевле, чем `left: Xpx`?"**

`left` — layout-свойство: его изменение запускает Layout → Paint → Composite на каждом кадре (работа CPU). `transform` — compositing-свойство: браузер повышает элемент до GPU-слоя, рисует его один раз, затем на каждом кадре применяет только матричную трансформацию (работа GPU, крайне дёшево). Позиция элемента в layout не меняется при использовании `transform` — GPU читает текстуру слоя и применяет к ней трансформацию, работа CPU не требуется.

---

**"Что на самом деле делает `will-change`?"**

Сигнализирует браузеру, что конкретное свойство скоро изменится. Браузер обычно реагирует, заранее повышая элемент до GPU-слоя — до начала анимации, чтобы переход к слою не вызвал рывок на первом кадре. Это подсказка, а не гарантия. Риски: накладные расходы GPU-памяти (каждый повышенный слой потребляет VRAM), возможный взрыв слоёв при перекрытии повышенного элемента. Применять непосредственно перед анимацией, убирать сразу после.

---

**"Что такое layout thrashing и как его предотвратить?"**

Layout thrashing — паттерн JavaScript, при котором чтение и запись layout-влияющих DOM-свойств чередуются в цикле. Каждое чтение принудительно сбрасывает ожидающие изменения стилей/layout для получения точного значения, вызывая синхронный reflow. Каждая запись затем инвалидирует layout снова. Исправление: сначала пакетное чтение всех значений, затем пакетная запись. Так браузер выполнит один reflow для всех чтений и одно обновление layout для всех записей.

---

**"Что делает `contain: layout`?"**

Объявляет, что изменения внутри элемента не влияют на внешний layout, и наоборот. Браузер может перестраивать поддерево элемента без учёта остального документа, а внешние изменения layout не распространяются внутрь. Побочные эффекты: создаёт stacking context, block formatting context и containing block для absolute/fixed потомков. Наиболее полезен для часто обновляемых компонентов (новостные ленты, чат, таблицы данных), где внутренние изменения DOM не должны вызывать reflow всей страницы.

---

**"Какие CSS-свойства запускают только Composite и почему?"**

Только `transform` и `opacity` (во всех основных браузерах). Причина: они не влияют на геометрию (нет Layout) и не меняют пиксельный вид самого элемента (нет Paint). Браузер загружает элемент как текстуру в GPU один раз, а последующие изменения полностью обрабатываются этапом GPU-композитинга — применение матричного умножения (`transform`) или альфа-смешивания (`opacity`). Это намеренное проектирование API: спецификация браузера гарантирует, что эти свойства не запускают layout, чтобы обеспечить производительные анимации.
