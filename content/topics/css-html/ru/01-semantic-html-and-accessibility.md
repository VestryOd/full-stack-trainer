# Семантический HTML и доступность

## Почему семантика важна — не только для SEO

Распространённое объяснение: "Google лучше ранжирует семантические страницы". Это правда, но это побочный эффект, а не причина существования семантики. Настоящая причина — **дерево доступности (accessibility tree)**.

Каждый браузер поддерживает два представления страницы: DOM-дерево (структурное) и **дерево доступности** (семантическое). Скринридеры, брайлевские дисплеи, программы голосового управления и другие вспомогательные технологии читают дерево доступности — не DOM и не визуальную раскладку. Дерево доступности строится из семантического смысла: роли элементов, их имена, состояния и свойства.

```html
<!-- Оба визуально идентичны — но деревья доступности кардинально различаются -->

<!-- Вариант 1: -->
<div class="btn" onclick="submit()">Отправить</div>

<!-- Вариант 2: -->
<button type="submit">Отправить</button>
```

Для `<div>`: дерево доступности видит обобщённый контейнер с текстом. Скринридер объявляет "Отправить" без роли. Элемент не входит в естественный порядок табуляции. Клавиатурная активация отсутствует. Нет неявного состояния `disabled`, которое могли бы распознать вспомогательные технологии.

Для `<button>`: дерево доступности видит role=`button`, name="Отправить", focusable=true, активация через Пробел/Enter. Скринридер объявляет "Отправить, кнопка". Браузер автоматически обрабатывает клавиатурное взаимодействие. `disabled` понимается вспомогательными технологиями.

Этот разрыв — **нативная семантика HTML против визуально оформленного div** — и есть суть доступности. Каждый `<div>` там, где уместен семантический элемент, — это ручная переимплементация поведения, которое браузер предоставляет бесплатно.

### Что нативная семантика даёт автоматически (без JavaScript)

| Элемент | Роль | Tab-стоп | Клавиатурная активация | Состояние |
|---|---|---|---|---|
| `<button>` | button | да | Пробел, Enter | disabled, pressed |
| `<a href>` | link | да | Enter | visited |
| `<input type="checkbox">` | checkbox | да | Пробел | checked, indeterminate |
| `<select>` | listbox | да | Стрелки | disabled |
| `<h1>`–`<h6>` | heading (уровень 1–6) | нет | – | – |
| `<nav>` | navigation | нет | – | – |
| `<main>` | main | нет | – | – |

Ориентиры (`<nav>`, `<main>`, `<header>`, `<footer>`, `<aside>`, `<section>`) позволяют пользователям скринридеров перемещаться между разделами страницы — это эквивалент того, как зрячий пользователь сканирует страницу визуально. Без ориентиров пользователи скринридеров вынуждены слушать страницу последовательно с начала до конца.

## Иерархия заголовков — навигационный скелет документа

Пользователи скринридеров часто перемещаются по заголовкам. Иерархия заголовков — это основная структура документа. Пропуск с `<h1>` сразу на `<h3>` — всё равно что оглавление с пропущенной главой: модель документа нарушается.

```html
<!-- Неправильно — визуальный размер определяет выбор заголовка -->
<h1>Название компании</h1>
<h3>Продукты</h3>  <!-- пропущен h2 -->
<h5>Продукт А</h5> <!-- пропущен h4 -->

<!-- Правильно — иерархия отражает структуру документа -->
<h1>Название компании</h1>
<h2>Продукты</h2>
<h3>Продукт А</h3>
```

На каждой странице должен быть ровно **один `<h1>`** (название страницы / основная тема). Уровни заголовков отражают вложенность, а не размер шрифта. Для визуального размера используйте CSS.

## ARIA: когда нативной семантики недостаточно

ARIA (Accessible Rich Internet Applications) — набор HTML-атрибутов, которые **изменяют дерево доступности**. ARIA не меняет поведение, стили или DOM-структуру — она только меняет то, что видят вспомогательные технологии.

Важнейший принцип ARIA: **Отсутствие ARIA лучше неправильной ARIA.**

Неверный `role` или `aria-*` атрибут активно вводит вспомогательные технологии в заблуждение. Скринридер, встречая `role="button"` на `<div>`, объявит "кнопка" — но если Enter/Пробел не обработаны в JavaScript, объявленная возможность — ложь. Плохая ARIA создаёт сломанный опыт, худший, чем его полное отсутствие.

### Когда ARIA действительно оправдана

**1. Роли для кастомных интерактивных виджетов без HTML-аналога**

```html
<!-- Кастомный виджет вкладок — нет нативного HTML-аналога -->
<div role="tablist" aria-label="Разделы настроек">
  <button role="tab" aria-selected="true" aria-controls="panel-general" id="tab-general">
    Основные
  </button>
  <button role="tab" aria-selected="false" aria-controls="panel-privacy" id="tab-privacy">
    Конфиденциальность
  </button>
</div>
<div role="tabpanel" id="panel-general" aria-labelledby="tab-general" tabindex="0">
  <!-- содержимое основных настроек -->
</div>
<div role="tabpanel" id="panel-privacy" aria-labelledby="tab-privacy" tabindex="0" hidden>
  <!-- содержимое настроек конфиденциальности -->
</div>
```

Заметьте: `<button role="tab">` вместо `<div role="tab">` сохраняет нативное клавиатурное поведение кнопки, переопределяя только объявляемую роль.

**2. Динамически изменяющиеся состояния**

```html
<button aria-expanded="false" aria-controls="menu" id="menu-btn">
  Меню
</button>
<ul id="menu" hidden>
  <li><a href="/about">О нас</a></li>
</ul>

<script>
  const btn = document.getElementById('menu-btn');
  const menu = document.getElementById('menu');

  btn.addEventListener('click', () => {
    const expanded = btn.getAttribute('aria-expanded') === 'true';
    btn.setAttribute('aria-expanded', String(!expanded));
    menu.hidden = expanded;
  });
</script>
```

`aria-expanded` сообщает скринридерам: "эта кнопка управляет областью, которая сейчас свёрнута/развёрнута" — критично для меню, аккордеонов и disclosure-виджетов.

**3. Живые регионы — объявление динамического контента**

```html
<!-- Обновления статуса, которые скринридер должен объявлять автоматически -->
<div aria-live="polite" aria-atomic="true" id="form-status">
  <!-- JavaScript обновляет это: "Форма успешно отправлена" -->
</div>

<!-- Для срочных, прерывающих объявлений (ошибки, предупреждения) -->
<div role="alert">
  <!-- Контент, добавленный сюда, объявляется немедленно -->
</div>
```

`aria-live="polite"` ждёт, пока пользователь завершит текущее действие, и только потом объявляет. `aria-live="assertive"` (или `role="alert"`) прерывает немедленно — используйте только для ошибок и срочных сообщений.

**4. Подписи для элементов, которым нельзя использовать `<label>`**

```html
<!-- Кнопка-иконка без видимого текста -->
<button aria-label="Закрыть диалог">
  <svg aria-hidden="true" focusable="false"><!-- иконка × --></svg>
</button>

<!-- Регион, описанный контентом в другом месте -->
<section aria-labelledby="section-heading">
  <h2 id="section-heading">Последние заказы</h2>
  <!-- ... -->
</section>

<!-- Дополнительное описание, выходящее за рамки подписи -->
<input
  type="password"
  id="pwd"
  aria-describedby="pwd-requirements"
/>
<div id="pwd-requirements">
  Минимум 8 символов, включая цифру и специальный символ.
</div>
```

`aria-hidden="true"` на SVG предотвращает чтение скринридером сырых SVG-заголовков и путей. `focusable="false"` необходим для предотвращения захвата фокуса клавиатуры SVG-элементом.

### Роли, состояния и свойства ARIA — различие

| Категория | Примеры | Что делает |
|---|---|---|
| Роли | `role="dialog"`, `role="tab"`, `role="alert"` | Переопределяет семантический тип элемента в дереве доступности |
| Состояния | `aria-expanded`, `aria-checked`, `aria-disabled`, `aria-selected` | Текущее состояние элемента (изменяется динамически) |
| Свойства | `aria-label`, `aria-labelledby`, `aria-describedby`, `aria-controls`, `aria-owns` | Связи и имена (относительно статичны) |

Состояния предназначены для изменения через JavaScript (`setAttribute`). Свойства обычно задаются в HTML и редко меняются.

## Управление фокусом

Пользователи клавиатуры и скринридеров перемещаются по странице через фокус. Неправильное управление фокусом — наиболее распространённая ошибка доступности в интерактивных приложениях.

### Естественный порядок табуляции

Порядок Tab следует порядку DOM, а не визуальному порядку. Если CSS позиционирует элемент визуально перед его DOM-позицией, пользователи клавиатуры встречают элементы в запутанной последовательности.

```html
<!-- Порядок DOM совпадает с визуальным — правильно -->
<nav>...</nav>
<main>
  <h1>Дашборд</h1>
  <button>Основное действие</button>
</main>

<!-- CSS `order` во flexbox меняет визуальный порядок, но НЕ порядок Tab -->
<!-- Это создаёт несоответствие — избегайте для интерактивных элементов -->
<div style="display: flex; flex-direction: row-reverse;">
  <button>Первый визуально</button>  <!-- Последний в DOM → последний по Tab -->
  <button>Второй визуально</button> <!-- Первый в DOM → первый по Tab -->
</div>
```

### `tabindex` — когда и как

```html
<!-- tabindex="0": добавляет неинтерактивный элемент в порядок Tab (на его DOM-позиции) -->
<div role="tabpanel" tabindex="0">...</div>

<!-- tabindex="-1": убирает из порядка Tab, но оставляет программно фокусируемым -->
<div id="modal" tabindex="-1">...</div>
<!-- document.getElementById('modal').focus() работает, Tab сюда не попадёт -->

<!-- tabindex > 0: ИЗБЕГАЙТЕ. Создаёт параллельный порядок Tab, невозможный для поддержки -->
<button tabindex="3">Не делайте так</button>
```

`tabindex > 0` почти никогда не оправдан. Он переопределяет естественный порядок DOM глобально — один элемент с `tabindex="1"` получает фокус раньше каждого `tabindex="0"` элемента на странице, независимо от его позиции в DOM.

### Ловушка фокуса в диалогах

Когда открывается модальный диалог, фокус должен быть заперт внутри него. Если пользователь может нажать Tab и выйти из модального окна на фоновый контент, он теряет контекст.

```javascript
function trapFocus(container) {
  const focusableSelectors = [
    'button:not([disabled])',
    'input:not([disabled])',
    'select:not([disabled])',
    'textarea:not([disabled])',
    '[href]',
    '[tabindex="0"]',
  ].join(', ');

  const focusable = [...container.querySelectorAll(focusableSelectors)];
  const first = focusable[0];
  const last = focusable[focusable.length - 1];

  container.addEventListener('keydown', (e) => {
    if (e.key !== 'Tab') return;

    if (e.shiftKey) {
      if (document.activeElement === first) {
        e.preventDefault();
        last.focus();
      }
    } else {
      if (document.activeElement === last) {
        e.preventDefault();
        first.focus();
      }
    }
  });
}

// При открытии диалога:
dialog.removeAttribute('hidden');
dialog.focus(); // переместить фокус в диалог
trapFocus(dialog);

// При закрытии диалога:
dialog.setAttribute('hidden', '');
triggerButton.focus(); // вернуть фокус на элемент, открывший диалог
```

Атрибут `inert` (теперь широко поддерживается) — более чистая альтернатива: установите `inert` на фоновый контент, и все фокусируемые элементы внутри него автоматически исключаются из порядка Tab и взаимодействия с указателем.

```html
<div id="page-content" inert>...</div>
<dialog open>...</dialog>
```

## Шаблоны клавиатурной навигации

Правильный шаблон клавиатуры зависит от типа виджета. Смешивание этих шаблонов ломает соглашения скринридеров.

### Tab vs стрелки — фундаментальное разделение

- **Tab/Shift+Tab**: перемещение между **виджетами** (поля форм, кнопки, ссылки, кастомные элементы управления)
- **Стрелки**: перемещение **внутри виджета** (между вкладками в tab list, между опциями в listbox, между пунктами меню)

Именно это разделение нарушается в большинстве реализаций. Список вкладок должен иметь **одну точку фокуса** для всего `role="tablist"`, а стрелки перемещают между отдельными вкладками. Если Tab перемещает между вкладками, пользователи скринридеров, использующие Tab для навигации по странице, вынуждены обходить каждую вкладку.

```javascript
// Паттерн roving tabindex — стандарт для tab list, toolbar, radio group
function initRovingTabindex(container, itemSelector) {
  const items = [...container.querySelectorAll(itemSelector)];
  let currentIndex = 0;

  // Только активный элемент находится в порядке Tab
  items.forEach((item, i) => {
    item.setAttribute('tabindex', i === 0 ? '0' : '-1');
  });

  container.addEventListener('keydown', (e) => {
    const lastIndex = items.length - 1;
    let nextIndex = currentIndex;

    if (e.key === 'ArrowRight' || e.key === 'ArrowDown') {
      nextIndex = currentIndex === lastIndex ? 0 : currentIndex + 1;
    } else if (e.key === 'ArrowLeft' || e.key === 'ArrowUp') {
      nextIndex = currentIndex === 0 ? lastIndex : currentIndex - 1;
    } else if (e.key === 'Home') {
      nextIndex = 0;
    } else if (e.key === 'End') {
      nextIndex = lastIndex;
    } else {
      return;
    }

    e.preventDefault();
    items[currentIndex].setAttribute('tabindex', '-1');
    items[nextIndex].setAttribute('tabindex', '0');
    items[nextIndex].focus();
    currentIndex = nextIndex;
  });
}
```

## Видимые индикаторы фокуса

Браузеры применяют рамку фокуса по умолчанию (стили `:focus`). Исторически многие дизайнеры убирали её через `outline: none` — это делает сайт непригодным для пользователей клавиатуры.

Современное решение: `:focus-visible`. Применяется только когда браузер определяет, что фокус получен с клавиатуры, а не через клик мыши.

```css
/* Убираем стандартную рамку для пользователей мыши (им она не нужна) */
:focus:not(:focus-visible) {
  outline: none;
}

/* Показываем заметную, хорошо спроектированную рамку для пользователей клавиатуры */
:focus-visible {
  outline: 3px solid #005fcc;
  outline-offset: 2px;
  border-radius: 2px;
}
```

Рамка должна иметь соотношение контраста не менее 3:1 по отношению к соседнему фону (требование WCAG 2.2). `outline-offset` отделяет рамку от границы элемента, улучшая видимость.

## Типичные ошибки на интервью

**"В чём разница между `aria-label` и `aria-labelledby`?"**

`aria-label` предоставляет строку непосредственно как доступное имя. `aria-labelledby` ссылается на содержимое другого элемента. `aria-labelledby` имеет приоритет над `aria-label`, когда оба присутствуют. Используйте `aria-labelledby`, когда текст подписи уже виден на экране (не дублируйте контент). Используйте `aria-label`, когда видимого текста подписи нет (кнопки-иконки, ориентиры, которым нужно имя, не отображаемое визуально).

---

**"Когда использовать `role="presentation"` или `role="none"`?"**

Это синонимы. Используются для удаления семантического смысла из элемента, который должен существовать в DOM по структурным причинам, но не должен появляться в дереве доступности. Классический пример: `<table role="presentation">` для таблиц вёрстки (эпоха до CSS Grid). Также для элементов-обёрток `<div>` внутри ARIA-составных виджетов, где лишний контейнер путает AT. Никогда не применяйте на интерактивных элементах.

---

**"Почему `display: none`, `visibility: hidden` и `opacity: 0` по-разному влияют на доступность?"**

`display: none`: убирает элемент из DOM-раскладки И из дерева доступности. Скринридеры полностью игнорируют его. `visibility: hidden`: убирает из визуальной раскладки, также скрывает из дерева доступности (элемент существует в DOM, но недоступен). `opacity: 0`: делает визуально невидимым, но элемент остаётся в дереве доступности И в порядке Tab. Скринридер прочитает кнопку с `opacity: 0`, и Tab на ней остановится — хотя пользователь её не видит. Типичная причина "призрачных" фокусируемых элементов.

---

**"Что делает `aria-hidden="true"` и в чём опасность?"**

`aria-hidden="true"` удаляет элемент из дерева доступности. Вспомогательные технологии пропускают его целиком. Опасность: применение на элементе, содержащем фокусируемые дочерние элементы. Элемент скрыт из дерева доступности, но Tab всё равно попадает на интерактивных потомков. Скринридер ничего не объявляет, когда те элементы получают фокус — дезориентирующая "чёрная дыра". Правило: никогда не ставьте `aria-hidden="true"` на элемент с фокусируемыми потомками, не сделав их `tabindex="-1"` или `disabled`.

---

**"Что такое псевдокласс `:focus-visible` и зачем он был введён?"**

`:focus-visible` применяется когда браузер определяет, что индикатор фокуса должен быть виден, основываясь на модальности ввода. Пользователи мыши, кликающие по кнопке, получают на ней фокус (для работы клавиатурных событий), но визуально не нуждаются в рамке фокуса — `:focus-visible` не применяется. Пользователи клавиатуры, переходящие на кнопку через Tab, нуждаются в рамке — `:focus-visible` применяется. До `:focus-visible` разработчики использовали `outline: none` для подавления рамки при кликах мыши, что одновременно убирало её и при клавиатурной навигации — нарушение доступности. `:focus-visible` решает это корректно без JavaScript-отслеживания модальности ввода.
