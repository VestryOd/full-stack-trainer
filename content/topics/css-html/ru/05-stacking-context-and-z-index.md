# Контексты наложения и z-index

## Почему z-index понимают неправильно — ментальная модель ошибочна

Большинство разработчиков воспринимают z-index как глобальный стек: "элемент с z-index: 100 всегда отображается поверх элемента с z-index: 10". Эта модель неверна. z-index — локальный: он конкурирует только внутри одного **контекста наложения**, а разные контексты наложения упорядочиваются относительно друг друга как единые атомарные единицы.

Правильная ментальная модель: браузер отрисовывает дерево контекстов наложения, а не плоский список z-index значений. Элемент с `z-index: 9999` внутри контекста наложения, отрисовываемого позади другого контекста, будет отображаться за каждым элементом того контекста — включая элементы с `z-index: 1`.

## Порядок отрисовки без контекстов наложения

Прежде чем вводить контексты наложения, разберём порядок отрисовки по умолчанию для документа без позиционированных элементов:

1. Фон и рамки корневого элемента
2. Блочные потомки в нормальном потоке (в порядке DOM)
3. Float-элементы
4. Строчные потомки в нормальном потоке (в порядке DOM)

Позже в DOM = нарисован поверх. z-index не нужен, контексты наложения не задействованы.

## Что такое контекст наложения

Контекст наложения — это элемент, формирующий независимый слой при отрисовке. Элементы внутри контекста наложения рисуются вместе и упорядочиваются относительно друг друга с помощью z-index. Сам контекст наложения затем размещается как единое целое относительно соседних контекстов.

Внутри контекста наложения порядок отрисовки:

1. Сам элемент (фон + рамка)
2. Потомки с отрицательным z-index (наименьший — первым)
3. Блочные потомки в нормальном потоке
4. Float-потомки
5. Строчные потомки в нормальном потоке
6. Потомки с z-index: 0 или z-index: auto и позиционированные
7. Потомки с положительным z-index (наименьший — первым)

Критическое ограничение: **значения z-index сравниваются только между элементами внутри одного контекста наложения.** Элемент с `z-index: 1000` в контексте A нельзя напрямую сравнить с элементом с `z-index: 1` в контексте B по их значениям z-index — важен порядок самих контекстов A и B.

## Полный список триггеров контекста наложения

Именно этот список большинство разработчиков не знают наизусть — и он является источником почти каждого бага "почему z-index не работает?".

**Позиционирование + z-index:**
- `position: absolute`, `relative`, `fixed` или `sticky` со значением `z-index`, отличным от `auto`

**Прозрачность:**
- `opacity` меньше `1`

**Трансформации:**
- `transform`, отличный от `none`
- `translate`, `rotate`, `scale` (отдельные свойства трансформаций, не `none`)
- `perspective`, отличный от `none`
- `transform-style: preserve-3d`

**Фильтры и эффекты:**
- `filter`, отличный от `none`
- `backdrop-filter`, отличный от `none`
- `clip-path`, отличный от `none`
- `mask` / `mask-image` / `mask-border`, отличный от `none`

**Композитинг:**
- `mix-blend-mode`, отличный от `normal`
- `isolation: isolate`

**Ограничение раскладки:**
- `contain: layout`, `contain: paint`, `contain: strict`, `contain: content`

**Will-change:**
- `will-change`, установленный на любое свойство, которое само по себе создаёт контекст наложения
  (например, `will-change: transform`, `will-change: opacity`)

**Прочее:**
- `position: fixed` или `position: sticky` (всегда, независимо от значения z-index)

```css
/* Все следующие создают контекст наложения: */
.a { position: relative; z-index: 1; }       /* ← классика */
.b { opacity: 0.99; }                         /* ← сюрприз */
.c { transform: translateX(0); }              /* ← побочный эффект GPU-оптимизации */
.d { filter: blur(0px); }                     /* ← ещё один сюрприз */
.e { will-change: transform; }                /* ← побочный эффект подсказки производительности */
.f { isolation: isolate; }                    /* ← явный, намеренный */
.g { position: fixed; }                       /* ← всегда, z-index не нужен */
```

## Почему `z-index: 9999` перестаёт работать — на примере

Наиболее распространённый реальный сценарий:

```html
<div class="card" style="position: relative; z-index: 1;">
  <div class="tooltip" style="position: absolute; z-index: 9999;">
    Должен отображаться поверх всего
  </div>
</div>

<div class="modal-overlay" style="position: relative; z-index: 2;">
  <!-- Весь этот контекст наложения — выше z-index: 1 -->
</div>
```

Тултип имеет `z-index: 9999`, но живёт внутри контекста наложения `.card` с `z-index: 1`. У `.modal-overlay` — `z-index: 2`. Поскольку `2 > 1`, весь контекст `.card` (включая тултип с 9999) отрисовывается позади `.modal-overlay`. Значение 9999 сравнивается только с другими потомками `.card`, но не с оверлеем.

**Другой распространённый сценарий — ловушка transform:**

```html
<style>
  .sidebar {
    transform: translateX(0); /* GPU-оптимизация */
    /* Это создаёт контекст наложения! */
  }

  .dropdown {
    position: absolute;
    z-index: 1000;
    /* Позиционирован относительно контекста наложения .sidebar */
  }

  .page-header {
    position: relative;
    z-index: 10;
  }
</style>

<header class="page-header">...</header>
<aside class="sidebar">
  <div class="dropdown">Должен отображаться поверх header?</div>
</aside>
```

Порядок контекста `.sidebar` vs контекста `.page-header` зависит от порядка DOM и их значений z-index. `z-index: 1000` у дропдауна не имеет отношения к этому сравнению — важен только z-index самого `.sidebar` (который равен `auto`, так как мы его не задали). Реальное исправление: убрать `transform` с `.sidebar` или дать `.sidebar` явный `z-index` выше, чем у `.page-header`.

## Свойство `isolation` — намеренный контекст наложения

`isolation: isolate` создаёт контекст наложения без каких-либо визуальных побочных эффектов. Это явный, явно выражающий намерение способ изолировать стек z-index компонента.

**Проблема, которую оно решает:**

```html
<style>
  .modal { position: fixed; z-index: 100; }

  /* Сторонний компонент создаёт контекст наложения с высоким z-index */
  .third-party-widget { position: relative; z-index: 50; }
  .third-party-widget .internal-popup { position: absolute; z-index: 999; }
  /* Внутренний попап с z-index: 999 находится внутри контекста z-index: 50.
     Если наш модал имеет z-index: 100 > 50, попап окажется за модалом.
     Но что если мы хотим, чтобы внутренний стек виджета был самодостаточным
     вне зависимости от окружения? */
</style>
```

**Использование `isolation: isolate` для изоляции компонента:**

```css
/* Оборачиваем свой компонент, чтобы предотвратить "утечку" z-index наружу */
.card-component {
  isolation: isolate;
  /* Теперь любой z-index внутри .card-component конкурирует
     только с другими элементами внутри .card-component.
     Сама карточка размещается в естественном порядке DOM
     без какого-либо значения z-index. */
}

.card-component .dropdown {
  position: absolute;
  z-index: 10;
  /* Это 10 не конкурирует с z-index: 5 снаружи карточки */
}
```

Это CSS-эквивалент границы модуля: внутренние значения z-index — это детали реализации, не экспонируемые во внешний мир.

**Когда `isolation: isolate` необходим:**

- Компоненты дизайн-системы, которые должны работать в любом z-index-окружении
- Сторонние встройки, где нельзя контролировать внутренние z-index
- Storybook или документационные страницы, где компоненты рендерятся изолированно
- Компоненты, использующие `mix-blend-mode`, которые не должны смешиваться с соседними

```css
/* mix-blend-mode без изоляции "просачивается" до несвязанных предков */
.text-overlay {
  mix-blend-mode: multiply;
  /* Без изоляции смешивается со всем, что находится позади */
}

/* isolation: isolate на обёртке предотвращает "выход" режима смешивания */
.image-card {
  isolation: isolate; /* режимы смешивания внутри влияют только друг на друга */
}
.image-card .text-overlay {
  mix-blend-mode: multiply; /* смешивается только с фоном .image-card */
}
```

## `z-index: auto` vs `z-index: 0` — важное различие

```css
.a { position: relative; z-index: auto; } /* НЕ создаёт контекст наложения */
.b { position: relative; z-index: 0; }   /* СОЗДАЁТ контекст наложения */
```

С `z-index: auto`: элемент позиционирован, но участвует в родительском контексте наложения. Значения z-index его потомков конкурируют с сиблингами самого элемента.

С `z-index: 0`: создаётся контекст наложения. Значения z-index потомков изолированы внутри него.

```html
<div style="position: relative; z-index: auto;">
  <span style="position: relative; z-index: 5;">
    <!-- z-index: 5 конкурирует с сиблингами родительского div -->
  </span>
</div>

<div style="position: relative; z-index: 0;">
  <span style="position: relative; z-index: 5;">
    <!-- z-index: 5 изолирован — конкурирует только внутри этого div -->
  </span>
</div>
```

## Отрицательный z-index — что он на самом деле делает

Отрицательный z-index помещает элемент позади фона его контекста наложения. Часто используется неправильно:

```css
.parent {
  position: relative;
  background: white;
}

.child {
  position: relative;
  z-index: -1;
  /* Отрисовывается позади фона .parent — фактически невидим,
     если у .parent непрозрачный фон */
}
```

Чтобы потомок появился между фоном родителя и другим контентом, родитель должен создавать контекст наложения (иначе потомок "вырвется" к контексту деда):

```css
/* Декоративный элемент позади контента карточки, но поверх её фона */
.card {
  position: relative;
  isolation: isolate; /* создаёт контекст наложения, удерживает отрицательный z-index */
  background: white;
}

.card::before {
  content: '';
  position: absolute;
  z-index: -1;
  /* Теперь: позади других потомков .card, поверх фона .card */
}
```

Без контекста наложения на `.card` псевдоэлемент `::before` с `z-index: -1` вырвался бы в корневой контекст наложения и потенциально оказался бы за фоном всей страницы.

## Практический подход к отладке

Когда z-index работает не так, как ожидается, следуйте этому алгоритму:

**Шаг 1: Найти контексты наложения в цепочке предков**

Для каждого задействованного элемента пройти вверх по DOM и найти каждого предка, создающего контекст наложения. Проверить: `position` + `z-index`, `opacity < 1`, `transform`, `filter`, `will-change`, `isolation`, `contain`.

```javascript
// Консоль DevTools: найти что создаёт контекст наложения для элемента
function findStackingContexts(el) {
  const contexts = [];
  let node = el.parentElement;
  while (node && node !== document.documentElement) {
    const style = getComputedStyle(node);
    const isContext =
      (style.position !== 'static' && style.zIndex !== 'auto') ||
      parseFloat(style.opacity) < 1 ||
      style.transform !== 'none' ||
      style.filter !== 'none' ||
      style.isolation === 'isolate' ||
      style.willChange !== 'auto';
    if (isContext) contexts.push({ node, style });
    node = node.parentElement;
  }
  return contexts;
}
```

**Шаг 2: Сравнивать контексты наложения, а не значения z-index**

Найдя содержащие контексты для обоих элементов, сравните значения z-index этих контекстов (или порядок DOM, если z-index одинаков).

**Шаг 3: Исправить на правильном уровне**

Исправлять z-index на уровне контекста наложения, а не внутреннего элемента. Если контекст элемента A находится позади контекста элемента B, никакое значение z-index у потомков A не поможет.

**Шаг 4: Использовать панель Layers в DevTools**

Chrome DevTools → панель Layers показывает все слои композитинга (которые часто соответствуют контекстам наложения). Можно увидеть дерево слоёв и определить, какой элемент вызывает неожиданное удержание.

**Типичные исправления:**

```css
/* Исправление 1: Дать контексту наложения более высокий z-index */
.sidebar {
  transform: translateX(0);
  z-index: 20; /* теперь контекст .sidebar выше контекста .header */
}

/* Исправление 2: Явная изоляция */
.component {
  isolation: isolate;
  position: relative;
  z-index: 5; /* конкурирует на этом уровне */
}

/* Исправление 3: Вынести элемент за пределы проблемного контекста */
/* Паттерн портала: рендерить модалы, дропдауны, тултипы как потомков <body>,
   чтобы не оказаться в ловушке контекста наложения компонента */
```

**Паттерн портала** — это правильное архитектурное решение для UI-оверлеев (модалы, дропдауны, тултипы): рендерить их как прямых потомков `<body>` (или выделенного портал-контейнера в корне), а не внутри дерева компонентов. Именно для этого существуют React Portal, Vue Teleport и Angular CDK Overlay.

## Типичные ошибки на интервью

**"Почему `z-index: 9999` не работает?"**

Почти всегда — проблема контекста наложения. Элемент находится внутри контекста наложения, чей собственный z-index (относительно соседних контекстов) ниже, чем у элемента, поверх которого нужно появиться. Значения z-index конкурируют только внутри одного контекста наложения. Исправление — на уровне контекста наложения, а не внутреннего элемента.

---

**"Что создаёт контекст наложения?"**

Интервьюеры ожидают больше, чем просто "position + z-index". Полный список включает: `opacity < 1`, `transform` (любое значение, кроме none), `filter` (любое, кроме none), `will-change` с compositing-свойствами, `isolation: isolate`, `contain: layout/paint/strict/content`, `mix-blend-mode` (не normal), `clip-path` (не none), `mask`, `backdrop-filter`, а также `position: fixed/sticky` (всегда, независимо от z-index).

---

**"В чём разница между `z-index: 0` и `z-index: auto` на позиционированном элементе?"**

`z-index: auto`: позиционированный элемент участвует в родительском контексте наложения, не создавая своего. Значения z-index его потомков конкурируют с сиблингами самого элемента. `z-index: 0`: создаётся новый контекст наложения. Значения z-index потомков изолированы внутри него и не могут конкурировать с внешними элементами.

---

**"Что делает `isolation: isolate` и когда его использовать?"**

Создаёт контекст наложения без визуальных побочных эффектов — элемент не трансформируется, не фильтруется, не становится полупрозрачным. Применяется для: (1) удержания z-index-значений внутри компонента, чтобы они не "вытекали" во внешний контекст; (2) предотвращения "выхода" `mix-blend-mode` за пределы компонента; (3) создания предсказуемого z-index-поведения для компонентов дизайн-системы, работающих в любом окружении.

---

**"У вас есть дропдаун, который отображается за sticky-шапкой — как отладить и исправить?"**

1. Проверить контекст наложения sticky-шапки (он всегда создаётся — `position: sticky` достаточно). Найти его z-index или позицию в DOM.
2. Проверить, в каком контексте наложения находится родитель дропдауна. Если родительский контекст дропдауна имеет меньший z-index, чем контекст шапки — дропдаун всегда будет за ней.
3. Исправление: дать родительскому контексту наложения дропдауна z-index выше, чем у контекста шапки, или рендерить дропдаун через портал вне родительского контекста.
