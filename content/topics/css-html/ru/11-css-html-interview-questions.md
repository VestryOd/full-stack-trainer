# CSS + HTML Advanced — Вопросы на интервью

## Группа 1: Блочная модель и раскладка

**Какова итоговая ширина элемента с `width: 300px`, `padding: 0 20px`, `border: 3px solid` и значением `box-sizing` по умолчанию?**

`346px`. По умолчанию `box-sizing: content-box` означает, что `width` определяет только область контента. Padding и border добавляются сверх: `300 + 20 + 20 + 3 + 3 = 346px`. С `box-sizing: border-box` итог составит `300px` — область контента сжимается до `254px`, чтобы вместить padding и border. Поэтому современный универсальный сброс применяет `box-sizing: border-box` ко всем элементам, включая `::before` и `::after`.

---

**Объясните коллапс margin. Когда он происходит, а когда нет?**

Коллапс margin объединяет смежные вертикальные отступы в единый, равный наибольшему из двух (не их сумме). Три сценария: (1) смежные блочные сиблинги — нижний margin первого и верхний margin второго схлопываются; (2) родитель и первый/последний потомок — если между ними нет border, padding или BFC, margin потомка "вытекает" из родителя; (3) пустые блоки — элемент без высоты, border и padding схлопывает собственные верхний и нижний margin. НЕ происходит во flex/grid-контейнерах, у float-элементов, у `position: absolute/fixed`, и когда BFC разделяет элементы. Частая ловушка: переключение с `display: block` на `display: flex` делает ранее схлопнутые margin внезапно складывающимися, увеличивая расстояние между элементами.

---

**Что такое Block Formatting Context (BFC) и что его создаёт?**

BFC — изолированная среда раскладки, где внутренние элементы не взаимодействуют с внешними для целей раскладки — margin не схлопывается через границы BFC, float-элементы ограничены внутри. Что создаёт BFC: `overflow` отличный от `visible`, `display: flow-root` (явный, без побочных эффектов), `display: flex/grid`, `float`, `position: absolute/fixed`, `contain: layout/paint`. Практическое различие между `overflow: hidden` (хак для BFC) и `display: flow-root` (намеренный BFC): `overflow: hidden` обрезает контент — это побочный эффект; `display: flow-root` создаёт BFC без побочных эффектов.

---

**Почему `position: fixed` иногда не фиксируется относительно вьюпорта?**

Потому что у предка есть `transform`, `filter`, `will-change: transform`, `backdrop-filter` или `perspective`. Любое из этих свойств создаёт новый containing block, захватывающий `position: fixed`-потомков и заставляющий их позиционироваться относительно этого предка вместо вьюпорта. Даже `transform: none` может вызвать это в некоторых браузерах. Архитектурное решение: рендерить фиксированные элементы (модалы, toast, drawer-ы) как прямых потомков `<body>` — паттерн портала, используемый React Portal и Vue Teleport.

## Группа 2: Flexbox

**В чём разница между `flex: 1` и `flex: 1 1 auto`?**

Значение `flex-basis`. `flex: 1` раскрывается в `flex: 1 1 0%` — все элементы начинают с нуля и растут пропорционально, давая равную ширину независимо от контента. `flex: 1 1 auto` использует ширину контента каждого элемента как отправную точку, затем пропорционально распределяет только оставшееся свободное пространство — элемент с бо́льшим контентом остаётся крупнее. Для по-настоящему равноширинных колонок: используйте `flex: 1` (или эквивалентно `flex-basis: 0`).

---

**Flex-элемент имеет `flex-shrink: 1`, но всё равно переполняет контейнер. Почему?**

Значение `min-width: auto` по умолчанию у flex-элементов. Когда flex-элемент содержит интринсический контент (текст, изображения, элементы с явной шириной), `min-width: auto` разрешается в минимальный размер контента — наименьший размер без переполнения. `flex-shrink` не может сжать элемент ниже этого порога. Исправление: добавить `min-width: 0` к flex-элементу, что убирает порог и позволяет `flex-shrink` работать корректно. Дополнить `overflow: hidden` для управления обрезанным контентом. Именно поэтому усечение текста через `text-overflow: ellipsis` требует `min-width: 0` на flex-предке.

---

**Как `flex-shrink` распределяет переполнение между элементами — пропорционально значениям `flex-shrink`?**

Не совсем. Сжатие взвешивается обоими параметрами — `flex-shrink` и `flex-basis`: вес каждого элемента = `flex-shrink × flex-basis`. Элемент с `flex-shrink: 2, flex-basis: 200px` имеет вес `400`; элемент с `flex-shrink: 1, flex-basis: 200px` имеет вес `200`. Переполнение распределяется пропорционально этим взвешенным коэффициентам. Это означает, что элементы с бо́льшим `flex-basis` сжимаются больше в абсолютных пикселях при равных `flex-shrink` — алгоритм спроектирован, чтобы избежать непропорционального сжатия относительно размера.

---

**Как выровнять один flex-элемент к дальнему концу главной оси, оставив остальные в начале?**

`margin-left: auto` (или `margin-inline-start: auto` для логических свойств) на элементе, который нужно сдвинуть. Во Flexbox `auto`-margin поглощает всё доступное свободное пространство в указанном направлении. `justify-self` не работает на flex-элементах — это свойство только для grid. Эквивалентно: `margin-right: auto` на последнем "левостороннем" элементе даёт тот же результат, выталкивая всё после него вправо.

## Группа 3: CSS Grid

**Что точно означает `1fr` — это одна доля ширины контейнера?**

Нет — одна доля **доступного свободного пространства**, а не ширины контейнера. Свободное пространство — то, что остаётся после разрешения всех фиксированных треков (px, em, %) и gap. В контейнере 900px с фиксированной колонкой 200px, gap 20px и двумя колонками `1fr`: свободное пространство = `900 - 200 - 2×20 = 660px`, каждая `fr` = `330px`. Поэтому `fr` предпочтительнее `%` — процентные колонки включают пространство gap и могут переполняться, а `fr` всегда вписывается.

---

**В чём разница между `auto-fill` и `auto-fit` в `repeat()`?**

Оба создают максимально возможное количество треков. Разница проявляется, когда элементов меньше, чем доступных слотов. `auto-fill` сохраняет пустые треки — элементы не растут за `max` в `minmax()`. `auto-fit` схлопывает пустые треки в ноль — элементы расширяются, заполняя всю ширину контейнера. При достаточном количестве элементов оба ведут себя одинаково. Используйте `auto-fit` для паттерна "адаптивная сетка без медиазапросов"; `auto-fill` — когда нужно, чтобы элементы сохраняли минимальный размер даже если остаются пустые колонки.

---

**Как работает `grid-column: 1 / -1` и что такое `-1`?**

Линия `-1` обозначает последнюю линию **явной** сетки. `grid-column: 1 / -1` охватывает от первой до последней явной колоночной линии — полная ширина. При `grid-template-columns: repeat(4, 1fr)` есть 5 линий (с 1 по 5), поэтому `1 / -1` охватывает все 4 колонки. Оговорка: `-1` работает только для явной сетки. Элементы в неявной сетке (автогенерируемые строки) не могут использовать `-1` для полной ширины, если эти строки не определены явно в `grid-template-rows`.

---

**Что такое subgrid и какую проблему он решает?**

Subgrid (`grid-template-columns: subgrid` или `grid-template-rows: subgrid`) позволяет grid-элементу, который сам является grid-контейнером, наследовать — а не копировать — структуру треков родителя. Решаемая проблема: в сетке карточек, где у каждой есть заголовок, тело и футер, если карточки имеют разную высоту контента, их футеры не выравниваются горизонтально. С subgrid все карточки делят строковые треки родителя, и заголовки, тела, футеры выравниваются автоматически независимо от высоты контента. Поддерживается во всех основных браузерах с Chrome 117 (август 2023).

## Группа 4: Контексты наложения и z-index

**Почему `z-index: 9999` не гарантирует, что элемент окажется поверх?**

Потому что z-index локален для контекста наложения. Элементы конкурируют по z-index только внутри одного контекста наложения. Если элемент A находится в контексте X (z-index: 1), а элемент B — в контексте Y (z-index: 2), то `z-index: 9999` у A не имеет значения — весь контекст X (включая A) рендерится позади всего контекста Y (включая B). Исправление всегда на уровне контекста наложения, а не внутреннего элемента.

---

**Назовите пять вещей, создающих контекст наложения, помимо `position` + `z-index`.**

`opacity` меньше 1 (даже `opacity: 0.99`), `transform` любое ненулевое значение (включая `translateZ(0)` для GPU), `filter` любое ненулевое, `will-change` с compositing-свойствами (`will-change: transform`), `isolation: isolate`, `contain: layout/paint/strict/content`, `mix-blend-mode` не normal, `clip-path` не none, `backdrop-filter`, `position: fixed` или `sticky` (всегда, независимо от z-index). Практическое следствие: безобидный `transform: translateX(0)` для GPU-оптимизации на сайдбаре захватит все абсолютно позиционированные потомки, ограничив их z-index внутри контекста наложения сайдбара.

---

**Что такое `isolation: isolate` и когда его использовать?**

Создаёт контекст наложения без визуальных побочных эффектов — нет изменения прозрачности, трансформации, фильтра. Основные сценарии: (1) изоляция внутренних z-index-значений компонента — компонент дизайн-системы может использовать `z-index: 10` внутри без конфликта с `z-index: 2` модала на странице; (2) предотвращение выхода `mix-blend-mode` за границы компонента. Это CSS-эквивалент границы модуля для контекстов наложения.

## Группа 5: Специфичность и каскад

**Какова специфичность `:is(div, .class, #id) p` и почему?**

`(1, 0, 1)`. Псевдокласс `:is()` берёт специфичность **наиболее специфичного аргумента** — `#id` это `(1, 0, 0)`, поэтому `:is(...)` вносит `(1, 0, 0)`. Типовой селектор `p` добавляет `(0, 0, 1)`. Итог: `(1, 0, 1)`. Это применяется даже когда элемент соответствует через `.class` или `div` — всегда используется максимальная специфичность из всего списка. Ключевое отличие от `:where()`, которое всегда вносит `(0, 0, 0)` независимо от аргументов.

---

**Что такое каскадные слои (`@layer`) и зачем они введены?**

`@layer` объявляет явно упорядоченные bucket-ы CSS. Правила в слое с более высоким приоритетом побеждают правила в слое с более низким независимо от специфичности. Порядок объявления задаёт приоритет: `@layer reset, base, components, utilities` — utilities имеют наивысший приоритет. Стили без слоёв всегда побеждают стилизованные. Введены для решения войн специфичности в масштабе: без слоёв весь CSS конкурирует по специфичности и порядку, что ведёт к бесконечной эскалации селекторов и `!important`. С слоями — кладёте код в правильный bucket и прекращаете бороться со специфичностью.

---

**Что происходит, когда CSS-переменная содержит недопустимое для контекста использования значение?**

Поведение invalid-at-computed-value-time: свойство разрешается в **унаследованное значение** (если наследуется) или **начальное значение** (если нет) — НЕ в браузерное умолчание. Например, `:root { --color: 16px }`, затем `p { color: var(--color) }` — свойство `color` получает недопустимое значение. Результат не `black` (браузерное умолчание), а унаследованное значение `color` от родителя или `initial` в корне. Поэтому баги с CSS-переменными сложно отлаживать — нет ошибки в консоли, и поведение запасного варианта неочевидно.

## Группа 6: Адаптивный и современный CSS

**Какую проблему решают container queries, которую медиазапросы не могут?**

Адаптацию компонентов к фактическому размеру их контейнера, а не к глобальному размеру вьюпорта. Карточка в сайдбаре 280px и та же карточка в основной области 600px нуждаются в разных раскладках при одном размере вьюпорта. Медиазапросы знают только ширину вьюпорта — не могут различить эти два размещения. Container queries позволяют карточке реагировать на занимаемое пространство: `@container (min-width: 400px) { .card { display: grid; } }`. Это делает компоненты по-настоящему переиспользуемыми — один компонент, один CSS, корректная раскладка в любом контейнере.

---

**Объясните `clamp(1rem, 2.5vw, 2rem)` точно.**

Значение равно `2.5vw` (предпочтительное, viewport-зависимое выражение), ограниченному минимумом `1rem` и максимумом `2rem`. При вьюпорте 400px: `2.5vw = 10px < 16px (1rem)` → используется `1rem`. При 800px: `2.5vw = 20px`, между мин и макс → используется `20px`. При 1400px: `2.5vw = 35px > 32px (2rem)` → используется `2rem`. Значение непрерывно переходит между крайними точками без скачков на брейкпоинтах. Одно объявление заменяет два медиазапроса-брейкпоинта.

---

**В чём разница между `margin-left` и `margin-inline-start`?**

`margin-left` — физическое свойство, всегда левая сторона независимо от направления письма. `margin-inline-start` — логическое свойство, отображается в начало inline-направления на основе `direction` и `writing-mode`. В LTR они идентичны. В RTL `margin-inline-start` отображается в `margin-right`. В вертикальных режимах письма (японский) — в верх или низ. Преимущество: RTL и вертикальные раскладки не требуют `[dir="rtl"]`-переопределений — браузер отображает логические свойства автоматически. Для международных приложений логические свойства должны быть стандартом.

## Группа 7: Конвейер рендеринга и производительность

**Назовите пять этапов конвейера рендеринга браузера и что каждый делает.**

(1) **Parse**: HTML → DOM, CSS → CSSOM, объединяются в Render Tree. CSS блокирует рендеринг; `<script>` без `async`/`defer` блокирует парсер. (2) **Style (Recalculate Style)**: вычисляет computed style каждого элемента — разрешает каскад, наследование, кастомные свойства, относительные единицы в абсолютные пиксели. (3) **Layout (Reflow)**: вычисляет геометрию — позицию и размер каждого элемента. Самый дорогой — каскадирует по документу. (4) **Paint**: записывает инструкции рисования (команды, а не пиксели) для каждого слоя — фоны, рамки, текст, тени. (5) **Composite**: GPU объединяет слои в финальный кадр, применяя трансформации и прозрачность. Только два CSS-свойства запускают исключительно этот последний этап: `transform` и `opacity`.

---

**Какие CSS-свойства запускают только Composite и почему? Почему не `left`/`top`?**

Только `transform` и `opacity`. Они не влияют на геометрию (Layout не нужен) и не меняют пиксельный вид самого элемента (Paint не нужен). Браузер повышает элемент до GPU-слоя, загружает текстуру один раз, и последующие изменения полностью обрабатываются GPU — применяя матричную трансформацию или альфа-смешивание на каждом кадре. `left`/`top` — layout-свойства: их изменение перемещает элемент в потоке документа, что требует пересчёта позиций окружающих элементов (Layout), перезаписи команд рисования (Paint), затем compositing. На каждом кадре — CPU-работа; отсюда плавность анимаций через `transform` и рывки через `left`.

---

**Что такое layout thrashing и как его устранить?**

Layout thrashing происходит, когда JavaScript чередует чтение layout-влияющих DOM-свойств и запись в них в цикле. Каждое чтение (например, `offsetWidth`, `getBoundingClientRect()`) принудительно сбрасывает ожидающие изменения стилей/layout и вычисляет актуальное значение — синхронный reflow. Каждая запись инвалидирует layout снова. Чередование вызывает N синхронных reflow за итерацию цикла. Исправление: пакетное чтение всех значений, затем пакетная запись. Браузер выполнит один reflow для чтений и одно обновление layout для записей. `requestAnimationFrame` помогает планировать записи в правильный момент конвейера.

---

**Что реально делает `will-change: transform` и каковы риски?**

Сигнализирует браузеру, что `transform` элемента скоро изменится. Браузер обычно реагирует, заранее повышая элемент до GPU-слоя — до начала анимации, чтобы первый кадр не дал рывок от создания слоя. Риски: каждый повышенный слой потребляет GPU-память (VRAM), и когда повышенный элемент перекрывает множество непромоутированных, браузер может повысить их все во избежание артефактов ("взрыв слоёв"). Правила ответственного использования: применять непосредственно перед анимацией, снимать (`will-change: auto`) сразу после. Никогда не применять глобально.

## Группа 8: CSS-архитектура и формы

**Почему BEM использует единственные классы вместо селекторов-потомков?**

Единственные классы имеют единообразную специфичность `(0, 1, 0)` — внутри BEM-кодовой базы конфликты специфичности невозможны. Селекторы-потомки (`.card .title`) имеют более высокую специфичность `(0, 2, 0)` и связывают селектор с DOM-структурой — если `.title` перемещается за пределы `.card`, стиль ломается. BEM кодирует отношение в имени (`.card__title`), а не в иерархии. Переопределение любого BEM-стиля тривиально: один класс выше `(0, 1, 0)` победит. Компромисс: BEM не решает проблему глобальной области — два разработчика могут создать `.card__title` с конфликтующими намерениями.

---

**Какова основная проблема производительности у runtime CSS-in-JS (Styled Components, Emotion)?**

Вставка стилей происходит в JavaScript на основном потоке при рендеринге. Каждый рендер с новыми значениями props генерирует новые CSS-правила и вставляет/обновляет теги `<style>`, добавляя к Time to Interactive. При SSR стили должны быть сериализованы в HTML и согласованы на клиенте при гидратации. Дополнительный размер бандла от определений стилей увеличивает время парсинга JS. Эти издержки стали блокирующей проблемой с React Server Components — runtime CSS-in-JS требует браузерного JS-окружения, которого в RSC нет. Zero-runtime альтернативы (vanilla-extract, StyleX) генерируют CSS при сборке, устраняя runtime-стоимость.

---

**Что делает `novalidate` на `<form>` и зачем это нужно?**

`novalidate` отключает нативный UI браузерной валидации (платформенные всплывающие пузыри и поведение фокусировки при отправке), сохраняя Constraint Validation API полностью активным. `input.validity`, `input.checkValidity()` и `input.setCustomValidity()` продолжают работать. Нужно, когда строится кастомный UX валидации: контроль над тем, когда появляются ошибки (при blur, а не только при submit), визуальный дизайн сообщений, управление фокусом на невалидных полях, межполевая валидация и интеграция серверных ошибок — ничего из этого нативный UI не поддерживает.

---

**Что такое `setCustomValidity` и какое критическое правило очистки?**

`setCustomValidity(message)` устанавливает кастомное сообщение об ошибке на элементе управления формой. Непустая строка помечает поле как невалидное (`validity.customError = true`), сообщение появляется в `validationMessage`. Критическое правило: необходимо вызывать `setCustomValidity('')` (пустая строка) при редактировании поля — иначе кастомная ошибка сохраняется бессрочно независимо от того, что вводит пользователь. Поле остаётся невалидным даже при абсолютно корректном вводе. Стандартный паттерн: устанавливать ошибку после ответа сервера, очищать в обработчике события `input`.

---

**Почему нативной HTML-валидации недостаточно для production, даже если сервер всё равно валидирует?**

Пять причин: (1) **UI неконтролируем** — нативные пузыри нельзя стилизовать под дизайн-систему. (2) **Тайминг неправильный** — нативная валидация только при отправке; лучшая практика требует валидации при blur (первый выход) и при input (после первой ошибки). (3) **Межполевая валидация невозможна** — совпадение паролей, диапазоны дат, условные обязательные поля не выразить в HTML-атрибутах. (4) **Серверные ошибки нельзя отобразить** — после отправки отказы сервера (дублирующийся email, недействительный купон) не имеют нативного механизма отображения; нужны `setCustomValidity` + JavaScript. (5) **Асинхронная валидация невозможна** — проверка доступности имени пользователя требует API-запроса, который атрибуты `pattern` не умеют делать.

## Группа 9: Доступность и семантика

**Что такое дерево доступности и почему оно важно?**

Дерево доступности — параллельное представление страницы, которое браузеры строят из семантического HTML и ARIA-атрибутов. Скринридеры, брайлевские дисплеи, программы голосового управления и другие вспомогательные технологии читают дерево доступности — не DOM и не визуальную раскладку. `<div onclick>` есть в DOM, но не имеет роли, клавиатурного поведения и состояния в дереве доступности. `<button>` имеет role=button, находится в порядке Tab, активируется Пробелом и Enter, имеет состояние disabled. Каждый `<div>` вместо семантического элемента — ручная переимплементация поведения, которое браузер предоставляет бесплатно.

---

**Когда ARIA полезна, а когда вредна?**

Полезна: (1) кастомные интерактивные виджеты без HTML-аналога (tab list, tree view, combobox); (2) динамически изменяемые состояния для скринридеров (`aria-expanded`, `aria-live`); (3) подписи для элементов, не имеющих `<label>` (кнопки-иконки, ориентиры). Вредна: переопределение корректной нативной семантики (`role="button"` на `<div>` без обработки клавиатуры), `aria-hidden="true"` на контейнере с фокусируемыми потомками (дыра для фокуса), некорректные роли, вводящие AT в заблуждение. Руководящий принцип: **отсутствие ARIA лучше плохой ARIA**. Всегда предпочитайте нативную семантику; добавляйте ARIA только когда нативной семантики недостаточно.

---

**Что такое паттерн roving tabindex и когда применяется?**

Roving tabindex используется для составных виджетов, где Tab перемещает между виджетами, а стрелки — внутри виджета: tab list, toolbar, radio group, tree view. Паттерн: в любой момент только один элемент имеет `tabindex="0"` (текущий активный); все остальные — `tabindex="-1"`. Обработчики стрелок перемещают фокус, устанавливая новый элемент в `tabindex="0"` и предыдущий обратно в `tabindex="-1"`. Это гарантирует, что виджет является единственной точкой Tab для пользователей клавиатуры — им не нужно объходить Tab-ом каждую вкладку в tab list чтобы выйти из компонента.

## Группа 10: Продвинутые и сквозные вопросы

**Как отладить проблему z-index, когда элемент оказывается позади другого несмотря на более высокий z-index?**

Шаг 1: определить цепочку контекстов наложения для обоих элементов. Пройти вверх по DOM от каждого элемента и найти всех предков, создающих контекст — проверить `position` + `z-index`, `opacity < 1`, `transform`, `filter`, `will-change`, `isolation`, `contain`. Шаг 2: найти общий предок-контекст наложения — важно сравнение контекстов наложения на этом уровне, а не собственные z-index элементов. Шаг 3: исправить на правильном уровне — скорректировать z-index содержащего контекста, а не внутреннего элемента. Шаг 4: если элемент находится внутри компонента, рассмотреть паттерн портала — рендерить оверлеи (модалы, тултипы, дропдауны) как потомков `<body>`, полностью избегая захвата контекстом наложения.

---

**В чём разница между `contain: layout`, `contain: paint` и `contain: strict`?**

`contain: layout`: внутренние изменения layout не влияют на внешние элементы и наоборот. Браузер может выполнять reflow поддерева независимо. Побочные эффекты: создаёт stacking context, BFC, containing block для позиционированных потомков. `contain: paint`: действует как вьюпорт для поддерева — потомки не могут визуально выходить за пределы. Также создаёт stacking context, BFC, containing block. Аналог `overflow: hidden` с дополнительными сигналами производительности. `contain: strict`: `layout + paint + size + style` — наиболее строгая форма. Требует явных `width` и `height` (size containment означает, что браузер игнорирует потомков при определении размера элемента). Используется для компонентов фиксированного размера, часто обновляющихся (миниатюры, элементы сетки).

---

**Сравните три подхода к адаптивной типографике: медиазапросы, единицы `vw` и `clamp()`.**

Медиазапросы: явные шаги — шрифт 16px до 768px, 18px выше. Просто, но создаёт резкие скачки на брейкпоинтах. Несколько брейкпоинтов умножают поверхность поддержки. Единицы `vw` (`font-size: 2vw`): непрерывное масштабирование, но без минимума и максимума — текст становится нечитаемо мелким на телефонах и нечитаемо крупным на широких мониторах. `clamp()` (`font-size: clamp(1rem, 2.5vw, 1.5rem)`): сочетает непрерывное масштабирование с заданными границами. Плавно масштабируется между минимумом (мобильный пол) и максимумом (десктопный потолок) без медиазапросов. Предпочтительный современный подход — одно объявление охватывает все размеры вьюпорта.

---

**Когда выбрать CSS Modules вместо Tailwind и наоборот?**

CSS Modules вместо Tailwind: когда стили компонента включают сложные дизайны псевдоэлементов, сложные анимации, глубоко условную логику, или когда стили имеют бо́льший смысл как связный блок, а не 15 отдельных utility-классов. Также когда команда имеет сильную CSS-экспертизу и предпочитает явный авторинг, или при интеграции с дизайн-системой на CSS-переменных. Tailwind вместо CSS Modules: когда важна скорость итерации (без переключения файлов, стили в JSX), нужна встроенная дизайн-система с ограничениями (`spacing-4`), проект использует компонентный фреймворк, абстрагирующий повторение списков классов, и когда размер CSS-бандла важен (Tailwind удаляет неиспользуемые utility).
