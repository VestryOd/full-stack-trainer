# Специфичность и каскад

## Каскад — что это на самом деле

**Каскад** — это алгоритм, который браузеры используют для разрешения конфликтов, когда несколько CSS-правил нацелены на один элемент и одно свойство. Это не просто "последнее правило побеждает" — это лишь финальный разрешитель ничьих. Каскад применяет пять критериев по порядку, останавливаясь как только один критерий выявляет победителя:

1. **Происхождение и важность** — откуда пришло правило и установлен ли `!important`
2. **Контекст** — границы инкапсуляции shadow DOM
3. **Каскадный слой** — порядок `@layer`
4. **Специфичность** — вес селектора
5. **Порядок появления** — позиция в исходном коде (финальный разрешитель)

Большинство разработчиков знают только критерии 4 и 5. Понимание всех пяти — разница между "CSS непонятен" и "CSS полностью логичен".

## Происхождение и важность — первый критерий

CSS-правила приходят из трёх источников:

1. **Таблица стилей user-agent** — браузерные стили по умолчанию (`<h1>` — жирный, `<a>` — синий с подчёркиванием, `<button>` — с отступами)
2. **Авторская таблица стилей** — ваш CSS (подключаемые файлы, теги `<style>`, инлайновые стили)
3. **Пользовательская таблица стилей** — кастомные стили, применяемые пользователем (инструменты доступности, расширения браузера)

Без `!important` приоритет: **авторские > пользовательские > user-agent**.

С `!important` порядок **обращается для этого свойства**: **user-agent `!important` > пользователь `!important` > автор `!important` > авторские обычные > пользовательские обычные > user-agent обычные**.

Это инверсия намеренная. Пользователь, применяющий `!important` (например, через расширение для высококонтрастного режима), должен иметь возможность переопределить даже авторский `!important` — потому что потребности пользователя приоритетнее.

Инлайновые стили (`style=""`) — это авторские стили с крайне высокой специфичностью, но они по-прежнему относятся к "авторскому" источнику и могут быть переопределены авторским `!important`.

## Специфичность — точный расчёт

Специфичность представлена тремя числами: **(A, B, C)** где:
- **A** — количество ID-селекторов (`#id`)
- **B** — количество классовых селекторов (`.class`), атрибутивных (`[attr]`) и псевдоклассов (`:hover`, `:focus`, `:nth-child()`)
- **C** — количество типовых селекторов (теги: `div`, `p`, `span`) и псевдоэлементов (`::before`, `::after`)

Универсальный селектор `*`, комбинаторы (`>`, `+`, `~`, ` `) и `:where()` вносят нулевую специфичность.

Сравнение — слева направо: большее A всегда побеждает любое значение B или C. **Это не десятичное число** — `(0, 1, 0)` не превращается в `0.10`. Класс `(0, 1, 0)` побеждает любое количество типовых селекторов `(0, 0, N)` — даже `(0, 0, 1000)`.

```css
/* Разбор специфичности: */

*                          /* (0, 0, 0) — универсальный, не вносит ничего */
div                        /* (0, 0, 1) — один типовой селектор */
.class                     /* (0, 1, 0) — один класс */
#id                        /* (1, 0, 0) — один ID */
div p                      /* (0, 0, 2) — два типовых */
div.class                  /* (0, 1, 1) — класс + типовой */
.nav .item                 /* (0, 2, 0) — два класса */
#header .nav a             /* (1, 1, 1) — ID + класс + типовой */
a:hover                    /* (0, 1, 1) — псевдокласс + типовой */
a::before                  /* (0, 0, 2) — псевдоэлемент + типовой */
input[type="text"]         /* (0, 1, 1) — атрибутивный + типовой */
:not(.active)              /* (0, 1, 0) — :not() берёт специфичность аргумента */
:is(.nav, #header)         /* (1, 0, 0) — :is() берёт наивысшую специфичность из аргументов */
:where(.nav, #header)      /* (0, 0, 0) — :where() всегда вносит ноль */
```

### `:is()`, `:not()`, `:has()` — специфичность списковых селекторов

Эти псевдоклассы "снисходительные" — принимают списки селекторов. Их специфичность определяется **наиболее специфичным селектором в списке аргументов**:

```css
/* Специфичность :is() = специфичность наиболее специфичного аргумента */
:is(div, .class, #id) p    /* (1, 0, 1) — #id наиболее специфичен в списке */

/* Даже если совпадающий элемент не соответствует #id: */
/* <p class="class"> соответствует :is(div, .class, #id) через .class */
/* но специфичность всё равно (1, 0, 1), потому что #id есть в списке */
```

Это ключевое отличие `:is()` от `:where()`: `:where()` всегда вносит нулевую специфичность, что делает его идеальным для базовых стилей, которые должно быть легко переопределить.

```css
/* Базовые стили через :where() — нулевая специфичность, легко переопределить */
:where(h1, h2, h3, h4, h5, h6) {
  font-weight: 700;
  line-height: 1.2;
}

/* Переопределение с любой специфичностью */
.article h2 { font-weight: 600; } /* (0, 1, 1) — побеждает */
```

### Инлайновые стили

Инлайновые стили имеют специфичность выше любого селектора: фактически `(1, 0, 0, 0)` — четвёртая колонка над A. Только `!important` переопределяет инлайновые стили.

```html
<p style="color: red;">текст</p>
```

```css
#specific.super-specific p { color: blue; } /* (1, 1, 1) — всё равно проигрывает инлайну */
```

## `!important` — механизм и когда он оправдан

`!important` перемещает объявление из нормальной конкуренции специфичности в отдельный "важный" bucket. Внутри важного bucket специфичность и порядок по-прежнему применяются, но важный bucket всегда побеждает обычный.

```css
p { color: blue !important; }          /* важный bucket, (0, 0, 1) */
#id p { color: red; }                  /* обычный bucket, (1, 0, 1) */
/* Результат: blue — важный побеждает обычный независимо от специфичности */

p { color: blue !important; }          /* важный bucket, (0, 0, 1) */
#id p { color: red !important; }       /* важный bucket, (1, 0, 1) */
/* Результат: red — внутри важного bucket побеждает более высокая специфичность */
```

### Когда `!important` действительно оправдан

`!important` имеет законные применения — проблема в том, что его используют как ленивое исправление, а не намеренно:

**1. Utility-классы, которые не должны переопределяться**

```css
/* Utility-классы в дизайн-системе — должны применяться всегда */
.sr-only {
  position: absolute !important;
  width: 1px !important;
  height: 1px !important;
  overflow: hidden !important;
  clip: rect(0, 0, 0, 0) !important;
  white-space: nowrap !important;
}
/* Стили компонентов могут иметь более высокую специфичность — !important гарантирует применение */
```

**2. Переопределение инлайновых стилей сторонних библиотек**

```css
/* Сторонний виджет добавляет инлайновые стили, которые мы не контролируем */
.widget-container {
  width: 100% !important; /* переопределяет style="width: 300px" виджета */
}
```

**3. Пользовательские переопределения для доступности**

```css
/* Пользовательская таблица стилей для высококонтрастного режима */
* {
  background-color: black !important;
  color: white !important;
}
/* Нужен !important для переопределения авторских стилей */
```

**4. Конечные состояния анимаций**

Редко, но `!important` в `@keyframes` предотвращает переопределение значения специфичностью в середине анимации.

**Антипаттерны:** использование `!important` для борьбы со специфичностью в собственной кодовой базе означает сломанную архитектуру специфичности. Исправление — в архитектуре (`@layer`, изоляция компонентов, низкоспецифичные селекторы), а не в `!important`.

## Каскадные слои — `@layer`

`@layer` появился в 2022 году (во всех основных браузерах к концу 2022) для решения ключевой проблемы CSS в масштабе: **войн специфичности**.

### Проблема, которую решает `@layer`

В большом приложении есть:
- Браузерные сбросы (например, normalize.css)
- Базовые стили фреймворка
- Токены и компоненты вашей дизайн-системы
- Стили страниц
- Стили сторонних виджетов

Без слоёв всё это конкурирует по специфичности и порядку. `.btn` из сброса, `.btn` из дизайн-системы и `.btn` из стороннего кода борются. В итоге — постоянно растущая специфичность селекторов или россыпь `!important`.

### Как работают слои

```css
/* Сначала объявляем порядок слоёв — позже в списке = ниже приоритет */
@layer reset, base, components, utilities;

@layer reset {
  *, *::before, *::after { box-sizing: border-box; }
  body { margin: 0; }
}

@layer base {
  h1 { font-size: 2rem; }
  a { color: inherit; text-decoration: underline; }
}

@layer components {
  .btn {
    display: inline-flex;
    padding: 8px 16px;
    border-radius: 4px;
    font-weight: 600;
  }
  .btn-primary {
    background: #0066cc;
    color: white;
  }
}

@layer utilities {
  .mt-0 { margin-top: 0; }
  .text-center { text-align: center; }
}
```

**Порядок приоритетов:** стили вне слоёв > `utilities` > `components` > `base` > `reset`.

Слой, объявленный последним в строке `@layer`, побеждает более ранние слои. Стили вне слоёв (за пределами `@layer`) всегда побеждают стилизованные — намеренно для постепенной миграции: ваш существующий CSS без слоёв имеет приоритет над слоистым кодом третьих сторон.

### Ключевой инсайт: слои побеждают специфичность

В слое с более высоким приоритетом **даже низкоспецифичный селектор побеждает высокоспецифичный из слоя с более низким приоритетом**:

```css
@layer base {
  #specific-id .nested-class p { color: blue; }  /* (1, 1, 1) */
}

@layer utilities {
  .text-red { color: red; }  /* (0, 1, 0) */
}
/* Результат: red — слой utilities побеждает слой base независимо от специфичности */
```

Это фундаментально меняет подход к написанию CSS. Вместо наращивания специфичности для победы — кладёте вещи в правильный слой. Utility-классы могут быть по-настоящему атомарными — `(0, 1, 0)` — и всё равно побеждать любые стили компонентов, поскольку находятся в слое с более высоким приоритетом.

### Импорт стороннего CSS в слой

```css
/* Сторонние стили помещаются в слой с более низким приоритетом */
@import url('normalize.css') layer(reset);
@import url('third-party-widget.css') layer(vendor);

@layer reset, vendor, components, utilities;
/* Ваши компоненты и утилиты всегда побеждают vendor-стили */
```

### Вложенные слои

```css
@layer components {
  @layer forms {
    .input { border: 1px solid #ccc; }
  }
  @layer buttons {
    .btn { padding: 8px 16px; }
  }
}
/* Обращение: components.forms, components.buttons */
/* components.buttons > components.forms по приоритету */
```

### `!important` внутри слоёв — инверсия

`!important` внутри `@layer` инвертирует приоритет слоёв — так же, как инвертируется приоритет происхождения для `!important`:

```css
@layer base, components;

@layer base {
  .text { color: blue !important; }
}
@layer components {
  .text { color: red; }
}
/* Обычный порядок: components побеждает → red */
/* Но !important в base: base's !important побеждает → blue */
/* Потому что !important инвертирует порядок слоёв */
```

Это согласуется с поведением происхождения: `!important` всегда инвертирует порядок каскада, чтобы источники/слои с более низким приоритетом могли защитить свои значения.

## CSS-переменные — взаимодействие с наследованием и каскадом

CSS-переменные (кастомные свойства) участвуют в каскаде и наследовании как любое другое свойство — но с уникальными особенностями.

### Разрешение в каскаде

Кастомные свойства следуют всем правилам каскада: специфичность, слой, происхождение, `!important`:

```css
:root { --color: blue; }
.card { --color: green; }
#special { --color: red; }

/* .card#special → --color: red (побеждает специфичность ID) */
```

### Наследование

Кастомные свойства наследуются по умолчанию — они текут вниз по DOM как `color` или `font-size`:

```css
:root {
  --spacing: 16px;
  --brand-color: #0066cc;
}

/* Все потомки могут использовать без переобъявления */
.button {
  padding: var(--spacing);
  background: var(--brand-color);
}

/* Переопределение для поддерева */
.compact-section {
  --spacing: 8px;
  /* Все потомки .compact-section используют 8px */
}
```

### Правило `@property` — зарегистрированные кастомные свойства

Незарегистрированные кастомные свойства — это по сути подстановка строк; браузер не знает тип. `@property` регистрирует тип, открывая возможности:

1. **Проверка типа** — недопустимые значения обнаруживаются
2. **Анимация** — кастомные свойства можно анимировать/транзишить
3. **Управление наследованием** — можно сделать кастомное свойство ненаследуемым

```css
@property --hue {
  syntax: '<number>';     /* CSS-тип: число */
  inherits: false;        /* НЕ наследуется по DOM */
  initial-value: 220;     /* запасное значение */
}

@property --card-bg {
  syntax: '<color>';
  inherits: true;
  initial-value: white;
}

/* Теперь --hue можно анимировать: */
.color-wheel {
  --hue: 0;
  background: hsl(var(--hue), 70%, 60%);
  transition: --hue 0.3s ease;
}

.color-wheel:hover {
  --hue: 180;
}
```

Без `@property` строка `transition: --hue 0.3s` не делает ничего — браузер не знает, как интерполировать строку. С `@property` и `syntax: '<number>'` происходит плавный переход числового значения.

### Концепция invalid-at-computed-value-time

В отличие от обычных свойств, недопустимое значение кастомного свойства не приводит к игнорированию объявления — оно приводит к разрешению свойства в **унаследованное значение** или **начальное значение**, а не в браузерное умолчание:

```css
:root { --color: 16px; } /* Не допустимый цвет, но допустимая строка кастомного свойства */

p {
  color: var(--color); /* --color равен 16px — недопустимо как цвет */
  /* Результат: НЕ браузерное умолчание (чёрный).
     Вместо этого: унаследованное значение 'color', или 'initial' в корне */
}
```

Поэтому баги с кастомными свойствами сложно диагностировать — недопустимые значения не генерируют предупреждений в DevTools.

### Запасное значение `var()`

```css
/* Второй аргумент — запасное значение */
color: var(--primary, #0066cc);

/* Запасные значения можно вкладывать */
color: var(--primary, var(--brand, #0066cc));

/* Запасное значение может быть другим кастомным свойством */
color: var(--button-color, var(--primary-color, blue));
```

Запасное значение используется только если переменная **не определена** (или определена как заведомо недопустимое значение). Пустая строка — это определённое значение: `color: var(--color, blue)` использует пустую строку, если задано `--color: ;`.

## Практические паттерны

### Избегание ада специфичности через низкоспецифичные селекторы

```css
/* Методология единственного класса — каждый селектор (0, 1, 0) */
.card { }
.card-header { }
.card-body { }
.card-footer { }

/* Без вложенности, без ID, без типовых селекторов для компонентов */
/* Легко переопределить: добавить ещё один класс */
.promotional.card { }  /* (0, 2, 0) — побеждает (0, 1, 0) */
```

### Архитектура дизайн-токенов с кастомными свойствами

```css
/* Уровень 1: Сырые значения (никогда не используются напрямую в компонентах) */
:root {
  --color-blue-500: #0066cc;
  --color-blue-600: #0052a3;
  --space-4: 16px;
  --space-8: 32px;
}

/* Уровень 2: Семантические токены (готовы к компонентам) */
:root {
  --color-action: var(--color-blue-500);
  --color-action-hover: var(--color-blue-600);
  --component-spacing: var(--space-4);
}

/* Уровень 3: Токены компонента */
.button {
  --button-bg: var(--color-action);
  --button-bg-hover: var(--color-action-hover);
  --button-padding: var(--component-spacing);

  background: var(--button-bg);
  padding: var(--button-padding);
}

/* Переключение темы — меняем только уровень 1 или 2 */
[data-theme="dark"] {
  --color-blue-500: #4da6ff;
  --color-blue-600: #3399ff;
}
```

## Типичные ошибки на интервью

**"Какова специфичность `:is(div, .class, #id)`?"**

`(1, 0, 0)` — специфичность `:is()` равна специфичности наиболее специфичного аргумента. `#id` имеет `(1, 0, 0)`, поэтому весь селектор `:is()` несёт эту специфичность, даже если соответствие произошло через `div` или `.class`. Это отличается от `:where()`, которое всегда вносит `(0, 0, 0)`.

---

**"ID-селектор всегда побеждает любое количество классовых?"**

Да — сравнение специфичности не является аддитивным между колонками. `(1, 0, 0)` побеждает `(0, 999, 999)`. ID-селектор всегда побеждает любую комбинацию классов, атрибутов и псевдоклассов — сколько бы их ни было. Колонки не переполняются в соседние.

---

**"Что делает `@layer` и зачем он был введён?"**

`@layer` объявляет каскадные слои — явно упорядоченные bucket-ы CSS. Стили в слое с более высоким приоритетом побеждают стили из слоя с более низким приоритетом, независимо от специфичности. Введён потому, что разрешение конфликтов на основе специфичности в большом масштабе приводит к нарастающим войнам селекторов, рассеянному `!important` и неподдерживаемому CSS. С слоями вещи кладутся в правильный bucket и борьба со специфичностью прекращается.

---

**"Стили вне слоёв побеждают стили внутри слоёв?"**

Да — стили без слоёв всегда побеждают стилизованные в том же происхождении. Это позволяет постепенную миграцию: обернуть сторонний CSS в слой, и ваш существующий CSS без слоёв будет иметь приоритет без изменений. Это также означает: utility-first фреймворки вроде Tailwind следует импортировать внутри слоя, если нужна возможность переопределения без `!important`.

---

**"Можно ли анимировать CSS-переменные?"**

Только если они зарегистрированы через `@property` с типизированным `syntax`. Незарегистрированные кастомные свойства трактуются как строки — у браузера нет алгоритма интерполяции для них. С `@property` и `syntax: '<color>'` можно делать transition цветового значения кастомного свойства. Без него `transition: --color 0.3s` тихо игнорируется.

**"Что происходит при `var(--color)`, когда `--color` содержит недопустимое для контекста значение?"**

Поведение invalid-at-computed-value-time: свойство использует своё **унаследованное значение** (если свойство наследуется) или **начальное значение** (если нет) — НЕ браузерное умолчание. Для `color` начальное значение технически `CanvasText` (браузерный цвет текста по умолчанию). Это тонкое различие важно для отладки: вы не увидите ожидаемого браузерного запасного значения.
