# Блочная модель и основы раскладки

## Блочная модель — что на самом деле вычисляет браузер

Каждый элемент в документе представлен прямоугольным боксом. Браузер вычисляет четыре вложенных прямоугольника для каждого бокса:

```
┌──────────────────────────────────────────┐
│                  margin                  │
│   ┌──────────────────────────────────┐   │
│   │             border               │   │
│   │   ┌──────────────────────────┐   │   │
│   │   │          padding         │   │   │
│   │   │   ┌──────────────────┐   │   │   │
│   │   │   │     content      │   │   │   │
│   │   │   │   width × height │   │   │   │
│   │   │   └──────────────────┘   │   │   │
│   │   └──────────────────────────┘   │   │
│   └──────────────────────────────────┘   │
└──────────────────────────────────────────┘
```

Эта структура фиксирована. Вопрос лишь в том, как интерпретируются `width` и `height` относительно неё.

## `content-box` vs `border-box` — реальное отличие

### `content-box` (значение по умолчанию, и контринтуитивное)

`width` и `height` определяют **только область контента**. Padding и border добавляются сверх этого:

```css
.box {
  box-sizing: content-box; /* по умолчанию */
  width: 200px;
  padding: 20px;
  border: 2px solid black;
}
/* Итоговая ширина на экране: 200 + 20 + 20 + 2 + 2 = 244px */
```

Это означает: `width: 200px` НЕ значит, что элемент будет 200px шириной на экране. 200px расходуются контентом, а элемент выходит за эти рамки. При подгонке элемента под контейнер нужно вычитать padding и border из желаемой ширины — арифметика, которая легко ошибается и ломается при изменении любого из этих значений.

### `border-box` (разумный стандарт)

`width` и `height` определяют **border box** — итоговый размер, включая padding и border. Область контента сжимается, чтобы их вместить:

```css
.box {
  box-sizing: border-box;
  width: 200px;
  padding: 20px;
  border: 2px solid black;
}
/* Итоговая ширина на экране: 200px (ровно) */
/* Область контента: 200 - 20 - 20 - 2 - 2 = 156px */
```

`width: 200px` теперь означает, что элемент ровно 200px шириной на экране. Изменение padding или border не влияет на внешние размеры.

### Почему `border-box` применяется как универсальный сброс стилей

Современный CSS-сброс применяет `border-box` ко всем элементам:

```css
*,
*::before,
*::after {
  box-sizing: border-box;
}
```

Включение `::before` и `::after` важно: псевдоэлементы не покрываются `*` и по умолчанию используют `content-box`.

Причина повсеместного распространения этого сброса: при построении колоночных раскладок (`width: 50%` для двух равных колонок) `content-box` делает невозможным добавление любого padding без поломки раскладки. С `border-box` конструкция `width: 50%; padding: 16px` работает ожидаемо.

**Единственный случай, когда `content-box` ещё полезен:** когда нужно, чтобы размеры элемента росли вместе с контентом при фиксированном padding. На практике встречается редко.

## Коллапс отступов — настоящий алгоритм

Коллапс margin — одно из наиболее неправильно понимаемых поведений CSS. Большинство объяснений сводятся к "смежные отступы схлопываются". Это неполно. Полные правила:

### Правило 1: Смежные сиблинги

Когда два блочных элемента расположены вертикально, нижний margin первого и верхний margin второго схлопываются в единый отступ. Результат — **максимум** из двух значений, а не их сумма.

```html
<p style="margin-bottom: 20px">Первый абзац</p>
<p style="margin-top: 30px">Второй абзац</p>
<!-- Расстояние между ними: 30px (max из 20 и 30), а НЕ 50px -->
```

```css
/* Отрицательные отступы схлопываются по-другому:
   max положительных + min отрицательных */
/* margin-bottom: 20px, margin-top: -10px → результат: 10px */
/* margin-bottom: -20px, margin-top: -10px → результат: -20px */
```

### Правило 2: Родитель и первый/последний потомок

Если родительский элемент не имеет border, padding, строчного контента или контекста блочного форматирования, разделяющего его от верхнего margin первого потомка — они схлопываются:

```html
<div class="parent">
  <!-- Нет padding-top, нет border-top, нет строчного контента перед потомком -->
  <p style="margin-top: 30px">Абзац-потомок</p>
</div>

<!-- margin-top родителя и margin-top потомка схлопываются.
     30px "вытекает" из родителя и становится его эффективным верхним отступом.
     Собственный margin-top родителя (если есть) сравнивается, и побеждает максимум. -->
```

Это явление называют "утечкой margin". Добавление `padding-top: 1px` к родителю прерывает коллапс — любое разделение (border или padding) предотвращает его.

**Что предотвращает коллапс между родителем и потомком:**
- У родителя есть `padding-top` или `border-top` (любое ненулевое значение)
- У родителя `overflow` отличается от `visible` (создаёт BFC)
- У родителя `display: flow-root`
- Родитель является flex- или grid-контейнером (потомки — flex/grid-элементы, не в потоке)

### Правило 3: Пустые блоки

Блочный элемент без высоты, padding, border и строчного контента схлопывает свои собственные верхний и нижний margin:

```css
.empty {
  margin-top: 20px;
  margin-bottom: 30px;
  /* Нет высоты, нет контента, нет padding, нет border */
}
/* Эффективный вклад в отступы: 30px (max из 20 и 30) */
```

### Когда коллапс margin НЕ происходит

- Элементы внутри flex-контейнеров (отступы flex-элементов не схлопываются)
- Элементы внутри grid-контейнеров
- Элементы с `float`
- Элементы с `position: absolute` или `position: fixed`
- Отступы корневого элемента
- Между родителем и потомком, когда между ними есть BFC, border или padding

Практическое следствие: если переключить раскладку с `display: block` на `display: flex`, ранее схлопнутые отступы внезапно складываются — элементы, бывшие на расстоянии 20px, теперь стоят на 50px. Типичный источник смещений раскладки при рефакторинге.

### Контекст блочного форматирования (BFC) — почему это важно для коллапса

BFC — это изолированная среда раскладки, где правила нормального потока применяются независимо. Элементы внутри BFC не схлопывают margin с элементами снаружи.

Что создаёт BFC:
- `overflow: hidden`, `overflow: auto`, `overflow: scroll` (любое значение, кроме `visible`)
- `display: flow-root` (явное создание BFC без побочных эффектов)
- `display: flex`, `display: grid`, `display: inline-flex`, `display: inline-grid`
- `float: left` или `float: right`
- `position: absolute` или `position: fixed`
- `contain: layout`, `contain: content`, `contain: paint`

`display: flow-root` — современный способ создать BFC без побочных эффектов. До его появления разработчики использовали `overflow: hidden` как хак — он создаёт BFC, но побочный эффект — обрезка контента.

## Позиционирование — правила содержащего блока

Свойство `position` определяет две вещи: (1) участвует ли элемент в нормальном потоке, и (2) каков его **содержащий блок** (containing block).

Содержащий блок — это опорный прямоугольник для процентных размеров и для смещений `top`/`right`/`bottom`/`left`. Непонимание этого — источник большинства багов "почему мой абсолютный элемент не там, где нужно?".

### `position: static` (по умолчанию)

Элемент участвует в нормальном потоке. `top`, `right`, `bottom`, `left` и `z-index` не имеют эффекта. Содержащий блок — ближайший блочный контейнер-предок (стандартные правила потока).

```css
.box {
  position: static; /* по умолчанию */
  top: 10px; /* игнорируется */
  z-index: 5; /* игнорируется */
}
```

### `position: relative`

Элемент участвует в нормальном потоке (занимает место как обычно). Свойства смещения (`top`, `left` и др.) перемещают **визуальный рендеринг** без влияния на раскладку — исходное место элемента в потоке сохраняется, другие элементы не перестраиваются.

```css
.box {
  position: relative;
  top: 20px;
  left: 10px;
  /* Визуально смещён на 20px вниз и 10px вправо,
     но исходное место в потоке сохраняется */
}
```

**Содержащий блок для потомков:** `position: relative` делает элемент содержащим блоком для любых абсолютно позиционированных потомков.

```html
<div style="position: relative; width: 400px; height: 200px;">
  <!-- Это содержащий блок для потомка ниже -->
  <div style="position: absolute; right: 0; bottom: 0;">
    <!-- Позиционирован в правый нижний угол РОДИТЕЛЯ, не вьюпорта -->
  </div>
</div>
```

### `position: absolute`

Элемент **вырывается из нормального потока** — не занимает места в документе. Позиционируется относительно **ближайшего позиционированного предка** (предка с `position`, отличным от `static`). Если такого предка нет — позиционируется относительно **начального содержащего блока** (вьюпорт в большинстве случаев, точнее — элемент `<html>`).

```html
<div class="container"> <!-- position: static — НЕ является содержащим блоком -->
  <div class="wrapper" style="position: relative;"> <!-- ЯВЛЯЕТСЯ содержащим блоком -->
    <div style="position: absolute; top: 0; right: 0;">
      <!-- Верхний правый угол .wrapper -->
    </div>
  </div>
</div>
```

**Частая ловушка:** содержащий блок определяется тем, что у предка `position` отличен от `static`, а НЕ визуальной вложенностью. Абсолютно позиционированный элемент внутри визуально вложенных div-ов "вырвется" из всех них, пока не найдёт один с `position: relative/absolute/fixed/sticky`.

**Что ещё создаёт содержащий блок для абсолютных потомков** (помимо `position`):
- `transform` (любое значение, кроме `none`)
- `filter` (любое значение, кроме `none`)
- `will-change: transform` или `will-change: filter`
- `contain: layout`, `contain: paint`, `contain: strict`
- `perspective`

Этот список застаёт врасплох даже опытных разработчиков. `transform: translateZ(0)` или `filter: blur(0)` на родителе молча захватят абсолютно позиционированных потомков, заставив их позиционироваться относительно этого родителя вместо дальнего предка.

```css
/* Это захватит абсолютно позиционированных потомков — часто неожиданно */
.sidebar {
  transform: translateX(0); /* оптимизация GPU-слоя */
}

/* Теперь любой position: absolute потомок внутри .sidebar
   позиционируется относительно .sidebar, а не ближайшего position: relative предка */
```

### `position: fixed`

Как `absolute`, вырывается из нормального потока. Но содержащий блок — всегда **вьюпорт**, независимо от `position` любого предка.

**Исключение:** `transform`, `filter`, `will-change`, `backdrop-filter` или `perspective` на предке ломают фиксированное позиционирование. Элемент захватывается трансформированным предком вместо вьюпорта. Одно из самых неожиданных поведений CSS.

```css
/* Это ломает position: fixed у любого потомка */
.parent {
  transform: none; /* Даже это! */
  /* Спецификация гласит: если transform имеет любое значение, кроме начального,
     создаётся содержащий блок для fixed-потомков. transform: none — это начальное значение,
     но у некоторых браузеров были баги с этим. */
}

/* Безопасный подход: сделать fixed-элементы прямыми потомками <body>,
   чтобы избежать случайного захвата */
```

### `position: sticky`

Гибрид: ведёт себя как `relative` внутри своего содержащего блока (**ближайшего прокручивающегося предка** с overflow, отличным от `visible`/`clip`), и становится похожим на `fixed`, когда достигается порог. Возвращается к `relative`, когда контейнер уходит из видимости при прокрутке.

```css
.sticky-header {
  position: sticky;
  top: 0; /* Расстояние от края прокручивающегося контейнера, при котором начинается "прилипание" */
}
```

**Почему sticky часто "не работает":**

1. **Содержащий блок не имеет высоты.** Sticky перемещается только в пределах своего содержащего блока. Если родитель не выше самого sticky-элемента, нет места для "прилипания".

```html
<!-- Сломано: у родителя нет высоты сверх sticky-элемента -->
<div class="section"> <!-- высота = высота h2 -->
  <h2 style="position: sticky; top: 0;">Заголовок</h2>
  <!-- нет другого контента в этой секции -->
</div>

<!-- Работает: родитель выше sticky-элемента -->
<div class="section"> <!-- высота = h2 + много абзацев -->
  <h2 style="position: sticky; top: 0;">Заголовок</h2>
  <p>...</p>
  <p>...</p>
  <!-- sticky "прилипает" пока прокручиваются эти элементы -->
</div>
```

2. **У предка есть `overflow: hidden` или `overflow: auto`.** Прокручивающийся предок для sticky — не вьюпорт, а ближайший предок с `overflow`, отличным от `visible`. Если этот предок обрезает контент, sticky-элемент заперт внутри него и прокрутка вьюпорта не вызывает "прилипание".

3. **Не задано `top`/`bottom`/`left`/`right`.** `position: sticky` без порогового значения ничего не делает — браузер не знает, когда начинать "прилипание".

## Inline vs block — контекст display, управляющий коллапсом

Блочные элементы (`<div>`, `<p>`, `<h1>` и др.) генерируют блочные боксы и участвуют в контексте блочного форматирования — именно здесь происходит коллапс margin.

Строчные элементы (`<span>`, `<a>`, `<strong>`) генерируют строчные боксы. У них есть горизонтальный padding/margin, но вертикальные margin не схлопываются и не отталкивают блочных сиблингов.

`display: inline-block` — гибрид: элемент обрабатывается как строчный родителем (находится в строке текста), но внутренне генерирует контекст блочного форматирования. Вертикальные margin и padding работают ожидаемо, создаётся BFC (нет коллапса margin с потомками).

```css
/* inline-block: занимает ровно столько ширины, сколько нужно,
   находится в потоке текста, но учитывает вертикальные размеры */
.badge {
  display: inline-block;
  padding: 4px 8px;
  vertical-align: middle;
}
```

## Типичные ошибки на интервью

**"Какова итоговая ширина этого элемента на экране?"**

```css
.box {
  width: 300px;
  padding: 0 20px;
  border: 3px solid;
  box-sizing: content-box; /* по умолчанию */
}
```

Ответ: `300 + 20 + 20 + 3 + 3 = 346px`. С `border-box`: `300px`. Вопрос проверяет знание значения по умолчанию для `box-sizing`.

---

**"Почему мой margin вытекает из родительского контейнера?"**

Классический коллапс margin между родителем и первым потомком. У margin-top потомка ничто не мешает на границе родителя (нет border-top, нет padding-top, нет строчного контента, нет BFC). Варианты исправления: добавить любой `padding-top` родителю, добавить `border-top: 1px solid transparent`, или сделать родителя `display: flow-root` / `overflow: hidden`.

---

**"Мой `position: fixed` элемент прокручивается вместе со страницей — что произошло?"**

На одном из предков применено CSS-свойство `transform`, `filter` или `will-change: transform`. Это создаёт новый содержащий блок для fixed-потомков, заставляя их позиционироваться относительно этого предка вместо вьюпорта. Переместите fixed-элемент в прямые потомки `<body>` или уберите transform с предка.

---

**"Почему переключение с `display: block` на `display: flex` на контейнере меняет расстояние между элементами?"**

Flex-элементы не находятся в контексте блочного форматирования — коллапс margin происходит только в BFC. Коллапс смежных margin блочных потомков перестаёт работать. Ранее схлопнутые отступы (например, 20px + 30px = 30px) теперь складываются (50px). То же происходит с grid-контейнерами.

---

**"В чём разница между `display: flow-root` и `overflow: hidden` для clearfix / предотвращения коллапса margin?"**

Оба создают контекст блочного форматирования (BFC). Разница: `overflow: hidden` обрезает контент, выходящий за пределы элемента — побочный эффект, который иногда нежелателен (выпадающие меню, тултипы, тени). `display: flow-root` создаёт BFC без побочных эффектов — он был введён именно потому, что `overflow: hidden` злоупотреблялся как BFC-хак.

---

**"Абсолютно позиционированный элемент не там, где нужно — как отладить?"**

Найти содержащий блок: пройти по DOM вверх и найти ближайшего предка с `position`, отличным от `static`. Также проверить `transform`, `filter` или `will-change` на предках — они тоже захватывают `position: absolute` потомков. В DevTools выбрать позиционированный элемент и посмотреть значения смещений в панели "Computed", затем определить, относительно чего происходит позиционирование.
