# Redis Interview Questions (Middle → Senior)

---

# 1. Что такое Redis?

Redis — высокопроизводительное in-memory хранилище данных.

Может использоваться как:

```txt
Cache
Database
Queue
Message Broker
Pub/Sub
```

---

# 2. Почему Redis такой быстрый?

Основные причины:

```txt
данные в памяти

однопоточная архитектура

нет SQL Parser

нет Query Planner

простые структуры данных
```

---

# 3. Что означает In-Memory Database?

Данные хранятся в оперативной памяти (RAM), а не читаются с диска при каждом запросе.

---

# 4. Redis — это база данных или кеш?

Оба варианта возможны.

На практике чаще используется как:

```txt
Cache Layer
```

поверх PostgreSQL.

---

# 5. Почему Redis использует один поток?

Чтобы избежать:

```txt
locks
mutexes
race conditions
```

---

Операции очень короткие и выполняются быстро.

---

# 6. Значит ли это, что Redis не использует несколько CPU?

Нет.

Современные версии используют дополнительные потоки для:

```txt
network IO

background tasks

persistence
```

---

Но команды выполняются одним главным потоком.

---

# 7. Какие структуры данных поддерживает Redis?

```txt
String
Hash
List
Set
Sorted Set
Stream
Bitmap
HyperLogLog
```

---

# 8. Что такое String?

Базовая структура Redis.

---

Пример:

```bash
SET user:1 John
GET user:1
```

---

# 9. Что такое Hash?

Набор полей внутри одного ключа.

---

Пример:

```bash
HSET user:1 name John
```

---

Часто используется для пользователей.

---

# 10. Что такое List?

Упорядоченная коллекция элементов.

---

Подходит для:

```txt
Queues
Jobs
```

---

# 11. Что такое Set?

Множество уникальных значений.

---

Подходит для:

```txt
Tags
Permissions
Followers
```

---

# 12. Что такое Sorted Set?

Set с дополнительным score.

---

Используется для:

```txt
Leaderboards
Ratings
Rankings
```

---

# 13. Как реализовать счетчик просмотров?

Через:

```bash
INCR page_views
```

---

# 14. Что такое TTL?

Time To Live.

---

Время жизни ключа.

---

Пример:

```bash
SET user:1 value EX 60
```

---

# 15. Что происходит после истечения TTL?

Ключ автоматически удаляется.

---

# 16. Что такое Cache Hit?

Данные найдены в Redis.

---

К БД обращаться не нужно.

---

# 17. Что такое Cache Miss?

Данных нет в Redis.

---

Нужно идти в БД.

---

# 18. Что такое Cache Aside Pattern?

Самый популярный паттерн кеширования.

---

Flow:

```txt
Redis
 ↓ miss
Database
 ↓
Redis
```

---

# 19. Что такое Write Through?

Обновление Redis и БД одновременно.

---

# 20. Что такое Write Behind?

Сначала запись в Redis.

---

Позже асинхронная запись в БД.

---

# 21. Что такое Cache Invalidation?

Удаление или обновление устаревших данных в кеше.

---

# 22. Что такое Cache Stampede?

Много запросов одновременно идут в БД после истечения TTL.

---

# 23. Как бороться с Cache Stampede?

```txt
Mutex Lock

Random TTL

Warm Cache
```

---

# 24. Что такое Cache Penetration?

Постоянные запросы к данным, которых не существует.

---

# 25. Как бороться с Cache Penetration?

Кешировать:

```txt
NULL
```

---

на короткое время.

---

# 26. Что такое Cache Avalanche?

Массовое истечение TTL большого количества ключей.

---

# 27. Что такое Eviction Policy?

Стратегия удаления данных при нехватке памяти.

---

# 28. Какие Eviction Policies знаете?

```txt
LRU
LFU
TTL
No Eviction
```

---

# 29. Что такое LRU?

Least Recently Used.

---

Удаляются давно не использовавшиеся ключи.

---

# 30. Что такое LFU?

Least Frequently Used.

---

Удаляются редко используемые ключи.

---

# 31. Что такое Pub/Sub?

Механизм обмена сообщениями через каналы.

---

# 32. Что такое Publisher?

Отправитель сообщений.

---

# 33. Что такое Subscriber?

Получатель сообщений.

---

# 34. Главный недостаток Pub/Sub?

Сообщения не сохраняются.

---

# 35. Что будет если Subscriber отключен?

Сообщение потеряется.

---

# 36. Что такое Redis Streams?

Надежная система обмена сообщениями Redis.

---

Сообщения сохраняются.

---

# 37. Чем Streams отличаются от Pub/Sub?

Pub/Sub:

```txt
без истории
```

---

Streams:

```txt
с историей
```

---

# 38. Что такое Consumer Group?

Группа consumers для распределения нагрузки.

---

# 39. Что такое ACK в Streams?

Подтверждение успешной обработки сообщения.

---

# 40. Что произойдет если Consumer упал?

Сообщение останется pending.

---

Может быть обработано повторно.

---

# 41. Что такое Distributed Lock?

Механизм синхронизации между несколькими инстансами приложения.

---

# 42. Как Redis реализует Lock?

Через:

```bash
SET key value NX EX
```

---

# 43. Что означает NX?

Создать ключ только если он отсутствует.

---

# 44. Зачем нужен EX?

Чтобы lock автоматически освободился.

---

# 45. Почему нельзя просто делать DEL lock?

Потому что lock мог уже перейти другому процессу.

---

# 46. Что такое Redlock?

Алгоритм распределенных блокировок через несколько Redis инстансов.

---

# 47. Что такое Persistence?

Сохранение данных на диск.

---

# 48. Какие механизмы Persistence существуют?

```txt
RDB
AOF
```

---

# 49. Что такое RDB?

Snapshot памяти Redis.

---

Создается файл:

```txt
dump.rdb
```

---

# 50. Недостаток RDB?

Можно потерять последние изменения между snapshot.

---

# 51. Что такое AOF?

Append Only File.

---

Лог всех операций записи.

---

# 52. Преимущество AOF?

Меньше риск потери данных.

---

# 53. Что такое fsync?

Политика записи AOF на диск.

---

# 54. Что такое Redis Replication?

Репликация данных с Master на Replica.

---

# 55. Что такое Redis Sentinel?

Система мониторинга и автоматического failover.

---

# 56. Что такое Redis Cluster?

Горизонтально масштабируемый кластер Redis.

---

# 57. Что такое Sharding?

Распределение данных между несколькими узлами.

---

# 58. Можно ли использовать Redis как основную БД?

Можно.

---

Но редко рекомендуется для бизнес-систем.

---

Для:

```txt
Cache

Sessions

Rate Limiting

Counters

Leaderboards
```

---

# 60. Когда PostgreSQL лучше Redis?

Для:

```txt
Transactions

Relations

JOIN

Complex Queries

Long-term Storage
```

---

# 61. Что такое Rate Limiting?

Ограничение количества запросов.

---

Часто реализуется через:

```txt
Redis + INCR
```

---

# 62. Как реализовать "100 запросов в минуту"?

Через:

```txt
INCR
+
TTL
```

---

# 63. Как реализовать хранение сессий?

```txt
session:{id}
```

---

с TTL.

---

# 64. Как реализовать leaderboard?

Через:

```txt
Sorted Set
```

---

# 65. Как реализовать очередь задач?

Через:

```txt
List

или

Streams
```

---

# 66. Почему Redis часто используют вместе с PostgreSQL?

Потому что они решают разные задачи:

Redis:

```txt
скорость
```

---

PostgreSQL:

```txt
надежность
```

---

# 67. Самый популярный Senior вопрос

Почему Redis быстрее PostgreSQL?

Ответ:

```txt
Данные находятся в памяти.

Нет SQL Parser.

Нет Query Planner.

Нет JOIN.

Нет MVCC.

Нет сложных транзакций.

Большинство операций имеют сложность O(1).
```

---

# 68. Самый сильный Senior ответ

Когда бы вы НЕ использовали Redis?

Ответ:

Я бы не использовал Redis как основное хранилище для финансовых операций, сложных бизнес-сущностей и систем, где критична консистентность данных. В таких случаях PostgreSQL подходит значительно лучше. Redis я бы использовал как дополнительный слой для кеширования, очередей, rate limiting и ускорения чтения.
