# Redis Caching Patterns

## Самый популярный кейс Redis

В большинстве проектов Redis используют как:

```txt
Cache Layer
```

---

# Зачем нужен Cache

Без кеша:

```txt
Request
 ↓
PostgreSQL
 ↓
Response
```

---

Каждый запрос:

```txt
идет в БД
```

---

# С кешем

```txt
Request
 ↓
Redis
 ↓
Hit
 ↓
Response
```

---

База данных не участвует.

---

# Cache Aside Pattern

Самый популярный паттерн.

---

Также называется:

```txt
Lazy Loading
```

---

Flow:

```txt
Request
 ↓
Redis
 ↓
Miss
 ↓
Database
 ↓
Redis
 ↓
Response
```

---

# Псевдокод

```ts
let user =
 await redis.get(key);

if (!user) {

 user =
  await db.user.find();

 await redis.set(
  key,
  user,
  ttl
 );
}

return user;
```

---

# Преимущества

```txt
простота

экономия памяти

данные кешируются только если нужны
```

---

# Недостаток

Первый запрос:

```txt
всегда медленный
```

---

# Write Through

Другой паттерн.

---

При записи:

```txt
Database
+
Redis
```

---

Обновляются одновременно.

---

Flow:

```txt
Update User
 ↓
Database
 ↓
Redis
```

---

# Плюсы

Кеш всегда актуален.

---

# Минусы

Запись становится медленнее.

---

# Write Behind

Редкий вопрос.

---

Записываем:

```txt
Redis
```

---

А БД обновляется позже.

---

```txt
Redis
 ↓
Async Flush
 ↓
Database
```

---

# Плюсы

Очень быстрые записи.

---

# Минусы

Можно потерять данные.

---

# Cache Invalidation

Один из двух самых сложных вопросов.

---

Есть шутка:

```txt
There are only two hard things:

Cache Invalidation

Naming Things
```

---

Проблема.

---

Пользователь обновил профиль.

---

Redis хранит старую версию.

---

# Решение №1

Удаляем кеш.

---

```ts
await redis.del(
 `user:${id}`
);
```

---

Следующий запрос:

```txt
Cache Miss
```

---

Получит новые данные из БД.

---

# Решение №2

Обновляем кеш сразу.

---

```ts
await redis.set(...)
```

---

После обновления БД.

---

# TTL Strategy

Очень популярный подход.

---

```txt
Cache живет

5 минут

10 минут

1 час
```

---

После чего удаляется.

---

# Cache Stampede

Очень популярный Senior вопрос.

---

Проблема.

---

TTL истек.

---

Одновременно:

```txt
1000 запросов
```

---

Все идут в БД.

---

Получаем:

```txt
DB Overload
```

---

# Решение №1

Mutex Lock.

---

```txt
Первый запрос
обновляет кеш

остальные ждут
```

---

# Решение №2

Random TTL.

---

Вместо:

```txt
60 секунд
```

---

Используем:

```txt
60 ± random
```

---

Ключи истекают не одновременно.

---

# Cache Penetration

Еще популярный вопрос.

---

Проблема.

---

Пользователь постоянно запрашивает:

```txt
user:99999999
```

---

Которого нет.

---

Каждый раз:

```txt
Redis Miss
 ↓
Database
```

---

# Решение

Кешировать:

```txt
NULL
```

---

На короткое время.

---

# Cache Avalanche

Редкий вопрос.

---

Много ключей истекли одновременно.

---

Вся нагрузка идет в БД.

---

Похоже на:

```txt
Cache Stampede
```

---

Но затрагивает много ключей.

---

# Частый вопрос

Какой паттерн кеширования используется чаще всего?

Ответ:

```txt
Cache Aside
```

---

# Частый вопрос

Что такое Cache Stampede?

Ответ:

Ситуация, когда после истечения TTL большое количество запросов одновременно начинает читать данные из БД.

---

# Частый вопрос

Как инвалидировать кеш?

Ответ:

Через удаление ключа после обновления данных либо через обновление кеша сразу после записи.

---

# Interview Answer

Наиболее популярный паттерн кеширования — Cache Aside. Приложение сначала пытается получить данные из Redis, а при отсутствии обращается к базе данных и сохраняет результат в кеш. Основные проблемы кеширования включают Cache Stampede, Cache Penetration и Cache Invalidation.
