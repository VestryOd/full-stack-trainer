# Distributed Locks

## Зачем вообще нужны блокировки

Представим ситуацию.

---

У нас есть:

```txt
Balance = 100$
```

---

Два запроса одновременно:

```txt
Withdraw 50$

Withdraw 70$
```

---

Оба читают:

```txt
100$
```

---

Получаем:

```txt
100 - 50 = 50

100 - 70 = 30
```

---

В итоге:

```txt
Balance = 30
```

---

Хотя должно было произойти:

```txt
одна операция успешна

вторая отклонена
```

---

# Distributed Lock

Нужен общий механизм.

---

Схема:

```txt
Service A
 ↓
Redis Lock
```

---

Пока lock занят:

```txt
Service B
```

---

не может выполнять операцию.

---

# Redis Lock

Самая популярная реализация.

---

Использует:

```bash
SET key value NX EX 30
```

---

```txt

```

---

```txt
TTL
```

---

```bash

```

---

Если ключа нет:

```txt

```

---

Если есть:

```txt
lock занят
```

```bash

```

---

Почему?

Потому что lock мог уже истечь.

# Частый вопрос

Почему нельзя использовать обычный mutex?

Ответ:

Mutex работает только внутри одного процесса, а distributed lock должен синхронизировать несколько сервисов или инстансов приложения.

---

# Частый вопрос

Зачем нужен TTL?

Ответ:

Чтобы избежать вечной блокировки, если процесс упал до освобождения lock.

---

# Частый вопрос

Что такое Redlock?

Ответ:

Алгоритм распределенной блокировки Redis, использующий несколько независимых Redis инстансов для повышения надежности.

---

# Interview Answer

Distributed Lock позволяет синхронизировать выполнение операций между несколькими инстансами приложения. В Redis обычно используется команда SET NX EX для атомарного получения блокировки. Для повышения надежности применяется алгоритм Redlock, работающий через несколько Redis узлов.
