# N+1 Problem и DataLoader

## Самый частый senior-вопрос по GraphQL — и почему он специфичен именно для GraphQL

N+1 — это не "GraphQL баг", это прямое следствие архитектуры резолверов, описанной в [Schema, Types, and Resolvers]: каждое поле может иметь СВОЙ резолвер, и резолвер для поля массива вызывается ОТДЕЛЬНО для КАЖДОГО элемента родительского массива.

```graphql
type User {
  id: ID!
  name: String!
  posts: [Post!]!
}

type Query {
  users: [User!]!
}
```

```ts
const resolvers = {
  Query: {
    users: (_p, _a, ctx) => ctx.db.user.findMany(),
  },
  User: {
    posts: (user, _a, ctx) => ctx.db.post.findMany({ where: { userId: user.id } }),
  },
};
```

```graphql
query {
  users {
    id
    name
    posts { title }
  }
}
```

```txt
Шаг 1: Query.users → 1 SQL-запрос, возвращает 100 пользователей
Шаг 2: User.posts вызывается ОТДЕЛЬНО для КАЖДОГО из 100
       пользователей → 100 SQL-запросов

Итого: 1 + 100 = 101 запрос вместо 2.

При 1000 пользователей — 1001 запрос. БД начинает страдать,
причём деградация ЛИНЕЙНАЯ от размера ответа — то есть проблема
незаметна на dev-данных (10 пользователей → 11 запросов
выглядят терпимо) и проявляется только в production-масштабе.
```

### Почему "просто используйте include/JOIN" не всегда решает проблему

```ts
// Наивное решение — работает для ОДНОГО уровня вложенности
prisma.user.findMany({ include: { posts: true } });
```

```txt
Это решает N+1 ДЛЯ ЭТОГО КОНКРЕТНОГО запроса — но GraphQL-сервер
не знает заранее, какой запрос придёт. include нужно было бы
прописывать в РЕЗОЛВЕРЕ Query.users — но резолвер Query.users
не знает, запросит ли клиент posts вообще (а если запросит — то
ещё и author каждого поста, и comments каждого поста...).

Прописывать include для ВСЕХ возможных вложенностей заранее —
это отказ от главного преимущества GraphQL ("клиент решает, что
получить", см. [GraphQL Fundamentals]) и оверфетчинг на уровне
БД для запросов, которым posts не нужны.

JOIN решает проблему ТОЛЬКО когда граф запроса известен заранее
(REST-эндпоинт с фиксированной формой ответа) — в GraphQL форма
запроса заранее не известна.
```

## DataLoader: Batching + Caching, созданный Facebook для именно этой проблемы

```ts
import DataLoader from 'dataloader';

const createUserLoader = (db: Db) =>
  new DataLoader<string, User | null>(async (ids) => {
    const users = await db.user.findMany({ where: { id: { in: ids as string[] } } });
    const byId = new Map(users.map((u) => [u.id, u]));
    // КРИТИЧНО: вернуть результаты в ТОМ ЖЕ ПОРЯДКЕ, что и ids,
    // и null/Error для ключей, которых нет — длина массива
    // ДОЛЖНА совпадать с длиной ids
    return ids.map((id) => byId.get(id) ?? null);
  });
```

```ts
// Resolver — вызывает .load(), а не findMany() напрямую
const resolvers = {
  Post: {
    author: (post, _args, ctx) => ctx.userLoader.load(post.authorId),
  },
};
```

### Почему обязателен порядок результатов, совпадающий с ids

```txt
DataLoader сопоставляет i-й элемент РЕЗУЛЬТАТА с i-м элементом
ВХОДНЫХ ключей ПОЗИЦИОННО, а не по значению. Если batch-функция
вернёт users в порядке, в котором их вернула БД (а БД НЕ
гарантирует порядок WHERE id IN (...) без явного ORDER BY),
DataLoader присвоит .load("3") результат, предназначенный для
.load("1").

Решение — всегда строить Map<key, value> из результата БД и
маппить по ВХОДНЫМ ids:

  return ids.map(id => byId.get(id) ?? null)

Если для ключа нет данных — вернуть null (для nullable полей)
или экземпляр Error (DataLoader пробросит его как ошибку именно
для ТЕХ .load(), что запросили этот ключ — остальные ключи в
batch'е не пострадают).
```

### Как работает batching на уровне event loop

```txt
.load(key) НЕ выполняет запрос немедленно — он:
  1. Добавляет key в очередь текущего "тика"
  2. Возвращает Promise, который резолвится позже
  3. На следующем тике event loop (process.nextTick /
     setImmediate, в зависимости от реализации) DataLoader
     собирает ВСЕ накопленные ключи и вызывает batch-функцию
     ОДИН раз со всеми ключами сразу

Поэтому 100 синхронных вызовов .load() из 100 параллельных
вызовов User.posts (см. "порядок выполнения вложенных
резолверов" в [Schema, Types, and Resolvers]) превращаются в
ОДИН вызов batch-функции с массивом из 100 ключей.
```

## Caching — вторая суперсила, и где она ломается

```txt
Внутри ОДНОГО DataLoader instance — повторный .load("1")
вернёт закэшированный Promise без нового запроса к batch-
функции, даже если между вызовами прошло несколько резолверов.

Senior-нюанс: этот кэш — НЕ кэш в обычном смысле (TTL,
инвалидация). Это request-scoped memoization. Если в рамках
ОДНОГО GraphQL-запроса есть Mutation, которая ИЗМЕНЯЕТ
пользователя, а ПОСЛЕ неё резолвер снова вызывает
userLoader.load(тот же id) — DataLoader вернёт СТАРОЕ
закэшированное значение, не отражающее изменение.

Решение: после мутации явно вызвать
ctx.userLoader.clear(id) или
ctx.userLoader.prime(id, обновлённыеДанные)
для конкретного ключа, который мутация изменила.
```

## DataLoader создаётся НА КАЖДЫЙ запрос — почему это не опционально

```ts
// NestJS / Apollo Server — НОВЫЙ instance в context каждого запроса
const server = new ApolloServer({
  schema,
  context: async () => ({
    userLoader: createUserLoader(db),
  }),
});
```

```txt
Если вынести createUserLoader() за пределы context (как
singleton, созданный один раз при старте сервера):

  1. Кэш не сбрасывается между запросами разных
     пользователей — Пользователь A может получить из кэша
     данные, которые видел при обработке запроса
     Пользователя B (data leak в многопользовательских
     системах с разными правами доступа)

  2. Кэш растёт неограниченно (memory leak) — нет точки,
     когда он очищается

Это тот же паттерн "request-scoped vs singleton", что и для
context в целом (см. [Schema, Types, and Resolvers]) —
DataLoader ДОЛЖЕН жить ровно один GraphQL-запрос.
```

## DataLoader не заменяет JOIN — когда что использовать

```txt
DataLoader = batching layer НА УРОВНЕ ПРИЛОЖЕНИЯ. Он превращает
N запросов "WHERE id = X" в 1 запрос "WHERE id IN (...)" —
но это всё ещё N+1 → 2 запроса, а не 1.

JOIN = SQL-операция, которая делает 1 запрос для ВСЕХ уровней
сразу — но требует, чтобы форма JOIN'а была известна ДО
выполнения запроса (то есть подходит для REST с фиксированной
формой ответа, а не для произвольных GraphQL-запросов).

Практика:
  - DataLoader — универсальное решение для GraphQL-резолверов,
    работает для ЛЮБОЙ формы запроса клиента
  - Если профилирование показывает, что КОНКРЕТНЫЙ частый
    запрос (например, User.posts почти ВСЕГДА запрашивается
    вместе с users) даёт заметный выигрыш от JOIN — можно
    добавить точечную оптимизацию в резолвер Query.users
    (eager-loading через include), и DataLoader для posts
    в этом случае просто не понадобится (кэш не тронут, т.к.
    .load() не вызывался)
```

## Как диагностировать N+1 в реальном проекте

```txt
1. Включить логирование SQL-запросов (Prisma: log: ['query'])
2. Выполнить GraphQL-запрос с массивом из N элементов на
   верхнем уровне
3. Если в логе вместо 1-2 запросов — N+1 (или больше, при
   нескольких уровнях вложенности — N*M+1) — это N+1

Инструменты: Apollo Server plugins для подсчёта резолверов,
Prisma query logging, APM-трейсинг (см. [Performance and
Security] про мониторинг GraphQL в целом).
```

## Вложенный N+1 — когда одного DataLoader недостаточно

```graphql
query {
  users {
    posts {
      comments {
        author { name }
      }
    }
  }
}
```

```txt
Каждый уровень вложенности — ОТДЕЛЬНАЯ потенциальная точка N+1,
и для каждой нужен СВОЙ DataLoader (postsByUserLoader,
commentsByPostLoader, userByIdLoader для author). Простого
"один DataLoader на всё" не бывает — DataLoader batched по
КОНКРЕТНОЙ ФОРМЕ batch-функции (один тип ключа → один тип
значения).

При глубокой вложенности суммарное количество SQL-запросов
становится "1 на уровень" (а не "1 на узел"), что и есть цель
DataLoader — но НЕ "1 запрос на весь граф".
```

## Типичные ошибки на интервью

- **"N+1 — это баг ORM/Prisma"** — не понимать, что причина в архитектуре резолверов GraphQL (каждое поле массива → отдельный вызов резолвера для каждого элемента), а не в конкретной ORM.

- **"include/JOIN решает N+1 для GraphQL так же, как для REST"** — не объяснять, что в GraphQL форма запроса заранее не известна, поэтому include нельзя статически прописать в резолвере без потери гибкости.

- **Возвращать результаты batch-функции не в порядке входных ids** — не знать, что DataLoader сопоставляет результаты с ключами ПОЗИЦИОННО, и неправильный порядок приводит к тому, что данные одного пользователя "приезжают" другому.

- **"DataLoader кэш — это обычный TTL-кэш"** — не понимать, что это request-scoped memoization, которое может вернуть устаревшие данные после мутации в рамках того же запроса, если не вызвать `.clear()`/`.prime()`.

- **Создавать DataLoader как singleton при старте сервера** — не видеть риск утечки кэша между пользователями и неограниченного роста памяти.

- **"DataLoader полностью заменяет JOIN"** — не понимать, что DataLoader даёт O(уровней вложенности) запросов, а не O(1), и для часто запрашиваемых связей точечный JOIN может быть быстрее.
