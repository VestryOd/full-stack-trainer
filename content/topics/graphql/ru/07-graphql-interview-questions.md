# GraphQL: Вопросы для интервью

Вопросы сгруппированы тематически. Внутри каждой группы — полный senior-ответ + типичные follow-up вопросы с краткими ответами. Цель: воспроизвести цепочку "базовый вопрос → уточнения → нюансы", которая реально происходит на интервью.

## Группа 1: Основы GraphQL — спецификация, схема, выполнение

### Что такое GraphQL и чем он отличается от REST на уровне спецификации?

GraphQL — это СПЕЦИФИКАЦИЯ (язык запросов + система типов + семантика выполнения), а не конкретная технология или транспортный протокол. Реализации: graphql-js (эталонная), Apollo Server, Mercurius, GraphQL Yoga — все реализуют одну спецификацию, но с разными расширениями. В отличие от REST (который вообще не является спецификацией — это архитектурный стиль), GraphQL строго определяет, как выглядит валидный запрос, как сервер должен его выполнить, и что должно содержаться в ответе.

Ключевая разница в контроле формы ответа: в REST СЕРВЕР определяет, что возвращает каждый эндпоинт; в GraphQL КЛИЕНТ определяет через selection set, какие именно поля нужны. Это решает overfetching (получаем только нужные поля) и underfetching (один запрос = несколько связанных сущностей).

```txt
Типичные follow-up:

Q: "GraphQL работает только через HTTP?"
A: Нет, спецификация транспорт-агностична. На практике — почти
   всегда HTTP POST /graphql, но GraphQL over WebSocket
   (подписки) и GraphQL over gRPC — также валидные реализации.

Q: "Что такое SDL?"
A: Schema Definition Language — формальный язык описания схемы
   GraphQL (type User { id: ID! name: String! }). Это часть
   спецификации, а не просто документационный формат — сервер
   обязан соответствовать SDL во время выполнения.

Q: "Почему GraphQL называют strongly typed, если runtime-ошибки
   всё равно возможны?"
A: "Strongly typed" означает, что НЕСООТВЕТСТВИЕ ЗАПРОСА И СХЕМЫ
   отклоняется на фазе Validate — до вызова резолверов, до
   обращения к БД. Но ошибки ВНУТРИ резолвера (БД недоступна,
   resolver вернул null для String!) — это runtime-ошибки фазы
   Execute, они происходят всегда.
```

### Опишите три фазы выполнения GraphQL-запроса.

1. **Parse** — текст запроса разбирается в AST (Abstract Syntax Tree). Синтаксическая ошибка (незакрытая скобка) отклоняется ЗДЕСЬ, до любого обращения к данным.

2. **Validate** — AST проверяется против СХЕМЫ: существуют ли запрошенные типы/поля/аргументы, совпадают ли типы аргументов. Ошибка валидации (запрошенное поле не существует в схеме) → резолверы НЕ вызываются ВООБЩЕ. Это позволяет отвергать невалидные запросы без нагрузки на БД.

3. **Execute** — для каждого поля в запросе вызывается соответствующий резолвер, результаты собираются в дерево, соответствующее форме запроса.

```txt
Типичные follow-up:

Q: "На каком этапе работает Depth Limiting / Query Complexity?"
A: На этапе Validate — они реализуются как дополнительные
   validation rules (validationRules в Apollo Server). Запрос
   отклоняется до выполнения резолверов.

Q: "На каком этапе GraphQL обнаруживает, что поле типа String!
   вернуло null?"
A: На этапе Execute — это runtime-ошибка внутри резолвера, а
   не нарушение схемы на уровне запроса. Схема — это обязательство
   СЕРВЕРА, а не гарантия, проверяемая на Validate.
```

## Группа 2: Операции — Query, Mutation, Subscription

### В чём разница между Query и Mutation с точки зрения спецификации (не только семантики)?

Семантически: Query — чтение, Mutation — запись. Но спецификация добавляет КРИТИЧЕСКИ ВАЖНОЕ ПОВЕДЕНЧЕСКОЕ отличие:

- **Query**: top-level поля МОГУТ (и по умолчанию — ДОЛЖНЫ, если async) выполняться ПАРАЛЛЕЛЬНО. Порядок полей в ответе соответствует запросу, но порядок ВЫЗОВА резолверов не гарантирован.
- **Mutation**: top-level поля выполняются СТРОГО ПОСЛЕДОВАТЕЛЬНО, в порядке их следования в запросе. Это требование спецификации, существующее для предотвращения race conditions при множественных изменениях данных в одном запросе.

```txt
Типичные follow-up:

Q: "Как ведут себя ВЛОЖЕННЫЕ поля внутри результата Mutation?"
A: Так же, как у Query — могут выполняться параллельно. Правило
   последовательности применяется только к TOP-LEVEL полям
   Mutation, а не ко всему дереву.

Q: "Что такое Payload Pattern и когда его использовать?"
A: Вместо type Mutation { createUser: User! } (где ошибки уходят
   в errors[] с null bubbling) — type Mutation { createUser:
   CreateUserPayload! }, где CreateUserPayload содержит
   user: User и errors: [UserError!]. Errors[] подходит для
   НЕОЖИДАННЫХ ошибок (сервер упал), Payload Pattern — для
   ОЖИДАЕМЫХ бизнес-ошибок (email занят). TypeScript тогда
   принуждает проверить errors перед доступом к user.

Q: "Когда подписки оверинжиниринг?"
A: Когда достаточно polling (счётчик уведомлений) или SSE
   (односторонний поток). Subscriptions оправданы при критичной
   задержке (чат, торговые котировки) или двунаправленных/очень
   частых событиях.
```

### Чем Variables отличаются от inline-аргументов и почему это важно?

На первый взгляд — вкусовщина. Но есть архитектурная причина:

- **Inline-аргументы** (`user(id: "42") { name }`) — каждый уникальный набор данных создаёт НОВУЮ строку запроса.
- **Variables** (`query GetUser($id: ID!) { user(id: $id) }` + `{"id": "42"}`) — текст запроса СТАБИЛЕН, меняются только переменные.

Стабильность текста критична для: 1) **Persisted Queries/APQ** — клиент отправляет SHA-256 хэш текста вместо всего текста; если текст меняется с каждым запросом, хэш меняется, кэш не работает; 2) **Query allowlisting** — белый список допустимых запросов работает только с фиксированными текстами; 3) **Безопасность** — значения в variables проходят через `parseValue` кастомных скаляров для валидации.

```txt
Типичные follow-up:

Q: "Как именно работает APQ (Automatic Persisted Queries)?"
A: 1) Клиент отправляет только хэш → сервер: "не знаю хэш" →
   2) Клиент отправляет хэш + полный текст → сервер сохраняет
   текст под хэшем → 3) Следующие запросы — только хэш.
   Плюс: через GET с хэшем в URL → CDN-кэш по URL.
```

## Группа 3: Схема и типы

### Что такое Null Bubbling и почему злоупотребление "!" опасно?

Если resolver возвращает null (или выбрасывает ошибку) для поля, помеченного как NON-NULL (`String!`), GraphQL не может нарушить контракт схемы и вернуть `{ "name": null }`. Поэтому ошибка "пробулькивает" вверх по дереву ответа до БЛИЖАЙШЕГО NULLABLE-предка, который становится null.

```graphql
type Query { user: User }    # nullable
type User { name: String! }  # non-null!
```

При ошибке в `User.name` → `data.user` становится `null` (весь объект), потому что `User` — ближайший nullable-предок. Если `Query.user` тоже был `User!` — `null` становится `data` целиком.

Практическое следствие: чем больше `!` в схеме — тем выше риск "провала" всего ответа из-за одного нестабильного резолвера глубоко в дереве. Типичная практика: `id` — всегда non-null, поля, зависящие от внешних сервисов — nullable.

```txt
Типичные follow-up:

Q: "Как null bubbling взаимодействует с authorization на уровне поля?"
A: Если поле email: String! недоступно текущему пользователю,
   и резолвер возвращает null вместо throw — это вызывает null
   bubbling объекта User. Поэтому поля с условным доступом
   ДОЛЖНЫ быть nullable в схеме — это архитектурное требование.
```

### Чем Interface отличается от Union, и когда использовать каждый?

- **Interface** — общий набор полей, ОБЯЗАТЕЛЬНЫХ для всех реализующих типов. Клиент может запросить общие поля без `... on TypeName`, а специфичные — через inline fragments.
- **Union** — объединение СОВЕРШЕННО несвязанных типов. Общих полей нет — только `... on TypeName` для каждого возможного типа.

Оба требуют `__resolveType` — функции, которая по объекту определяет, к какому конкретному типу он относится:

```ts
Notification: {
  __resolveType(obj) {
    if ('likedBy' in obj) return 'LikeNotification';
    if ('comment' in obj) return 'CommentNotification';
  }
}
```

Практически: Interface — когда типы концептуально похожи (`Notification`, `Animal`), Union — когда поле может быть одним из принципиально разных вариантов (`SearchResult = User | Post | Comment`).

```txt
Типичные follow-up:

Q: "Почему нельзя использовать Object Type как тип аргумента?"
A: Object Type может содержать поля с резолверами, interfaces,
   unions — концепции, бессмысленные для входных данных. Для
   аргументов нужен Input Type — "плоская" структура без
   резолверов. (Input Unions через @oneOf появились только в
   последних версиях спецификации.)
```

## Группа 4: N+1 Problem и DataLoader

### Что такое N+1 Problem в контексте GraphQL и почему она возникает именно здесь?

Каждое поле GraphQL может иметь свой резолвер. Резолвер для поля-массива вызывается ОТДЕЛЬНО для КАЖДОГО элемента родительского массива:

```txt
Query.users → 1 SQL (возвращает 100 пользователей)
User.posts  → вызывается 100 раз → 100 SQL
Итого: 101 запрос вместо 2
```

В REST у каждого эндпоинта фиксированная форма → можно прописать JOIN заранее. В GraphQL форма запроса динамическая → JOIN в резолвере Query.users работает только если клиент ВСЕГДА запрашивает posts. GraphQL-специфика в том, что это линейная деградация, невидимая на малых dev-данных (10 → 11 запросов) и убивающая в production (1000 → 1001).

### Как DataLoader решает N+1?

DataLoader работает через **batching** + **request-scoped caching**:

- `.load(key)` не выполняет запрос немедленно — добавляет ключ в очередь и возвращает Promise.
- На следующем тике event loop DataLoader вызывает batch-функцию ОДИН раз со ВСЕМИ накопленными ключами → один SQL `WHERE id IN (1,2,...,100)` вместо 100 отдельных.
- Внутри одного запроса повторный `.load("1")` возвращает закэшированный Promise без нового обращения к БД.

```ts
const createUserLoader = (db: Db) =>
  new DataLoader<string, User | null>(async (ids) => {
    const users = await db.user.findMany({ where: { id: { in: [...ids] } } });
    const byId = new Map(users.map(u => [u.id, u]));
    return ids.map(id => byId.get(id) ?? null); // порядок ОБЯЗАТЕЛЕН!
  });
```

```txt
Типичные follow-up:

Q: "Почему нужно возвращать результаты в том же порядке, что ids?"
A: DataLoader сопоставляет i-й результат с i-м ключом
   ПОЗИЦИОННО. Если порядок нарушен — данные одного пользователя
   "приедут" другому.

Q: "Почему DataLoader нельзя создать как singleton?"
A: Его кэш не очищается между запросами → данные пользователя A
   утекают в ответ пользователю B (data leak), плюс неограниченный
   рост памяти. Каждый GraphQL-запрос должен получать НОВЫЙ
   instance в context.

Q: "Что произойдёт, если в рамках одного запроса Mutation
   изменит пользователя, а потом резолвер снова запросит его
   через DataLoader?"
A: DataLoader вернёт закэшированное СТАРОЕ значение. Нужно явно
   вызвать userLoader.clear(id) или userLoader.prime(id, newData)
   после мутации.

Q: "DataLoader полностью заменяет JOIN?"
A: Нет. DataLoader даёт O(уровней вложенности) запросов, а не
   O(1). Для часто запрашиваемых связей точечный JOIN в
   конкретном резолвере может быть быстрее.
```

## Группа 5: Производительность и безопасность

### Почему GraphQL API потенциально опаснее REST с точки зрения DoS?

В REST у каждого эндпоинта фиксированная стоимость (сервер контролирует JOIN'ы и глубину). В GraphQL клиент формирует запрос динамически, и ЛЮБАЯ синтаксически валидная комбинация должна быть выполнена:

```graphql
query {
  users {
    posts {
      comments {
        author {
          posts {        # циклический граф — глубина растёт
            comments { text }
          }
        }
      }
    }
  }
}
```

Защита — несколько уровней:
- **Depth Limiting** (graphql-depth-limit) — грубая, обходится через Query Aliasing (1000 алиасов одного поля при глубине 1).
- **Query Complexity** с multipliers — `users(first: 100) { posts(first: 100) }` стоит `100 * (1 + 100*2) = 20100` — точнее, учитывает перемножение limit-аргументов.
- **Persisted Queries + allowlisting** — только известные запросы проходят на сервер.
- **Rate limiting по complexity** — не по количеству HTTP-запросов, а по сумме complexity за период.

```txt
Типичные follow-up:

Q: "Отключение introspection в production — достаточная мера?"
A: Нет, это "security through obscurity". Атакующий с доступом
   к bundle.js фронтенда видит все запросы. Отключение introspection
   НЕ заменяет authorization на уровне резолвера.

Q: "Почему authorization нужна на уровне резолвера, а не middleware?"
A: В GraphQL один endpoint обслуживает всё. Middleware на /graphql
   проверяет "залогинен ли пользователь", но не "может ли этот
   пользователь видеть User.email ДРУГОГО пользователя". Это
   проверяется только внутри резолвера конкретного поля.
```

### Почему cursor-based pagination лучше offset на больших данных?

`OFFSET 100000 LIMIT 20` — БД всё равно сканирует и отбрасывает 100 000 строк (seq scan или index scan до нужной точки).

`WHERE id > cursor LIMIT 20` — использует индекс напрямую, без сканирования пропущенных строк. Плюс: устойчив к вставкам/удалениям (offset "сдвигает" страницы, cursor привязан к записи).

Relay Connection spec формализует cursor pagination: `UserConnection` с `edges: [UserEdge]` (node + cursor), `pageInfo` (hasNextPage, endCursor) и аргументами `first`/`after`.

## Группа 6: Архитектура — Federation, BFF, GraphQL vs REST

### Как бы вы масштабировали большой GraphQL API в микросервисной архитектуре?

Ключевой инструмент — **Apollo Federation**: каждый микросервис владеет своей частью GraphQL-графа (`@key(fields: "id")`, `extend type`, `__resolveReference`), Gateway/Router объединяет их в единую схему для клиента. Команды работают независимо, без единого monolithic BFF.

Дополнительно:
- **DataLoader** на уровне каждого сервиса (для N+1 к собственной БД)
- Gateway также батчит `_entities`-запросы между сервисами — аналог DataLoader на межсервисном уровне
- **Query complexity + depth limiting** на Gateway
- **Persisted Queries** для кэша и allowlisting
- **Cursor-based pagination** для всех коллекций

```txt
Типичные follow-up:

Q: "Что такое BFF и когда Federation предпочтительнее?"
A: BFF (Backend For Frontend) — единый GraphQL-сервис,
   агрегирующий данные из нескольких REST/gRPC downstream-сервисов.
   Прост в начале, но становится monolith. Federation — когда
   команды растут и нужна независимость между подграфами. Federation
   добавляет сложность Gateway, но убирает bottleneck монолитного BFF.

Q: "Когда выбрать REST вместо GraphQL?"
A: Публичный API (CDN-кэш, не нужен GraphQL-клиент), простое
   CRUD без вложенных связей, частая загрузка файлов, webhooks.
   Наиболее реалистичный production-ответ: REST + GraphQL в одном
   проекте — REST для публичного API, GraphQL как BFF для
   собственного frontend.

Q: "GraphQL заменяет REST?"
A: Нет. Они решают разные задачи. Большинство публичных API
   останутся REST из-за простоты консьюминга, HTTP-кэша и
   отсутствия зависимости от GraphQL-клиента. GraphQL — оптимален
   как aggregation layer над сервисами для сложных клиентов.
```
