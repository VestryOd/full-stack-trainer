# GraphQL Fundamentals

## Что такое GraphQL — точнее, чем "альтернатива REST"

GraphQL — это СПЕЦИФИКАЦИЯ (язык запросов + система типов + семантика выполнения), а не конкретная технология или протокол транспорта. Реализаций множество: `graphql-js` (референсная реализация), Apollo Server, Mercurius (Fastify), GraphQL Yoga и т.д. — все они реализуют ОДНУ спецификацию, но с разными расширениями (кэширование, federation, subscriptions transport).

```txt
Из спецификации GraphQL:
  - Schema Definition Language (SDL) — язык описания типов
  - Query Language — язык запросов (query/mutation/subscription)
  - Execution — алгоритм обхода резолверов и сборки ответа
  - Введение НЕ входит в спецификацию: транспорт (обычно
    HTTP POST, но не обязательно), способ аутентификации,
    кэширование
```

Это важно для интервью: вопрос "GraphQL работает только по HTTP?" — нет, спецификация транспорт-агностична. На практике почти всегда HTTP POST на `/graphql`, но GraphQL поверх WebSocket (subscriptions) или даже gRPC — тоже валидные реализации.

## Проблема, которую решает GraphQL: overfetching и underfetching — с конкретным числовым примером

```txt
REST: экран профиля пользователя показывает имя,
аватар, 3 последних поста и счётчик подписчиков.

GET /users/1        → { id, name, email, avatar, bio,
                          createdAt, updatedAt, settings, ... }
                        (overfetching: используется 2 поля из 9)
GET /users/1/posts?limit=3
GET /users/1/followers/count

= 3 round-trip'а, каждый со своей латентностью сети
```

```graphql
# GraphQL: один запрос, ровно нужные поля
query ProfileScreen {
  user(id: 1) {
    name
    avatar
    posts(limit: 3) { title, createdAt }
    followersCount
  }
}
```

```txt
Главная идея в одной фразе:

REST  — СЕРВЕР определяет форму ответа (через структуру
        endpoint'ов), клиент подстраивается
GraphQL — КЛИЕНТ определяет форму ответа (через структуру
        запроса), сервер предоставляет ВОЗМОЖНОСТИ через схему
```

## Конвейер выполнения запроса — три отдельные фазы

```txt
1. Parse — текст запроса разбирается в AST (Abstract Syntax
   Tree). Синтаксическая ошибка (например, незакрытая
   скобка) — отклоняется ЗДЕСЬ, до любого обращения к данным.

2. Validate — AST сверяется со SCHEMA: существуют ли
   запрошенные типы/поля/аргументы, совпадают ли типы
   аргументов. Ошибка ВАЛИДАЦИИ (запрошено несуществующее
   поле) — отклоняется ЗДЕСЬ, резолверы НЕ ВЫЗЫВАЮТСЯ вообще.

3. Execute — для каждого поля в запросе вызывается
   соответствующий resolver, результаты собираются в дерево,
   соответствующее форме запроса.
```

Этот трёхфазный конвейер объясняет, почему "GraphQL строго типизирован" — НЕ означает "ошибки типов невозможны в runtime". Это означает: НЕСООТВЕТСТВИЕ ЗАПРОСА СХЕМЕ отклоняется на фазе Validate, ДО выполнения каких-либо resolver'ов — то есть до обращения к БД. Но ошибки ВНУТРИ resolver'а (например, БД недоступна, или resolver вернул `null` для поля с типом `String!`) — это ошибки EXECUTE-фазы, и они происходят в runtime, как в любом другом API.

## Schema + Resolvers — разделение "что доступно" и "как получить"

```graphql
# Schema (SDL) — декларация ВОЗМОЖНОСТЕЙ API
type User {
  id: ID!
  name: String!
  posts: [Post!]!
}

type Query {
  user(id: ID!): User
}
```

```ts
// Resolvers — РЕАЛИЗАЦИЯ того, как получить данные
// для каждого поля схемы
const resolvers = {
  Query: {
    user: (_parent, { id }, context) => context.db.users.findById(id),
  },
  User: {
    posts: (parent, _args, context) => context.db.posts.findByUserId(parent.id),
  },
};
```

```txt
Schema — это КОНТРАКТ, который клиенты видят и на который
полагаются (через introspection, codegen). Resolvers —
ДЕТАЛИ РЕАЛИЗАЦИИ, полностью скрытые от клиента. Можно
полностью переписать резолвер User.posts (сменить БД,
добавить кэширование) — клиент НЕ заметит изменений, пока
форма данных в ответе соответствует схеме.

Эта развязка — основа для дальнейших тем: [N+1 Problem and
DataLoader] (оптимизация резолверов без изменения схемы),
[Schema, Types, and Resolvers] (детали выполнения резолверов).
```

## Один endpoint и HTTP-семантика — где GraphQL ломает привычные паттерны REST

```txt
Большинство GraphQL-серверов отвечают на ОДНОМ endpoint
(обычно POST /graphql), независимо от запрошенных данных.
```

### Senior-нюанс №1: HTTP статус почти всегда 200, даже при ошибке

```json
// Ответ на запрос, где resolver выбросил ошибку
// (например, БД недоступна) — HTTP 200 OK!
{
  "data": { "user": null },
  "errors": [
    { "message": "Database connection failed", "path": ["user"] }
  ]
}
```

```txt
Это ОЖИДАЕМОЕ поведение спецификации: ошибки выполнения —
ЧАСТЬ ответа GraphQL, а не HTTP-уровня. HTTP-статус 4xx/5xx
зарезервирован для ошибок ТРАНСПОРТНОГО уровня (невалидный
JSON, сервер недоступен, превышен размер запроса).

Практическое следствие: HTTP-мониторинг ("alerting на 5xx")
НЕ ЛОВИТ ошибки бизнес-логики GraphQL — нужен анализ поля
"errors" в теле ответа, что требует GraphQL-aware
инструментов (Apollo Studio, специфичные дашборды), а не
стандартного APM по статус-кодам.
```

### Senior-нюанс №2: один endpoint усложняет HTTP-кэширование на уровне инфраструктуры

```txt
REST: GET /users/1 — кэшируется CDN/браузером по URL
"бесплатно" (HTTP-кэш работает на уровне URL + метода).

GraphQL: POST /graphql с телом {"query": "..."} — POST-запросы
НЕ кэшируются стандартными HTTP-кэшами, и даже если бы
кэшировались, ключ кэша должен включать ВЕСЬ body (query +
variables), а не просто URL.

Решения (детали — в [Performance and Security]):
  - Persisted Queries — клиент отправляет ХЭШ запроса вместо
    полного текста; сервер может кэшировать по хэшу,
    и теоретически — обслуживать через GET с хэшем в URL
  - Кэширование на уровне РЕЗОЛВЕРОВ/полей (Apollo Cache
    Control directives), а не HTTP-уровня
```

## Introspection — мощь и риск одновременно

```graphql
{
  __schema {
    types { name, fields { name, type { name } } }
  }
}
```

```txt
Introspection — встроенная возможность GraphQL описывать
СВОЮ СОБСТВЕННУЮ схему через запрос. На этом построены:
  - GraphiQL/Apollo Studio Playground (автодополнение)
  - Codegen (генерация TS-типов из схемы)
  - Schema diffing в CI (обнаружение breaking changes)

Senior-нюанс — security trade-off: introspection в
production РАСКРЫВАЕТ ВСЮ структуру API, включая поля,
которые могут быть "internal only" (например,
type AdminMutation), но всё ещё технически достижимые.
Многие команды ОТКЛЮЧАЮТ introspection в production
(`introspection: false` в Apollo Server), оставляя её
только в staging — но это создаёт trade-off: внешние
инструменты (Apollo Studio Explorer, codegen для партнёров)
тоже перестают работать против production-схемы.
```

## Сильные и слабые стороны — без "GraphQL всегда лучше"

```txt
Сильные стороны:
  - клиент контролирует форму ответа → меньше round-trip'ов
    для сложных UI с вложенными данными
  - единая строго типизированная схема как контракт между
    фронтендом и бэкендом, с автогенерацией типов
  - встроенная эволюция API без версионирования (см.
    [GraphQL vs REST] — добавление полей не ломает старых
    клиентов)

Слабые стороны (требуют отдельной инженерной работы):
  - HTTP-кэширование сложнее (см. выше)
  - произвольные клиентские запросы → риск дорогих/глубоко
    вложенных запросов (см. [N+1 Problem and DataLoader] и
    [Performance and Security] — query complexity limits,
    depth limiting)
  - мониторинг/алертинг не сводится к HTTP статус-кодам
  - file uploads, batch-операции — не "из коробки", требуют
    расширений спецификации (multipart request spec и т.п.)
```

## Типичные ошибки на интервью

- **"GraphQL — это база данных/протокол замена REST"** — путать спецификацию языка запросов с конкретным транспортом или хранилищем; GraphQL — слой между клиентом и ЛЮБЫМИ источниками данных (БД, REST API, gRPC).

- **"Строгая типизация GraphQL означает отсутствие ошибок в runtime"** — не различать ошибки фазы Validate (запрос не соответствует схеме, резолверы не вызываются) и ошибки фазы Execute (резолвер упал, БД недоступна — обычные runtime-ошибки).

- **"GraphQL всегда возвращает HTTP 200, значит он не различает успех/неуспех"** — не знать про массив `errors` в теле ответа и про то, что мониторинг должен анализировать его, а не HTTP статус.

- **"Introspection — просто удобная фича без рисков"** — не упоминать, что раскрытие полной схемы в production может быть нежелательным с точки зрения security, и что отключение introspection — это trade-off, а не "бесплатное улучшение".

- **Не упоминать, что один endpoint усложняет HTTP-кэширование** — типичный "скрытый" минус GraphQL, который не очевиден, пока не попробуешь настроить CDN-кэш для GraphQL API.
