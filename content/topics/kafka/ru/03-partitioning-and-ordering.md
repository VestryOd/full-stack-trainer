# Партишнирование и порядок сообщений

## Как продюсер выбирает партишн

Когда продюсер отправляет сообщение, ему нужно решить, в какой из N партишнов топика его записать. Это решение принимается **на стороне клиента** (а не брокером) по одной из трёх стратегий:

### Стратегия 1: Key-based (ключевое партишнирование)

Если сообщение отправлено с ключом (key ≠ null), Kafka вычисляет партишн по формуле:

```txt
partition = murmur2(key) % numPartitions
```

`murmur2` — нефриптографическая хеш-функция. Важно: один и тот же ключ **всегда** попадает в один и тот же партишн (при неизменном числе партишнов).

```txt
Топик "order-events" (4 партишна):

Сообщение { key: "order-101", value: "placed"  } → hash("order-101") % 4 = 2 → Partition 2
Сообщение { key: "order-101", value: "shipped" } → hash("order-101") % 4 = 2 → Partition 2
Сообщение { key: "order-101", value: "delivered"} → hash("order-101") % 4 = 2 → Partition 2
Сообщение { key: "order-202", value: "placed"  } → hash("order-202") % 4 = 0 → Partition 0
```

Результат: все события для `order-101` гарантированно попадают в Partition 2 → их порядок гарантирован, они будут обработаны одним consumer'ом в правильной последовательности.

### Стратегия 2: Round-robin (без ключа)

Если ключ не задан (key = null), продюсер распределяет сообщения по партишнам равномерно по кругу:

```txt
Топик "logs" (3 партишна), key = null:

Сообщение 1 → Partition 0
Сообщение 2 → Partition 1
Сообщение 3 → Partition 2
Сообщение 4 → Partition 0
Сообщение 5 → Partition 1
...
```

Преимущество: равномерная нагрузка. Недостаток: нет гарантий порядка — сообщения 1, 4, 7 (в partition 0) и 2, 5, 8 (в partition 1) читаются параллельно разными consumer'ами.

> **Заметка о sticky partitioner**: начиная с Kafka 2.4, при key=null по умолчанию используется sticky partitioner — продюсер накапливает батч в одном партишне, затем переключается. Это снижает задержку за счёт уменьшения количества запросов к брокеру. На поведение с точки зрения порядка это не влияет.

### Стратегия 3: Явное указание партишна

Продюсер может задать партишн напрямую. Используется редко (специальные сценарии маршрутизации, тестирование):

```ts
await producer.send({
  topic: 'order-events',
  messages: [
    { partition: 2, key: 'order-101', value: JSON.stringify(event) }
  ]
});
```

## Почему порядок гарантирован только внутри партишна

Это самая частая тема на интервью по Kafka, и ответ нужно понимать механически, а не просто запомнить.

### Один партишн = один лог = один порядок

Партишн — это последовательный append-only файл. Сообщения записываются строго одно за другим, offset монотонно растёт. Читать их можно только в том порядке, в котором они записаны.

```txt
Partition 2:  [off:0: "placed"] [off:1: "payment-ok"] [off:2: "shipped"] [off:3: "delivered"]
              ────────────────────────────────────────────────────────────────────────────►
              Consumer читает строго слева направо: 0 → 1 → 2 → 3
```

Consumer, назначенный на Partition 2, получит эти сообщения строго в этом порядке. Гарантия абсолютная.

### Несколько партишнов — нет глобального порядка

Когда топик имеет несколько партишнов, разные partition-и читаются параллельно разными consumer'ами. Нет механизма, который бы синхронизировал их прогресс:

```txt
Топик "order-events" (3 партишна):

Partition 0: [order-A: placed] [order-D: placed] [order-A: shipped]
Partition 1: [order-B: placed] [order-B: payment-ok]
Partition 2: [order-C: placed] [order-C: shipped] [order-D: payment-ok]

Consumer A читает Partition 0: placed→A, placed→D, shipped→A
Consumer B читает Partition 1: placed→B, payment-ok→B
Consumer C читает Partition 2: placed→C, shipped→C, payment-ok→D

Вопрос: что произошло раньше — "order-A shipped" (P0, offset 2)
        или "order-D payment-ok" (P2, offset 2)?
Ответ: Kafka не знает. Offset 2 в P0 и offset 2 в P2 — независимые счётчики.
```

Глобального timestamp'а, который позволял бы сравнить сообщения из разных партишнов, у Kafka нет (хотя timestamp записывается в каждое сообщение, он не гарантирует упорядоченность при параллельной записи с нескольких продюсеров).

### Практический вывод: ключ как инструмент изоляции порядка

Если порядок важен для определённой сущности (заказ, пользователь, сессия), используй её ID как ключ:

```txt
✓ Правильно — порядок событий заказа гарантирован:
  key = "order-101" → все события order-101 в одном партишне

✗ Неправильно — события одного заказа могут оказаться в разных партишнах:
  key = null (или key = случайный UUID каждый раз)
```

## Как выбрать хороший ключ партишна

Выбор ключа — это баланс между тремя факторами:

### 1. Гарантия порядка (correctness)

Ключ должен группировать сообщения, порядок которых имеет смысл. Вопрос: "Если эти два события поменять местами, сломается ли логика?"

```txt
✓ Хорошие ключи:
  - userId    (события пользователя: регистрация → логин → покупка)
  - orderId   (события заказа: создан → оплачен → отгружен → доставлен)
  - sessionId (события сессии)
  - deviceId  (телеметрия устройства)

✗ Плохие ключи для порядка:
  - null             (нет порядка вообще)
  - timestamp        (у каждого сообщения уникальный ключ → не группирует)
  - случайный UUID   (то же самое)
```

### 2. Равномерность распределения (cardinality)

Плохое распределение → один партишн перегружен, остальные простаивают. Это называют **hot partition** (горячий партишн):

```txt
Топик "events" (4 партишна), key = countryCode:

Russia:  ████████████████████ (70% трафика) → P0 перегружен
Germany: ████ (15%)                          → P1 в норме
France:  ███ (10%)                           → P2 в норме
Other:   ██ (5%)                             → P3 простаивает

→ Consumer, читающий P0, не успевает. Lag растёт.
  Consumer'ы P1, P2, P3 простаивают.
  Добавление consumer'ов не помогает — bottleneck в P0.
```

```txt
✓ Высокая кардинальность (много уникальных значений):
  - userId (миллионы пользователей → равномерно по партишнам)
  - orderId
  - sessionId

✗ Низкая кардинальность (мало уникальных значений):
  - countryCode (10–200 значений → легко создаёт hot partition)
  - status ("active"/"inactive" — только 2 значения)
  - isVip (boolean)
```

### 3. Неизменность числа партишнов (stability)

После увеличения числа партишнов `hash(key) % numPartitions` даст другой результат для тех же ключей. Сообщения с одним ключом начнут попадать в другой партишн → гарантия порядка нарушается для новых сообщений относительно старых.

```txt
Было: 4 партишна
  hash("order-101") % 4 = 2  → Partition 2

Стало: 6 партишнов
  hash("order-101") % 6 = 5  → Partition 5 (другой!)

Старые события order-101 в Partition 2,
новые события order-101 в Partition 5 →
consumer'ы читают их независимо → порядок нарушен.
```

**Вывод**: число партишнов нужно планировать заранее с запасом. Увеличение партишнов — болезненная операция, если важен порядок.

Практическое правило: закладывай в 2–4 раза больше партишнов, чем планируемое число consumer'ов на старте.

## Rebalancing — что происходит при изменении состава группы

**Rebalancing (ребалансировка)** — это процесс перераспределения партишнов между consumer'ами группы. Во время ребалансировки **вся группа приостанавливает обработку** — это важно.

### Что вызывает rebalancing

```txt
1. Consumer присоединяется к группе
   (новый экземпляр запущен, горизонтальное масштабирование)

2. Consumer покидает группу штатно
   (graceful shutdown, деплой новой версии)

3. Consumer считается мёртвым
   (не прислал heartbeat в течение session.timeout.ms — по умолчанию 45 сек)

4. Добавление партишнов в топик
   (изменение конфигурации топика)

5. Смена подписки consumer'а
   (изменение списка топиков)
```

### Механика rebalancing

До Kafka 2.3 использовался Eager Rebalancing:

```txt
Eager Rebalancing (старый протокол):

  До:   P0→C1, P1→C2, P2→C3
  
  C4 присоединяется:
  1. Все consumer'ы REVOKE все партишны (коммитят offset, освобождают)
  2. Group Coordinator (один из брокеров) пересчитывает назначения
  3. Все consumer'ы получают новые назначения

  После: P0→C1, P1→C2, P2→C3, P3→C4 (если партишнов 4)
         или P0→C1, P1→C2, P2→C4     (если C3 вышел)

  Проблема: пауза равна времени ревокации + пересчёта для ВСЕХ partition
```

С Kafka 2.3+: Cooperative (Incremental) Rebalancing:

```txt
Cooperative Rebalancing (новый протокол):

  До:   P0→C1, P1→C2, P2→C3

  C4 присоединяется:
  1. Group Coordinator определяет минимальные изменения
  2. Только P2 revoked у C3 (остальные продолжают работу)
  3. P2 назначается C4

  После: P0→C1, P1→C2, P2→C4

  Преимущество: только затронутые partition'ы приостанавливаются,
                остальные обрабатывают сообщения без паузы
```

В `kafkajs` cooperative rebalancing включается через настройку `partitionAssigners`.

### session.timeout.ms vs heartbeat.interval.ms

```txt
heartbeat.interval.ms (по умолчанию: 3000 мс)
  — как часто consumer шлёт "я живой" брокеру
  — должен быть значительно меньше session.timeout.ms

session.timeout.ms (по умолчанию: 45000 мс, min: 6000 мс)
  — если в течение этого времени нет heartbeat → consumer считается мёртвым
  — брокер инициирует rebalancing

max.poll.interval.ms (по умолчанию: 300000 мс = 5 минут)
  — максимальное время между двумя вызовами poll()
  — если consumer обрабатывает сообщение дольше этого → считается зависшим
  — брокер инициирует rebalancing, даже если heartbeat идут
```

Типичная проблема: consumer обрабатывает тяжёлое сообщение 6 минут → превышает `max.poll.interval.ms` (5 минут) → брокер исключает его из группы → rebalancing → другой consumer начинает обрабатывать то же сообщение → дублирование обработки.

Решение: либо увеличить `max.poll.interval.ms`, либо уменьшить `max.poll.records` (читать меньше сообщений за один poll), либо ускорить обработку.

## Итоговая схема: ключ → партишн → порядок

```txt
Producer отправляет события заказа:

  { key: "ord-1", event: "placed"   } ─────┐
  { key: "ord-2", event: "placed"   } ──┐  │   hash % 3
  { key: "ord-1", event: "paid"     } ─────┤──► P0: [ord-2:placed][ord-2:paid]
  { key: "ord-3", event: "placed"   } ─┐  │    P1: [ord-1:placed][ord-1:paid][ord-1:shipped]
  { key: "ord-1", event: "shipped"  } ─────┘    P2: [ord-3:placed][ord-3:shipped]
  { key: "ord-2", event: "paid"     } ──┘
  { key: "ord-3", event: "shipped"  } ─┘

Consumer Group (3 consumer'а):
  Consumer A читает P0 → ord-2 события строго по порядку ✓
  Consumer B читает P1 → ord-1 события строго по порядку ✓
  Consumer C читает P2 → ord-3 события строго по порядку ✓

  Глобальный порядок между ord-1, ord-2, ord-3 — не гарантирован ✗
  (но нам он и не нужен: заказы независимы)
```

## Типичные ошибки на интервью

**"Kafka гарантирует порядок сообщений"**

Неполный ответ. Kafka гарантирует порядок **внутри партишна**. При нескольких партишнах глобальный порядок не гарантируется. Правильный ответ: "Kafka гарантирует порядок внутри партишна. Для гарантии порядка событий конкретной сущности используют её ID как ключ — тогда все её события попадают в один партишн."

**"Можно увеличить число партишнов без последствий для порядка"**

Нет. После увеличения `hash(key) % newNum` даст другие партишны для тех же ключей. Новые события одной сущности будут писаться в другой партишн, чем старые → порядок относительно старых событий нарушен. Число партишнов нужно планировать заранее.

**"Hot partition — это нормально, просто один consumer будет медленнее"**

Это хуже, чем кажется. Consumer на hot partition станет bottleneck для всей группы: он накапливает consumer lag. Нельзя "помочь" ему, добавив ещё consumer'ов в тот же партишн — один партишн = один consumer. Нужно либо менять ключ партишнирования, либо делить горячий поток на несколько топиков.

**"Rebalancing — мгновенная операция"**

Нет. Во время eager rebalancing вся группа приостанавливает обработку: сначала все consumer'ы ревокают партишны, ждут координатора, получают новые назначения. На большом кластере с медленными consumer'ами это может занять секунды и более. Именно поэтому стоит использовать cooperative rebalancing (Kafka 2.3+).

**"Если consumer падает и переподнимается, он теряет неподтверждённые сообщения"**

Нет — при правильно настроенных offset'ах. Consumer возобновляет работу с последнего закоммиченного offset. Если он упал до коммита, он перечитает сообщения с момента последнего коммита. Это at-least-once семантика — сообщения будут обработаны повторно, не потеряны. Подробнее — в статье о гарантиях доставки.
