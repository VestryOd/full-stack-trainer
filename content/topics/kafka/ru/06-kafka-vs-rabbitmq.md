# Kafka vs RabbitMQ

## Философская разница: лог против очереди

Это не сравнение "что быстрее" или "что надёжнее". Kafka и RabbitMQ решают принципиально разные задачи, потому что построены на разных моделях хранения данных.

```txt
RabbitMQ — модель очереди:               Kafka — модель лога:
┌────────────────────────────┐           ┌────────────────────────────────────┐
│  Broker хранит сообщения   │           │  Broker хранит лог (append-only)   │
│  до тех пор, пока их не    │           │  до истечения срока хранения,      │
│  прочитают и не подтвердят │           │  независимо от того, прочитано     │
│  (ack). После ack —        │           │  ли сообщение или нет.             │
│  сообщение удаляется.      │           │                                    │
└────────────────────────────┘           └────────────────────────────────────┘

  Producer ──► [msg1][msg2][msg3]           offset: 0     1     2     3     4
  Consumer читает msg1 →                    [msg1][msg2][msg3][msg4][msg5]
  
  ──► [msg2][msg3]                          Consumer A: offset=2 (прочитал 0,1)
  msg1 УДАЛЁН                               Consumer B: offset=0 (читает с начала)
                                            Consumer C: offset=4 (почти в реальном времени)
                                            msg1,2,3,4,5 ВСЕ В ЛОГЕ
```

Из этой разницы вытекает всё остальное: поведение при сбоях, масштабирование, поддержка нескольких consumer'ов, возможность replay.

## Retention: как долго хранятся данные

```txt
RabbitMQ — event-driven retention:
  Сообщение существует ТОЛЬКО пока не обработано.
  Прочитано + ack → удалено немедленно.
  Нет ack в течение N секунд → requeue (вернуть в очередь для retry).
  Опционально: Dead Letter Queue для failed messages.

  Проблема: нельзя "перемотать назад". Если analytics-сервис
  не читал очередь 2 часа — эти 2 часа событий потеряны навсегда.

Kafka — time-based retention:
  Сообщение хранится N дней (retention.ms) или до объёма M байт (retention.bytes),
  НЕЗАВИСИМО от того, прочитали его или нет.

  По умолчанию: 7 дней. Конфигурируется на уровне топика:
    retention.ms=-1 → хранить вечно (compact topics)
    retention.ms=3600000 → 1 час
    retention.bytes=1073741824 → удалять старые при превышении 1GB

  Преимущество: analytics-сервис упал на 2 часа → перезапустился
  с последнего offset → догнал всё, что пропустил. Без потерь.
```

Это не просто техническая деталь — это архитектурная ценность Kafka. Лог является источником правды (source of truth). Новый сервис может подключиться и получить всю историю событий с самого начала. RabbitMQ такой возможности не даёт.

## Модель доставки: push vs pull

```txt
RabbitMQ — push-модель:
  Broker активно "толкает" сообщения к подписчикам.
  Broker управляет backpressure через prefetch count.
  Consumer не контролирует скорость получения напрямую.

  Хорошо: низкая задержка (sub-millisecond для simple tasks)
  Плохо: consumer может получить больше, чем успевает обработать

Kafka — pull-модель:
  Consumer сам опрашивает брокер (poll).
  Consumer полностью контролирует скорость обработки.
  Broker не знает о состоянии consumer'а (кроме offset'ов).

  Хорошо: consumer никогда не перегружен; читает в своём темпе
  Плохо: чуть выше latency для real-time задач (но обычно <10ms)
```

## Маршрутизация: где живёт логика

```txt
RabbitMQ — умная маршрутизация в брокере:
  ┌─────────────────────────────────────────────┐
  │              RabbitMQ Broker                 │
  │                                              │
  │  Producer ──► Exchange ──► Bindings ──►      │
  │              (direct/fanout/               Queue A ──► Consumer A
  │               topic/headers)              Queue B ──► Consumer B
  │                                            Queue C ──► Consumer C
  └─────────────────────────────────────────────┘

  Exchange types:
  - direct:  точное совпадение routing key
  - fanout:  всем очередям (broadcast)
  - topic:   wildcards в routing key (orders.*, *.critical)
  - headers: маршрутизация по заголовкам сообщения

  Богатая маршрутизация — сильная сторона RabbitMQ.
  Можно выстроить сложные топологии прямо в брокере.

Kafka — маршрутизация в клиентском коде:
  Брокер просто хранит топики.
  Вся логика "какой consumer что читает" — в коде приложения.
  
  Нужен fanout? → несколько consumer groups читают один топик.
  Нужна фильтрация? → consumer читает всё, фильтрует сам.
  Нужна маршрутизация? → producer пишет в разные топики по условию.

  Это не недостаток — это осознанная простота брокера ("dumb broker").
```

## Когда выбирать Kafka

```txt
✓ Kafka — правильный выбор:

  1. Event Streaming / Event Sourcing
     Поток событий как источник правды.
     Нужен replay для новых сервисов или отладки.
     "Что произошло с этим заказом за последние 30 дней?"

  2. Несколько независимых потребителей одного потока
     Order события → search + analytics + recommendations + audit
     Все читают один топик независимо.
     Добавление нового потребителя не требует изменений продюсера.

  3. Высокий throughput
     Миллионы событий в секунду.
     Аналитика в реальном времени, IoT-телеметрия, финансовые потоки.

  4. Log Aggregation
     Централизованный сбор логов из микросервисов.
     Logstash/Filebeat → Kafka → Elasticsearch.

  5. Change Data Capture (CDC)
     Стриминг изменений из БД (Debezium → Kafka).
     Синхронизация между хранилищами данных.

  6. Долгосрочное хранение событий
     История событий нужна для compliance, аудита, ML-обучения.
```

## Когда выбирать RabbitMQ

```txt
✓ RabbitMQ — правильный выбор:

  1. Task Queues / Work Queues
     "Отправить письмо", "изменить размер изображения", "сгенерировать PDF".
     Одна задача — один воркер — результат.
     Нет нужды читать задачу повторно после выполнения.

  2. RPC-like паттерны
     Запрос → обработка → ответ в reply queue.
     Caller ждёт результата от worker'а.
     Kafka для этого неудобна.

  3. Сложная маршрутизация на уровне брокера
     Разные очереди для разных типов событий с wildcards.
     Conditional routing без кода в consumer'е.

  4. Приоритизация сообщений
     RabbitMQ поддерживает очереди с приоритетами (priority queues).
     Kafka не поддерживает приоритизацию нативно.

  5. Короткоживущие задачи с немедленным результатом
     Timeout на обработку имеет смысл.
     Retry с DLQ встроен в брокер.

  6. Низкая latency для простых задач
     Sub-millisecond доставка при небольших объёмах.
```

## Честное сравнение по параметрам

```txt
┌───────────────────────┬──────────────────────────┬──────────────────────────┐
│ Параметр              │ Kafka                    │ RabbitMQ                 │
├───────────────────────┼──────────────────────────┼──────────────────────────┤
│ Модель хранения       │ Лог (append-only)        │ Очередь (удаляется       │
│                       │                          │ после ack)               │
├───────────────────────┼──────────────────────────┼──────────────────────────┤
│ Retention             │ По времени/объёму        │ До чтения + ack          │
│                       │ (независимо от чтения)   │                          │
├───────────────────────┼──────────────────────────┼──────────────────────────┤
│ Replay                │ Да — перемотай offset    │ Нет                      │
│                       │ назад                    │                          │
├───────────────────────┼──────────────────────────┼──────────────────────────┤
│ Несколько потребит.   │ Да — consumer groups,    │ Ограничено — каждая      │
│                       │ каждая группа получает   │ очередь читается одним   │
│                       │ все сообщения            │ набором consumer'ов       │
├───────────────────────┼──────────────────────────┼──────────────────────────┤
│ Throughput            │ Очень высокий            │ Средний-высокий          │
│                       │ (млн msg/сек)            │ (тыс–сотни тыс msg/сек)  │
├───────────────────────┼──────────────────────────┼──────────────────────────┤
│ Latency               │ Миллисекунды (pull)      │ Sub-millisecond (push)   │
├───────────────────────┼──────────────────────────┼──────────────────────────┤
│ Маршрутизация         │ В клиентском коде        │ В брокере (Exchange)     │
│                       │ (простой брокер)         │ (умный брокер)           │
├───────────────────────┼──────────────────────────┼──────────────────────────┤
│ Порядок               │ Гарантирован внутри      │ Гарантирован внутри      │
│                       │ партишна                 │ одной очереди            │
├───────────────────────┼──────────────────────────┼──────────────────────────┤
│ Операционная          │ Выше: брокеры,           │ Ниже: проще в настройке  │
│ сложность             │ партишны, репликация,    │ и управлении             │
│                       │ ZooKeeper/KRaft          │                          │
├───────────────────────┼──────────────────────────┼──────────────────────────┤
│ Dead Letter           │ Паттерн DLT через код    │ Встроен в брокер (DLX)   │
│                       │ (не встроен в брокер)    │                          │
├───────────────────────┼──────────────────────────┼──────────────────────────┤
│ Приоритизация         │ Не поддерживается        │ Priority queues          │
│ сообщений             │                          │ (нативно)                │
└───────────────────────┴──────────────────────────┴──────────────────────────┘
```

## Честная заметка про выбор в реальном мире

Книжный ответ на вопрос "Kafka или RabbitMQ?" — "смотри на требования: нужен replay? нужен высокий throughput? нужна сложная маршрутизация?". Это правильный ответ. Но в реальности всё немного иначе.

**Многие команды выбирают инструмент не по техническому fit, а по тому, что предлагает их облачный провайдер:**

```txt
AWS:
  → Amazon MSK (Managed Streaming for Kafka) — Kafka
  → Amazon SQS — простая очередь (не RabbitMQ, но аналог для task queues)
  → Amazon SNS — pub/sub поверх SQS

GCP:
  → Google Cloud Pub/Sub — managed messaging (семантика ближе к Kafka,
    но не Kafka)

Azure:
  → Azure Event Hubs — совместимый с Kafka API (managed Kafka)
  → Azure Service Bus — RabbitMQ-like messaging

Confluent Cloud:
  → Полностью managed Kafka с дополнительными инструментами
    (Schema Registry, ksqlDB, Kafka Connect)
```

Выбор "Kafka" часто означает "MSK" или "Confluent Cloud", а не собственный кластер. Это снижает операционную нагрузку, но добавляет vendor lock-in и стоимость.

**Ещё одна реальность**: многие стартапы начинают с RabbitMQ (проще поднять, меньше оверхед) и переходят на Kafka при росте нагрузки или когда появляется нужда в replay/event sourcing. Kafka — не автоматически лучший выбор для любого нового проекта.

## Сценарий: order-сервис — что выбрать?

Чтобы прояснить, когда что использовать, рассмотрим конкретный сценарий:

**Требования**: при размещении заказа нужно отправить email, зарезервировать товар на складе и записать в аналитику.

```txt
Вариант A: RabbitMQ
  Order Service
    │── publish → "email" exchange → Email Queue → Email Worker
    │── publish → "inventory" exchange → Inventory Queue → Inventory Worker
    └── publish → "analytics" exchange → Analytics Queue → Analytics Worker

  Плюсы: просто, задачи независимы, каждый воркер делает одно
  Минусы: Order Service знает о трёх downstream-системах;
          новый потребитель (например, recommendations) требует изменений
          в Order Service; нет replay при сбое analytics на 2 часа

Вариант B: Kafka
  Order Service
    └── publish → topic "order-placed" (key: orderId)

  Email Consumer Group     ← читает "order-placed"
  Inventory Consumer Group ← читает "order-placed"
  Analytics Consumer Group ← читает "order-placed"

  Плюсы: Order Service не знает о downstream-системах;
         новый потребитель (recommendations) — просто новая group,
         Order Service не меняется; analytics упала на 2 часа →
         перезапустилась → обработала всё с последнего offset
  Минусы: больше инфраструктуры; избыточно, если нужны только
          простые fire-and-forget задачи без replay
```

**Правило выбора**: если у вас **одно событие → несколько независимых потребителей** и/или **нужен replay** — Kafka. Если **одна задача → один воркер** без необходимости перечитывать — RabbitMQ (или SQS).

## Типичные ошибки на интервью

**"Kafka быстрее RabbitMQ — поэтому нужно выбирать Kafka"**

Неправильный критерий выбора. RabbitMQ имеет sub-millisecond latency и отлично справляется с сотнями тысяч сообщений в секунду — для большинства task-queue сценариев этого более чем достаточно. Kafka действительно быстрее при очень высоком throughput (миллионы msg/сек), но это не аргумент для задачи "отправить 1000 emails в день".

**"Kafka заменяет RabbitMQ"**

Нет. Kafka не умеет то, что умеет RabbitMQ: приоритизация очередей, встроенный DLX, богатая маршрутизация через Exchange, RPC-паттерн с reply queue. Это разные инструменты для разных задач, а не версии одного и того же.

**"RabbitMQ устарел, все переходят на Kafka"**

Это индустриальный хайп, не факт. RabbitMQ активно развивается (RabbitMQ 3.x → 4.x), используется в миллионах продакшн-систем и остаётся правильным выбором для task queues, простых messaging-паттернов и сред с невысоким throughput.

**"Нужен pub/sub → нужна Kafka"**

Не обязательно. RabbitMQ fanout exchange реализует pub/sub. Amazon SNS + SQS реализует pub/sub без Kafka. Kafka — один из способов реализации pub/sub, но не единственный. Выбор Kafka оправдан, если помимо pub/sub нужен replay, долгосрочное хранение или очень высокий throughput.

**"Если мы используем AWS — нужно брать SQS вместо Kafka"**

SQS и Kafka — разные инструменты. SQS — managed task queue (ближе к RabbitMQ по модели). Amazon MSK — managed Kafka. Выбор между ними определяется той же логикой: replay нужен? → MSK. Простые task queues? → SQS.
