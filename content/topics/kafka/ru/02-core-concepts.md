# Ключевые концепции Kafka

## Топик (Topic)

**Топик (topic)** — это именованный канал, в который продюсеры пишут сообщения, а консьюмеры их читают. Аналог: папка с журналом событий определённого типа.

```txt
topic: "order-placed"       topic: "payment-completed"    topic: "user-registered"
┌──────────────────────┐    ┌──────────────────────┐    ┌──────────────────────┐
│ [order1][order2]...  │    │ [pay1][pay2][pay3]...│    │ [usr1][usr2]...      │
└──────────────────────┘    └──────────────────────┘    └──────────────────────┘
```

Топик — логическая единица организации данных. Физически топик разбит на **партишны (partitions)**, которые и хранят реальные данные.

Важные свойства топика:
- Имя топика — строка (например, `"order-placed"`, `"user-events"`)
- Количество партишнов задаётся при создании (можно увеличить, но уменьшить — нет)
- Срок хранения (`retention.ms`) настраивается на уровне топика
- Один топик может читать любое число consumer group — данные общие

## Партишн (Partition)

**Партишн (partition)** — это физическое подразделение топика: упорядоченный, неизменяемый лог сообщений, который хранится на диске одного брокера.

Каждый топик разбивается на N партишнов (N ≥ 1). Партишны — это и есть то, что позволяет Kafka масштабироваться горизонтально.

```txt
Topic "order-placed" с 3 партишнами:

Partition 0:  [order-A] [order-B] [order-C] [order-D] ──► новые сообщения добавляются сюда
              offset: 0    1         2         3

Partition 1:  [order-E] [order-F] [order-G] ──► ...
              offset: 0    1         2

Partition 2:  [order-H] [order-I] ──► ...
              offset: 0    1
```

**Зачем вообще нужны партишны?**

Один файл-лог — один CPU-поток на запись/чтение. Это bottleneck. Партишны разрывают это ограничение:

1. **Параллелизм записи**: продюсер может писать в partition 0, 1, 2 одновременно
2. **Параллелизм чтения**: каждый партишн может читать отдельный consumer — 3 партишна = 3 параллельных consumer'а в одной группе
3. **Распределение по брокерам**: каждый партишн живёт на определённом брокере (с репликами на других), что позволяет распределить нагрузку по кластеру

```txt
Kafka Cluster с 3 брокерами, Topic с 3 партишнами:

Broker 1          Broker 2          Broker 3
┌─────────────┐   ┌─────────────┐   ┌─────────────┐
│ P0 (leader) │   │ P1 (leader) │   │ P2 (leader) │
│ P1 (replica)│   │ P2 (replica)│   │ P0 (replica)│
│ P2 (replica)│   │ P0 (replica)│   │ P1 (replica)│
└─────────────┘   └─────────────┘   └─────────────┘
```

**Leader** — партишн, который принимает записи от продюсеров. **Replica** — копия, которая синхронизируется с лидером. Если брокер с лидером падает, один из replica-брокеров автоматически становится новым лидером.

## Оффсет (Offset)

**Оффсет (offset)** — это порядковый номер сообщения внутри конкретного партишна. Начинается с 0, монотонно возрастает.

```txt
Partition 0:
  offset 0: { key: "user-1", value: "order placed" }
  offset 1: { key: "user-2", value: "order placed" }
  offset 2: { key: "user-1", value: "order shipped" }
  offset 3: { key: "user-3", value: "order placed" }
              ↑
        consumer читает отсюда → до сих пор прочитано до offset 2 включительно
```

**Ключевое свойство**: offset — это **per-partition**, не глобальный по топику. Offset 5 в partition 0 и offset 5 в partition 1 — это совершенно разные сообщения.

Кроме того, Kafka хранит offset **per-consumer-group** (об этом ниже). Это означает, что две разные группы могут читать один и тот же топик независимо, каждая со своим набором offset'ов.

```txt
Topic "order-placed", Partition 0:
[msg0][msg1][msg2][msg3][msg4][msg5]

Consumer Group "search-service":   offset = 5 (прочитала всё)
Consumer Group "analytics":        offset = 3 (отстаёт на 2 сообщения)
Consumer Group "audit-log":        offset = 0 (только запустилась, читает с начала)
```

Consumer самостоятельно **коммитит (commit) оффсет** — сохраняет в служебный топик Kafka `__consumer_offsets`, сигнализируя "я обработал до этого места". Подробнее о стратегиях коммита — в статье о Kafka в Node.js.

## Продюсер (Producer)

**Продюсер (producer)** — клиент, который пишет сообщения в топик.

Продюсер решает:
1. **В какой топик** писать (обязательно)
2. **С каким ключом** (key) — влияет на то, в какой партишн попадёт сообщение
3. **С какими заголовками (headers)** — метаданные (версия схемы, trace ID и т.п.)

```txt
Producer → [topic: "order-placed", key: "user-123", value: { orderId: 42, ... }]
                │
                ▼
         Kafka выбирает партишн:
         - key задан → hash(key) % numPartitions → всегда один партишн для одного ключа
         - key = null → round-robin по партишнам (равномерное распределение)
```

Ключ — это не уникальный идентификатор сообщения. Это инструмент **управления партишнированием**: все сообщения с одним ключом гарантированно попадают в один партишн → их порядок гарантирован.

## Консьюмер (Consumer)

**Консьюмер (consumer)** — клиент, который читает сообщения из топика. Consumer подписывается на топик и Kafka назначает ему партишны.

Consumer'ы работают в **pull-модели**: consumer сам опрашивает брокер (`poll()`), а не брокер "толкает" (push) сообщения к consumer'у. Это отличие от некоторых других систем и от HTTP webhooks. Pull-модель позволяет consumer'у читать в собственном темпе и не перегружаться.

```ts
// Упрощённая схема работы consumer'а (псевдокод)
while (true) {
  const messages = consumer.poll(timeout: 1000ms); // pull у брокера
  for (const message of messages) {
    await processMessage(message);
  }
  consumer.commitOffset(); // сохранить прогресс
}
```

## Consumer Group — как Kafka объединяет pub/sub и queue в одном

**Consumer group (группа консьюмеров)** — это именованная группа consumer-процессов, которые читают один топик сообща.

Правило распределения партишнов: **каждый партишн топика назначается ровно одному consumer'у внутри группы**.

Это правило — ключ к пониманию того, как Kafka реализует два принципиально разных паттерна в зависимости от конфигурации:

### Паттерн 1: Параллельная обработка (queue-like)

Одна группа, несколько consumer'ов — каждый читает свой партишн:

```txt
Topic "orders" (3 партишна), Consumer Group "order-processor":

Partition 0 ──► Consumer A  (обрабатывает треть всех заказов)
Partition 1 ──► Consumer B  (обрабатывает треть всех заказов)
Partition 2 ──► Consumer C  (обрабатывает треть всех заказов)

→ Горизонтальное масштабирование: добавить ещё один consumer → добавить партишн → нагрузка делится
→ Каждое сообщение обрабатывается РОВНО ОДНИМ consumer'ом из группы
```

Это аналог классической очереди с несколькими воркерами. Но с важным отличием: если consumer A падает, его партишны перераспределяются на B и C (rebalance) — автоматически.

### Паттерн 2: Pub/Sub (каждый получает копию)

Несколько групп, каждая получает все сообщения:

```txt
Topic "order-placed" (3 партишна):

Consumer Group "search-service":
  Partition 0 ──► Search Consumer 1
  Partition 1 ──► Search Consumer 2
  Partition 2 ──► Search Consumer 3

Consumer Group "analytics-service":
  Partition 0 ──► Analytics Consumer 1
  Partition 1 ──► Analytics Consumer 2
  Partition 2 ──► Analytics Consumer 3

→ Каждая группа получает ВСЕ сообщения топика
→ Группы абсолютно независимы: их offset'ы не связаны
```

Это pub/sub: одно событие "заказ размещён" → и search, и analytics получают его полностью.

### Лимит параллелизма

Максимальное число "полезных" consumer'ов в группе = число партишнов. Если consumer'ов больше, лишние сидят без дела:

```txt
3 партишна, 5 consumer'ов в одной группе:

Partition 0 ──► Consumer A  ✓ активен
Partition 1 ──► Consumer B  ✓ активен
Partition 2 ──► Consumer C  ✓ активен
(нет партишна) Consumer D  ✗ idle (не получает сообщений)
(нет партишна) Consumer E  ✗ idle (не получает сообщений)
```

Consumer D и E готовы взять на себя партишн при отказе A, B или C — это резерв. Но больше партишнов, чем есть в топике, добавить нельзя без изменения топика.

## Брокер (Broker)

**Брокер (broker)** — это один узел (сервер/процесс) Kafka. Брокер принимает запросы от продюсеров, хранит партишны на диске, отдаёт данные consumer'ам.

Один брокер может хранить сотни партишнов от разных топиков. Брокеры взаимодействуют между собой для репликации данных.

```txt
Broker — это:
  - Процесс kafka (JVM)
  - Хранилище партишнов (сегменты файлов на диске)
  - Сервер для продюсеров и consumer'ов
  - Участник репликации

Broker — это НЕ:
  - Отдельная база данных
  - Знающий о состоянии consumer'ов (кроме offset'ов в __consumer_offsets)
  - "Умный" маршрутизатор (это делает клиент)
```

## Кластер (Cluster)

**Кластер (cluster)** — это группа брокеров, работающих вместе. Один из брокеров исторически (при ZooKeeper) выполнял роль Controller — координировал выборы лидеров партишнов и отслеживал состояние кластера.

```txt
Минимальный production-кластер (3 брокера):

  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ Broker 1 │◄──►│ Broker 2 │◄──►│ Broker 3 │
  │          │    │(Controller│    │          │
  └──────────┘    └──────────┘    └──────────┘
        │              │               │
        └──────────────┴───────────────┘
                    Репликация

  3 брокера → кластер переживает отказ 1 брокера (при replication factor = 3)
```

Почему минимум 3? Принцип кворума (quorum): для согласования нужно большинство (2 из 3). С 2 брокерами кластер не переживёт потерю одного — нет кворума.

## ZooKeeper vs KRaft — что изменилось и почему

Исторически Kafka использовала **ZooKeeper** (Apache ZooKeeper) — отдельный координационный сервис — для:
- Хранения метаданных кластера (какие брокеры существуют, какой партишн на каком брокере)
- Выборов лидера Controller
- Хранения конфигураций топиков

```txt
Kafka с ZooKeeper (до Kafka 3.x):

  ┌─────────────────────┐    ┌──────────┐
  │    ZooKeeper        │    │ Broker 1 │
  │   (отдельный        │◄──►│ Broker 2 │
  │    кластер!)        │    │ Broker 3 │
  └─────────────────────┘    └──────────┘
  
  → Нужно поднимать и поддерживать 2 отдельных кластера
  → ZooKeeper становился узким местом при больших кластерах
  → Усложнённые операции: смена конфигурации требовала ZooKeeper
```

**KRaft** (Kafka Raft, KIP-500) — встроенный протокол консенсуса Kafka, который заменяет ZooKeeper:

```txt
Kafka с KRaft (Kafka 3.3+, default с Kafka 4.0):

  ┌──────────────────────────────────────────────┐
  │           Kafka Cluster                       │
  │                                               │
  │  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
  │  │ Broker 1 │  │ Broker 2 │  │ Broker 3 │   │
  │  │(Controller│  │          │  │          │   │
  │  │  mode)   │  │          │  │          │   │
  │  └──────────┘  └──────────┘  └──────────┘   │
  │                                               │
  │  Метаданные хранятся внутри кластера          │
  │  в выделенном внутреннем топике              │
  └──────────────────────────────────────────────┘
  
  → Один кластер вместо двух
  → Faster controller failover (миллисекунды vs секунды)
  → Поддержка миллионов партишнов (ZooKeeper масштабировался плохо)
```

**Практически**: если вы запускаете новый проект сегодня — используйте KRaft. ZooKeeper deprecated и будет удалён в Kafka 4.x. В большинстве managed-сервисов (Confluent Cloud, AWS MSK) переход на KRaft уже произошёл прозрачно для пользователя.

## Типичные ошибки на интервью

**"Consumer Group — это несколько consumer'ов, которые дублируют друг друга"**

Нет. Внутри группы каждый партишн назначен ровно одному consumer'у — дублирования нет. Дублирование возникает между группами: группа A и группа B оба получают все сообщения топика, но это не дублирование, а pub/sub паттерн.

**"Offset — это глобальный счётчик по всему топику"**

Нет. Offset уникален в пределах партишна. У топика с 3 партишнами есть три независимых последовательности offset'ов: 0..N в partition 0, 0..M в partition 1, 0..K в partition 2. Нет единого "offset 42 в топике".

**"Добавить consumer'ов в группу → линейный рост производительности"**

Только до числа партишнов. После этого — лишние consumer'ы просто ждут. Реальный рост производительности = увеличение числа партишнов (плюс consumer'ов).

**"В одном партишне несколько consumer'ов из одной группы читают параллельно"**

Нет. Одному партишну — один consumer в группе. Это ограничение дизайна: именно оно гарантирует порядок обработки внутри партишна.

**"ZooKeeper — часть Kafka, которую всегда надо поднимать"**

Устаревшая информация. С Kafka 3.3+ KRaft стал production-ready и рекомендуется по умолчанию. С Kafka 4.0 ZooKeeper-режим удалён.
