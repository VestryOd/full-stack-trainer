# Что такое Kafka на самом деле

## Один вопрос, который путает большинство

"Kafka — это очередь сообщений, как RabbitMQ, только быстрее?"

**Нет.** Это самое распространённое заблуждение, и именно с него начинают ломать голову на интервью. Kafka — это **распределённый лог (distributed log)**. Разница не терминологическая — она архитектурная, и влияет на всё: как данные хранятся, кто и когда их читает, что происходит после чтения.

Чтобы понять, зачем Kafka вообще появилась, нужно сначала понять, какую проблему она решает — и почему RabbitMQ с этой проблемой не справляется.

## Проблема, которую не решает очередь

Представьте LinkedIn в 2010 году. Пользователь обновляет профиль. Это событие нужно:
- Проиндексировать в поиске (немедленно)
- Обновить рекомендации связей (фоново, через секунды)
- Отправить аналитику поведения (фоново, через минуты)
- Добавить в журнал аудита (долгосрочное хранение)

С классической очередью (RabbitMQ, SQS) это выглядит так:

```txt
[Profile Service]
     │
     ├──► [Queue: search-index]  ──► [Search Consumer]    (читает, удаляет)
     ├──► [Queue: recommendations] ──► [Reco Consumer]    (читает, удаляет)
     ├──► [Queue: analytics]    ──► [Analytics Consumer]  (читает, удаляет)
     └──► [Queue: audit-log]    ──► [Audit Consumer]      (читает, удаляет)
```

Проблемы:
1. **Масштабирование потребителей**: появляется новый сервис (например, ML-модель для ранжирования) — нужна новая очередь, и Profile Service должен знать о ней и писать в неё
2. **Нет replay**: сервис analytics упал на 2 часа → все события за это время потеряны, в очереди их уже нет (сообщение удаляется после подтверждения (ack))
3. **Отладка и аудит**: если завтра обнаружится баг в Search Consumer, нет способа переиграть события с нуля — они уже ушли из очереди

Это и есть проблема, которую решает Kafka.

## Kafka как распределённый лог

Kafka хранит сообщения не в очереди, которая опустошается при чтении, а в **логе** (log) — последовательном, неизменяемом файле записей. Сообщение остаётся в логе после того, как его прочитали. Оно остаётся там, пока не истечёт настроенный срок хранения (по умолчанию 7 дней, но это конфигурируется вплоть до "хранить вечно").

```txt
Очередь (RabbitMQ, SQS):          Лог (Kafka):
┌─────────────────────────┐        ┌─────────────────────────────────┐
│ [msg1][msg2][msg3][msg4]│        │ [msg1][msg2][msg3][msg4][msg5] │
└─────────────────────────┘        └─────────────────────────────────┘
Consumer читает msg1 →               Consumer A читает msg1 (offset=0)
                                     Consumer B читает msg1 (offset=0)
┌──────────────────────┐             Consumer A потом читает msg3 (offset=2)
│ [msg2][msg3][msg4]   │             Сообщение msg1 ОСТАЁТСЯ в логе
└──────────────────────┘
msg1 УДАЛЁН из очереди
```

Это меняет всё:
- **Любой новый потребитель может читать с самого начала** — можно добавить новый сервис и "воспроизвести" всю историю событий
- **Независимые потребители читают независимо** — Search Service и Analytics Service читают один и тот же лог, не мешая друг другу
- **Replay после сбоя** — если Analytics Service упал, он просто продолжит с того offset, на котором остановился

```txt
Kafka Topic "user-profile-updated":

  offset: 0        1        2        3        4
         [evt-A] [evt-B] [evt-C] [evt-D] [evt-E]

  Search Consumer ────────────────────────── читает offset 4 (почти в реальном времени)
  Reco Consumer ──────────────────── читает offset 3 (немного отстаёт)
  Analytics Consumer ────── читает offset 1 (упал, восстанавливается с offset=1)
  Новый ML-сервис ─────── читает offset 0 (запущен только что, догоняет всю историю)
```

## "Умный потребитель, тупой брокер" — в чём философия

В RabbitMQ брокер умный: он знает, кому доставить сообщение, отслеживает подтверждения (ack), управляет маршрутизацией через exchange, знает о состоянии каждой очереди.

Kafka устроена принципиально иначе — это называют **"dumb broker, smart consumer"** (тупой брокер, умный потребитель):

```txt
RabbitMQ — умный брокер:           Kafka — умный потребитель:
┌──────────────────────┐            ┌─────────────┐
│       Broker         │            │   Broker    │
│  - Знает о consumers │            │  - Хранит   │
│  - Маршрутизирует    │            │    лог      │
│  - Удаляет после ack │            │  - Больше   │
│  - Отслеживает ack   │            │    ничего   │
└──────────────────────┘            └─────────────┘
                                          │
                                    Consumer сам:
                                    - помнит свой offset
                                    - решает, когда коммитить
                                    - читает в своём темпе
                                    - перечитывает при необходимости
```

**Broker (брокер)** — узел Kafka, который принимает и хранит данные. Он не знает, прочитал ли consumer сообщение — брокер просто хранит лог.

**Consumer (потребитель)** сам отслеживает, до какого места он прочитал лог. Это число называется **offset** — порядковый номер следующего сообщения, которое нужно прочитать. Consumer коммитит (сохраняет) свой offset самостоятельно, и именно это даёт возможность replay: достаточно сбросить offset назад.

Практический результат этой философии — Kafka **горизонтально масштабируется** гораздо лучше: брокер не хранит состояние per-consumer, он просто последовательно пишет и читает данные с диска (что очень быстро из-за sequential I/O).

## Почему Kafka так быстра: sequential I/O

Kafka пишет все сообщения **последовательно в конец файла**. Последовательная запись на обычный HDD быстрее случайной в 100–1000 раз. Kafka использует механизм ОС под названием `sendfile` (zero-copy): данные передаются прямо с дискового буфера в сетевой сокет, минуя копирование в пространство пользователя.

```txt
Традиционное чтение файла + отправка:
  Диск → kernel buffer → user buffer → kernel socket buffer → сеть
                         (копирование в userspace — дорого)

Kafka с zero-copy (sendfile):
  Диск → kernel buffer ──────────────────────────────────► сеть
         (данные никогда не покидают ядро ОС — быстро)
```

Это не академическая деталь — именно это позволяет одному брокеру Kafka обрабатывать миллионы сообщений в секунду на обычном железе.

## Когда Kafka — правильный выбор

Kafka решает конкретный класс задач лучше традиционных очередей:

```txt
Kafka хорошо подходит для:
  ✓ Event streaming — поток событий, который нужно читать нескольким сервисам
  ✓ Replay — воспроизведение событий (отладка, новые сервисы, восстановление)
  ✓ Высокий throughput — миллионы сообщений в секунду
  ✓ Долгосрочное хранение событий как источника правды (event sourcing)
  ✓ Log aggregation — сбор логов из множества сервисов
  ✓ Change Data Capture (CDC) — стриминг изменений из базы данных
  ✓ Real-time аналитика — несколько consumer'ов читают один поток по-разному

Kafka — плохой выбор для:
  ✗ Простые task queues (отправить письмо, изменить размер изображения)
  ✗ RPC-подобные паттерны (запрос → ответ с результатом)
  ✗ Маршрутизация на основе содержимого сообщения (routing by content)
  ✗ Команды с немедленным результатом ("обработай это и верни статус")
  ✗ Маленькие команды, где отдельные шаги транзакционно связаны
```

## Типичные ошибки на интервью

**"Kafka — это продвинутый RabbitMQ"**

Неправильная постановка. Это разные инструменты с разной философией. RabbitMQ — брокер сообщений (message broker): получил, маршрутизировал, доставил, удалил. Kafka — лог событий (event log / event streaming platform): получил, сохранил в порядке, дал читать всем желающим столько раз, сколько нужно. Нельзя сказать, что одно "лучше" другого — нужно объяснять, для каких задач.

**"В Kafka сообщение удаляется после прочтения"**

Нет. Это именно то, чего Kafka НЕ делает — в этом её ключевое отличие от очереди. Сообщение удаляется только по истечении срока хранения (`retention.ms`) или при превышении объёма (`retention.bytes`). Потребитель только двигает свой offset вперёд, не удаляя данные.

**"Kafka гарантирует глобальный порядок сообщений"**

Kafka гарантирует порядок только внутри одного партишна (partition). Если топик имеет несколько партишнов, глобальный порядок НЕ гарантируется. Это фундаментальная компромисс между порядком и параллелизмом — подробнее в статье о партишнах.

**"Offset — это глобальный счётчик"**

Offset уникален в пределах конкретного партишна, а не всего топика. Offset 5 в partition 0 и offset 5 в partition 1 — это два разных сообщения.
