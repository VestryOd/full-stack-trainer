## Группа 1: Принципы SOLID

**1. Что на самом деле означает SRP — и в чём типичное заблуждение?**

SRP говорит, что класс должен иметь одну *причину для изменения*, а не один *метод*. Заблуждение: дробить всё на классы с одним методом. Правильное прочтение: "кто из стейкхолдеров может запросить изменение этого класса?". `UserService`, который и отправляет email, и работает с БД, имеет двух стейкхолдеров (маркетинг vs. команда БД) → две причины для изменения → нарушение SRP. Класс с 10 методами, все вокруг персистентности пользователя, — одна причина (схема/запросы) → SRP в порядке.

---

**2. Объясните OCP на реальном примере, а не через "открыт для расширения, закрыт для изменения".**

OCP: добавлять поведение не редактируя существующий код. Классическое нарушение — `switch` по типу, который растёт с каждым новым типом. Исправление: выделить интерфейс `Notification`, добавить `EmailNotification`, `SmsNotification`, `PushNotification`. Новый канал = новый класс, ноль правок в существующем коде. В TypeScript:

```ts
interface Notifier { send(message: string): Promise<void>; }

class EmailNotifier implements Notifier { /* ... */ }
class SmsNotifier  implements Notifier { /* ... */ }

// NotificationService никогда не меняется при добавлении нового канала
class NotificationService {
  constructor(private notifiers: Notifier[]) {}
  async notify(msg: string) {
    await Promise.all(this.notifiers.map(n => n.send(msg)));
  }
}
```

---

**3. Что такое LSP и что он означает на практике?**

LSP: подтип должен быть взаимозаменяем с базовым типом без нарушения корректности программы. Классическое нарушение: `Square extends Rectangle` — установка ширины меняет высоту (чтобы квадрат оставался квадратом), что ломает код, ожидающий независимых измерений. Тест: если нужна проверка `instanceof` или `try/catch` вокруг метода базового класса при вызове подкласса — LSP нарушен.

```ts
// ❌ Нарушение: Square ломает контракт Rectangle
class Rectangle {
  setWidth(w: number)  { this.width = w; }
  setHeight(h: number) { this.height = h; }
  area() { return this.width * this.height; }
}
class Square extends Rectangle {
  setWidth(w: number)  { this.width = this.height = w; } // ломает ожидание
}

// ✅ Исправление: раздельные абстракции
interface Shape { area(): number; }
class Rectangle implements Shape { /* независимые ширина и высота */ }
class Square    implements Shape { /* одна сторона */ }
```

**4. ISP — когда "жирный интерфейс" создаёт реальные проблемы?**

ISP: клиенты не должны зависеть от методов, которые они не используют. Реальная проблема: интерфейс `IRepository` с методами `findOne`, `findAll`, `create`, `update`, `delete`, `bulkInsert`, `runReport`. Модуль отчётности, которому нужны только `findAll` и `runReport`, вынужден зависеть от методов мутации — любое изменение `create` требует перекомпиляции/перетестирования модуля отчётности. Решение: разбить на `IReadRepository`, `IWriteRepository`, `IReportRepository`.

---

**5. DIP — в чём разница между Dependency Inversion и Dependency Injection?**

DIP — это *принцип*: модули высокого уровня не должны зависеть от модулей низкого уровня — оба должны зависеть от абстракций. DI — это *механизм* реализации DIP: вместо `new EmailService()` внутри класса зависимость передаётся через конструктор. DI без DIP возможно (инжектировать конкретный класс — всё ещё тесная связь). DIP без DI — возможно, но неудобно. В NestJS, `@Injectable()` + `@Module` даёт DI; зависимость от токена интерфейса (не класса) даёт DIP.

```ts
// ❌ Нарушение DIP — зависимость от конкретного класса
class OrderService {
  private mailer = new SendGridMailer(); // жёстко задано
}

// ✅ DIP через DI — зависимость от абстракции
interface IMailer { sendEmail(to: string, body: string): Promise<void>; }

class OrderService {
  constructor(private mailer: IMailer) {} // инжектируется, не инстанцируется
}
```

## Группа 2: Порождающие паттерны

**6. Какую проблему решает Factory Method, которую не решает обычный `new`?**

`new ConcreteClass()` связывает вызывающий код с конкретной реализацией. Factory Method развязывает: вызывающий код работает с базовым типом, а подклассы или фабричные функции решают, что инстанцировать. В Node.js, `http.createServer()` и `https.createServer()` — фабричные методы, возвращающие разные реализации сервера через один интерфейс — вызывающий код не знает и не должен знать, какой транспорт используется.

---

**7. Когда Builder выгоднее конструктора с опциональными параметрами?**

Когда объект требует более 3-4 опциональных параметров, или когда конструирование включает кросс-валидацию полей (нельзя иметь одновременно `oauth` и `password`). Builder также задаёт порядок шагов (паттерн Director) и позволяет создавать разные "конфигурации" одного объекта из одного builder'а.

```ts
class QueryBuilder {
  private table = '';
  private conditions: string[] = [];
  private limitVal?: number;

  from(table: string)   { this.table = table; return this; }
  where(cond: string)   { this.conditions.push(cond); return this; }
  limit(n: number)      { this.limitVal = n; return this; }

  build(): string {
    let q = `SELECT * FROM ${this.table}`;
    if (this.conditions.length) q += ` WHERE ${this.conditions.join(' AND ')}`;
    if (this.limitVal)          q += ` LIMIT ${this.limitVal}`;
    return q;
  }
}

const query = new QueryBuilder().from('users').where('age > 18').limit(10).build();
```

**8. Singleton — какие проблемы он создаёт в контексте Node.js, и как их смягчить?**

Синглтоны в Node.js — это синглтоны на уровне модуля (кэш модулей держит один экземпляр). Проблемы: (1) скрытое глобальное состояние — сложно тестировать (тесты загрязняют друг друга); (2) модульный синглтон — один на процесс, не на запрос — stateful синглтоны это баг конкурентности. Смягчение: dependency injection вместо `module.exports = new Service()` — каждый тест создаёт свой экземпляр; DI-контейнер управляет временем жизни.

## Группа 3: Структурные паттерны

**9. Adapter vs Decorator — объясните на конкретном TypeScript-примере.**

Adapter конвертирует *несовместимый* интерфейс в ожидаемый (нельзя изменить ни одну из сторон). Decorator добавляет *поведение* к существующему совместимому интерфейсу, не изменяя класс.

```ts
// Adapter — у легаси-логгера другая сигнатура
interface ILogger { log(level: string, msg: string): void; }
class LegacyLogger { write(msg: string) { console.log(msg); } }

class LoggerAdapter implements ILogger {
  constructor(private legacy: LegacyLogger) {}
  log(level: string, msg: string) {
    this.legacy.write(`[${level}] ${msg}`);
  }
}

// Decorator — добавить временную метку, не меняя оригинал
class TimestampLogger implements ILogger {
  constructor(private inner: ILogger) {}
  log(level: string, msg: string) {
    this.inner.log(level, `${new Date().toISOString()} ${msg}`);
  }
}
```

**10. Что делает Facade и где он встречается в React/Node.js-кодовых базах?**

Facade предоставляет упрощённый интерфейс над сложной подсистемой. В React: кастомный хук (`useAuth`), скрывающий логику обновления токена, обработки редиректов и проверки прав за `{ user, login, logout }`. В Node.js: `PaymentService`, координирующий `StripeClient`, `InvoiceRepository`, `EmailService` и `AuditLogger` за единственным методом `charge(orderId, amount)`.

**11. Паттерн Proxy — три основных сценария использования.**

(1) **Virtual proxy** — ленивая загрузка (инстанциировать реальный объект только при первом обращении). (2) **Protection proxy** — контроль доступа (проверка прав перед делегированием). (3) **Logging/caching proxy** — перехват вызовов для добавления сквозного поведения. Нативный объект `Proxy` в JavaScript — это паттерн, реализованный на уровне языка. MobX использует JS Proxy как virtual proxy над observable-объектами для перехвата чтений и записей при реактивном отслеживании.

```ts
function withCache<T extends object>(target: T, ttlMs: number): T {
  const cache = new Map<string | symbol, { value: unknown; expiresAt: number }>();
  return new Proxy(target, {
    get(obj, key) {
      const cached = cache.get(key);
      if (cached && Date.now() < cached.expiresAt) return cached.value;
      const value = (obj as any)[key];
      cache.set(key, { value, expiresAt: Date.now() + ttlMs });
      return value;
    },
  });
}
```

## Группа 4: Поведенческие паттерны

**12. Observer vs EventEmitter vs Pub/Sub — в чём разница?**

Observer: субъект держит прямые ссылки на наблюдателей, вызывает их синхронно. EventEmitter (Node.js): именованные события, по умолчанию синхронный, слабая связность (эмиттер не знает, кто слушает). Pub/Sub: брокер посередине (Redis, RabbitMQ), publisher и subscriber в разных процессах, полная развязка, асинхронность. Каждый шаг добавляет развязку и инфраструктуру.

---

**13. Паттерн Strategy — реальный кейс, а не учебный пример с сортировкой.**

Стратегии аутентификации: интерфейс `PassportStrategy` с методом `validate(token)`. `JwtStrategy`, `GoogleOAuthStrategy`, `ApiKeyStrategy` реализуют его. `AuthGuard` не знает, какую стратегию использует — она инжектируется. Добавить новый метод аутентификации = добавить класс, ноль изменений в guard. Именно так работает `passport.js` внутри.

```ts
interface AuthStrategy {
  validate(token: string): Promise<User | null>;
}

class JwtStrategy implements AuthStrategy {
  async validate(token: string) { /* верифицировать JWT */ }
}

class ApiKeyStrategy implements AuthStrategy {
  async validate(token: string) { /* найти API-ключ в БД */ }
}

class AuthGuard {
  constructor(private strategy: AuthStrategy) {}
  async authenticate(token: string) {
    const user = await this.strategy.validate(token);
    if (!user) throw new UnauthorizedException();
    return user;
  }
}
```

**14. Паттерн Command — почему он позволяет сделать undo/redo, и где используется на фронтенде?**

Command инкапсулирует действие как объект с методами `execute()` и опционально `undo()`. Поскольку объект команды знает, как себя отменить, стек истории даёт undo/redo бесплатно. На фронтенде: объекты action в Redux — это Commands. Они сериализуемы, диспатчатся, воспроизводимы. Time-travel в Redux DevTools работает, потому что каждое изменение состояния — Command, хранящийся в массиве.

---

**15. Template Method vs Strategy — когда что применять?**

Template Method: определить *скелет* алгоритма в базовом классе, позволить подклассам переопределять отдельные шаги. Связь — через наследование. Strategy: инкапсулировать *весь* алгоритм как заменяемый объект. Связь — через композицию. Правило: если шаги неизменны и кастомизируются лишь несколько — Template Method. Если весь алгоритм нужно менять во время выполнения или на уровне экземпляра — Strategy. Template Method нарушает OCP, когда новая вариация требует правок в базовом классе; Strategy — нет.

---

**16. Что такое Chain of Responsibility и где он встречается в backend-коде?**

Запрос проходит цепочку обработчиков; каждый либо обрабатывает его, либо передаёт следующему. Встречается в: Express/Koa middleware, NestJS guards + interceptors + pipes + exception filters (каждый — звено цепочки), AWS Lambda handler chains. Ключевое свойство: отправитель не знает, какой обработчик обработает запрос, обработчики можно добавлять/удалять без изменения отправителя.

## Группа 5: Распознавание паттернов в реальном коде

**17. Посмотрите на этот код — назовите паттерн(ы) и объясните, что не так:**

```ts
class ReportGenerator {
  generate(type: string, data: unknown[]) {
    if (type === 'pdf')   return this.generatePdf(data);
    if (type === 'excel') return this.generateExcel(data);
    if (type === 'csv')   return this.generateCsv(data);
    throw new Error('Unknown type');
  }
  private generatePdf(data: unknown[])   { /* ... */ }
  private generateExcel(data: unknown[]) { /* ... */ }
  private generateCsv(data: unknown[])   { /* ... */ }
}
```

Это нарушение OCP и пропущенная возможность для Strategy / Factory Method. Каждый новый тип отчёта требует редактирования этого класса. Исправление: выделить интерфейс `ReportStrategy`, перенести каждый формат в отдельный класс, инжектировать или искать по типу. В `ReportGenerator` тогда будет ноль `if`-выражений и он никогда не потребует изменений.

**18. Где паттерны проектирования используются в самом React (не в пользовательском коде)?**

- **Composite**: дерево React-элементов — каждый узел либо лист (`<span>`), либо композит (`<div>` с дочерними). `React.Children` работает с обоими одинаково.
- **Observer**: `useContext` + `createContext` — React автоматически распространяет изменения контекста на всех потребителей (наблюдателей).
- **Proxy**: система Synthetic Events — React оборачивает нативные DOM-события в единый объект-прокси.
- **Template Method**: жизненный цикл `React.Component` — `render()` абстрактный (обязан быть переопределён), хуки жизненного цикла (`componentDidMount` и т.д.) — опциональные шаги.
- **HOC (Higher-Order Component)**: паттерн Decorator — оборачивает компонент для добавления поведения без его изменения.

## Группа 6: Анти-паттерны

**19. Что такое "God Object" и какой принцип SOLID он нарушает?**

God Object знает слишком много и делает слишком много — накапливает бизнес-логику, доступ к данным и представление. Нарушает SRP (много причин для изменения) и обычно OCP (нельзя расширить без модификации). В Node.js: контроллер на 1000 строк, который валидирует ввод, делает запросы к БД, отправляет email, пишет audit-лог и форматирует ответ. Решение: разбить по ответственности на сервисы.

---

**20. Что такое "Shotgun Surgery" и как это связано с SRP?**

Shotgun Surgery: одно логическое изменение требует правок во многих разных классах одновременно. Обычно вызывается ответственностью, разбросанной по многим местам (обратное God Object). Добавление нового поля в ответ API означает правку контроллера, DTO, маппера, тестовой фикстуры и документации по отдельности. Решение: определить разбросанную ответственность и локализовать её в одном месте.

---

**21. Что такое "Primitive Obsession" и как исправить?**

Использование сырых примитивов (string, number) там, где доменная концепция заслуживает собственного типа. Классический пример: email как `string` везде — валидация дублируется, опечатки невидимы. Решение: value object `Email`, который валидирует при создании и передаётся по типу:

```ts
// ❌ Primitive obsession
function sendWelcome(email: string) { /* email может быть "" или "not-an-email" */ }

// ✅ Value object
class Email {
  readonly value: string;
  constructor(raw: string) {
    if (!/.+@.+\..+/.test(raw)) throw new Error(`Некорректный email: ${raw}`);
    this.value = raw.toLowerCase();
  }
}
function sendWelcome(email: Email) { /* всегда валиден по конструкции */ }
```

---

**22. Что такое "Anemic Domain Model" и когда он допустим?**

Anemic Domain Model: классы — контейнеры данных (геттеры/сеттеры) без поведения; вся логика в сервисах. Нарушает принцип OOP, что объекты должны иметь и состояние, и поведение. Однако *допустим*: в CRUD-приложениях с минимальной бизнес-логикой, для DTO (они намеренно — контейнеры данных), в функциональном стиле, где данные и функции намеренно разделены. Метка "анти-паттерн" применима, когда анемичная модель используется в контексте с насыщенной доменной логикой и логика размазывается по сервисам.

## Группа 7: Паттерны в React и Node.js

**23. Что такое Compound Component в React и на какой GoF-паттерн он похож?**

Compound Components: родительский компонент управляет состоянием и передаёт его через Context; дочерние компоненты — отдельные экспорты, потребляющие этот контекст неявно. Пользователь компонует их:

```tsx
<Select value={val} onChange={setVal}>
  <Select.Trigger />
  <Select.Options>
    <Select.Option value="a">Вариант A</Select.Option>
  </Select.Options>
</Select>
```

Похож на **Composite** (родитель + дочерние свободно компонуются) и **Mediator** (родитель посредничает между дочерними, которые ничего не знают друг о друге).

---

**24. Как модульная система NestJS реализует Dependency Inversion?**

NestJS-модули объявляют `providers` (что создают) и `exports` (чем делятся). Другие модули объявляют `imports`, чтобы потреблять экспортированное. DI-контейнер разрешает граф зависимостей при старте. Ключевой момент DIP: можно зарегистрировать провайдер как токен интерфейса (`{ provide: IMailer, useClass: SendGridMailer }`) — потребитель зависит от токена (абстракции), не от конкретного класса. Замена `SendGridMailer` на `ResendMailer` в тестах или для другого окружения не требует никаких изменений в потребляющем классе.

**25. Где паттерн Repository встречается в кодовой базе Prisma + NestJS, и почему он спорен?**

Repository: абстракция над доступом к данным, которая выглядит как коллекция в памяти с точки зрения домена. В NestJS + Prisma: класс `UserRepository`, оборачивающий `prisma.user.*` и предоставляющий `findById`, `save`, `delete`. Почему спорен: клиент Prisma уже является хорошо типизированной абстракцией над доступом к данным — добавление слоя Repository может быть избыточным бойлерплейтом без реальной выгоды для простых случаев. Паттерн *оправдан* когда: (1) интерфейс Repository позволяет тест-даблы без реальной БД; (2) домен сложный и нужно скрыть детали Prisma от доменной логики; (3) нужно поддерживать несколько источников данных за одним интерфейсом.

## Типичные follow-up вопросы (Группа 7)

```txt
"Render Props vs HOC vs Custom Hooks — что победило в 2025?" →
  Custom Hooks — компонуются без wrapper hell, естественно работают
  с TypeScript, не добавляют узлы в дерево компонентов. HOC полезны
  для классовых компонентов (легаси) и сторонних компонентов, которые
  нельзя изменить. Render Props в основном заменены хуками.

"Является ли useReducer паттерном?" →
  Да — это паттерн Command (actions — команды) в сочетании с конечным
  автоматом (reducer — функция перехода). Для сложного локального
  состояния useReducer предпочтительнее useState, потому что переходы
  состояния явные и тестируемые как чистые функции.
```
