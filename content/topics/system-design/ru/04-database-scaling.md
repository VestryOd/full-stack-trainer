# Database Scaling

## Почему именно БД — самое частое узкое место

App-серверы stateless, их легко масштабировать горизонтально — добавил инстансы за load balancer'ом, и готово. БД — это, по определению, **stateful** компонент: данные физически живут в одном месте, и просто "добавить ещё одну БД" не работает так же просто, как с app-сервером, потому что нужно решить, **где** какие данные находятся и как поддерживать консистентность между копиями.

Путь масштабирования БД почти всегда идёт в таком порядке, и каждый следующий шаг сложнее предыдущего:

```txt
1. Vertical scaling (больше RAM/CPU/диска под БД)
2. Индексы и оптимизация запросов (часто решает 80% проблем без инфраструктурных изменений)
3. Read Replicas (масштабирование чтения)
4. Caching layer (см. отдельную тему)
5. Sharding / Partitioning (масштабирование записи и объёма данных)
6. Database per service / CQRS (архитектурное разделение)
```

Senior-ответ начинается не с шардинга, а с замечания, что **большая часть "проблем с БД" на практике решается индексами, нормализацией запросов и кешем** — переход к шардингу — это значимая архитектурная инвестиция со своими долгосрочными издержками, и её стоит делать только когда более простые шаги исчерпаны.

## Read Replicas — синхронная vs асинхронная репликация

```txt
Write
 ↓
Primary DB ──replication──→ Replica 1
                ──────────→ Replica 2
                ──────────→ Replica 3

App: записи → Primary
     чтения → Replicas (round-robin / least-lag)
```

Это работает, потому что в типичных read-heavy системах read:write часто 10:1-1000:1 (см. [System Design Fundamentals]) — несколько read-реплик снимают основную нагрузку с primary, оставляя ему только записи.

**Синхронная репликация**: primary ждёт подтверждения от реплики(к) перед тем как считать запись успешной.

```txt
Плюсы: реплика гарантированно содержит последние данные (zero data loss при failover)
Минусы: latency записи = latency самой медленной синхронной реплики;
        если реплика недоступна — запись может блокироваться
```

**Асинхронная репликация**: primary подтверждает запись клиенту немедленно, реплика применяет изменения "когда сможет".

```txt
Плюсы: запись быстрая, не зависит от состояния реплик
Минусы: Replication Lag — реплика временно отстаёт от primary
        (обычно мс, но под нагрузкой может быть секунды)
```

### Replication Lag и проблема "read-your-own-writes"

Классический сценарий, который почти всегда всплывает на интервью:

```txt
1. Пользователь обновляет профиль → запись идёт в Primary
2. Сразу после этого делает GET своего профиля → запрос идёт на Replica
3. Replication Lag = 200ms, Replica ещё не применила изменение
4. Пользователь видит СТАРЫЕ данные сразу после сохранения
```

Решения, каждое со своим trade-off:

```txt
- Читать сразу после записи у того же пользователя — с Primary
  (часто реализуется как "read-your-writes" routing: после write
  следующие N секунд читать у primary для этого пользователя/сессии)

- Read-after-write через session affinity к конкретной реплике,
  которая гарантированно получила обновление (сложно отследить)

- Версионирование: клиент передаёт "минимальную приемлемую версию данных",
  реплика, отставшая больше — не используется для этого запроса

- Принять staleness как часть UX (для большинства данных это нормально —
  "лайк" не обязан появляться мгновенно у самого автора)
```

Это не "теоретическая" проблема — она реально проявляется в продакшене как баг "сохранил — а старые данные показались" и часто становится причиной тикетов поддержки.

## Sharding (Partitioning) — разделение данных между узлами

Когда даже с репликами **объём записи** или **объём данных** превышает возможности одного узла, данные физически разделяются между независимыми БД-узлами (шардами), каждый из которых хранит свою часть данных.

```txt
Range-based sharding:
  Users A-M  → Shard 1
  Users N-Z  → Shard 2

  Плюсы: просто, легко выполнять range-запросы внутри одного шарда
  Минусы: неравномерное распределение (hot shard, если, например,
          много пользователей с фамилиями на одну букву)

Hash-based sharding:
  shard = hash(user_id) % N

  Плюсы: равномерное распределение
  Минусы: range-запросы ("все пользователи с ID 1000-2000") требуют
          обращения ко ВСЕМ шардам; ресharding при изменении N —
          массовое перемещение данных (см. consistent hashing
          в теме про Load Balancing)

Geo-based sharding:
  EU users → Shard EU, US users → Shard US

  Плюсы: соответствие требованиям data residency (GDPR и т.п.),
         низкая latency (данные ближе к пользователю)
  Минусы: cross-region запросы (например, друзья в разных регионах)
          становятся сложными
```

### Главная проблема шардинга: операции, которые раньше были "бесплатными", становятся дорогими

```txt
До шардинга (одна БД):              После шардинга:

JOIN users, orders                   JOIN между шардами — невозможен
  → один SQL-запрос                  на уровне БД, нужно делать
                                      на уровне приложения (несколько
                                      запросов + merge в коде)

Транзакция: списать со счёта A,      Распределённая транзакция —
зачислить на счёт B (один шард)      A и B могут быть в разных шардах
  → ACID гарантирует атомарность     → нужен 2-Phase Commit или Saga
                                      (со своими сложностями и latency)

SELECT COUNT(*) WHERE status = X     Нужно агрегировать результат
  → один запрос                      со всех шардов и просуммировать
                                      в приложении
```

**Выбор shard key — самое важное решение в шардинге**, и его почти невозможно изменить без полной миграции данных. Хороший shard key:

- обеспечивает равномерное распределение нагрузки (избегает hot shard — например, не стоит шардировать по `tenant_id`, если один tenant генерирует 90% трафика);
- покрывает большинство запросов (если 95% запросов идут с `user_id`, шардирование по `user_id` означает, что эти запросы попадают в один шард; но запросы "по order_id" теперь требуют похода во все шарды, если order_id не связан с user_id предсказуемо).

## SQL vs NoSQL — это не "какая БД лучше", а "какие гарантии нужны"

```txt
SQL (PostgreSQL, MySQL):
  + строгая схема, ACID-транзакции, JOIN, зрелая экосистема
  - горизонтальное масштабирование (шардинг) сложнее и менее "из коробки"

NoSQL — Document (MongoDB):
  + гибкая схема, удобно для denormalized/nested данных
  - нет JOIN на уровне БД, ACID обычно только в рамках одного документа

NoSQL — Key-Value (DynamoDB, Redis):
  + предсказуемая latency при больших объёмах, встроенный шардинг
  - запросы только по ключу (или ограниченным индексам)

NoSQL — Wide-Column (Cassandra):
  + write-heavy нагрузка, встроенная multi-region репликация
  - eventual consistency по умолчанию, модель данных заточена
    под конкретные query patterns (denormalization обязательна)
```

Senior-ответ на "SQL или NoSQL?" — это вопрос про **паттерн доступа к данным и требования к консистентности**, а не про "NoSQL масштабируется лучше" (современный PostgreSQL с репликами и партиционированием прекрасно держит огромные нагрузки — Instagram много лет работал на шардированном PostgreSQL).

## CQRS и Database per Service

**CQRS (Command Query Responsibility Segregation)** — разделение модели данных для записи и для чтения:

```txt
Write Model (нормализованная, оптимизирована для целостности данных)
  ↓ события/синхронизация
Read Model (денормализованная, оптимизирована под конкретные запросы UI,
            может быть в другой БД — например, Elasticsearch для поиска)
```

Это позволяет независимо масштабировать и оптимизировать пути чтения и записи, но добавляет задержку синхронизации между write- и read-моделью (обычно через очередь событий) — то есть осознанный переход к eventual consistency для чтения.

**Database per Service** (микросервисная практика) — у каждого сервиса своя БД, никто не лезет в чужую БД напрямую. Это даёт изоляцию (изменение схемы одного сервиса не ломает другие) и независимое масштабирование, но порождает классическую проблему: запрос, требующий данных из нескольких сервисов, требует **либо** нескольких запросов с merge на уровне API Gateway/BFF, **либо** денормализованной копии данных (синхронизируемой событиями) в сервисе, которому это нужно.

## Типичные ошибки на интервью

- **Предлагать шардинг как первый шаг масштабирования БД** — не упомянув индексы, read-реплики и кеш, которые решают значительно больший процент реальных проблем с гораздо меньшей сложностью.

- **Не упоминать replication lag и read-your-writes** при добавлении read-реплик — это один из самых ожидаемых follow-up вопросов, и его отсутствие в ответе сразу заметно.

- **Считать шардинг "бесплатным" масштабированием** — не проговаривая, что JOIN'ы, транзакции и агрегации становятся кратно сложнее.

- **Выбирать shard key без объяснения** — особенно не упоминая риск hot shard и проблему миграции при изменении схемы шардирования.

- **"NoSQL масштабируется, SQL — нет"** — упрощение, игнорирующее, что современный PostgreSQL с партиционированием и репликами масштабируется до очень больших объёмов, а выбор должен основываться на паттернах доступа и требованиях к консистентности.

- **CQRS как "просто разделить чтение и запись" без упоминания eventual consistency** между write- и read-моделью — это центральный trade-off паттерна.
