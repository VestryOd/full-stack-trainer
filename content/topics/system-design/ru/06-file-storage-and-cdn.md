# File Storage и CDN

## Почему "файл через backend" — это плохая архитектура

Наивная реализация загрузки файлов:

```txt
Frontend → Backend → File System / S3
```

Проблема не только в "нагрузке" в абстрактном смысле — она конкретна:

```txt
- Backend держит HTTP-соединение открытым на всё время загрузки
  (для видео в 1 GB это могут быть минуты) — занимает поток/worker,
  который мог бы обслуживать другие запросы
- Файл проходит через память/диск backend'а дважды:
  принять от клиента → переслать в S3
- Backend должен масштабироваться под нагрузку ЗАГРУЗКИ ФАЙЛОВ,
  хотя его основная роль — бизнес-логика
```

## Pre-Signed URLs — правильный паттерн

```txt
1. Frontend → Backend: "хочу загрузить файл report.pdf, 5MB, application/pdf"
2. Backend проверяет права доступа, генерирует pre-signed URL
   (подписанный временный URL с ограниченным сроком действия,
   например 15 минут, и конкретными разрешёнными параметрами)
3. Backend → Frontend: возвращает pre-signed URL
4. Frontend → S3: загружает файл НАПРЯМУЮ по этому URL (PUT-запрос)
5. Backend узнаёт о завершении загрузки через:
   - callback/webhook от S3 (S3 Event Notifications), или
   - явный запрос от frontend "я загрузил, вот его id"
```

```ts
// Backend: генерация pre-signed URL (AWS SDK v3)
import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3';
import { getSignedUrl } from '@aws-sdk/s3-request-presigner';

async function createUploadUrl(userId: string, fileName: string, contentType: string) {
  const key = `uploads/${userId}/${crypto.randomUUID()}-${fileName}`;

  const command = new PutObjectCommand({
    Bucket: 'my-app-uploads',
    Key: key,
    ContentType: contentType, // ограничивает, какой Content-Type примет S3
  });

  const url = await getSignedUrl(s3Client, command, { expiresIn: 900 }); // 15 минут
  return { uploadUrl: url, key };
}
```

Почему это решает проблему:

```txt
+ Backend больше не передаёт через себя сами байты файла —
  только метаданные запроса (миллисекунды CPU вместо минут I/O)
+ S3 масштабируется горизонтально "из коробки" — backend'у
  не нужно об этом думать
+ Дешевле — трафик идёт напрямую в объектное хранилище,
  минуя compute-ресурсы backend'а
```

Senior-нюанс про безопасность: pre-signed URL должен **ограничивать** параметры загрузки (`ContentType`, максимальный размер через `Content-Length-Range` в policy, конкретный `key`), иначе пользователь, получивший URL для загрузки своего аватара, теоретически может загрузить произвольный файл произвольного размера в произвольное место бакета в течение TTL ссылки.

## Метаданные: что в S3, что в БД

```txt
S3 (объектное хранилище):
  - сами байты файла
  - URL/key для доступа

PostgreSQL (метаданные):
  - id, userId, originalFileName
  - s3Key, size, mimeType
  - status (uploading / processing / ready / failed)
  - createdAt, processedAt
```

Почему не хранить файлы в БД (BLOB-поля): реляционные БД физически не предназначены для хранения больших бинарных объектов — это раздувает размер БД, замедляет бэкапы (бэкап БД с TB видео занимает часы), плохо работает с репликацией и кешированием на уровне БД. Объектное хранилище спроектировано именно под это: дешёвое хранение, встроенное geo-распределение, прямая отдача через CDN.

Поле `status` — важная деталь, которую часто упускают: после получения pre-signed URL клиент может вообще не загрузить файл (закрыл вкладку) — запись в БД должна отражать это состояние, чтобы не показывать пользователю "файл загружен", когда его нет в S3. Также нужна периодическая очистка "осиротевших" записей со статусом `uploading`, которые никогда не перешли в `ready`.

## CDN: как кеш, но для файлов, и с нюансами приватности

```txt
Без CDN:
  User (Australia) → Origin (S3 bucket в eu-west-1)
  → latency определяется расстоянием до Ирландии

С CDN:
  User (Australia) → CDN Edge (Sydney)
  → cache hit: ответ с edge-сервера, минимальная latency
  → cache miss: edge идёт в Origin, кеширует результат, отдаёт пользователю
```

### Публичный контент vs приватный контент

Для публичных файлов (аватары, изображения товаров) — всё просто: CDN кеширует по URL, `Cache-Control` заголовки определяют TTL.

Для **приватного** контента (документы пользователя, платный видео-контент) — прямое кеширование по URL не подходит, потому что URL не должен быть доступен без авторизации. Решения:

```txt
- Signed CDN URLs (CloudFront Signed URLs/Cookies) — CDN сам проверяет
  подпись/срок действия перед тем как отдать закешированный контент
- Кеш на уровне CDN всё равно работает (контент закеширован на edge),
  но доступ к нему контролируется срок-ограниченной подписью,
  а не публичностью URL
```

### Cache invalidation для CDN

```txt
Проблема: пользователь обновил аватар, но CDN ещё 24 часа
(TTL из Cache-Control) отдаёт старую версию всем пользователям

Решения:
  1. Versioned URLs: avatar-v2.jpg вместо avatar.jpg —
     новый URL = гарантированный cache miss = новый контент.
     Это лучший подход — НЕ требует явной инвалидации.
  2. Explicit invalidation API (CloudFront Invalidation) —
     явный запрос "сбросить кеш для этого пути на всех edge-нодах"
     — медленнее (минуты) и стоит денег при частом использовании
```

Versioned URLs — практически всегда лучший выбор для пользовательского контента: он превращает проблему "когда инвалидировать кеш" в "кеш живёт вечно, потому что URL уникален для каждой версии контента".

## Image/Video Processing Pipeline

```txt
1. Client → pre-signed URL → S3 (оригинал)
2. S3 Event Notification → Queue (SQS/SNS)
3. Worker читает событие, скачивает оригинал из S3
4. Worker генерирует производные версии:
   - thumbnails разных размеров
   - конвертация формата (HEIC → JPEG, видео → несколько разрешений)
5. Worker загружает результаты обратно в S3 (отдельные keys/bucket)
6. Worker обновляет статус в БД (status: ready, добавляет URLs производных)
7. Frontend (через polling/WebSocket/SSE) узнаёт, что обработка завершена
```

Очередь здесь решает ту же задачу, что и в [Message Queues]: 10 000 одновременных загрузок не создают 10 000 параллельных тяжёлых задач обработки видео — они становятся 10 000 сообщений в очереди, обрабатываемых с постоянной скоростью пулом worker'ов. Без очереди резкий всплеск загрузок (например, после вирусного поста) уронит processing-сервис.

## Multipart Upload — для больших файлов

```txt
Файл 5 GB:
  - разбивается на части по 5-100 MB
  - каждая часть загружается отдельным PUT-запросом
    (можно параллельно, можно с retry для конкретной части)
  - после загрузки всех частей — CompleteMultipartUpload собирает их в один объект
```

Зачем: при загрузке файла в один запрос любой сетевой сбой на 99% — это retry **всего файла с нуля**. Multipart позволяет повторить только неудачную часть, параллелить загрузку (быстрее на быстром интернете) и поддерживать файлы больше лимита на один PUT-запрос (для S3 — 5GB на один PUT).

## Безопасность загрузок — частый "follow-up"

```txt
- Валидация Content-Type на backend при выдаче pre-signed URL —
  не доверять расширению файла от клиента
- Сканирование на вирусы/malware ПОСЛЕ загрузки, ДО того как файл
  становится доступен другим пользователям (статус "scanning" → "ready")
- Ограничение размера через policy в pre-signed URL,
  не только проверка на клиенте (клиент можно обойти)
- Изоляция загруженного пользовательского контента от
  исполняемого кода — отдельный домен/bucket без возможности
  выполнения (избегаем XSS через загруженный "image.svg" с JS внутри)
```

## Типичные ошибки на интервью

- **Предлагать "загрузка через backend" как финальный ответ** — не упоминая pre-signed URLs, что является ожидаемым "правильным" решением для этой темы.

- **Хранить файлы как BLOB в реляционной БД** — без объяснения, почему это плохо масштабируется (размер БД, бэкапы, репликация).

- **Не разделять метаданные и сам файл** — путать "где хранится файл" с "где хранится информация о файле".

- **CDN как "просто кеш"** — не упоминая разницу между публичным и приватным контентом, и что приватный контент требует signed URLs, а не просто отсутствия CDN.

- **Cache invalidation через явный API вызов как единственное решение** — не упоминая versioned URLs, которые элегантно избегают проблемы инвалидации вообще.

- **Не упоминать очередь для обработки медиа** — синхронная генерация thumbnail при загрузке означает, что пользователь ждёт обработку видео в HTTP-запросе.

- **Игнорировать безопасность загрузок** — pre-signed URL без ограничений на размер/тип, отсутствие проверки контента перед публикацией.
