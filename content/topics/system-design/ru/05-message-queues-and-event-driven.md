# Message Queues и Event-Driven Architecture

## Главная идея: decoupling во времени и по доступности

Без очереди синхронный вызов связывает время жизни и доступность двух сервисов:

```txt
API → Email Service (синхронный HTTP-вызов)

Если Email Service медленный → API медленный
Если Email Service упал     → API падает (или возвращает ошибку клиенту)
```

С очередью производитель (producer) и потребитель (consumer) развязаны:

```txt
API → Queue → Email Worker

API записал сообщение в очередь и сразу ответил клиенту 200 OK.
Email Worker обрабатывает сообщение когда сможет — даже если
он сейчас недоступен, сообщение просто ждёт в очереди.
```

Это даёт четыре конкретные выгоды, и на интервью важно не просто их перечислить, а связать с конкретной задачей:

```txt
Асинхронность  — клиент не ждёт результат долгой операции
Буферизация    — всплеск из 10 000 запросов "размазывается" во времени
                 для consumer'а, который обрабатывает с постоянной скоростью
Изоляция сбоев — падение consumer'а не приводит к падению producer'а
Независимое масштабирование — можно добавить больше worker'ов
                 без изменений в API
```

## Queue vs Pub/Sub vs Event Streaming — это разные модели, не синонимы

Это различие путают чаще всего, и оно принципиально влияет на архитектуру:

| | Point-to-Point Queue (SQS, RabbitMQ) | Pub/Sub (SNS, Redis Pub/Sub) | Event Streaming (Kafka, Kinesis) |
|---|---|---|---|
| Кто получает сообщение | **Один** consumer (из группы) забирает и удаляет сообщение | **Все** подписчики получают копию | Все consumer-группы читают независимо, сообщение **не удаляется** после чтения |
| Хранение после обработки | Удаляется | Не сохраняется (если нет подписчика — потеряно) | Хранится N дней/часов (retention), можно "перечитать" |
| Порядок | Обычно не гарантирован (или FIFO-очереди с ограничениями throughput) | Не гарантирован | Гарантирован **в рамках партиции** |
| Типичный кейс | Задачи на обработку (отправить email, сгенерировать отчёт) | Уведомить несколько сервисов о событии | Event sourcing, аналитика в реальном времени, replay событий |

Senior-нюанс: "очередь" в духе SQS — это **task queue** (задача обрабатывается ровно одним worker'ом и исчезает), а Kafka — это **журнал событий** (log), который многие consumer'ы читают независимо и который можно "перечитать с начала". Путать их в дизайне — частая ошибка: если несколько сервисов должны знать о "User Created", обычная очередь (где сообщение исчезает после обработки одним consumer'ом) не подходит — нужен Pub/Sub или event log.

## Fan-Out: одно событие → много подписчиков

```txt
User Service
   │ публикует "UserCreated"
   ▼
 Event Bus / Topic
   ├──→ Email Service     (отправить welcome email)
   ├──→ Analytics Service (записать в аналитику)
   └──→ CRM Service        (создать карточку клиента)
```

Ключевое архитектурное преимущество: **User Service не знает**, сколько у "UserCreated" подписчиков и зачем они им нужны. Добавление нового подписчика (например, Fraud Detection Service) не требует изменений в User Service — это снижает coupling между командами/сервисами кратно сильнее, чем прямые HTTP-вызовы "на всякий случай".

## Delivery Guarantees — at-most-once, at-least-once, exactly-once

```txt
At-most-once:  сообщение доставляется 0 или 1 раз.
               Может быть потеряно, но никогда не дублируется.
               (например, UDP-подобная "fire and forget" доставка)

At-least-once: сообщение доставляется 1 или более раз.
               Никогда не теряется, но может дублироваться.
               (стандарт для большинства очередей — SQS, Kafka by default)

Exactly-once:  сообщение доставляется ровно 1 раз.
```

Senior-нюанс про exactly-once: в распределённых системах **истинный exactly-once на уровне сети практически недостижим** — то, что маркетируется как "exactly-once" (например, Kafka transactions), на практике реализуется как **at-least-once доставка + идемпотентная обработка на стороне consumer'а**, что даёт *эффект* exactly-once. Хороший ответ на "как обеспечить exactly-once" — не "включить флаг exactly-once", а "спроектировать consumer идемпотентным, потому что доставка с повторами неизбежна".

### Идемпотентность — практическая реализация

```ts
// ❌ Не идемпотентно: повторная обработка того же сообщения
// списывает деньги дважды
async function processPayment(message: PaymentMessage): Promise<void> {
  await chargeCard(message.userId, message.amount);
}

// ✅ Идемпотентно: idempotency key проверяется перед побочным эффектом
async function processPaymentIdempotent(message: PaymentMessage): Promise<void> {
  const alreadyProcessed = await db.processedMessages.findUnique({
    where: { messageId: message.id },
  });
  if (alreadyProcessed) {
    return; // уже обработано — безопасный no-op
  }

  await db.$transaction(async (tx) => {
    await chargeCard(message.userId, message.amount);
    await tx.processedMessages.create({ data: { messageId: message.id } });
  });
}
```

Ключевая деталь: запись в `processedMessages` и побочный эффект (`chargeCard`) должны быть атомарны (в одной транзакции или с компенсирующей логикой) — иначе появляется новое окно для дублирования между "выполнили операцию" и "записали, что выполнили".

## Ordering — гарантии порядка стоят дорого

По умолчанию большинство очередей **не гарантируют порядок** при параллельной обработке несколькими consumer'ами — два сообщения об одном пользователе могут быть обработаны "не в том порядке", если их забрали разные worker'ы.

```txt
Kafka: гарантирует порядок ТОЛЬКО внутри одной партиции.

  Partition key = user_id
  → все события для user_id=42 всегда попадают в одну партицию
  → обрабатываются одним consumer'ом строго по порядку

  Но: события для разных user_id могут обрабатываться
  параллельно разными партициями — порядок между ними не гарантирован
  (и обычно не нужен)
```

Практический вывод: если порядок важен ("UserUpdated" должно быть обработано после "UserCreated" для того же пользователя), нужно выбрать partition key так, чтобы связанные события **всегда** попадали в одну партицию — обычно это entity ID.

## Dead Letter Queue и retry с backoff

```txt
Message → Worker → ошибка обработки
                       ↓
                  Retry с экспоненциальным backoff
                  (1s, 2s, 4s, 8s, ...)
                       ↓
              После N неудачных попыток
                       ↓
                 Dead Letter Queue (DLQ)
```

DLQ решает конкретную проблему: "ядовитое сообщение" (poison message) — сообщение, которое **не может** быть обработано из-за бага в данных (например, невалидный JSON) — без DLQ такое сообщение будет бесконечно повторяться (retry loop), занимая ресурсы consumer'а и блокируя обработку остальных сообщений (если очередь FIFO).

Senior-нюанс: DLQ — это не "конец истории", это сигнал для **человека или алертинга**. Сообщения в DLQ нужно мониторить — рост DLQ часто означает баг в коде consumer'а или изменение формата данных producer'ом (несовместимое изменение схемы события).

## Backpressure — что происходит, когда producer быстрее consumer'а

```txt
Producer публикует 10 000 msg/sec
Consumer обрабатывает 1 000 msg/sec

→ очередь растёт без ограничений
→ растёт latency обработки (сообщение из конца очереди ждёт всё дольше)
→ в крайнем случае — очередь исчерпывает память/диск брокера
```

Решения: увеличить число consumer'ов (horizontal scaling worker'ов — это "масштабирование", о котором говорилось в начале), ограничить размер очереди с явным отбрасыванием/возвратом ошибки producer'у (back-pressure "наружу"), или применить rate limiting на стороне producer'а.

## Когда очередь — не то решение

```txt
✅ Подходит:
  - отправка email/push-уведомлений
  - генерация отчётов, обработка видео (долгие операции)
  - синхронизация данных между сервисами (event-driven)
  - сглаживание всплесков нагрузки (batch-обработка заказов)

❌ Не подходит:
  - запросы, требующие немедленного ответа в том же HTTP-цикле
    (например, "проверить наличие товара перед показом цены")
  - real-time чат/игры — там нужен WebSocket/прямое соединение,
    очередь добавляет latency, неприемлемую для интерактивности
  - операции, где порядок и атомарность критичны "из коробки" —
    проще сделать синхронной транзакцией в одной БД,
    чем городить distributed-логику вокруг очереди
```

Главный признак "не используй очередь здесь" — если ответ пользователю **зависит** от результата операции в реальном времени. Очередь подходит, когда можно сказать "приняли, обработаем" (202 Accepted), а не когда нужно "вот результат прямо сейчас" (200 OK с данными).

## Типичные ошибки на интервью

- **Путать queue и pub/sub** — предлагать SQS-подобную очередь для сценария "несколько сервисов должны узнать о событии", не замечая, что сообщение в point-to-point очереди достанется только одному consumer'у.

- **Говорить "exactly-once delivery" как о решённой проблеме** — не упоминая, что на практике это всегда at-least-once + идемпотентность.

- **Не упоминать идемпотентность consumer'а вообще** — если очередь обсуждается, идемпотентность — обязательная часть ответа, а не "опционально".

- **Считать, что Kafka гарантирует глобальный порядок** — порядок гарантирован только внутри партиции; выбор partition key — это решение, влияющее на ordering.

- **DLQ как "мусорка, и забыли"** — без мониторинга DLQ означает, что часть сообщений теряется незаметно для команды.

- **Предлагать очередь для синхронных операций** — "проверить остаток товара перед оплатой через очередь" добавляет latency и сложность там, где нужен прямой синхронный запрос.

- **Не обсуждать backpressure** — что происходит, если producer стабильно быстрее consumer'а, и как система деградирует (рост latency, переполнение очереди).
