# Scalability и Load Balancing

## Масштабирование: напоминание про vertical vs horizontal

Базовое различие vertical vs horizontal scaling разобрано в [System Design Fundamentals]. Здесь важно зафиксировать практическое следствие: горизонтальное масштабирование **невозможно без load balancer'а** — без него у тебя просто N независимых серверов, между которыми клиент должен выбирать сам (или у каждого свой DNS-адрес), что не масштабируется и не даёт отказоустойчивости.

Поэтому "добавить ещё серверов" как ответ на вопрос о масштабировании — половина ответа. Вторая половина — **как трафик распределяется между ними** и **что происходит, когда один из них падает**. Это и есть тема Load Balancing.

## L4 vs L7 Load Balancer — разный уровень принятия решений

Это одно из самых важных различий, которое часто упускают:

**L4 (Transport Layer) Load Balancer** работает на уровне TCP/UDP — видит только IP-адреса и порты, не заглядывает в содержимое запроса.

```txt
Решение о маршрутизации принимается по:
  source IP, destination IP, source port, destination port

Преимущества: очень быстрый (не парсит payload), низкий overhead
Ограничения: не может маршрутизировать по URL, headers, cookies
```

**L7 (Application Layer) Load Balancer** работает на уровне HTTP — видит URL, headers, cookies, тело запроса.

```txt
Решение о маршрутизации принимается по:
  path (/api/* → backend A, /static/* → CDN/backend B)
  headers (Authorization, User-Agent)
  cookies (sticky sessions)

Преимущества: гибкая маршрутизация, можно делать A/B тесты, canary deploy
Минусы: выше latency (нужно терминировать TLS и парсить HTTP), дороже в вычислениях
```

Senior-нюанс: L7 LB обычно **терминирует TLS** (HTTPS → HTTP внутри сети), что даёт ему доступ к содержимому запроса, но добавляет шаг расшифровки/шифрования на каждый запрос — это нужно учитывать в latency-бюджете. Реальные системы часто комбинируют оба уровня: L4 (например, AWS NLB) на входе для грубого распределения по регионам/availability zones, и L7 (AWS ALB, Nginx, Envoy) внутри — для маршрутизации по сервисам.

## Алгоритмы балансировки

| Алгоритм | Как работает | Когда хорош | Когда плох |
|---|---|---|---|
| **Round Robin** | По кругу: A → B → C → A → ... | Серверы одинаковой мощности, запросы примерно одной "тяжести" | Не учитывает текущую нагрузку — медленный сервер получит столько же запросов, сколько быстрый |
| **Weighted Round Robin** | Как Round Robin, но с весами (сервер мощнее → больше запросов) | Серверы разной мощности (например, постепенный canary rollout) | Веса статичны, не реагируют на реальную нагрузку в реальном времени |
| **Least Connections** | Запрос идёт на сервер с наименьшим числом активных соединений | Запросы сильно различаются по длительности обработки | Требует от LB отслеживать состояние соединений — чуть дороже |
| **IP Hash / Consistent Hashing** | Маршрутизация — функция от хеша клиентского IP (или другого ключа) | Нужна "липкость" без cookies, или для cache-aware routing | При добавлении/удалении серверов с обычным хешем (`hash % N`) перераспределяется **почти весь** трафик |

### Consistent Hashing — почему это отдельная большая тема

Простой `hash(key) % N` ломается при изменении N: если у тебя 4 сервера и ты добавляешь пятый, у **большинства** ключей меняется `hash % N`, и они "переезжают" на другой сервер — для кеша это означает массовый cache miss, для шардированной БД — масштабную перебалансировку данных.

**Consistent hashing** решает это, размещая и серверы, и ключи на одном "кольце" хешей — при добавлении/удалении сервера перемещается только доля ключей `~1/N`, а не весь набор:

```ts
// Упрощённая реализация consistent hashing с виртуальными узлами
class ConsistentHashRing {
  private ring = new Map<number, string>(); // hash -> server
  private sortedHashes: number[] = [];
  private readonly virtualNodesPerServer = 150; // снижает неравномерность распределения

  addServer(serverId: string): void {
    for (let i = 0; i < this.virtualNodesPerServer; i++) {
      const hash = this.hash(`${serverId}#${i}`);
      this.ring.set(hash, serverId);
    }
    this.sortedHashes = [...this.ring.keys()].sort((a, b) => a - b);
  }

  getServer(key: string): string {
    const hash = this.hash(key);
    // находим первый узел на кольце с hash >= hash(key) (по кругу)
    const index = this.sortedHashes.findIndex((h) => h >= hash);
    const ringIndex = index === -1 ? 0 : index;
    return this.ring.get(this.sortedHashes[ringIndex])!;
  }

  private hash(input: string): number {
    // в реальной реализации — стабильная хеш-функция (например, MurmurHash)
    let h = 0;
    for (const char of input) h = (h * 31 + char.charCodeAt(0)) >>> 0;
    return h;
  }
}
```

**Virtual nodes** (несколько точек на кольце для каждого реального сервера) решают проблему неравномерного распределения, когда серверов мало — без них один сервер может случайно получить значительно больший диапазон кольца, чем другой.

Это не только про load balancing — тот же принцип лежит в основе шардинга в Cassandra, DynamoDB, и в распределённых кешах (memcached с consistent hashing на клиенте).

## Health Checks: active vs passive

LB должен знать, какие backend-инстансы живы, **до** отправки трафика на них.

```txt
Active health checks:
  LB сам периодически шлёт запрос (например, GET /health)
  Backend отвечает 200 OK, если готов принимать трафик
  Если N проверок подряд провалились → инстанс убирается из пула

Passive health checks:
  LB следит за реальными запросами
  Если процент ошибок/таймаутов превышает порог → инстанс временно убирается
```

Senior-нюанс: эндпоинт `/health` не должен быть тривиальным "always 200" — но и не должен проверять **всё** (например, если он проверяет соединение с БД, а БД временно перегружена, health check может начать "снимать с пула" все инстансы одновременно — это называется **cascading failure**, когда механизм, призванный защищать систему, сам её обрушивает). Хороший health check проверяет именно то, без чего инстанс не может обслуживать запросы, и не более того.

## Sticky Sessions — когда это всё-таки нужно

Sticky sessions (привязка пользователя к конкретному серверу, обычно через cookie с ID инстанса) — почти всегда антипаттерн при stateless-архитектуре (см. [System Design Fundamentals]), но есть легитимные случаи:

```txt
- WebSocket-соединения: соединение физически держится одним сервером,
  "переключить" его на другой сервер без разрыва невозможно
- Серверный кеш в памяти процесса (in-memory cache конкретного инстанса) —
  но это нужно явно обосновать, обычно лучше Redis
```

Если sticky sessions используются — нужно явно проговорить последствия: при падении инстанса все "прилипшие" к нему клиенты теряют соединение/состояние и должны переподключиться (для WebSocket это означает явную реконнект-логику на клиенте).

## CDN как "масштабирование на уровне сети"

CDN (Content Delivery Network) — это, по сути, географически распределённый layer кеширования **перед** твоим backend'ом:

```txt
User (Berlin) → CDN edge (Frankfurt) → cache hit → ответ без запроса к origin
User (Tokyo)  → CDN edge (Tokyo)     → cache miss → запрос к origin → кешируется
```

CDN снижает нагрузку на origin (статика, изображения, видео, иногда целые HTML-страницы для SSG-сайтов) и снижает latency за счёт географической близости к пользователю. Для динамического API CDN обычно не кеширует напрямую, но может использоваться для **edge-вычислений** (проверка авторизации, geo-роутинг, A/B-тесты до похода к origin).

## Auto Scaling — не только "добавить сервер при высокой нагрузке"

```txt
Reactive scaling:
  metric (CPU > 70%, queue depth > 1000, p99 latency > 500ms)
  → scale-out (новые инстансы)
  → cooldown period (например, 5 минут — не масштабироваться повторно,
     пока новые инстансы не "разогрелись")

Predictive/scheduled scaling:
  заранее известные паттерны нагрузки (утренний пик, Black Friday)
  → масштабирование ПЕРЕД пиком, а не в реакции на него
```

Senior-нюансы по auto scaling:

- **Cold start**: новый инстанс может занимать минуты на запуск (особенно если требует прогрева кеша/соединений с БД) — reactive scaling "опаздывает" на этот промежуток, и именно в это время происходит деградация. Поэтому для предсказуемых пиков (распродажи) используется scheduled scaling.
- **Метрика CPU не всегда правильный сигнал** — для I/O-bound сервисов (много ожидания ответов от БД/внешних API) CPU может оставаться низким при перегрузке. Более показательные метрики: глубина очереди запросов, latency, число активных соединений.
- **Connection limits на БД** — если auto scaling добавляет app-инстансы без ограничения, каждый открывает пул соединений к БД, и можно упереться в `max_connections` БД раньше, чем в CPU app-серверов. Это частый практический "gotcha", который показывает глубину понимания.

## Cache Layer как способ масштабирования

```txt
Client → App Server → Redis (cache) → Database
                         ↑
                  cache hit: ответ за ~1ms, без похода в БД
                  cache miss: читаем из БД, кладём в кеш
```

Кеш — это не "ускорение", это **снятие нагрузки с БД**, что позволяет обслуживать больше запросов тем же количеством ресурсов БД. Детали стратегий кеширования (cache-aside, write-through, TTL, инвалидация) разбираются отдельно в теме про кеширование — здесь важно зафиксировать, что кеш — один из главных рычагов горизонтального масштабирования системы в целом, а не только "оптимизация по мелочи".

## Типичные ошибки на интервью

- **Путать L4 и L7 балансировку** или не упоминать, что L7 нужен для маршрутизации по URL/cookies/headers, а L4 — для грубой быстрой балансировки на транспортном уровне.

- **Предлагать `hash(key) % N` как решение для consistent hashing** — не понимая, что именно изменение N при таком подходе и является проблемой, которую consistent hashing решает.

- **Считать sticky sessions "плохими всегда"** — для WebSocket-соединений они архитектурно необходимы (соединение физически держится одним инстансом); важно понимать *когда* они нужны, а не отвергать концепцию целиком.

- **Health check, который проверяет слишком много** — приводящий к cascading failure, когда временная проблема одной зависимости (БД) выводит из пула все инстансы одновременно.

- **Auto scaling по CPU для I/O-bound сервисов** — не замечая, что сервис может быть перегружен (по latency/queue depth) при низком CPU.

- **"CDN решит проблему динамического API"** — CDN эффективен для статики и кешируемого контента; для персонализированных/динамических ответов нужен другой подход (кеш на уровне приложения, edge compute).

- **Игнорировать connection limits БД при auto scaling app-серверов** — классический случай, когда "масштабирование одного слоя" создаёт bottleneck в другом.
