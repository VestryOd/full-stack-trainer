# Caching

## Зачем нужен кеш — снятие нагрузки, а не "ускорение"

Базовая мотивация: запрос к Redis/memcached занимает ~0.1-1ms, запрос к реляционной БД с диском — единицы-десятки миллисекунд, а сложный JOIN/агрегация — может быть сотни миллисекунд. Кеш сокращает не только latency конкретного запроса, но и **общую нагрузку на БД**, что позволяет той же БД обслуживать кратно больше пользователей.

```txt
Без кеша:                          С кешом (cache hit):
Client → API → Database            Client → API → Redis → Response
10 000 req/s все идут в БД          9 000 req/s обслужены из Redis,
                                     1 000 req/s (miss) идут в БД
```

Главный вопрос на интервью не "что такое кеш", а **где именно в архитектуре он стоит, что в нём хранится, как он инвалидируется и что происходит при cache miss на масштабе**. Именно эти детали отличают senior-ответ от "добавим Redis".

## Cache-Aside (Lazy Loading) — паттерн по умолчанию

Приложение явно управляет кешем: читает из кеша, при miss идёт в БД и сохраняет результат.

```ts
async function getUser(userId: string): Promise<User> {
  const cached = await redis.get(`user:${userId}`);
  if (cached) {
    return JSON.parse(cached); // cache hit
  }

  const user = await db.user.findUnique({ where: { id: userId } });
  if (user) {
    // TTL обязателен — иначе устаревшие данные могут жить вечно
    await redis.set(`user:${userId}`, JSON.stringify(user), 'EX', 300);
  }
  return user;
}
```

- **Плюсы**: кешируется только то, что реально запрашивается (lazy); если Redis упал — система продолжает работать (просто медленнее, идёт прямо в БД).
- **Минусы**: первый запрос после miss/инвалидации всегда "медленный" (cache miss penalty); есть короткое окно несогласованности между БД и кешом.

## Write-Through и Write-Behind — альтернативы для записи

| Паттерн | Как работает | Плюсы | Минусы |
|---|---|---|---|
| **Cache-Aside** | Чтение через кеш, запись — напрямую в БД (с инвалидацией кеша) | Просто, кеш не "грязный" | Cache miss penalty |
| **Write-Through** | Запись синхронно идёт и в кеш, и в БД | Кеш всегда согласован с БД | Каждая запись медленнее (две операции) |
| **Write-Behind (Write-Back)** | Запись идёт в кеш, в БД сбрасывается асинхронно (batched) | Очень быстрая запись | Риск потери данных при падении кеша до flush; сложность |

Write-behind — это явный trade-off durability за latency записи. На собеседовании важно сказать прямо: "write-behind подходит для метрик/счётчиков, где потеря последних нескольких секунд данных приемлема, но не для платежей".

## Eviction Policies — что выкидывать, когда кеш заполнен

```txt
LRU (Least Recently Used)  — выкидываем то, что дольше всего не запрашивали
                              (хорошо для "горячих" данных с временной локальностью)
LFU (Least Frequently Used) — выкидываем то, что запрашивали реже всего
                              (лучше, когда популярность стабильна во времени)
TTL-based                    — выкидываем по истечении времени жизни,
                              независимо от частоты использования
```

Redis по умолчанию поддерживает несколько политик (`allkeys-lru`, `allkeys-lfu`, `volatile-ttl` и др.) — выбор политики должен соответствовать паттерну доступа: для feed/trending-данных LFU часто лучше LRU, потому что популярный пост может временно "не запрашиваться" несколько минут, но всё равно остаётся горячим в среднем.

## Cache Invalidation — "одна из двух действительно сложных вещей в CS"

Известная шутка ("There are only two hard things in Computer Science: cache invalidation and naming things") — не просто шутка. Проблема в том, что у тебя есть **два источника правды** (БД и кеш), и любая запись создаёт окно, в котором они расходятся.

```txt
Стратегии инвалидации:

1. TTL (Time To Live) — самый простой, но данные могут быть устаревшими
   до TTL минут. Подходит, когда staleness допустима бизнес-логикой.

2. Explicit invalidation (DEL после UPDATE) — кеш удаляется/обновляется
   синхронно с изменением данных. Точнее, но требует, чтобы ВСЕ пути
   записи данных не забывали инвалидировать кеш — легко забыть
   при росте кодовой базы.

3. Event-based invalidation — изменение данных публикует событие
   (через очередь/CDC), кеш инвалидируется асинхронным обработчиком.
   Развязывает запись от инвалидации, но добавляет задержку
   (eventual consistency для кеша).
```

Senior-нюанс — **race condition при cache-aside записи**:

```txt
T1: читает из БД старое значение V1 (cache miss)
T2: обновляет БД на V2, инвалидирует кеш
T1: записывает в кеш V1 (устаревшее!) — после того, как T2 уже всё сделал

Результат: кеш содержит V1, БД содержит V2 — несогласованность до TTL
```

Это редкий, но реальный edge case в высоконагруженных системах. Решения: короткий TTL как "страховочная сетка" (даже если инвалидация не сработала, данные не "зависнут" навечно), либо версионирование ключей кеша (`user:123:v{version}`), либо distributed lock на запись.

## Cache Stampede (Thundering Herd) — классический вопрос

Проблема: ключ с очень высоким трафиком (например, главная страница) истекает по TTL — и **тысячи одновременных запросов** видят cache miss и **все** одновременно идут в БД, создавая внезапную пиковую нагрузку, способную "уронить" БД именно в момент истечения TTL.

```ts
// ❌ Без защиты: при истечении TTL все параллельные запросы
// одновременно бьют в БД
async function getHomepage(): Promise<Homepage> {
  const cached = await redis.get('homepage');
  if (cached) return JSON.parse(cached);
  const data = await db.buildHomepage(); // 10 000 параллельных вызовов!
  await redis.set('homepage', JSON.stringify(data), 'EX', 60);
  return data;
}

// ✅ С distributed lock: только один запрос идёт в БД,
// остальные либо ждут, либо получают немного устаревшие данные
async function getHomepageSafe(): Promise<Homepage> {
  const cached = await redis.get('homepage');
  if (cached) return JSON.parse(cached);

  const lockAcquired = await redis.set('homepage:lock', '1', 'NX', 'EX', 10);
  if (!lockAcquired) {
    // другой инстанс уже пересчитывает — отдаём слегка устаревшие данные
    // или ждём и повторяем (с коротким polling)
    const stale = await redis.get('homepage:stale');
    if (stale) return JSON.parse(stale);
    await sleep(50);
    return getHomepageSafe();
  }

  const data = await db.buildHomepage();
  await redis.set('homepage', JSON.stringify(data), 'EX', 60);
  await redis.set('homepage:stale', JSON.stringify(data), 'EX', 3600); // fallback на час
  await redis.del('homepage:lock');
  return data;
}
```

Другие решения:

```txt
Random TTL jitter:     TTL = 60 + random(0, 10) секунд
                        → ключи истекают не одновременно, нагрузка размазывается

Probabilistic early expiration (XFetch):
                        чем ближе к истечению TTL, тем выше шанс,
                        что один из запросов "досрочно" пересчитает значение,
                        пока остальные продолжают использовать старое

Background refresh:    отдельный воркер обновляет горячие ключи ДО истечения TTL,
                        пользовательские запросы никогда не видят miss для них
```

## Multi-Level Caching — кеш как иерархия, а не один слой

```txt
Browser Cache (HTTP cache-control, Service Worker)
  ↓ miss
CDN Edge Cache (статика, иногда полные HTML-страницы)
  ↓ miss
Application-level Cache (in-process, например LRU-кеш в памяти инстанса)
  ↓ miss
Distributed Cache (Redis/Memcached — общий для всех инстансов)
  ↓ miss
Database
```

Каждый уровень отсекает часть трафика для следующего. Senior-нюанс: **in-process кеш на каждом инстансе** — это самый быстрый уровень (нет сетевого round-trip), но он "размножается" по инстансам и сложно инвалидируется (нужно broadcast-сообщение всем инстансам, например через Redis Pub/Sub, чтобы синхронно сбросить локальные кеши). Поэтому in-process кеш обычно используют для данных, которые либо почти никогда не меняются (конфигурация, feature flags), либо допускают staleness в несколько секунд.

## Что кешировать, а что — нет

```txt
Хорошие кандидаты:
  - профили пользователей, каталоги товаров — read-heavy, меняются редко
  - результаты тяжёлых агрегаций/JOIN-ов
  - сессии (если не stateless через JWT)
  - rate-limit счётчики, фиче-флаги

Плохие кандидаты (или требуют особой осторожности):
  - балансы счетов, состояние платежей — здесь staleness = бизнес-риск,
    источник истины должен запрашиваться напрямую (или CP-кеш с явной
    инвалидацией в той же транзакции)
  - данные, уникальные почти для каждого запроса (низкий hit ratio —
    кеш не помогает, а только добавляет накладные расходы)
```

Метрика, на которую стоит ориентироваться — **cache hit ratio**: если он ниже ~70-80% для конкретного типа ключей, кеш, возможно, не оправдывает добавленную сложность для этих данных.

## Типичные ошибки на интервью

- **"Добавим Redis" без указания паттерна (cache-aside vs write-through) и стратегии инвалидации** — это самый частый поверхностный ответ; интервьюер ожидает конкретику.

- **Не упоминать cache stampede** при обсуждении высоконагруженного популярного ключа с TTL — это один из самых ожидаемых "follow-up" вопросов.

- **Путать TTL и eviction policy** — TTL — это явное время жизни конкретного ключа; eviction policy (LRU/LFU) — это что происходит, когда кеш **переполнен**, независимо от TTL.

- **Игнорировать race condition при cache-aside** — считать, что "прочитал из БД → записал в кеш" атомарно и безопасно при конкурентных записях.

- **Кешировать данные, требующие строгой консистентности** (балансы, инвентарь с лимитом), без обсуждения, что staleness здесь может означать прямые денежные потери.

- **Забывать про in-process кеш при горизонтальном масштабировании** — если кеш живёт в памяти каждого инстанса, инвалидация на одном инстансе не затрагивает остальные.

- **Не давать числовую оценку cache hit ratio** — "кеш поможет" без объяснения, при каком соотношении read:write и при каком hit ratio это действительно снижает нагрузку на БД на порядок.
