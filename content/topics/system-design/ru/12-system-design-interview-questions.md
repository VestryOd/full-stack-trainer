# System Design Interview Questions

## Как использовать этот обзор

Это не 12 новых тем, а каталог самых частых формулировок задач на Senior Fullstack интервью, с указанием **главной развилки** каждой задачи и связью с темами, разобранными ранее. Три задачи (URL Shortener, Chat System, Notification System) уже разобраны полностью отдельными статьями — здесь для них только краткая навигация. Остальные 9 разобраны компактно: проблема → ключевая развилка → архитектура → follow-up с обоснованным ответом (применяя [Universal System Design Interview Framework]).

Цель — не выучить 12 диаграмм наизусть, а увидеть, что **за разными формулировками часто стоит одна и та же развилка** (например, "fan-out on write vs on read" встречается и в Chat System, и в Instagram Feed, и в News Feed).

## 1. Design URL Shortener (bit.ly, tinyurl)

Полностью разобран в [URL Shortener]. Главная развилка — генерация уникальных коротких кодов в распределённой системе (Base62 от auto-increment + Ticket Server для распределённой генерации, vs hash-based с коллизиями, vs random+retry) и read-heavy архитектура с кешем как центральным компонентом, а не опциональной оптимизацией.

## 2. Design Chat System (Telegram, Slack, WhatsApp)

Полностью разобран в [Chat System] (плюс транспортный слой в [WebSockets]). Главная развилка — fan-out on write vs fan-out on read для групповых чатов, и порядок persist → ack → deliver для durability сообщений.

## 3. Design Notification System

Полностью разобран в [Notification System] (плюс [Message Queues] для event-driven основы). Главная развилка — Decision Layer (preferences, приоритеты каналов, rate limiting) как отдельный слой между событием и доставкой, и идемпотентность при at-least-once доставке.

## 4. Design File Upload Service (фото/видео/документы)

### Проблема и ключевая развилка

Наивный подход — файл идёт через ваш backend (`Client → Backend → S3`). Проблема: backend становится узким местом для большого бинарного трафика — память и сетевой канал API-сервера тратятся на проксирование байтов, которые ему не нужны для бизнес-логики.

```txt
✅ Client → Backend: запрос на pre-signed URL (метаданные, права)
   Backend → Client: подписанный URL с TTL (см. [File Storage and CDN])
   Client → S3: загружает файл напрямую, минуя backend
   S3 → Backend (через событие/webhook): "файл загружен" → дальнейшая обработка
```

Backend участвует только в выдаче разрешения и в постобработке — само перемещение байт идёт по прямому каналу Client↔S3. Это прямое применение pre-signed URL из [File Storage and CDN].

### Follow-up: как генерировать thumbnail/превью?

```txt
S3 (event: ObjectCreated) → Queue → Image Processing Worker
  → resize/convert → загрузка thumbnail обратно в S3 (другой prefix/bucket)
  → обновление metadata DB (thumbnailUrl)
```

Это та же причина, что и для аналитики в URL Shortener: обработка изображения — потенциально медленная операция, которая не должна блокировать ответ пользователю на "файл загружен". Senior-нюанс: нужна **идемпотентность** воркера — если S3 event придёт повторно (at-least-once у большинства систем событий), повторная генерация того же thumbnail не должна создавать дубликаты или приводить к ошибке.

### Follow-up: как обеспечить безопасность аплоада?

```txt
- Pre-signed URL ограничен по времени (TTL ~15 мин) и по операции
  (PutObject на конкретный key, не на весь bucket)
- Ограничение размера файла — через Content-Length в подписи
  ИЛИ через S3 bucket policy
- Валидация типа файла — на этапе постобработки (после загрузки),
  т.к. до загрузки backend не видит содержимое; "сырой" upload
  попадает во временный prefix, после валидации перемещается
  в постоянное хранилище
```

## 5. Design Instagram Feed

### Проблема и ключевая развилка — та же, что в Chat System, но для другого паттерна нагрузки

```txt
Подход 1: Fan-out on read
  При открытии ленты → собрать посты от всех, на кого подписан
  пользователь, отсортировать по времени → "на лету"

  + O(1) при публикации поста
  - дорогое чтение: JOIN/merge постов от сотен авторов при
    каждом открытии ленты

Подход 2: Fan-out on write
  При публикации поста → сразу записать его в "предрасчитанную
  ленту" каждого подписчика (precomputed feed)

  + чтение ленты — это просто SELECT по одной таблице (быстро)
  - публикация поста у пользователя с 10M подписчиков = 10M
    записей — "celebrity problem"
```

### Follow-up: что выберет реальный Instagram?

**Гибрид** — и это сильный ответ только если объяснить, КАК именно делится:

```txt
Обычные пользователи (десятки-сотни подписчиков):
  → fan-out on write — стоимость записи минимальна,
    чтение ленты быстрое

"Celebrity"-аккаунты (миллионы подписчиков):
  → fan-out on read ИЛИ отложенный fan-out — пост celebrity
    подмешивается в ленту читателя в момент открытия,
    а не рассылается всем подписчикам сразу
```

Это прямая параллель с гибридом fan-out для групповых чатов из [Chat System]: критерий выбора — не тип контента, а **соотношение write:read для конкретного автора/чата**. Сильный кандидат проговаривает именно этот общий принцип, а не "для Instagram гибрид, потому что так делает Instagram".

## 6. Design YouTube (видео-хостинг)

### Проблема и ключевая развилка

```txt
Upload → S3 (raw video, может быть несколько GB)
  → Encoding Queue → Workers (транскодирование в несколько
    разрешений: 360p/480p/720p/1080p и форматов)
  → CDN (готовые видео раздаются оттуда, не из origin S3)
```

Транскодирование — классический пример Heavy Computation из [Universal System Design Interview Framework]: операция занимает минуты, не должна блокировать ответ пользователю "видео загружено", и должна быть retry-able при сбое воркера (если воркер упал на 80% транскодирования 1080p, перезапуск с нуля — приемлемая цена за простоту, в отличие от попытки сохранять промежуточный прогресс).

### Follow-up: почему нельзя отдавать видео напрямую из origin?

```txt
- Bandwidth: один популярный видеофайл, отдаваемый напрямую из
  одного origin-сервера миллионам пользователей — мгновенно
  насыщает сетевой канал этого сервера
- Latency: пользователи географически распределены; без CDN
  каждый запрос идёт в один датацентр, что добавляет сотни мс
- Adaptive bitrate streaming (HLS/DASH): видео отдаётся чанками
  разного качества в зависимости от сети пользователя — это
  требует множества небольших файлов, для которых CDN — это
  не "ускорение", а единственный практичный способ раздачи
```

Это применение паттерна CDN из [File Storage and CDN] к самому write-heavy по объёму трафика типу контента — видео.

## 7. Design Dropbox / Google Drive (синхронизация файлов)

### Проблема и ключевая развилка

Разделение **метаданных** (структура папок, версии, права доступа — реляционная БД) и **содержимого файлов** (object storage, S3-подобный):

```txt
Client
 ↓
API (auth, разрешения, метаданные)
 ↓                    ↓
Metadata DB        Object Storage (S3)
(файлы, папки,     (содержимое файлов,
 версии, владельцы) адресуется по content hash)
```

Главная идея, отличающая "просто файловое хранилище" от Dropbox — **синхронизация между устройствами**, и именно она почти всегда становится follow-up.

### Follow-up: как синхронизировать изменения между устройствами?

```txt
Наивно: каждое устройство периодически опрашивает "что изменилось?"
  → задержка синхронизации = интервал опроса, лишняя нагрузка на API

Лучше: Change Events + long-lived соединение
  - при изменении файла на устройстве A → API создаёт
    событие "FileChanged" с версией
  - устройства B, C подписаны на эти события (WebSocket/long
    polling, см. [WebSockets]) → получают уведомление и
    скачивают изменённую часть файла
```

### Follow-up: как избежать повторной передачи всего файла при небольшом изменении?

```txt
Делим файл на блоки (chunks) фиксированного размера,
адресуем по hash содержимого (content-addressed storage):

  - изменился 1 абзац в большом документе → изменился только
    1-2 chunk'а → передаются только они
  - если несколько пользователей загружают одинаковый файл
    (одинаковый набор chunks) → дедупликация на уровне storage
```

Это применение принципа "версионирование" к данным, разбитым на content-addressed блоки — похожий принцип используется в Git (тот же подход к хранению объектов по hash).

## 8. Design Ride Sharing (Uber, Bolt)

### Проблема и ключевая развилка

Главная техническая сложность — не "хранить поездки в БД", а **постоянно обновляемая геопозиция тысяч водителей** и быстрый поиск "кто рядом":

```txt
Driver app → периодически (каждые 3-5 сек) отправляет
  координаты → Location Service

Location Service хранит позиции в структуре, оптимизированной
для geo-запросов, НЕ в обычной реляционной таблице с
полным сканом по lat/lng
```

### Follow-up: как искать ближайших водителей эффективно?

```txt
Geohash: кодирует (lat, lng) в строку — близкие координаты
  имеют общий префикс строки
  → "найти водителей рядом" = поиск по префиксу

  geohash("50.450, 30.523") → "ucewef..."
  → все водители с префиксом "ucewe" — в той же области

Альтернатива: spatial index (PostGIS GiST index, или
  Redis GEOADD/GEORADIUS — Redis хранит геоданные как
  отсортированное множество по geohash под капотом)
```

Сильный ответ объясняет ПОЧЕМУ обычный B-tree индекс по (lat, lng) не работает для запроса "в радиусе 2 км": диапазонные запросы по двум независимым измерениям одновременно не сводятся к одному range scan, а geohash/spatial index превращает 2D-близость в 1D-близость (общий префикс / отсортированная структура).

### Follow-up: как происходит matching водителя и пассажира?

```txt
1. Пассажир запрашивает поездку → Matching Service находит
   N ближайших доступных водителей (через geo-индекс выше)
2. Запрос отправляется водителям (по одному или пачкой) через
   realtime-канал (WebSocket/push)
3. Первый принявший — назначается; остальным отправляется "отмена"
4. Race condition: что если 2 водителя одновременно нажали "принять"?
   → атомарная операция (DB transaction с условным UPDATE,
     или distributed lock на ride_id) — аналогично проблеме
     double-booking в Booking System ниже
```

## 9. Design Booking System (Airbnb, кинотеатр, авиабилеты)

### Проблема и ключевая развилка

Главная проблема — **double booking**: два пользователя одновременно бронируют последнее свободное место/дату.

```txt
❌ Наивно:
  1. SELECT — проверить, свободно ли
  2. если свободно → INSERT booking

  Между шагами 1 и 2 другой запрос может пройти ту же
  проверку → оба INSERT-а успешны → овербукинг
```

### Решение и follow-up: всегда ли нужен Redis Lock?

```txt
✅ Решение по умолчанию — атомарность на уровне БД:

  BEGIN;
  SELECT * FROM seats WHERE id = ? FOR UPDATE;  -- row-level lock
  -- если seat.status = 'available' → UPDATE seats SET status = 'booked'
  COMMIT;

  Второй параллельный запрос на тот же seat блокируется на
  SELECT FOR UPDATE до завершения первой транзакции —
  гарантия даётся самой БД, без дополнительной инфраструктуры.
```

`SELECT FOR UPDATE` достаточен для бронирования с относительно низким write-throughput на конкретный ресурс (одно место в кинотеатре редко пытаются забронировать тысячи раз в секунду). Redis distributed lock становится нужен, когда: (а) операция бронирования распределена между несколькими сервисами/шагами (например, "удержать место → подождать оплату → подтвердить" — нельзя держать DB-транзакцию открытой на время ожидания оплаты), или (б) ресурс настолько горячий (распродажа билетов на популярный концерт), что даже row-level lock в БД становится узким местом — тогда лок в Redis с коротким TTL "придерживает" место на время оформления заказа, а финальное подтверждение всё равно идёт через атомарную операцию в БД.

```txt
Удержание места (hold) с TTL — частый паттерн:
  Redis: SET seat:123:hold user_456 EX 600 NX
  → если пользователь не оплатит за 10 минут, hold истекает
    автоматически, место освобождается без явной "отмены"
```

## 10. Design News Feed (аналогично Instagram Feed)

Главная развилка идентична Instagram Feed (см. выше) — fan-out on write vs read, гибрид по соотношению write:read для конкретного источника контента. Дополнительная специфика News Feed — **ranking**: лента не просто хронологическая, а отсортирована по релевантности (engagement-сигналы, recency, отношения между пользователями).

### Follow-up: что станет bottleneck при добавлении ranking?

```txt
Хронологическая лента: сортировка по одному полю (created_at) —
  тривиально для БД/precomputed feed

Ranked лента: score = f(recency, author_affinity, engagement, ...)
  → score не статичен (engagement постов меняется со временем) →
    "предрасчитанная лента" из fan-out on write устаревает быстрее

Практическое решение: двухэтапный подход —
  1. Candidate generation — получить ~500 кандидатов
     (через fan-out/precomputed feed, как раньше)
  2. Ranking — пересчитать score и отсортировать ТОЛЬКО эти
     500 кандидатов в момент запроса (дешевле, чем ранжировать
     весь корпус постов)
```

Это снова применение общего принципа из [Universal System Design Interview Framework]: дорогая операция (ranking по множеству сигналов) выполняется над маленьким уже отфильтрованным набором, а не над всеми данными.

## 11. Design Rate Limiter

### Проблема и решение

```txt
Redis INCR + TTL — простейший рабочий вариант (Fixed Window Counter):

  key = `ratelimit:{userId}:{currentMinute}`
  INCR key
  EXPIRE key 60 (только при первом INCR)
  if value > limit → 429 Too Many Requests
```

### Follow-up: в чём проблема Fixed Window и что лучше?

```txt
Fixed Window: пользователь может сделать `limit` запросов
  в konце минуты N и ещё `limit` запросов в начале минуты N+1
  → 2x limit за короткий промежуток на границе окон

Sliding Window Log / Sliding Window Counter:
  - хранит точные timestamps запросов (или счётчики по
    под-интервалам) и считает запросы за последние 60 секунд
    ОТ ТЕКУЩЕГО МОМЕНТА, а не от начала фиксированного окна
  - точнее, но дороже по памяти/вычислениям

Token Bucket:
  - "ведро" токенов пополняется с фиксированной скоростью,
    каждый запрос тратит токен
  - позволяет короткие всплески (burst) до размера ведра,
    что часто ближе к реальным требованиям API (один клиент
    может слать пачку запросов сразу после паузы)
```

### Follow-up: как сделать rate limiter распределённым (несколько API-серверов)?

```txt
Redis INCR — атомарная операция сама по себе уже решает
эту проблему: все инстансы API инкрементируют один и тот же
ключ в общем Redis, поэтому race condition между серверами
не возникает без дополнительных locks.

Senior-нюанс: Redis становится единой точкой отказа для
rate limiting — если Redis недоступен, решение по умолчанию
обычно "fail open" (пропускать запросы без лимита), а не
"fail closed" (отклонять все запросы), т.к. отказ rate
limiter'а не должен останавливать весь API.
```

## 12. Design Search System (Google-подобный, product search)

### Проблема и ключевая развилка

```txt
PostgreSQL → CDC/Events → Elasticsearch (или OpenSearch)
  - PostgreSQL остаётся source of truth для транзакционных данных
  - Elasticsearch — read-side индекс, оптимизированный для
    full-text поиска (это CQRS из [Database Scaling]:
    write-модель и read-модель разделены)
```

### Follow-up: почему не `WHERE name LIKE '%query%'` в PostgreSQL?

```txt
- LIKE '%...%' (с wildcard в начале) НЕ может использовать
  B-tree индекс — full table scan на каждый поисковый запрос
- Full-text возможности PostgreSQL (tsvector/GIN индекс)
  существуют, но не покрывают: релевантность с учётом
  опечаток (fuzzy matching), синонимы, ранжирование по
  множеству полей с весами, фасетный поиск (агрегации по
  атрибутам) — это специализация Elasticsearch/Lucene
- При большом объёме документов и высокой частоте поисковых
  запросов, специализированный поисковый индекс масштабируется
  горизонтально (sharding по умолчанию в Elasticsearch)
  независимо от транзакционной БД
```

Senior-нюанс, который стоит упомянуть: синхронизация PostgreSQL → Elasticsearch — это **eventually consistent** (через очередь событий или CDC) — то есть сразу после создания записи она может на короткое время не находиться в поиске. Если это неприемлемо для конкретного use case (например, "я только что создал объявление и хочу сразу найти его сам") — нужен явный fallback на прямой запрос к PostgreSQL для "своих" записей, либо синхронная индексация для критичных операций.

## Общие паттерны across всех 12 задач

Если обобщить — почти все задачи сводятся к комбинации одних и тех же строительных блоков, разобранных в предыдущих темах:

```txt
Load Balancer → API (stateless) → Redis (cache/lock/rate-limit)
                                 → PostgreSQL (source of truth)
                                 → Queue → Workers (async/heavy)
                                 → S3/CDN (файлы, статика)
                                 → WebSocket (realtime)
                                 → Elasticsearch (поиск, опционально)
```

Но **набор блоков — это не ответ**. Ответ — это последовательность решений по [Universal System Design Interview Framework]: requirements → scale → данные/access patterns → какие из этих блоков нужны ИМЕННО для этой задачи и почему, с явными trade-offs.

## Типичные ошибки на интервью

- **Воспринимать каждую задачу как новую** — большинство "новых" формулировок (Instagram Feed, News Feed, Chat groups) сводятся к одной и той же развилке fan-out on write vs read; кандидат, который видит эту связь, демонстрирует более глубокое понимание, чем тот, кто решает каждую задачу "с нуля".

- **Перечислять блоки без привязки к конкретной задаче** — "Load Balancer, Redis, PostgreSQL, Queue, S3, CDN" одинаково подходит почти любой системе, и именно поэтому само по себе ничего не доказывает.

- **Redis Lock как ответ на любую concurrency-задачу** — для большинства booking-сценариев `SELECT FOR UPDATE` достаточен и проще; распределённый лок нужен в конкретных случаях (операция распределена между сервисами, либо ресурс экстремально горячий).

- **"Fail closed" по умолчанию для вспомогательных систем** — rate limiter, recommendation engine, search — недоступность Redis/Elasticsearch не должна останавливать основной функционал (можно отдать заказ без рекомендаций, нельзя — без оплаты).

- **Игнорировать eventual consistency между write-моделью и read-моделью** (PostgreSQL → Elasticsearch, или precomputed feed) — не упоминать, что эти данные могут быть на короткое время не синхронизированы, и не обсуждать, критично ли это для конкретного use case.

- **Не проговаривать общий принцип за решением** — например, объяснить geohash как "магическую строку для Uber", а не как общий приём "превратить 2D-близость в 1D-близость через общий префикс", который применим шире.
