# Chat System Design

## Requirements Clarification и scope

"Спроектируй чат как WhatsApp/Slack" — слишком широко. Сильный ответ начинается с сужения scope:

**Functional Requirements (для обсуждения выбираем 1:1 + группы небольшого размера):**

```txt
- 1:1 сообщения
- групповые чаты (уточнить: лимит участников — 10 человек или 10 000,
  это РАЗНЫЕ архитектурные задачи, см. ниже)
- история сообщений
- online/offline статус, read receipts, typing indicator
- (отдельно уточнить, "вне scope" на первой итерации) голос/видео,
  end-to-end шифрование, файлы/медиа
```

**Non-Functional Requirements:**

```txt
- Latency доставки сообщения: десятки-сотни мс
- Сообщение не должно теряться (durability) — это явное отличие
  от "лайка" или presence-статуса, которые можно потерять без катастрофы
- Поддержка миллионов одновременных WebSocket-соединений
- Ordering: сообщения в рамках одного чата должны приходить
  в консистентном порядке у всех участников
```

Транспортный слой (WebSocket, проблема привязки соединения к серверу, Pub/Sub для межсерверной доставки, presence через TTL) уже разобран в [WebSockets и Realtime Systems] — здесь эти механизмы используются как данность, а фокус — на специфике чата: хранение сообщений, порядок, доставка, группы.

## Send Message Flow — порядок шагов критичен

```txt
User A → WebSocket → Chat Service
                          │
                          ▼
                 1. Persist в БД (source of truth)
                          │
                          ▼
                 2. ACK отправителю ("delivered to server")
                          │
                          ▼
                 3. Попытка доставки User B (если online)
                          │
                          ▼
                 4. Если offline → push-уведомление через очередь
```

Почему **persist строго до ACK**, а не наоборот: если сначала отправить ACK, а потом упасть при записи в БД — отправитель думает, что сообщение доставлено, но оно потеряно навсегда. Это прямое применение принципа "WebSocket — транспорт, БД — источник истины" из темы про WebSocket. Любая "оптимизация", меняющая этот порядок ради latency, превращает durability в "обычно работает".

## Message Ordering — почему "просто timestamp" не работает

Наивный подход — сортировать сообщения по `created_at`. Проблема: при высокой нагрузке несколько сообщений могут получить **одинаковый timestamp** (особенно если время хранится с точностью до миллисекунды и БД принимает тысячи записей в секунду), а часы на разных серверах слегка рассинхронизированы (clock skew).

```txt
Решение: составной упорядочивающий ключ —

  (chat_id, sequence_number) — монотонно возрастающий номер
  ВНУТРИ конкретного чата, а не глобально

  message_id = snowflake-подобный ID:
    [timestamp][server_id][sequence] — упорядочен по времени
    создания и глобально уникален без координации между серверами
```

Snowflake ID (подход Twitter/Discord) — это, по сути, тот же принцип, что Ticket Server из темы URL Shortener: каждый сервер генерирует ID локально из своего диапазона/namespace, без блокирующего похода в центральную БД на каждое сообщение, но итоговые ID остаются упорядочиваемыми по времени.

## Хранение сообщений: схема и партиционирование

```sql
messages (
  id           BIGINT PRIMARY KEY,  -- snowflake ID, сортируем по нему
  chat_id      BIGINT NOT NULL,
  sender_id    BIGINT NOT NULL,
  content      TEXT,
  created_at   TIMESTAMP,
  INDEX (chat_id, id)  -- основной паттерн чтения: "сообщения чата X, по порядку"
)
```

Главный паттерн чтения — "последние N сообщений чата X" — это почти всегда **range scan по индексу `(chat_id, id)`**, что делает PostgreSQL/MySQL вполне адекватным выбором даже при больших объёмах, если данные правильно индексированы.

**Партиционирование по `chat_id`** (а не по времени) становится нужным, когда объём сообщений превышает возможности одного узла — это прямое применение hash-based sharding из [Database Scaling]: shard key = `chat_id` гарантирует, что все сообщения одного чата (основной паттерн чтения) остаются в одном шарде, избегая cross-shard запросов для самой частой операции.

## Групповые чаты: fan-out — главный senior-вопрос темы

Это разница между "просто доставить сообщение одному человеку" и реальной сложностью группового чата.

### Fan-out on write (push model)

```txt
User отправляет сообщение в группу из 500 человек
  → Chat Service создаёт 500 "записей о доставке"
    (или публикует в Pub/Sub-каналы всех 500 участников)
  → каждый участник получает push немедленно при подключении
```

```txt
Плюсы: чтение истории — простая выборка "мои сообщения" (если используется
       модель per-user inbox), быстрая отдача непрочитанных
Минусы: ОДНО сообщение в группе из 500 человек = 500 операций записи/публикации.
        Для очень больших групп (10 000+ участников, как публичные каналы
        Telegram) — это становится узким местом на запись.
```

### Fan-out on read (pull model)

```txt
Сообщение пишется ОДИН раз в общую таблицу группы (shared timeline)
  → каждый участник при подключении/запросе читает
    "сообщения группы X с моего last_read_id"
```

```txt
Плюсы: запись O(1) независимо от размера группы
Минусы: чтение дороже (нужно вычислять "что я ещё не видел"
        для каждого пользователя при каждом запросе)
```

**Практическое решение, используемое крупными системами**: гибрид — fan-out on write для небольших групп (большинство чатов, размер до сотен участников — write-стоимость приемлема), fan-out on read для огромных каналов/broadcast-групп (Telegram-каналы с миллионами подписчиков — здесь "разослать всем при каждом сообщении" немыслимо, читатели сами подтягивают новые сообщения общего канала).

Это прямая параллель с темой Read-Heavy vs Write-Heavy из [System Design Fundamentals]: маленькая группа — фактически read-heavy с точки зрения каждого участника (он читает свой inbox чаще, чем пишет), огромный broadcast-канал — структурно ближе к "один writer, миллионы readers", где push-модель не масштабируется.

## Read Receipts и Typing Indicators — разная durability

```txt
Read Receipts (прочитано/доставлено):
  ХРАНЯТСЯ в БД — это часть истории переписки,
  пользователь ожидает, что статус "прочитано" сохранится
  между сессиями.

  message_status (message_id, user_id, status, updated_at)
  status: sent | delivered | read

Typing Indicator ("печатает..."):
  НЕ хранится в БД — это эфемерное состояние с TTL в несколько секунд.
  Передаётся как WebSocket-событие напрямую между клиентами
  (через тот же Pub/Sub-механизм, что и сообщения).
  Если событие "потеряется" — последствие нулевое
  (просто индикатор не появится/не исчезнет на секунду раньше).
```

Это иллюстрирует общий принцип: **не всё в realtime-системе требует одинаковых гарантий доставки**. Применение durability "сообщения" (persist-then-ack-then-deliver) к typing indicator — избыточная сложность; применение "fire and forget" к самому сообщению — потеря данных.

## Offline Delivery и Push Notifications

```txt
User B offline в момент отправки сообщения
  → сообщение сохранено в БД (status: sent)
  → Chat Service публикует в Notification Queue
  → Push Worker → APNs/FCM → push-уведомление на устройство

При следующем подключении User B:
  → клиент запрашивает "сообщения чатов X, Y, Z с last_synced_id"
  → догружает пропущенное, помечает delivered/read
```

Очередь для push здесь — снова применение паттерна из [Message Queues]: отправка push-уведомления может быть медленной/нестабильной (внешний сервис APNs/FCM), и она не должна блокировать основной message flow.

## Финальная архитектура

```txt
Clients
  ↓ (WebSocket)
Load Balancer (L4, передаёт Upgrade-заголовки)
  ↓
WebSocket Gateway servers (stateless для HTTP, но держат
                            долгоживущие соединения)
  ↓                                    ↑
Chat Service ──→ PostgreSQL/MySQL    Redis Pub/Sub
  (persist,         (messages,        (межсерверная доставка,
   sequence IDs)     chat_members,     presence — см. WebSocket-тему)
                      message_status)
  ↓
Notification Queue → Push Worker → APNs/FCM
```

## Типичные ошибки на интервью

- **Не обсуждать порядок persist → ack → deliver** — самый частый "детальный" follow-up, и его пропуск выглядит как незнание, почему сообщения не теряются при сбоях.

- **Сортировка по `created_at` без обсуждения коллизий timestamp'ов и clock skew** — не предлагать snowflake-ID или составной `(chat_id, sequence)`.

- **Не различать fan-out on write и fan-out on read для групп** — это центральный вопрос для "что если группа из 100 000 человек", и ответ "просто разошлём всем" не масштабируется.

- **Хранить typing indicator в БД** — или, наоборот, пытаться сделать сообщения "fire and forget" как presence — путать требования к durability разных типов данных.

- **Партиционировать сообщения по времени, а не по `chat_id`** — это ломает основной паттерн чтения "история конкретного чата", требуя cross-shard запросов для самой частой операции.

- **Игнорировать межсерверную доставку** — повторять архитектуру без Redis Pub/Sub/Kafka, как если бы все WebSocket-соединения были на одном сервере (см. [WebSockets и Realtime Systems]).

- **Push-уведомления синхронно в основном потоке** — блокировать ответ пользователю на время вызова внешнего APNs/FCM API.
