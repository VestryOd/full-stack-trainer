# URL Shortener Design

## Requirements Clarification

Это один из самых "стандартных" вопросов, именно поэтому интервьюер ждёт структурированный подход, а не "сразу архитектуру" (см. [System Design Fundamentals]).

**Functional Requirements:**

```txt
- по длинному URL создать короткий код (POST /shorten)
- по короткому коду редиректить на оригинальный URL (GET /:code)
- (уточнить) пользовательские алиасы ("app.com/my-link")?
- (уточнить) срок жизни ссылки (expiration)?
- (уточнить) аналитика по переходам — нужна ли, в каком виде?
```

**Non-Functional Requirements:**

```txt
- Read:Write ratio — обычно 100:1 или выше (одну ссылку создают раз,
  переходят по ней многократно) → read-heavy, нужен кеш
- Latency редиректа — должна быть минимальной (это в "горячем" пути UX)
- Уникальность кодов — обязательна, коллизии недопустимы
- Доступность — высокая (редирект не должен падать,
  даже если аналитика временно недоступна)
```

### Быстрая оценка масштаба

```txt
100M новых ссылок/месяц → ~40 writes/sec (средний)
Read:Write = 100:1     → ~4000 reads/sec (средний), пики выше

Storage: 100M ссылок/месяц * 500 байт (URL + метаданные) * 5 лет
  ≈ 100M * 12 * 5 * 500B ≈ 3 TB за 5 лет — не проблема для одной БД
  (объём НЕ требует шардинга сам по себе — требует только индекс по shortCode)
```

Вывод: эта система **не** требует sharding по объёму данных — ключевая сложность в другом: генерация уникальных кодов на масштабе и кеширование read-пути.

## Генерация коротких кодов — главный вопрос темы

### Вариант 1: Hash (MD5/SHA256) + обрезка

```ts
function generateCode(longUrl: string): string {
  const hash = crypto.createHash('md5').update(longUrl).digest('hex');
  return hash.slice(0, 7); // первые 7 символов hex
}
```

- Плюсы: детерминированно (тот же URL → тот же код — можно использовать как дедупликацию).
- Минусы: **коллизии неизбежны** при обрезке хеша — нужна проверка в БД и стратегия "если занято — взять следующие N символов или добавить соль". Это превращает простую операцию в потенциально многошаговую с retry.

### Вариант 2: Base62-кодирование auto-increment ID

```ts
const BASE62 = '0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ';

function toBase62(num: number): string {
  if (num === 0) return BASE62[0];
  let result = '';
  while (num > 0) {
    result = BASE62[num % 62] + result;
    num = Math.floor(num / 62);
  }
  return result;
}
// toBase62(125) → "cb"
// 7 символов Base62 покрывают 62^7 ≈ 3.5 * 10^12 комбинаций
```

- Плюсы: гарантированная уникальность (ID уникален по определению — `PRIMARY KEY AUTO_INCREMENT`), код становится короче по мере роста, без коллизий.
- Минусы — и это **главный senior-нюанс**: auto-increment ID из одной БД — это **single point of contention** на запись. На масштабе много инстансов API не могут безопасно "сами" генерировать следующий ID без похода в БД на каждый запрос.

### Решение для распределённой генерации: Pre-allocated ID ranges (Ticket Server)

```txt
Отдельный сервис (или таблица в БД с атомарным UPDATE) выдаёт
КАЖДОМУ инстансу API целый ДИАПАЗОН ID за раз:

  Instance A: запрашивает диапазон → получает [1..1000]
  Instance B: запрашивает диапазон → получает [1001..2000]

Каждый инстанс генерирует короткие коды из своего диапазона
ЛОКАЛЬНО, без обращения к БД на каждый запрос —
обращение к "тикет-серверу" происходит раз на 1000 запросов.
```

```sql
-- атомарное выделение следующего диапазона
UPDATE id_ranges SET current_max = current_max + 1000
WHERE id = 1
RETURNING current_max - 1000 AS range_start, current_max AS range_end;
```

Это классическое решение (используется, например, в Flickr's Ticket Server, Instagram ID generation) — оно превращает "БД как bottleneck на каждую запись" в "БД как bottleneck раз на 1000 записей", что снижает нагрузку на порядки.

### Вариант 3: Random + проверка коллизий

```ts
async function generateUniqueCode(): Promise<string> {
  for (let attempt = 0; attempt < 5; attempt++) {
    const code = randomBase62String(7);
    const exists = await db.shortUrl.findUnique({ where: { code } });
    if (!exists) return code;
  }
  throw new Error('Failed to generate unique code after 5 attempts');
}
```

- При длине 7 символов Base62 (~3.5 * 10^12 комбинаций) вероятность коллизии при миллионах записей крайне мала (см. "birthday paradox"), но **не нулевая** — поэтому всегда нужна проверка + retry, и важно явно проговорить, что это вероятностный, а не гарантированный подход (в отличие от Base62 от auto-increment).

## Read Path: redirect — самый горячий путь системы

```txt
GET /abc123
  ↓
Redis: GET shortcode:abc123 → cache hit → 302 Redirect (быстро)
  ↓ (miss)
PostgreSQL: SELECT longUrl WHERE shortCode = 'abc123'
  ↓
Redis: SET shortcode:abc123 (с TTL)
  ↓
302 Redirect
```

### 301 vs 302 — неочевидный, но реальный senior-вопрос

```txt
301 Moved Permanently:
  Браузер КЕШИРУЕТ редирект локально — повторные обращения
  к /abc123 могут вообще не доходить до сервера.
  + меньше нагрузка на сервер
  - невозможно посчитать клики (браузер не делает повторный запрос)
  - невозможно изменить целевой URL для существующего короткого кода

302 Found (Temporary Redirect):
  Браузер НЕ кеширует — каждый переход доходит до сервера.
  + точная аналитика кликов
  + можно изменить целевой URL позже
  - больше нагрузка на сервер (компенсируется кешем в Redis)
```

Для URL-шортенера с аналитикой правильный выбор — **302**, и это нужно явно обосновать (часто кандидаты выбирают 301 "потому что permanent звучит правильно", не подумав про аналитику).

## Analytics — асинхронно, не в hot path

```txt
GET /abc123
  ↓ (синхронно, минимальная latency)
302 Redirect отправлен пользователю
  ↓ (асинхронно, fire-and-forget)
Queue: { shortCode: 'abc123', timestamp, userAgent, referrer, ip }
  ↓
Worker → batched insert в Analytics DB (ClickHouse/аналитический store)
```

Почему не `UPDATE clicks SET count = count + 1` синхронно на каждый редирект: это лишняя запись в БД **на критическом пути** редиректа — при пиковой нагрузке именно она может стать bottleneck'ом, замедляя то, что должно быть максимально быстрым. Очередь (см. [Message Queues]) развязывает "ответить пользователю" от "посчитать статистику".

## Custom Aliases и Expiration

```txt
Custom alias ("app.com/my-campaign"):
  - проверка уникальности перед созданием (UNIQUE constraint на shortCode)
  - не покрывается генерацией Base62/Ticket Server —
    отдельный путь записи с явной проверкой коллизий

Expiration:
  - поле expiresAt в таблице ShortUrls
  - проверка при редиректе: если expiresAt < now() → 404/410 Gone
  - периодическая job для удаления истёкших записей
    (или TTL на уровне Redis-кеша + lazy-проверка в БД)
```

## Финальная архитектура

```txt
                    ┌─────────────┐
Client ──────────→ │ Load Balancer│
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │  API Servers │ (stateless)
                    └──────┬──────┘
              ┌────────────┼────────────────┐
              ▼            ▼                ▼
           Redis      PostgreSQL        Queue (clicks)
        (hot codes)  (source of truth)       │
                                              ▼
                                       Analytics Worker
                                              │
                                              ▼
                                       Analytics DB
```

Read replicas для PostgreSQL добавляются, если redirect-нагрузка после Redis-кеша всё ещё значима — но в большинстве реалистичных сценариев hit ratio Redis для популярных ссылок (Zipf-распределение: малая доля ссылок получает большую часть трафика) делает это не первоочередной задачей.

## Типичные ошибки на интервью

- **Использовать `hash % N` или простую обрезку MD5 без обсуждения коллизий** — без проверки + retry-стратегии это не "решение", а источник скрытых багов на масштабе.

- **Auto-increment ID без обсуждения распределённой генерации** — на одном сервере это просто, но "как это работает с 50 API-инстансами" — ожидаемый follow-up, и Ticket Server/pre-allocated ranges — ожидаемый ответ.

- **Выбирать 301 redirect "по умолчанию"** — без понимания, что это делает аналитику невозможной и пользователь может не доходить до сервера повторно.

- **Синхронный инкремент счётчика кликов** — добавление записи в БД на каждый редирект в hot path, без очереди.

- **Не оценивать масштаб данных** — пытаться обосновать sharding для системы, где 5 лет данных занимают пару TB и прекрасно живут в одной БД с индексом.

- **Игнорировать кеш как "опциональное улучшение"** — для read:write 100:1+ кеш — это не оптимизация, это центральный элемент архитектуры.
