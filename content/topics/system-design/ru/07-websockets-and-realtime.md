# WebSockets и Realtime Systems

## Главная проблема HTTP

HTTP работает по модели:

```txt
Request
 ↓
Response
```

---

Сервер не может сам написать клиенту.

---

# Пример

Чат.

---

Пользователь А отправил сообщение.

---

Как пользователь Б узнает об этом?

---

Вариант №1

Постоянно спрашивать сервер.

---

```txt
GET /messages
GET /messages
GET /messages
```

---

Это называется:

```txt
Polling
```

---

# Недостаток

Очень много лишних запросов.

---

# Long Polling

Улучшенная версия.

---

Клиент отправляет запрос.

---

Сервер держит соединение открытым.

---

Пока не появятся данные.

---

После ответа:

```txt
новый запрос
```

---

# Недостаток

Все равно много HTTP запросов.

---

# WebSocket

Решение проблемы.

---

```txt
Client
 ↕
 Server
```

---

Связь двусторонняя.

```txt
NATS
```

# Частый вопрос

Когда использовать WebSocket?

Ответ:

Когда сервер должен отправлять данные клиенту в реальном времени.

---

# Частый вопрос

Чем WebSocket лучше HTTP?

Ответ:

Не нужно постоянно создавать новые соединения.

---

# Частый вопрос

Как масштабировать WebSocket?

Ответ:

Через Redis Pub/Sub или брокер сообщений.

---

# Interview Answer

WebSocket обеспечивает постоянное двустороннее соединение между клиентом и сервером и используется для чатов, уведомлений, совместного редактирования документов и других realtime систем.
