# WebSockets и Realtime Systems

## Проблема: HTTP — это request/response, а серверу нужно "говорить первым"

HTTP по своей модели — клиент инициирует, сервер отвечает. Сервер не может сам отправить данные клиенту вне ответа на запрос. Но многим продуктам (чаты, live-уведомления, котировки, совместное редактирование) нужно именно это: **сервер инициирует отправку**, когда происходит событие.

## Сравнение подходов к realtime

| Подход | Как работает | Latency | Overhead | Когда подходит |
|---|---|---|---|---|
| **Short Polling** | Клиент каждые N секунд шлёт `GET /messages` | До N секунд | Очень высокий — большинство запросов возвращают "ничего нового" | Почти никогда не оптимален, но прост в реализации |
| **Long Polling** | Клиент шлёт запрос, сервер держит его открытым до появления данных или таймаута, затем клиент сразу шлёт новый | Низкая | Средний — соединение держится, но переоткрывается | Когда WebSocket недоступен (старые прокси/файрволы), низкая частота событий |
| **Server-Sent Events (SSE)** | Одно HTTP-соединение, сервер пишет события в поток (`text/event-stream`) | Низкая | Низкий, но **только сервер → клиент** | Уведомления, live-фиды, прогресс long-running задач — где клиенту не нужно "отвечать" по тому же каналу |
| **WebSocket** | Полноценное двунаправленное соединение после handshake | Минимальная | Низкий, постоянное соединение | Чаты, игры, совместное редактирование — нужна **обоюдная** низколатентная передача |

Senior-нюанс: SSE часто упускают, а это значительно более простое решение, чем WebSocket, когда коммуникация **односторонняя** (server → client) — оно работает по обычному HTTP (проще проходит через корпоративные прокси/балансировщики, автоматический реконнект встроен в браузерный API `EventSource`). Предлагать WebSocket для "показывать пользователю live-уведомления" — рабочее, но избыточное решение; SSE решает это проще.

## WebSocket Handshake — почему это важно для архитектуры

```http
Клиент:
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==

Сервер:
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: ...
```

Ключевой практический момент: соединение **начинается как обычный HTTP-запрос** и "переключается" в WebSocket-протокол через `101 Switching Protocols`. Это значит:

- L7 load balancer должен явно поддерживать и пропускать `Upgrade`-заголовки (не все конфигурации Nginx/ALB делают это по умолчанию — частая причина "WebSocket работает локально, не работает за балансировщиком");
- после handshake соединение **долгоживущее** (stateful) — это принципиально отличается от обычных коротких HTTP-запросов и влияет на то, как балансировщик и инфраструктура должны его обрабатывать (таймауты, keep-alive, лимиты на количество одновременных соединений на инстанс).

## Главная архитектурная сложность: соединение привязано к конкретному серверу

Это **самый важный senior-вопрос** в этой теме, и старая версия статьи правильно его обозначала, но без деталей решения.

```txt
10 серверов за load balancer'ом

User A подключен по WebSocket к Server #3
User B подключен по WebSocket к Server #8

User A отправляет сообщение для User B
  → попадает на Server #3 (его соединение)
  → но WebSocket-соединение User B живёт на Server #8

Server #3 физически не может "написать" в socket, которым владеет Server #8
```

### Решение: Pub/Sub как шина между инстансами

```txt
Server #3 (получил сообщение от User A)
   ↓ публикует в Redis Pub/Sub channel "user:B:messages"
Redis Pub/Sub
   ↓ доставляет всем подписанным инстансам
Server #8 (подписан на "user:B:messages", т.к. держит соединение User B)
   ↓ пишет в WebSocket-соединение User B
User B получает сообщение
```

```ts
// Server #8 при подключении User B подписывается на его канал
const subscriber = redis.duplicate();
await subscriber.subscribe(`user:${userId}:messages`);
subscriber.on('message', (channel, payload) => {
  const socket = activeConnections.get(userId);
  socket?.send(payload); // доставка в конкретный открытый WebSocket
});

// Любой сервер, получивший сообщение для userId, публикует его
await redis.publish(`user:${recipientId}:messages`, JSON.stringify(message));
```

Это тот же паттерн, что и Pub/Sub из темы Message Queues — Redis Pub/Sub подходит для небольшого/среднего масштаба (низкая latency, простота); для крупного масштаба используют Kafka/NATS, которые дают персистентность (если consumer временно отвалился — Redis Pub/Sub сообщение теряется безвозвратно, у Kafka — нет).

### Connection Registry — "кто на каком сервере"

Для каналов/комнат (а не индивидуальных пользователей) нужен ещё один компонент — реестр, отвечающий "какие серверы держат соединения участников комнаты X":

```txt
Redis: room:42:servers = {server-3, server-8, server-15}

При подключении к комнате: SADD room:42:servers <server-id>
При отключении:            SREM room:42:servers <server-id>

Публикация в комнату:
  for server in room:42:servers:
    publish to "server:<id>:broadcast" channel
```

Без этого реестра серверу пришлось бы публиковать в **все** инстансы "на всякий случай" — это работает на маленьком масштабе, но не масштабируется на тысячи комнат.

## Presence (online/offline) — TTL вместо явного disconnect

```txt
Проблема: WebSocket-соединение может "тихо" умереть
(потеря сети, закрытие ноутбука) без явного close-события —
сервер не всегда узнаёт, что клиент отключился
```

Решение — heartbeat + TTL в Redis:

```ts
// Клиент каждые 30 секунд отправляет heartbeat
// Сервер при каждом heartbeat обновляет TTL
await redis.set(`presence:user:${userId}`, 'online', 'EX', 60); // TTL 60 сек

// Если heartbeat не пришёл за 60 секунд — ключ протухает сам,
// presence:user:123 перестаёт существовать → пользователь "offline"
// (нет необходимости в явном disconnect-обработчике)
```

TTL-подход надёжнее, чем "обработать onDisconnect", потому что disconnect-событие может не сработать при аварийном разрыве сети — а TTL гарантированно "истечёт" независимо от того, получил ли сервер уведомление об отключении.

## Delivery Guarantees в чате — offline-доставка

WebSocket доставляет сообщение только если получатель **сейчас онлайн**. Для офлайн-пользователей нужен отдельный путь:

```txt
1. Сообщение всегда сначала пишется в БД (источник истины) — Message Service
2. Затем пытаемся доставить через WebSocket (если пользователь online)
3. Если пользователь offline:
   - сообщение остаётся в БД со статусом "unread"
   - (опционально) push-уведомление через APNs/FCM
4. При следующем подключении клиент запрашивает
   "все сообщения с last_seen_message_id" — догружает пропущенное
```

Senior-нюанс: WebSocket — это **транспорт для доставки в реальном времени**, а не источник истины и не гарантия доставки. Любая надёжная realtime-система имеет "под капотом" обычную модель persist-then-notify, а WebSocket — это просто самый быстрый из путей notify.

## Load Balancing для WebSocket

```txt
Round Robin на новые соединения — нормально, выбор сервера
происходит один раз при установке соединения.

После установки — соединение живёт на этом сервере до разрыва.
Это НЕ "sticky session" в смысле HTTP cookie — это просто
физическое свойство TCP-соединения: оно держится одним сервером
по определению, пока не закрыто.
```

Главное следствие для capacity planning: при рестарте/деплое сервера **все** WebSocket-соединения на нём рвутся одновременно — клиенты должны иметь реконнект-логику (с backoff), а graceful shutdown должен давать клиентам сигнал на переподключение до полного отключения сервера, а не просто рвать TCP.

## Типичные ошибки на интервью

- **Не упоминать, что соединение "застряло" на конкретном сервере** — это центральная сложность масштабирования WebSocket, и именно это интервьюер хочет услышать.

- **Предлагать sticky sessions как решение масштабирования WebSocket** — это не решение проблемы межсерверной доставки, это просто описание того, как и так устроено TCP-соединение; реальная проблема — доставка между серверами через Pub/Sub.

- **Игнорировать SSE** — для односторонних обновлений (уведомления, live-фиды) предлагать WebSocket как единственный вариант, не упоминая значительно более простой SSE.

- **Считать WebSocket надёжным транспортом** — не объяснять persist-then-notify и offline-доставку; WebSocket сам по себе не гарантирует доставку.

- **Presence через явный disconnect-обработчик без TTL** — не учитывая, что соединения "тихо умирают" без события disconnect.

- **Не упоминать connection registry для room-based broadcast** — пытаться "разослать всем серверам на всякий случай" вместо целевой доставки через Redis Set/реестр.

- **Не обсуждать reconnect-логику клиента** — при деплое/рестарте сервера все соединения рвутся одновременно, и без backoff-реконнекта тысячи клиентов одновременно переподключаются (thundering herd на уровне соединений).
