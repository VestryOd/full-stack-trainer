# HTTP/REST — Вопросы для интервью

25–30 вопросов, сгруппированных по темам. Уровень указан: 🟢 junior · 🟡 middle · 🔴 senior.

---

## Основы HTTP

### 1. 🟢 Что такое HTTP и почему он stateless?

HTTP (HyperText Transfer Protocol) — протокол прикладного уровня поверх TCP/IP. Текстовый запрос → текстовый ответ. Stateless означает, что каждый запрос независим: сервер не хранит информации о предыдущих запросах. Именно поэтому существуют cookies, сессии и JWT — они переносят состояние явно.

Stateless — не ограничение, а осознанный выбор: упрощает горизонтальное масштабирование (любой инстанс обработает любой запрос), кэширование и отладку.

---

### 2. 🟢 Какие HTTP-методы ты знаешь? В чём разница между GET и POST?

GET — безопасный и идемпотентный: читает ресурс, не меняет состояние сервера. POST — небезопасный и не идемпотентный: создаёт ресурс или выполняет действие.

Полная картина по safety и idempotency:

```txt
Метод   | Safe | Idempotent
--------|------|------------
GET     |  ✅  |    ✅
HEAD    |  ✅  |    ✅
OPTIONS |  ✅  |    ✅
PUT     |  ❌  |    ✅
DELETE  |  ❌  |    ✅
POST    |  ❌  |    ❌
PATCH   |  ❌  |  зависит
```

Идемпотентность важна для повторных запросов: если сеть сбросила PUT — его можно повторить безопасно. POST нельзя — создастся дубликат.

---

### 3. 🟢 Что означают коды 201, 204, 301, 304, 401, 403, 409, 429?

- **201 Created** — ресурс создан (POST). Обычно + `Location` заголовок.
- **204 No Content** — успех, тела нет (DELETE, PUT без возврата).
- **301 Moved Permanently** — ресурс переехал навсегда, браузер кэширует.
- **304 Not Modified** — кэш актуален, тела нет (ответ на `If-None-Match`).
- **401 Unauthorized** — не аутентифицирован (нет/просрочен токен).
- **403 Forbidden** — аутентифицирован, но нет прав.
- **409 Conflict** — конфликт (дубликат email при регистрации).
- **429 Too Many Requests** — rate limiting, обычно + `Retry-After`.

Ловушка: 401 называется "Unauthorized", но означает "Unauthenticated". Историческая путаница в HTTP-стандарте.

---

### 4. 🟡 Что такое идемпотентность и почему это важно на практике?

Метод идемпотентен, если многократное применение одного запроса даёт тот же результат, что и однократное.

Практическое значение: при потере соединения клиент не знает, дошёл ли запрос. Если метод идемпотентный (PUT, DELETE) — безопасно повторить. Если нет (POST) — повтор создаст дубликат заказа/платежа.

PATCH идемпотентен только если операция абсолютная: `{ "status": "active" }` — да; `{ "views": { "increment": 1 } }` — нет.

Именно поэтому платёжные системы требуют `Idempotency-Key` заголовок на POST /payments — это превращает неидемпотентный метод в идемпотентный через механизм дедупликации на сервере.

---

### 5. 🟡 В чём разница между PUT и PATCH?

PUT заменяет ресурс **целиком**: клиент отправляет полное представление. Не отправить поле = удалить его.

PATCH обновляет **частично**: только изменённые поля.

```http
PUT /users/42     { "name": "Alice", "email": "new@example.com", "role": "admin" }
                   ↑ обязательно все поля

PATCH /users/42   { "email": "new@example.com" }
                   ↑ только изменяемое
```

PUT идемпотентен по определению: `PUT /users/42` дважды с теми же данными — результат одинаковый.

## REST и дизайн API

### 6. 🟢 Что такое REST? Является ли обычный JSON API RESTful?

REST — архитектурный стиль из 6 ограничений (Fielding, 2000): Client-Server, Stateless, Cacheable, Uniform Interface, Layered System, Code on Demand. Не протокол, не стандарт.

Большинство "REST API" — это Richardson Maturity Level 2: ресурсы + HTTP-методы. Это "REST-ish", не полный REST. Полный REST требует HATEOAS (Level 3) — когда клиент следует ссылкам из ответа, не зная URL заранее. Почти никто не реализует Level 3.

---

### 7. 🟡 Что такое HATEOAS? Почему его не используют?

HATEOAS (Hypermedia As The Engine Of Application State) — ограничение REST: клиент не знает URL заранее, он следует гиперссылкам из ответов сервера, как браузер переходит по ссылкам.

```json
{
  "id": 42,
  "_links": {
    "self":    { "href": "/users/42" },
    "orders":  { "href": "/users/42/orders" },
    "suspend": { "href": "/users/42/suspend", "method": "POST" }
  }
}
```

Не используют потому что: клиенты всё равно жёстко прошивают логику навигации; документация (OpenAPI) даёт то же обнаружение API проще; нет единого стандарта (HAL, JSON:API, Siren несовместимы); динамическое построение UI на основе ссылок — очень сложно.

---

### 8. 🟡 Стратегии версионирования API. Какую выбрать и почему?

Три подхода:

- **URL** (`/v1/users`) — самый распространённый. Просто кэшировать, легко тестировать, роутить на nginx. Минус: нарушает REST (URI должен идентифицировать ресурс, а не версию API).
- **Заголовок** (`Accept: application/vnd.api.v2+json`) — REST-корректно, но сложно кэшировать (нужен `Vary: Accept`), неочевидно.
- **Query param** (`?version=2`) — удобно для optional, но путает с фильтрами.

Рекомендация для большинства проектов: `/v1` в URL. Проще, надёжнее кэшируется, понятнее команде.

## Кэширование

### 9. 🟡 Что означает `Cache-Control: no-cache`? Чем отличается от `no-store`?

Частая ловушка интервью:

- `no-cache` — кэш **хранить можно**, но перед использованием обязательно **валидировать** с сервером. Если сервер отвечает 304 — используем кэш. Это экономит трафик, но не RTT.
- `no-store` — кэш **не хранить вообще**. Каждый раз полный запрос. Для чувствительных данных (банк, медицина).

Разница критична: `no-cache` — про freshness, `no-store` — про privacy/security.

---

### 10. 🟡 Что такое ETag и как работают conditional requests?

ETag — уникальный идентификатор версии ресурса. Сервер включает его в ответ: `ETag: "v3-abc123"`.

При следующем запросе клиент отправляет `If-None-Match: "v3-abc123"`. Если ресурс не изменился — сервер возвращает `304 Not Modified` без тела. Экономия трафика при частых повторных запросах.

`Last-Modified` / `If-Modified-Since` — аналогичный механизм на основе времени. ETag точнее: файл может пересохраниться с тем же содержимым, изменив время.

`If-Match` используется при PUT/PATCH для оптимистичной блокировки: "обновить только если версия совпадает". При несовпадении → `412 Precondition Failed`.

---

### 11. 🔴 Что такое `stale-while-revalidate`? В каких сценариях оправдан?

```http
Cache-Control: public, max-age=60, stale-while-revalidate=30
```

Ресурс считается свежим 60 сек. Следующие 30 сек кэш отдаёт устаревшее пока в фоне обновляется. Максимальная несвежесть данных — 90 секунд.

Оправдан там, где важнее воспринимаемая скорость, чем абсолютная свежесть: лента новостей, публичные API, CDN с динамическим контентом.

Не подходит: банковские операции, инвентарь (где несвежесть приводит к oversell), персонализированные данные.

## CORS

### 12. 🟢 Что такое CORS и зачем он нужен?

CORS — механизм браузера, позволяющий серверу явно разрешить запросы с других origin'ов. Без него Same-Origin Policy блокирует JavaScript от чтения ответов чужих сайтов.

Ключевое: CORS не защищает сервер — запрос выполняется физически. CORS контролирует что браузер передаёт JavaScript. Именно поэтому curl не знает о CORS — это браузерная политика, не серверная.

---

### 13. 🟡 Когда браузер делает preflight OPTIONS-запрос?

Preflight отправляется при:
- Методах PUT, DELETE, PATCH
- Кастомных заголовках (`Authorization`, `X-Custom-Header`)
- `Content-Type: application/json` (!)
- Запросах с credentials на другой origin

Следствие: большинство API-запросов с JSON или токеном триггерят preflight = двойное число HTTP-запросов. `Access-Control-Max-Age: 86400` кэширует preflight-ответ в браузере на сутки.

---

### 14. 🟡 Почему нельзя использовать `Access-Control-Allow-Origin: *` с cookies?

Wildcard разрешает любому сайту читать ответ. Credentials (cookies) несут данные залогиненного пользователя. Вместе: любой сайт читает персональные данные — очевидная уязвимость.

Браузер принудительно блокирует комбинацию `*` + `credentials: "include"`. При credentials нужен конкретный origin.

---

### 15. 🔴 Зачем нужен `Vary: Origin` и что происходит без него?

Когда сервер динамически отражает `Access-Control-Allow-Origin` в ответе (один origin из списка), кэш должен знать: один URL может возвращать разный ACAO для разных клиентов.

Без `Vary: Origin`: CDN кэширует ответ с `ACAO: https://a.com` и отдаёт его клиенту с `Origin: https://b.com`. Браузер блокирует — CORS-ошибка для b.com.

С `Vary: Origin`: кэш хранит отдельные копии для каждого уникального origin.

## Пагинация

### 16. 🟡 Offset vs cursor пагинация. Когда что выбрать?

Offset (`LIMIT N OFFSET M`): прост в реализации, позволяет прыгать на страницу, есть total. Проблемы: при больших offset'ах БД читает и отбрасывает M строк (O(n) по индексу), при concurrent DML данные "дрейфуют" (дубликаты/пропуски).

Cursor (`WHERE id < lastId LIMIT N`): стабильна, O(log n) всегда, идеальна для бесконечного скролла. Нельзя прыгнуть на страницу 42, нет total.

Правило: классическая таблица с навигацией → offset. Бесконечный скролл / real-time лента / >100K записей → cursor.

---

### 17. 🔴 Почему `OFFSET 5000000` медленно работает даже с индексом?

`ORDER BY created_at DESC LIMIT 20 OFFSET 5000000` — БД проходит по индексу и **пропускает** 5 000 000 записей, только потом берёт 20. Это O(n) по offset, не O(log n).

Cursor-based (`WHERE created_at < lastValue ORDER BY created_at DESC LIMIT 20`) использует индекс напрямую — O(log n) независимо от позиции.

На таблице 10M строк offset=5M может занимать секунды; cursor — миллисекунды.

## Аутентификация

### 18. 🟡 Как работает JWT? Что содержит и как верифицируется?

JWT = `base64url(header).base64url(payload).signature`. Header содержит алгоритм (`alg: HS256`), payload — claims (`sub`, `exp`, `role`), signature — HMAC или RSA подпись.

Важно: payload не зашифрован — его может прочитать любой. Signature гарантирует **целостность** (никто не изменил), но не **конфиденциальность**.

Верификация: сервер пересчитывает signature из header + payload + secret и сравнивает. Затем проверяет `exp`. Без signature-проверки — любой может подделать токен.

---

### 19. 🟡 Access token + refresh token — зачем два токена?

Access token: короткий TTL (15 мин — 1 час), stateless, несёт данные пользователя. При компрометации — малое окно уязвимости.

Refresh token: долгий TTL (7–90 дней), хранится **на сервере** (Redis/БД), передаётся только на `/auth/refresh` через HttpOnly cookie. Хранение на сервере позволяет инвалидировать (logout = удалить из Redis).

При выходе: refresh token удаляется с сервера. Access token продолжает действовать до `exp` — именно поэтому его TTL короткий.

---

### 20. 🟡 В чём разница между сессиями и JWT?

Сессия: сервер хранит состояние в Redis/БД. Клиент хранит только ID (маленький cookie). Мгновенная инвалидация. Требует session store — дополнительный RTT на каждый запрос.

JWT: состояние внутри токена, сервер ничего не хранит (stateless). Горизонтально масштабируется без Redis. Инвалидация до истечения — только через blacklist (= обратно к stateful).

Выбор: традиционный web на одном домене → сессии. SPA/мобильные/микросервисы → JWT + refresh token.

---

### 21. 🔴 Что такое PKCE и зачем он нужен в OAuth 2.0?

PKCE (Proof Key for Code Exchange) — расширение для защиты Authorization Code Flow в публичных клиентах (SPA, мобильные), где нельзя безопасно хранить `client_secret`.

Клиент генерирует случайный `code_verifier`, отправляет его SHA256-хэш (`code_challenge`) в authorization request. При обмене code → token отправляет оригинальный `code_verifier`. Authorization Server проверяет: `SHA256(verifier) == challenge`.

Если authorization code перехвачен — без `code_verifier` он бесполезен. В SPA без PKCE — перехваченный code напрямую конвертируется в токены.

---

### 22. 🔴 Почему JWT нельзя хранить в localStorage?

localStorage доступен любому JS на странице. XSS-атака на любую зависимость или CDN-ресурс → кража токена → полный компромисс сессии.

Правильно: access token — в памяти JS (переменная), теряется при перезагрузке. Refresh token — в HttpOnly cookie (недоступен JS, браузер отправляет автоматически). При перезагрузке — тихо получить новый access token через refresh.

## Real-Time коммуникация

### 23. 🟡 В чём разница между SSE и WebSockets? Когда что выбрать?

SSE — однонаправленный HTTP-поток (сервер → клиент). Встроенное автопереподключение с `Last-Event-ID`. Работает через любой прокси. Текст только.

WebSocket — полнодуплексное TCP-соединение. Клиент и сервер шлют независимо. Поддерживает бинарные данные. Нет автопереподключения — нужно реализовать. Сложнее масштабировать.

Выбор: если нужен только push от сервера (уведомления, прогресс, стриминг LLM) → SSE. Если нужен push от клиента к серверу в реальном времени (чат, игры, совместное редактирование) → WebSockets.

---

### 24. 🟡 Как работает Server-Sent Events? Что такое `Last-Event-ID`?

EventSource открывает один HTTP GET с `Accept: text/event-stream`. Сервер держит соединение открытым, отправляет события в формате:
```
event: orderUpdate\n
data: {"id":42}\n
id: 55\n\n
```

При разрыве браузер автоматически переподключается (через `retry` мс) и отправляет `Last-Event-ID: 55`. Сервер видит это и может отдать пропущенные события с ID > 55. Без обработки `Last-Event-ID` события теряются при кратком разрыве.

---

### 25. 🔴 Как масштабировать WebSocket-сервер горизонтально?

Проблема: клиент на инстансе A хочет отправить сообщение пользователю на инстансе B. В памяти инстанса A нет соединения этого пользователя.

Решение: Redis Pub/Sub. При получении сообщения от клиента → сохранить + `publish("user:42", data)` в Redis. Все инстансы подписаны на `user:*`. Инстанс B получает из Redis и отправляет по WebSocket.

Альтернативы: Kafka (высокая пропускная способность), Socket.IO с Redis adapter (готовое решение), sticky sessions (нарушает horizontal scaling, но проще).

---

### 26. 🔴 Сравни polling, long polling, SSE и WebSockets по нагрузке при 10 000 клиентах.

Polling (5 сек интервал): 2 000 req/sec постоянно, 95% пустых. Нагрузка предсказуема и независима от числа событий.

Long polling: в idle — 10 000 открытых HTTP-соединений ожидают. При событии — 10 000 ответов + 10 000 новых подключений мгновенно. Пики нагрузки.

SSE: 10 000 постоянных HTTP-потоков. События только когда есть данные. Минимальный overhead (keepalive комментарии раз в 30 сек).

WebSocket: 10 000 постоянных TCP-соединений. Минимальный per-message overhead (без HTTP-заголовков). Ping/pong каждые 30 сек. Самый эффективный по трафику при высокой частоте событий.

## HTTP/2 и HTTP/3

### 27. 🔴 Что такое head-of-line blocking? HTTP/2 решил эту проблему?

HoL-блокировка в HTTP/1.1: одно TCP-соединение = один запрос за раз. Медленный ответ блокирует последующие. Браузеры открывали 6 соединений — частичное решение.

HTTP/2 решил HoL **на уровне HTTP**: мультиплексирование стримов в одном TCP-соединении. Стримы независимы на уровне HTTP.

**Но**: TCP-level HoL остался. Потеря одного TCP-пакета блокирует все HTTP/2 стримы (TCP гарантирует порядок байт, не зная о стримах).

HTTP/3 (QUIC поверх UDP) решил и TCP-level HoL: стримы независимы на уровне транспорта. Потеря пакета одного стрима не блокирует другие.

---

### 28. 🔴 В чём уязвимость 0-RTT в HTTP/3?

0-RTT позволяет клиенту отправить данные вместе с первым пакетом установки соединения (нет ожидания handshake). Это возможно потому что QUIC запоминает параметры предыдущей сессии.

Уязвимость: **replay attack**. Перехваченный 0-RTT пакет можно отправить повторно. Сервер не может отличить оригинал от replay в рамках одного 0-RTT.

Защита: 0-RTT только для **идемпотентных** запросов (GET). POST/PUT в 0-RTT — сервер должен иметь механизм дедупликации (nonce, timestamp check). QUIC-серверы могут отвечать `RETRY` чтобы форсировать 1-RTT handshake для чувствительных операций.

---

### 29. 🟡 Что изменится в подходе к оптимизации фронтенда при переходе с HTTP/1.1 на HTTP/2?

HTTP/1.1 оптимизации, которые **вредят** в HTTP/2:
- Domain sharding (множество поддоменов для 6×N соединений) → в HTTP/2 мультиплексирование в одном соединении лучше
- CSS/JS спрайты (слияние файлов ради меньшего числа запросов) → много маленьких файлов ≈ один большой
- Inline CSS/JS → лишает кэширования

HTTP/2 оптимизации:
- Granular caching: маленькие файлы кэшируются независимо
- 103 Early Hints вместо Server Push
- HTTP/2 приоритизация для критических ресурсов

---

### 30. 🔴 Как работает HMAC-подпись запросов для API? Зачем она нужна?

HMAC (Hash-based Message Authentication Code) подписывает весь запрос (метод + путь + timestamp + тело) с помощью секретного ключа. Сервер пересчитывает подпись и сравнивает.

Преимущества перед простым API-ключом:
- Защита от man-in-the-middle: изменение любого байта запроса делает подпись невалидной
- Защита от replay: timestamp в подписи + сервер проверяет "не старше 5 минут"
- Нельзя переиспользовать перехваченный запрос на другой endpoint

Используется: AWS SDK (`AWS4-HMAC-SHA256`), Stripe webhook verification, платёжные системы.

```typescript
const message = `${method}\n${path}\n${timestamp}\n${body}`;
const signature = crypto.createHmac("sha256", secret).update(message).digest("hex");
// Сервер: проверяет timestamp, пересчитывает signature, сравнивает
```
