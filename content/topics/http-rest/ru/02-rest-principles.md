# Принципы REST

## Что такое REST на самом деле

REST (Representational State Transfer) — не протокол и не стандарт. Это **архитектурный стиль**, описанный Роем Филдингом в его диссертации 2000 года. Филдинг был одним из авторов HTTP/1.1 и описал REST как набор ограничений, которым должна удовлетворять система, чтобы получить определённые свойства (масштабируемость, независимость компонентов, кэшируемость).

```txt
REST — это не:
  ❌ HTTP API с JSON
  ❌ "CRUD через HTTP"
  ❌ набор соглашений об именовании URL
  ❌ стандарт с официальной спецификацией

REST — это:
  ✅ набор из 6 архитектурных ограничений
  ✅ стиль, из которого следуют свойства системы
  ✅ идея о том, как должна выглядеть масштабируемая
     распределённая система в Web
```

Большинство API называет себя "RESTful", реально удовлетворяя 3–4 из 6 ограничений. Это нормально — это "практический REST" или "REST-ish". На интервью важно отличать теоретический REST от того, что реально строится.

## Шесть ограничений Филдинга — простым языком

### 1. Client-Server (Клиент-Сервер)

Клиент и сервер разделены: клиент не знает о хранении данных, сервер не знает о пользовательском интерфейсе. Это разделение позволяет развивать их независимо.

```txt
Что это даёт:
  - Клиент (браузер, мобильное приложение) меняется независимо от сервера
  - Сервер не хранит состояние UI — это забота клиента
  - Один API обслуживает несколько клиентов (web, mobile, CLI)
```

### 2. Stateless (Отсутствие состояния)

Каждый запрос содержит всю информацию, необходимую серверу для его обработки. Сервер не хранит контекст между запросами.

```txt
Что это даёт:
  - Горизонтальное масштабирование: любой экземпляр сервера
    обработает любой запрос (нет sticky sessions)
  - Отказоустойчивость: падение инстанса не теряет состояние
  - Видимость: каждый запрос самодостаточен для мониторинга

Цена:
  - Клиент отправляет токен аутентификации с каждым запросом
  - Немного больше трафика (повторные заголовки)
```

### 3. Cacheable (Кэшируемость)

Ответы должны явно указывать, можно ли их кэшировать. Кэшируемые ответы могут повторно использоваться клиентом или промежуточными узлами (CDN, прокси).

```http
Cache-Control: max-age=3600, public   ← можно кэшировать 1 час, включая CDN
Cache-Control: no-store               ← нельзя кэшировать (персональные данные)
ETag: "abc123"                        ← версия для conditional requests
```

### 4. Uniform Interface (Единый интерфейс)

Это центральное ограничение REST. Оно само состоит из 4 подограничений:

```txt
4a. Resource Identification:
    Ресурсы идентифицируются URI: /users/42, /orders/100/items

4b. Resource Manipulation through Representations:
    Клиент работает с представлениями ресурса (JSON, XML),
    а не с ресурсом напрямую. Ресурс — это концепция,
    представление — то, что передаётся по сети.

4c. Self-Descriptive Messages:
    Каждое сообщение содержит достаточно информации для обработки:
    Content-Type, статус-код, семантика метода.

4d. HATEOAS:
    Клиент переходит между состояниями через ссылки, которые
    возвращает сервер. (Подробнее ниже.)
```

### 5. Layered System (Многоуровневая система)

Клиент не знает, с кем общается напрямую: сервер, CDN, load balancer, API gateway. Промежуточные узлы прозрачны.

```txt
Клиент → CDN (cache hit, ответ без бэкенда)
Клиент → Load Balancer → Один из N серверов
Клиент → API Gateway → Microservice A или B
```

### 6. Code on Demand (опциональное)

Сервер может передавать исполняемый код клиенту (JavaScript). Единственное необязательное ограничение — именно поэтому web-браузеры загружают JS и это "RESTful".

## Ресурсно-ориентированное проектирование URL

Центральная идея REST: **URL идентифицирует ресурс** (существительное), **HTTP-метод описывает действие** (глагол).

### Базовые правила

```txt
✅ Правильно — существительные, иерархия ресурсов:
GET    /users            — список пользователей
GET    /users/42         — конкретный пользователь
POST   /users            — создать пользователя
PUT    /users/42         — заменить пользователя целиком
PATCH  /users/42         — обновить частично
DELETE /users/42         — удалить пользователя

GET    /users/42/orders          — заказы конкретного пользователя
GET    /users/42/orders/7        — конкретный заказ пользователя
POST   /users/42/orders          — создать заказ для пользователя

❌ Неправильно — глаголы в URL (RPC-стиль):
GET  /getUser?id=42
POST /createUser
POST /deleteUser/42
POST /updateUserEmail
```

### Коллекции и отдельные ресурсы

```txt
/users          — коллекция (множественное число)
/users/42       — отдельный ресурс (идентификатор)
/users/42/posts — подколлекция (посты пользователя 42)

Глубина вложенности: не более 2–3 уровней
/users/42/posts/7/comments — допустимо
/users/42/posts/7/comments/3/replies/1 — слишком глубоко,
  лучше: /comments/3/replies или /replies/1
```

### Действия, не укладывающиеся в CRUD

Не всё в API — CRUD. Как моделировать действия:

```txt
Вариант 1 — Суб-ресурс (предпочтительно):
POST /orders/42/cancel         — создать "отмену" заказа
POST /users/42/password-reset  — запустить процесс сброса
POST /articles/7/publish       — опубликовать статью

Вариант 2 — Изменение поля через PATCH:
PATCH /orders/42
{ "status": "cancelled" }

Вариант 3 — Отдельная коллекция событий:
POST /order-cancellations
{ "orderId": 42, "reason": "customer_request" }
```

### Фильтрация, сортировка, пагинация — query params

```txt
Фильтрация:
GET /users?role=admin&status=active

Сортировка:
GET /users?sort=createdAt&order=desc
GET /users?sort=-createdAt          (минус = desc — популярная конвенция)

Пагинация:
GET /users?page=2&limit=20          (offset-based)
GET /users?cursor=eyJpZCI6NDJ9&limit=20 (cursor-based)

Поиск:
GET /users?q=alice
GET /users?search=alice&fields=name,email

Выборка полей (sparse fieldsets):
GET /users?fields=id,name,email
```

## Честная картина: REST в реальности — это "RPC с соглашениями"

Большинство "REST API" на практике — это RPC (Remote Procedure Call) с HTTP-транспортом и JSON, с некоторыми REST-конвенциями. Вот почему:

```txt
Теоретический REST требует:
  1. HATEOAS — клиент следует ссылкам из ответов, не знает URL заранее
  2. Content negotiation — сервер отдаёт JSON или XML по Accept заголовку
  3. Самоописывающие сообщения — каждый ответ содержит всю семантику

Практика:
  1. Клиенты знают URL заранее (жёстко прошито в коде или документации)
  2. JSON и только JSON (XML давно умер для большинства API)
  3. Документация — OpenAPI/Swagger, а не HATEOAS
```

Это нормально. "Уровень зрелости REST" (Richardson Maturity Model) описывает это явно:

```txt
Уровень 0: Один URL, один метод (SOAP, XML-RPC)
  POST /api {"action": "getUser", "id": 42}

Уровень 1: Ресурсы (разные URL, но один метод)
  POST /users/42

Уровень 2: HTTP-глаголы (методы несут семантику)
  GET /users/42   → читаем
  DELETE /users/42 → удаляем
  ↑ БОЛЬШИНСТВО "REST API" находится здесь ↑

Уровень 3: HATEOAS (ответ содержит ссылки на следующие действия)
  ↑ редко встречается в реальных API ↑
```

## HATEOAS — объяснение и честная оценка

HATEOAS (Hypermedia As The Engine Of Application State) — самое радикальное и наименее реализованное ограничение REST.

Идея: клиент не должен знать URL заранее. Он получает стартовую точку (`/api`) и следует гиперссылкам, которые возвращает сервер — как человек переходит по ссылкам в браузере.

```json
// Ответ с HATEOAS (HAL-формат):
{
  "id": 42,
  "name": "Alice",
  "status": "active",
  "_links": {
    "self":    { "href": "/users/42" },
    "orders":  { "href": "/users/42/orders" },
    "suspend": { "href": "/users/42/suspend", "method": "POST" },
    "delete":  { "href": "/users/42", "method": "DELETE" }
  }
}
```

Клиент смотрит на `_links` и понимает: какие действия доступны прямо сейчас, не зная URL заранее.

```txt
Теоретические преимущества HATEOAS:
  - Сервер может менять URL без поломки клиентов (клиент следует ссылкам)
  - Клиент видит, какие действия доступны в текущем состоянии
    (suspend только если status=active)
  - Настоящее разделение клиента и сервера

Почему почти никто не реализует полностью:
  1. Клиенты всё равно жёстко прошивают логику:
     "после POST /login иду на /dashboard"
  2. Парсить _links и динамически строить UI — очень сложно
  3. Документация (OpenAPI) даёт те же преимущества обнаружения API,
     но проще в реализации
  4. Браузер следует ссылкам сам. REST API обычно используется
     JavaScript-кодом, которому нужна предсказуемость, а не открытие
  5. Нет устоявшегося стандарта: HAL, JSON:API, Siren, Collection+JSON —
     несовместимые форматы
```

**Практический вывод:** знайте что такое HATEOAS для интервью, понимайте зачем оно нужно теоретически, но не ждите его в реальных проектах.

## Версионирование API

Когда API меняется несовместимо (breaking change), нужна версионность. Три подхода:

### 1. Версия в URL (наиболее распространено)

```txt
https://api.example.com/v1/users
https://api.example.com/v2/users
```

**Плюсы:**
- Видно сразу, без инспекции заголовков
- Легко тестировать в браузере/curl
- Просто кэшировать (URL уникален)
- Легко роутить на nginx/API gateway

**Минусы:**
- Нарушает принцип REST: URI должен идентифицировать ресурс, а не версию API
- `/v1/users/42` и `/v2/users/42` — один и тот же ресурс
- Клиенты хардкодят версию, миграция требует изменений кода

### 2. Версия в заголовке (Accept или кастомный)

```http
GET /users/42 HTTP/1.1
Accept: application/vnd.myapi.v2+json

# Или кастомный заголовок:
GET /users/42 HTTP/1.1
API-Version: 2
```

**Плюсы:**
- URL остаётся чистым — ресурс идентифицируется правильно
- Теоретически ближе к REST (content negotiation)

**Минусы:**
- Сложнее тестировать (нужен curl/Postman, не браузер)
- Кэширование проблематично: один URL, разный контент для разных версий
  (нужен `Vary: Accept` заголовок)
- Менее очевидно для разработчиков, читающих код клиента

### 3. Версия в query-параметре

```txt
GET /users/42?api-version=2
GET /users/42?v=2
```

**Плюсы:**
- Опционален: без параметра — дефолтная версия
- Легко добавить в существующий API

**Минусы:**
- Query params традиционно используются для фильтрации, а не версионирования
- Создаёт путаницу (`?version=2&status=active`)
- Кэши могут игнорировать query params

### Сравнение и рекомендация

```txt
┌────────────────┬──────────┬──────────┬──────────┐
│                │ URL /v1  │ Header   │ Query ?v │
├────────────────┼──────────┼──────────┼──────────┤
│ Видимость      │ ✅ Сразу  │ ❌ Скрыт │ ✅ Видно  │
│ Кэширование    │ ✅ Просто │ ❌ Сложно │ ⚠️ Риски │
│ REST-корректно │ ❌ Нет   │ ✅ Да     │ ❌ Нет   │
│ DX (удобство)  │ ✅ Просто │ ⚠️ Сложно│ ✅ Просто │
│ Распространён  │ ✅✅✅     │ ✅        │ ⚠️        │
└────────────────┴──────────┴──────────┴──────────┘

Рекомендация для большинства проектов: /v1 в URL.
Проще, надёжнее, понятнее команде и клиентам.
```

### Стратегии выхода из старых версий

```txt
1. Deprecation-период: документируйте устаревание, ставьте
   заголовок Deprecation/Sunset:

   Deprecation: true
   Sunset: Sat, 31 Dec 2026 23:59:59 GMT
   Link: <https://api.example.com/v2/users>; rel="successor-version"

2. Параллельные версии: /v1 и /v2 работают одновременно
   минимум 6–12 месяцев

3. Не делайте /v1 бессмертной: устаревшие версии — технический долг
```

## Типичные ошибки на интервью

- **"REST — это HTTP + JSON"** — нет. REST — архитектурный стиль с 6 ограничениями, не зависящий от протокола. Теоретически REST можно реализовать поверх FTP или email. На практике — HTTP, но суть в ограничениях, а не в протоколе.

- **"Наш API RESTful"** — уточните уровень. Большинство API находится на уровне 2 по Ричардсону (ресурсы + HTTP-методы). HATEOAS (уровень 3) встречается редко. Знание этого разграничения показывает глубину понимания.

- **"URL `POST /users/42/activate` нарушает REST"** — нет, это допустимый паттерн суб-ресурса. REST не запрещает действия — он говорит думать ресурсно. "Активация" — это создание состояния "активный" или суб-ресурса "активация".

- **"Версия в URL нарушает REST, нужно использовать заголовки"** — технически верно (URI должен идентифицировать ресурс), но на практике версия в URL — стандарт индустрии по причинам кэширования, отладки и простоты. Филдинг сам говорил, что прагматизм важнее чистоты.

- **"HATEOAS не нужен вообще"** — нужно понимать зачем он существует. Его цель — настоящее разделение клиента и сервера, возможность менять URL без перекомпиляции клиента. Что это обычно решается документацией и версионированием — честный ответ.

- **"DELETE /users — удалить всех пользователей"** — технически по REST правилам это корректно (DELETE на коллекцию). На практике — почти никогда не реализуется из-за очевидных рисков. Правильный ответ: знать теорию и понимать прагматичное отклонение.
