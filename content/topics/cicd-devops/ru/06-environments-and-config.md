# Environments and Configuration

## Что означает "окружение" в DevOps

В DevOps **окружение** (environment) — это автономное развёртывание вашего приложения со своей инфраструктурой, конфигурацией и данными. Разные окружения служат разным целям в жизненном цикле разработки.

Канонический набор окружений для веб-приложения:

```txt
Ноутбук разработчика     → локальное окружение (local)
                            цель: написать и вручную протестировать код
                            данные: примеры/фиктивные данные, локальная БД
                            кто использует: разработчик

CI-пайплайн              → тестовое окружение (ephemeral — временное, создаётся под каждый запуск)
                            цель: автоматизированное тестирование (unit, integration)
                            данные: тестовые фикстуры, in-memory или throwaway БД
                            кто использует: CI-раннер (автоматически)

Общий сервер             → dev-окружение (development)
                            цель: интеграция между несколькими сервисами,
                            тестирование фич до staging
                            данные: синтетические/анонимизированные данные
                            кто использует: разработчики, иногда QA

QA / пред-прод сервер    → staging-окружение (также называется "pre-prod")
                            цель: финальная проверка перед production
                            должно быть максимально близко к prod
                            данные: копия prod-данных (анонимизированная) или
                            реалистичные синтетические данные
                            кто использует: QA, продакт-менеджеры, демо для клиентов

Боевые серверы           → production-окружение ("prod")
                            цель: обслуживает реальных конечных пользователей
                            данные: реальные пользовательские данные
                            кто использует: конечные пользователи
```

Не все команды используют все окружения. Многие стартапы работают только с тремя: local → staging → production. В крупных организациях могут быть отдельные QA, UAT (User Acceptance Testing — стадия, на которой клиент или владелец продукта проверяет фичу перед запуском), нагрузочное тестирование и окружения для восстановления после катастроф.

### Почему "staging должен зеркалить production"

Staging существует для поиска багов, которые проявляются только в production-условиях: данные в production-масштабе, production-размер инфраструктуры, реальные ответы сторонних API, production TLS-сертификаты, production CDN-конфигурация. Если staging существенно отличается от production, он не выполняет свою главную задачу.

Распространённые расхождения staging-production, вызывающие баги "работает на staging, сломано в prod":

```txt
- Разные переменные окружения (staging использует mock-провайдер платежей, prod — реальный)
- Разные характеристики серверов (staging: 1 CPU / 1 ГБ RAM, prod: 4 CPU / 16 ГБ RAM)
- Разные версии БД (staging: Postgres 14, prod: Postgres 15)
- Разное поведение CDN/кэширования (staging без CDN, prod использует CloudFront)
- Один экземпляр на staging vs несколько на prod
  (выявляет гонки состояний, проблемы с session affinity)
```

## Переменные окружения

**Переменная окружения** (environment variable) — это пара ключ-значение, доступная процессу во время выполнения, устанавливаемая операционной системой или лаунчером процесса — не захардкоженная в исходном коде приложения.

```bash
# Установка переменной окружения в shell
export DATABASE_URL="postgres://user:password@localhost:5432/myapp"
export NODE_ENV="production"
export PORT="3000"

# Запущенное приложение читает их
node dist/server.js
```

В Node.js:

```ts
const dbUrl = process.env.DATABASE_URL;
const port = parseInt(process.env.PORT ?? '3000', 10);
const isProd = process.env.NODE_ENV === 'production';
```

Почему переменные окружения — не захардкоженные значения или конфиг-файлы, закоммиченные в репозиторий:

```txt
1. Безопасность: учётные данные (пароли БД, API-ключи) не должны быть в version control.
   Любой с доступом на чтение к репозиторию увидит их. История коммитов постоянна.

2. Гибкость: один и тот же Docker-образ запускается на staging с staging-учётными данными,
   и на prod с prod-учётными данными — сам образ не меняется.
   "Собрать однажды, запускать везде."

3. Принцип 12-factor app: один из основополагающих принципов (с 12factor.net)
   построения переносимых, cloud-native приложений — "хранить конфиг в окружении".
   Конфиг = всё, что меняется между деплоями (dev/staging/prod).
```

### `NODE_ENV` — специальная конвенция

`NODE_ENV` — де-факто стандарт в экосистеме Node.js. Его значение меняет поведение многих библиотек:

```txt
NODE_ENV=development  → React включает полные сообщения об ошибках и dev-предупреждения
                        Express показывает stack trace в ответах на ошибки
                        Webpack/Vite не минифицируют, включают source maps
                        Многие библиотеки пропускают оптимизации для удобства отладки

NODE_ENV=production   → React убирает dev-предупреждения, минифицирует имена компонентов
                        Express скрывает stack trace (безопасность)
                        Webpack/Vite минифицируют, tree-shake'ят, оптимизируют бандл
                        Многие библиотеки включают кэширование и оптимизированные пути
```

```ts
// Распространённый паттерн: разное поведение по окружению
if (process.env.NODE_ENV !== 'production') {
  app.use(morgan('dev'));          // подробное HTTP-логирование в dev
  app.use(errorHandler());        // детальные ответы на ошибки в dev
}
```

**Никогда не устанавливайте `NODE_ENV=development` на staging.** Staging должен работать с `NODE_ENV=production`, чтобы ловить production-mode баги (проблемы минификации, отсутствующие error boundary и т.д.).

## Конвенция файла `.env`

Файл `.env` (читается "dot env" — "точка env") — это обычный текстовый файл в корне проекта, хранящий переменные окружения в формате `КЛЮЧ=ЗНАЧЕНИЕ`, по одной на строку:

```bash
# .env  (пример)
DATABASE_URL=postgres://postgres:password@localhost:5432/myapp_dev
REDIS_URL=redis://localhost:6379
JWT_SECRET=local-dev-secret-not-for-production
PORT=3000
NODE_ENV=development
STRIPE_SECRET_KEY=sk_test_...
```

Библиотеки вроде **dotenv** (Node.js) читают этот файл при запуске приложения и загружают переменные в `process.env`:

```ts
// В самом начале точки входа (до любых других импортов, использующих process.env)
import 'dotenv/config';
// или
import dotenv from 'dotenv';
dotenv.config();
```

**`.env` только для локальной разработки. НИКОГДА не должен коммититься в version control.**

Правильная настройка `.gitignore`:

```gitignore
# .gitignore
.env
.env.local
.env.*.local
```

Что вместо этого КОММИТИТСЯ в репозиторий — файл `.env.example` (также называемый `.env.template`) с именами переменных, но без реальных значений:

```bash
# .env.example  — коммитится в репозиторий, безопасно для публикации
DATABASE_URL=postgres://user:password@localhost:5432/dbname
REDIS_URL=redis://localhost:6379
JWT_SECRET=change-this-to-a-random-secret
PORT=3000
NODE_ENV=development
STRIPE_SECRET_KEY=sk_test_your_key_here
```

Этот файл сообщает каждому разработчику, какие переменные нужны, не раскрывая реальные учётные данные. Новые члены команды копируют его (`cp .env.example .env`) и вносят реальные локальные значения.

### Несколько `.env`-файлов

В некоторых проектах используется несколько `.env`-файлов для разных окружений:

```txt
.env              → значения по умолчанию (наименьший приоритет, обычно безопасные дефолты)
.env.local        → локальные переопределения (наивысший приоритет, не коммитится)
.env.development  → значения для dev-окружения
.env.test         → значения для тестового окружения (загружается при NODE_ENV=test)
.env.production   → значения для prod (часто просто заглушка — реальные значения из CI-секретов)
```

Эта конвенция встроена во фреймворки Next.js, Vite и Create React App. Порядок загрузки и правила приоритетов специфичны для фреймворка — знайте правила того фреймворка, который используете.

## Секреты vs переменные окружения vs конфиг-файлы

Эти три механизма часто путают. Они отличаются как тем, что хранят, так и тем, как управляются:

```txt
                 Переменные     Конфиг-       Secrets
                 окружения      файлы         Manager
─────────────────────────────────────────────────────────────────
Что хранит       Runtime-конфиг Статический   Чувствительные
                 (нечувств.)    конфиг приложения учётные данные
                                (флаги фич,   (пароли, API-ключи,
                                таймауты...)  TLS-сертификаты,
                                              ключи шифрования)

Где живёт        Переменная     Файл в        Внешний сервис
                 процесса, CI-  репозитории   (AWS Secrets Manager,
                 секреты,       или деплое    HashiCorp Vault,
                 платформа                    GCP Secret Manager)

Кто устанавливает Ops / CI      Разработчик   Ops / команда
                 пайплайн                     безопасности

Версионируется?  Нет (в CI-     Да            Нет (секреты
                 хранилище      (в репо)      ротируются отдельно
                 секретов)                    от кода)

Ротируется без   Вручную        Требует       Да (автоматическая
деплоя?          (обновить CI-  деплоя        ротация поддерж.
                 переменную)                  большинством менедж.)
```

### Когда что использовать

**Переменные окружения** для значений, которые:
- Меняются между окружениями (URL dev ≠ URL prod)
- Нечувствительны (номера портов, флаги фич, уровни логирования)
- Или чувствительны, но будут инжектированы платформой во время выполнения

**Конфиг-файлы** для значений, которые:
- Одинаковы во всех окружениях (дефолтные таймауты запросов, лимиты пагинации, константы алгоритмов)
- Безопасны для нахождения в кодовой базе
- Выигрывают от версионирования и ревью как код

```ts
// config/defaults.ts — коммитится в репозиторий
export const config = {
  pagination: { defaultLimit: 20, maxLimit: 100 },
  cache: { ttlSeconds: 300 },
  upload: { maxFileSizeMb: 10 },
};
```

**Secrets manager** для значений, которые:
- Высоко чувствительны (prod пароль БД, приватные ключи, учётные данные платёжного процессора)
- Требуют audit log (кто обращался к этому секрету и когда)
- Требуют ротации (автоматическая периодическая ротация без переразвёртывания)
- Требуют мелкогранулярного контроля доступа (только эта Lambda-функция может читать этот секрет)

## Управление секретами

### Почему нельзя коммитить секреты в репозиторий

```txt
Причина 1 — История Git постоянна
  Даже если вы удалили секрет из файла и закоммитили удаление,
  секрет всё равно виден в git log / git blame на старом коммите.
  Любой с доступом к репозиторию (текущим или историческим) может его найти.
  Инструменты truffleHog, gitleaks сканируют историю git на предмет утечек.

Причина 2 — Доступ к репозиторию ≠ доступ к production
  Разработчики, контрактники, боты GitHub Actions и Dependabot — все
  имеют доступ на чтение к репозиторию. Они НЕ должны автоматически
  иметь доступ к prod-паролям БД или API-ключам платёжных систем.

Причина 3 — Секреты в публичных репозиториях = немедленная компрометация
  Боты сканируют GitHub на предмет утечших AWS-ключей, Stripe-секретов и т.д.
  в течение секунд после пуша коммита. AWS сами запускают автоматические
  сканеры и уведомляют вас — но к тому времени может быть уже поздно.
```

Что делать, если вы случайно закоммитили секрет:

```bash
# Шаг 1: НЕМЕДЛЕННО ротируйте секрет (измените пароль/перегенерируйте ключ)
# История git не может быть полностью очищена в общем репозитории без force-push
# на все ветки и повторного клонирования всеми — это непрактично.
# Единственное безопасное средство — ротация.

# Шаг 2: Удалить из истории git (необязательно, но не заменяет ротацию)
git filter-repo --path .env --invert-paths
# или: BFG Repo Cleaner (внешний инструмент)
# Внимание: требует force-push; все коллаборанты должны перезаклонировать репо
```

### Что такое secrets manager

**Secrets manager** (менеджер секретов) — специализированный сервис для хранения, доступа и ротации чувствительных учётных данных. Он обеспечивает:

- **Шифрование at rest**: секреты хранятся зашифрованными, не в открытом виде
- **Audit log**: каждый доступ записывается (какой сервис, какой пользователь, в какое время)
- **Контроль доступа**: мелкогранулярные права (только этот EC2-инстанс может читать этот секрет)
- **Автоматическая ротация**: ротировать секрет по расписанию без изменения кода или переразвёртывания
- **Версионирование**: хранить предыдущие версии секрета для отката

Распространённые secrets manager'ы:

```txt
AWS Secrets Manager    → управляется AWS; интегрируется с Lambda, ECS, RDS;
                         встроенная автоматическая ротация паролей RDS;
                         стоимость: ~$0.40/секрет/месяц

HashiCorp Vault        → open-source, self-hosted или cloud-managed (HCP Vault);
                         поддерживает множество бэкендов секретов (БД, PKI, SSH, облако);
                         сложнее в эксплуатации, чем managed-сервисы

GCP Secret Manager     → эквивалент Google Cloud
Azure Key Vault        → эквивалент Microsoft Azure

Doppler / Infisical    → developer-friendly SaaS менеджеры секретов;
                         интегрируются с CI/CD-пайплайнами и локальной разработкой
```

Как секреты попадают в работающее приложение:

```txt
Вариант A: Инжекция как переменные окружения во время деплоя
  CI-пайплайн получает секрет из AWS Secrets Manager
  → инжектирует его как env-переменную в контейнер/функцию
  → приложение читает process.env.DATABASE_URL
  Недостаток: секрет виден как env-переменная (читается из /proc на Linux)

Вариант B: Приложение получает при запуске
  Приложение вызывает SDK AWS Secrets Manager при старте
  → получает секрет напрямую
  → хранит в памяти (никогда не в env-переменной)
  Лучше для: высокочувствительных секретов, ротации без перезапуска

Вариант C: Инжекция через sidecar / агент
  Sidecar-контейнер (напр., Vault Agent) работает рядом с app-контейнером,
  получает секреты и записывает их в общий in-memory volume
  Приложение читает из файлов в /vault/secrets/
  Лучше для: Kubernetes-окружений с Vault
```

Пример — получение секрета из AWS Secrets Manager в Node.js:

```ts
import {
  SecretsManagerClient,
  GetSecretValueCommand,
} from '@aws-sdk/client-secrets-manager';

const client = new SecretsManagerClient({ region: 'us-east-1' });

async function getDbPassword(): Promise<string> {
  const response = await client.send(
    new GetSecretValueCommand({ SecretId: 'prod/myapp/db-password' }),
  );
  if (!response.SecretString) throw new Error('Secret not found');
  const { password } = JSON.parse(response.SecretString);
  return password;
}
```

## Infrastructure as Code (IaC)

**Infrastructure as Code** (IaC — инфраструктура как код, произносится "ай-эк") — это практика определения и управления инфраструктурой — серверами, сетями, базами данных, load balancer'ами, DNS-записями — в файлах кода (как правило, YAML, JSON или предметно-ориентированный язык), а не путём кликов в веб-консоли или ручного выполнения shell-команд.

```txt
Без IaC (ручной подход, "ClickOps"):
  Инженер идёт в консоль AWS
  → кликает "Create EC2 instance"
  → выбирает тип инстанса, AMI, security group
  → вручную настраивает load balancer
  → вручную обновляет DNS
  Проблема: не воспроизводимо, не аудируемо, configuration drift
             между окружениями, невозможно проверить в PR

С IaC:
  Инженер пишет конфигурационный файл
  → файл коммитится в version control
  → изменения проверяются в PR
  → CI-пайплайн применяет изменения автоматически
  → staging и production определены идентично в коде
  Выгода: воспроизводимо, аудируемо, паритет окружений, восстановление после катастроф
```

### Configuration drift (дрейф конфигурации)

Без IaC **configuration drift** (дрейф конфигурации) неизбежен: окружения, начавшиеся идентичными, постепенно расходятся, потому что кто-то внёс ручное изменение в production ("только на этот раз"), забыл применить его на staging, и теперь два окружения незаметно отличаются. Это главный источник "работает на staging, сломано в prod".

IaC решает это, делая код единственным источником истины — если чего-то нет в коде, этого не существует.

### Инструменты IaC — краткая карта

```txt
HashiCorp Terraform  → наиболее широко используется; не привязан к провайдеру
                        (AWS, GCP, Azure, Cloudflare, Vercel...); синтаксис HCL
                        (HashiCorp Configuration Language); state-файл отслеживает
                        что задеплоено

AWS CDK              → AWS Cloud Development Kit; определение AWS-инфраструктуры
(Cloud Development     на TypeScript/Python/Java; компилируется в CloudFormation;
Kit)                   ощущается как написание кода приложения

AWS CloudFormation   → нативный IaC для AWS; JSON/YAML-шаблоны; зрелый и глубоко
                        интегрированный с AWS; многословный, но надёжный

Pulumi               → похож на CDK, но не привязан к провайдеру; инфраструктура
                        пишется на TypeScript/Python/Go и т.д.

Ansible              → управление конфигурацией (установка пакетов, управление
                        файлами, запуск команд на серверах); иной фокус по
                        сравнению с Terraform — Terraform подготавливает
                        инфраструктуру, Ansible настраивает то, что на ней работает
```

### Что fullstack-разработчику реально нужно знать об IaC

Скорее всего, вы не будете писать Terraform или CDK с нуля как fullstack-инженер — это обычно работа DevOps/platform-инженера. Но вы должны понимать:

```txt
1. Зачем существует IaC — альтернатива (ручные клики в консоли) не масштабируется,
   не воспроизводима и не аудируема

2. Рабочий процесс "plan before apply":
   terraform plan   → показывает что изменится (как "diff" для инфраструктуры)
   terraform apply  → фактически вносит изменения
   Аналогично git diff перед мёржем

3. State-файлы — Terraform хранит state-файл, который записывает что сейчас задеплоено.
   Этот файл должен храниться удалённо (S3-бакет с блокировкой) в командных окружениях —
   не коммититься в git (может содержать чувствительные выходные данные)

4. Модули — переиспользуемые инфраструктурные компоненты, аналогичные функциям или npm-пакетам

5. Разделение окружений — staging и production обычно являются отдельными Terraform-workspace'ами
   или отдельными state-файлами, даже если разделяют один и тот же код модуля
```

Базовый пример Terraform (создание S3-бакета — Simple Storage Service AWS, хранилище объектов):

```hcl
# main.tf

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region = "us-east-1"
}

resource "aws_s3_bucket" "app_assets" {
  bucket = "my-app-assets-${var.environment}"   # напр. "my-app-assets-staging"

  tags = {
    Environment = var.environment
    ManagedBy   = "terraform"
  }
}

variable "environment" {
  type    = string
  default = "staging"
}

output "bucket_name" {
  value = aws_s3_bucket.app_assets.bucket
}
```

```bash
terraform init       # скачать провайдеры
terraform plan       # предварительный просмотр изменений
terraform apply      # создать бакет
terraform destroy    # уничтожить (осторожно с prod!)
```

## Всё вместе: поток конфигурации в реальном проекте

```txt
Локальная разработка:
  .env файл (не коммитится) → process.env → приложение читает конфиг

CI-пайплайн (GitHub Actions / GitLab CI):
  GitHub/GitLab UI секретов → инжектируются как env-переменные в раннер
  → тесты приложения выполняются с URL тестовой БД, mock API-ключами

Деплой на staging:
  CI получает секреты из AWS Secrets Manager
  → инжектируются как env-переменные контейнера (ECS task definition) или K8s secrets
  → приложение читает DATABASE_URL, REDIS_URL из process.env

Деплой на production:
  Тот же механизм, разные значения секретов
  → приложение на prod читает prod URL БД, реальные ключи платежей

Инфраструктура (серверы, БД, сети):
  Terraform / CDK → определяет что существует в AWS/GCP/Azure
  → проверяется в PR, применяется CI при мёрже в main
```

## Типичные ошибки на интервью

- **"Мы храним конфиг в config.json в репозитории"** — если этот файл содержит что-либо, меняющееся между окружениями (особенно учётные данные), это красный флаг. Нечувствительный конфиг, одинаковый во всех окружениях, может жить в репозитории; всё что меняется или чувствительно должно быть переменной окружения или секретом.

- **"Мы кладём секреты в `.env` и они в безопасности, потому что добавили `.env` в `.gitignore`"** — `.gitignore` только предотвращает будущие коммиты. Если `.env` был когда-либо закоммичен до добавления в `.gitignore`, секрет уже в истории git. И `.gitignore` не защищает от случайного коммита файла с другим именем, содержащего те же секреты.

- **Путать "secrets manager" с "password manager"** — менеджер паролей (1Password, Bitwarden) — для людей, хранящих личные учётные данные. Secrets manager (AWS Secrets Manager, Vault) — это программный сервис для приложений и CI-систем, позволяющий хранить и получать учётные данные во время выполнения с API-доступом, audit-логами и автоматической ротацией.

- **Устанавливать `NODE_ENV=development` на staging** — staging должен работать точно как production. `NODE_ENV=development` активирует development-режим во многих библиотеках (подробные ошибки, неминифицированный код, отключённые кэши), что может скрывать production-баги.

- **Не знать, что такое IaC** — на собеседовании уровня senior fullstack ответ "DevOps-команда настраивает серверы" слабый. Вы должны знать что такое IaC, зачем оно существует, и уметь назвать хотя бы один инструмент (Terraform, CDK, CloudFormation).

- **"Мы используем переменные окружения для всего, включая секреты"** — обычные переменные окружения читаются из среды процесса на хосте (`/proc/<pid>/environ` на Linux). Для высокочувствительных секретов в production лучший подход — получение на уровне приложения из secrets manager или хотя бы инжекция через механизм, не раскрывающий значение в листинге окружения.

- **Не знать паттерн `.env.example`** — сильный сигнал того, что разработчик работал в зрелых проектах — это знание, что `.env` не коммитится, а `.env.example` коммитится. Интервьюеры ищут это.
