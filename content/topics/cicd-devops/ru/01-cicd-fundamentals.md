# CI/CD Fundamentals

## Проблема, которую решает CI/CD

До появления CI/CD команды работали в долгоживущих feature-ветках, которые расходились с основной кодовой базой на дни и недели. Когда наступало время мёржа, начинался так называемый "integration hell" (ад интеграции): мучительный, полный ошибок процесс разрешения конфликтов и отладки взаимодействия между кодом, который развивался независимо — зачастую в условиях горящего дедлайна релиза.

**CI/CD** (Continuous Integration / Continuous Delivery — Непрерывная интеграция / Непрерывная доставка) — это практика, подкреплённая инструментами, которая делает интеграцию и доставку изменений частой, автоматизированной и малорисковой. Чем короче разрыв между написанием кода и его тестированием и деплоем — тем меньше каждое отдельное изменение, и тем дешевле найти и исправить каждую ошибку.

```txt
Без CI/CD:
  Код пишется 2 недели → "big bang" мёрж → integration hell → ручное тестирование → рискованный релиз

С CI/CD:
  Код пишется 2 часа → автомёрж + тест → staged deploy → маленькие, частые релизы
```

## Три понятия: CI, CD (Delivery) и CD (Deployment)

Эти три аббревиатуры постоянно путают в вакансиях, на собеседованиях и в разговорах. Это три разные концепции, каждая строится поверх предыдущей.

### Continuous Integration (CI — Непрерывная интеграция)

**CI** — Continuous Integration — означает, что каждый разработчик мёржит (интегрирует) свои изменения в общую основную ветку часто — в идеале несколько раз в день — и каждый мёрж автоматически запускает пайплайн (pipeline — последовательность автоматических шагов), который собирает и тестирует код.

```txt
Разработчик пушит код в репозиторий
              ↓
Автоматически запускается пайплайн:
  1. Скачать код коммита, который был запушен
  2. Установить зависимости
  3. Запустить линтер (поймать синтаксические ошибки и нарушения стиля)
  4. Запустить проверку типов (если TypeScript)
  5. Запустить unit- и интеграционные тесты
  6. Собрать (скомпилировать, забандлить)
              ↓
Пайплайн ПРОШЁЛ → ветка в рабочем состоянии
Пайплайн УПАЛ  → разработчик получает уведомление через минуты,
                 пока сломанный код ещё свеж в памяти
```

Ключевая идея: **ошибки ловятся через минуты, а не через дни**. Баг, найденный через 5 минут после появления, требует секунд для понимания и исправления. Баг, найденный через 2 недели — когда автор уже переключился на другую задачу — может потребовать часов только для воспроизведения.

### Continuous Delivery (CD — Непрерывная доставка)

**Continuous Delivery** (Непрерывная доставка) расширяет CI: каждое изменение кода, прошедшее автоматический пайплайн, упаковывается в развёртываемый **артефакт** (Docker-образ, скомпилированный бандл, zip-архив — подробнее об артефактах ниже) и **готовится к деплою на продакшн** — но сам продакшн-деплой требует ручного подтверждения.

```txt
CI пайплайн прошёл (lint + test + build)
              ↓
Сборка артефакта (например, Docker-образ с тегом SHA коммита)
              ↓
Автоматический деплой на staging-окружение
              ↓
Автоматические smoke/acceptance-тесты на staging
              ↓
        ⏸ РУЧНЫЕ ВОРОТА (HUMAN APPROVAL GATE)
          Кто-то проверяет и нажимает "Задеплоить на продакшн"
              ↓
Деплой на продакшн
```

Главная идея: код *всегда находится в развёртываемом состоянии* (это и есть "delivery" — доставка), но человек решает, *когда* его фактически задеплоить. Это подходит командам, которым нужна координация релизов, знаки согласования по compliance или запланированные окна технических работ.

### Continuous Deployment (CD — Непрерывное развёртывание)

**Continuous Deployment** (Непрерывное развёртывание) убирает ручные ворота полностью. Каждое изменение, прошедшее все автоматические тесты, деплоится на продакшн автоматически, без участия человека.

```txt
Тесты прошли → деплой на staging → smoke-тесты прошли → ДЕПЛОЙ НА ПРОД (полностью автоматически)
```

Это требует очень высокого уровня доверия к автоматическим тестам и мониторингу. Не все команды достигают этого уровня, и это нормально — Continuous Delivery — вполне подходящий конечный результат для большинства продуктов.

```txt
Итоговая таблица:
  Continuous Integration  = частые мёржи + автоматическая сборка + автоматические тесты
  Continuous Delivery     = CI + артефакт + авто-деплой на staging + ручной триггер на прод
  Continuous Deployment   = CI + артефакт + авто-деплой на staging + авто-деплой на прод
```

## Что такое пайплайн

**Пайплайн** (pipeline) — последовательность автоматических шагов, которые выполняются в заданном порядке при срабатывании события (обычно push коммита или открытие pull request). Пайплайн описывается как код — как правило, в YAML-файле, который хранится в репозитории вместе с кодом приложения. Эта практика называется **pipeline as code** (пайплайн как код).

Каждый пайплайн состоит из **джобов** (jobs). Джоб — логическая единица работы, выполняющаяся на одной машине. Каждый джоб содержит отдельные **шаги** (steps) — команды оболочки или готовые действия, которые делают реальную работу.

```txt
Пайплайн (триггер: git push в ветку main)
│
├── Job: lint                    ← выполняется параллельно с "test"
│   ├── Step: checkout code
│   ├── Step: npm ci             (установить точные версии зависимостей из lockfile)
│   └── Step: npm run lint
│
├── Job: test                    ← выполняется параллельно с "lint"
│   ├── Step: checkout code
│   ├── Step: npm ci
│   └── Step: npm test
│
└── Job: build                   ← стартует ТОЛЬКО когда lint + test оба прошли
    ├── Step: checkout code
    ├── Step: npm ci
    ├── Step: npm run build
    └── Step: docker build + push
```

Джобы в пайплайне часто выполняются **параллельно** для экономии времени. Но некоторые джобы имеют зависимости — джоб `build` должен запускаться только если `lint` и `test` прошли успешно; `deploy` — только если `build` прошёл.

## Что такое артефакт

**Артефакт** (artifact) в CI/CD — это любой файл или набор файлов, созданный на этапе сборки и переданный на следующий этап или сохранённый для дальнейшего использования.

Примеры:
- Скомпилированный JavaScript/TypeScript бандл (папка `dist/`, `build/`)
- **Docker-образ** (image) — упакованный снимок вашего приложения и его среды выполнения
- `.zip`-архив кода AWS Lambda-функции
- Отчёт о покрытии тестами (HTML/XML-файлы)
- Бинарный исполняемый файл

Критически важное свойство: **собирайте один раз, деплойте везде**. Один и тот же артефакт должен деплоиться на staging и затем на продакшн — вы никогда не пересобираете из исходников на каждом этапе. Пересборка создаёт риск того, что на прод попадёт немного другая версия, если между двумя сборками обновилась зависимость.

```txt
Джоб сборки → создаёт Docker-образ с тегом SHA коммита "abc1234"
                              ↓
               Пуш в реестр артефактов
                              ↓
  Деплой на staging ←── pull образа "abc1234" ──→ Деплой на прод
        ↓                                                ↓
  Те же биты                                       Те же биты
  (гарантировано)                                  (гарантировано)
```

**Реестр артефактов** (artifact registry) — или **реестр контейнеров** (container registry), если речь именно о Docker-образах — это хранилище, куда артефакты загружаются, версионируются и откуда загружаются при деплоях. Примеры: GitHub Container Registry (ghcr.io), Docker Hub, AWS ECR (Elastic Container Registry — управляемый реестр контейнеров Amazon), Google Artifact Registry.

## Что такое runner (агент)

**Runner** (GitHub Actions) или **agent** (Jenkins и некоторые другие инструменты) — машина, физическая или виртуальная, на которой фактически выполняются джобы пайплайна.

Когда вы пушите код и пайплайн срабатывает, кто-то же должен на самом деле выполнить команды `npm test`, `docker build` и т.д. Этот "кто-то" и есть runner.

```txt
Серверы CI-провайдера           Ваш репозиторий с кодом
  ┌──────────────────┐        ┌──────────────────────────┐
  │  Планировщик     │  читает│  .github/workflows/       │
  │  пайплайнов      │ ──────→│  ci.yml                   │
  └────────┬─────────┘        └──────────────────────────┘
           │ назначает джоб на
           ↓
  ┌──────────────────┐
  │     Runner       │  ← виртуальная машина, которая выполняет
  │  (ubuntu-latest) │    шаги из YAML-файла:
  │                  │    npm ci, npm test, docker build...
  └──────────────────┘
```

**Hosted-runners (управляемые)** — виртуальные машины, которые CI-провайдер полностью управляет сам. Для каждого запуска пайплайна вы получаете чистую VM с предустановленными инструментами. Удобно, нулевое обслуживание с вашей стороны, но: вы платите за каждую минуту вычислений, железо разделено с другими пользователями, и у раннеров нет доступа к вашей приватной сети.

**Self-hosted runners (собственные)** — ваши VM или физические машины с установленным ПО раннера. Они регистрируются у CI-провайдера и забирают джобы. Полезны когда:
- Пайплайну нужен доступ к внутренним ресурсам (приватная БД, on-premises реестр, VPN)
- Нужно специфическое железо (GPU-машина для тестов ML-моделей, macOS для iOS-сборок)
- Объём пайплайнов настолько большой, что managed-раннеры становятся слишком дорогими

## Типичные этапы пайплайна для fullstack-проекта

Реальный пайплайн для Node.js + TypeScript проекта обычно выглядит так:

```txt
┌────────────────────────────────────────────────────────────────┐
│                         CI Пайплайн                             │
│                                                                  │
│  [lint] ──┐                                                      │
│            ├──→ [test] ──→ [type-check] ──→ [build] ──→ [push] │
│  [format]─┘                                    ↓                │
│                                    (только при мёрже в main)    │
│                                     [деплой на staging]         │
│                                            ↓                    │
│                                   [smoke-тесты на staging]      │
│                                            ↓                    │
│                            (ручные ворота ИЛИ автоматически)   │
│                                   [деплой на продакшн]          │
└────────────────────────────────────────────────────────────────┘
```

**lint** — запуск ESLint (и/или Stylelint для CSS) для поиска синтаксических ошибок, неиспользуемых переменных, нарушений стиля. Занимает 5–30 секунд. Падает быстро и дёшево.

**type-check** — для TypeScript-проектов запуск `tsc --noEmit`, чтобы проверить корректность типов без создания выходных файлов. Выделяется в отдельный шаг, потому что современные бандлеры (esbuild, swc, Vite) часто пропускают проверку типов ради скорости — они транспилируют, но не проверяют типы. `tsc --noEmit` в CI гарантирует, что ошибки типов никогда не будут молча пропущены.

**test** — полный запуск тест-сьюта (unit-тесты, интеграционные тесты). Обычно самый долгий шаг. Для больших проектов распараллеливается на несколько раннеров.

**build** — компиляция TypeScript в JavaScript, бандлинг фронтенд-ассетов, сборка Docker-образа. Выход — развёртываемый артефакт.

**push** (публикация артефакта) — отправка собранного Docker-образа в реестр артефактов с тегом SHA коммита и опционально тегом версии.

**deploy** — получение артефакта из реестра и выкатка в целевое окружение. Этот шаг использует **секреты** — учётные данные, API-ключи, URL баз данных, — которые хранятся в хранилище секретов CI-системы, а не в коде.

## Почему порядок важен: fail fast

Пайплайны проектируются по принципу **fail fast** (падать быстро): самые дешёвые проверки выполняются первыми. Если разработчик пропустил точку с запятой или допустил ошибку типов в TypeScript, пайплайн упадёт меньше чем за минуту — без трат 10 минут на Docker-сборку или 30 минут на end-to-end тесты.

```txt
Лестница стоимости (от дешёвого к дорогому):
  ESLint / проверка формата    →   5–30 секунд
  TypeScript type-check        →  15–60 секунд
  Unit-тесты                   →  30 сек – 3 минуты
  Интеграционные тесты         →   2–10 минут
  Docker build                 →   3–15 минут
  End-to-end (E2E) тесты       →  10–60 минут
  
Запускайте дешёвое первым. Первое падение останавливает весь пайплайн.
```

## Типичные ошибки на интервью

- **Путать CI/CD с конкретным инструментом** — CI/CD это *практика*; GitHub Actions, GitLab CI, Jenkins, CircleCI — это *инструменты*, которые эту практику реализуют. "У нас есть CI" — описывает процесс; "мы используем GitHub Actions" — называет инструмент.

- **Путать Continuous Delivery с Continuous Deployment** — одна из самых распространённых ошибок на собеседованиях. Разница в одном слове и ключевой концепции: Delivery = ручные ворота перед продакшном; Deployment = полностью автоматически. Интервьюеры специально спрашивают об этом.

- **"Мы пересобираем из исходников для продакшна"** — красный флаг. Артефакт должен быть собран один раз, и этот же самый артефакт деплоится на staging и на прод. Пересборка означает, что два окружения могут получить немного разный код (например, если зависимость обновилась между сборками).

- **Воспринимать YAML пайплайна и раннер как одно и то же** — YAML это *определение* (что делать); раннер — это *машина, которая это выполняет* (где это делается). Одно определение пайплайна может работать на managed-раннерах, self-hosted раннерах или их комбинации.

- **Не понимать, зачем lint запускается до тестов** — это не конвенция, а экономика. Lint стоит в 10 раз дешевле, чем полный тест-сьют. Быстрое падение на ошибке линтера сохраняет минуты вычислений на каждый PR и даёт разработчику быструю обратную связь.

- **"CI/CD значит, что мы деплоим автоматически"** — только Continuous *Deployment* деплоит автоматически. Многие зрелые команды используют Continuous *Delivery* с ручными воротами, особенно для regulated-продуктов или продуктов с запланированными окнами технических работ.
