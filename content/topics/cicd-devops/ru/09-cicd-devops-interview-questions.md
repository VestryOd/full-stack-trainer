# CI/CD и DevOps — Вопросы для собеседования

## Группа 1: Основы CI/CD

**В чём разница между Continuous Delivery и Continuous Deployment?**

Оба начинаются одинаково: каждое изменение кода, прошедшее автоматический пайплайн, упаковывается в деплоябельный артефакт и автоматически выкатывается на staging. Разница — в финальном шаге. **Continuous Delivery** останавливается здесь и требует ручного подтверждения деплоя в production — кнопка есть, но человек её нажимает. **Continuous Deployment** убирает эту кнопку полностью: каждая успешная сборка уходит в production автоматически. Разница — один ручной шаг подтверждения. Оба подхода требуют достаточно надёжного пайплайна, чтобы упавшая сборка действительно означала «не деплоить».

---

**Что такое CI-пайплайн и какие шаги он обычно включает?**

CI-пайплайн — последовательность автоматизированных шагов, запускаемых при изменении кода (push или pull request). Типичные шаги в порядке выполнения: (1) получить исходный код; (2) установить зависимости; (3) запустить линтер и проверку типов; (4) запустить юнит- и интеграционные тесты; (5) собрать приложение; (6) опубликовать артефакт (Docker-образ в реестр, бандл в S3 и т.д.); (7) опционально задеплоить на staging. Шаги упорядочены от самых дешёвых к самым дорогим: быстрый дешёвый сбой (ошибка линтера) ловит проблему ещё до запуска медленных дорогих шагов (полного набора тестов или Docker-сборки).

---

**Что такое артефакт и почему его следует собирать один раз и продвигать по окружениям?**

Артефакт — любой файл, созданный на этапе сборки: скомпилированный JS-бандл, Docker-образ, Lambda-архив, скомпилированный бинарник. «Собери один раз, деплой везде» означает, что один и тот же артефакт деплоится на staging, а затем в production — никакой пересборки из исходников под каждое окружение. Причина: пересборка вносит вариативность (npm-пакеты резолвятся по-другому, переменные окружения вшиваются иначе, случайная загрузка падает). То, что вы тестировали на staging, должно быть побайтово идентично тому, что уходит в production. Артефакт загружается в artifact registry (Docker Hub, ECR, S3) с тегом по хешу коммита, чтобы любую версию можно было воспроизвести точно.

---

**Что означает «pipeline as code» и какие у этого подхода преимущества?**

Pipeline as code означает, что CI/CD-пайплайн определяется в конфигурационном файле (YAML), хранящемся в том же репозитории, что и код приложения. Преимущества: (1) изменения пайплайна проходят тот же процесс ревью через pull request, что и изменения кода — никаких «случайных» правок пайплайна; (2) история изменений пайплайна видна в `git log`; (3) пайплайн воспроизводим — каждая ветка может иметь свой конфиг пайплайна, и он будет работать одинаково для всех разработчиков; (4) откат — это `git revert`. Для сравнения: Jenkins эпохи 2010-х, где пайплайны настраивались кликами в веб-интерфейсе — невидимые, не поддающиеся аудиту, невозможные для версионирования.

---

**Что такое раннер (агент) и каковы компромиссы между hosted и self-hosted раннерами?**

Раннер — машина, на которой фактически выполняются задания пайплайна. При срабатывании задания планировщик CI назначает его свободному раннеру: тот клонирует репозиторий, выполняет шаги и сообщает результат. **Hosted runners** (GitHub-hosted, GitLab.com shared): предоставляются платформой, свежая VM на каждое задание, нулевое обслуживание, оплата поминутно, нет доступа к приватным сетям. **Self-hosted runners**: ваши собственные машины, зарегистрированные в CI-платформе, могут обращаться к приватным БД и VPC, железо полностью под вашим контролем, фиксированная стоимость (ваши серверы), но вы сами обслуживаете обновления ОС, патчи безопасности и очистку диска. Важно: никогда не используйте self-hosted runners в публичных репозиториях — злоумышленник может изменить YAML воркфлоу в PR, чтобы выполнить произвольный код на вашей машине.

## Группа 2: GitHub Actions и GitLab CI

**В чём разница между триггерами `on: push` и `on: pull_request` в GitHub Actions?**

`on: push` срабатывает при каждом пуше коммита в репозиторий — включая main, feature-ветки или любую ветку по паттерну. `on: pull_request` срабатывает при открытии PR, его обновлении (новый коммит в ветку источника) или синхронизации. Типичная схема: `on: pull_request` запускает тесты и линтер (проверка предлагаемых изменений), а `on: push` в `main` (после мержа PR) запускает пайплайн деплоя. Важное отличие: `on: pull_request` из форка выполняется с ограниченным доступом к секретам по соображениям безопасности — воркфлоу имеет только права на чтение и не получает доступа к секретам репозитория.

---

**Как работают matrix builds и когда они полезны?**

Matrix build — определение одного задания, которое GitHub Actions разворачивает в несколько параллельных заданий, комбинируя значения из матрицы. Пример: `matrix: { node: [18, 20, 22] }` порождает три параллельных задания, каждое с разной версией Node.js. Матрицы могут быть многомерными: `matrix: { os: [ubuntu-latest, windows-latest], node: [18, 20] }` → 4 задания. Сценарии использования: тестирование на нескольких версиях Node.js/Python/Java, тестирование на нескольких ОС, тестирование с несколькими версиями базы данных. Каждая комбинация матрицы — полностью изолированное задание со своим раннером.

---

**В чём разница между reusable workflow и composite action в GitHub Actions?**

**Reusable workflow** (определяется с `on: workflow_call`) — полноценный воркфлоу, запускаемый как отдельное задание на своём раннере. Вызывается через `uses: ./.github/workflows/deploy.yml`. Может включать несколько шагов, собственные `env:` и `with:` входы, работает в полной изоляции. **Composite action** (определяется в `action.yml` с `runs.using: composite`) — повторно используемая последовательность шагов, выполняемая *внутри существующего задания*, разделяя одну среду раннера. Используйте composite action, когда хотите абстрагировать повторяющиеся шаги установки в рамках задания. Используйте reusable workflow, когда хотите совместно использовать целое задание или флоу деплоя между репозиториями.

---

**В чём разница между `only/except` и `rules` в GitLab CI? Что использовать?**

`only/except` — устаревший синтаксис: `only: [main, merge_requests]` — задание выполняется только на указанных ветках или событиях. Ограничения: нельзя комбинировать условия, нельзя использовать CI-переменные в условиях, нельзя условно задавать `when: manual`. `rules` — современная замена: упорядоченный список условий `if/when/changes`, оцениваемых сверху вниз; первое совпавшее правило выигрывает. `rules` поддерживает: `if: $CI_COMMIT_BRANCH == "main"`, `changes: [src/**/*]` (запускать только при изменении исходных файлов), `when: manual`, `allow_failure: true` на уровне правила. Всегда используйте `rules` — `only/except` устарел и будет удалён.

---

**Что делает ключевое слово `needs:` в GitLab CI и какую структуру оно создаёт?**

`needs:` определяет явные зависимости между заданиями, превращая пайплайн из последовательности этапов в DAG (Directed Acyclic Graph — направленный ациклический граф). Без `needs:` задания каждого этапа ждут завершения всего предыдущего этапа. С `needs:` задание стартует сразу после завершения конкретных зависимостей — независимо от того, в каком этапе они находятся. Пример: `deploy-preview` может зависеть только от `build`, не ожидая `test-e2e` в том же этапе. Это существенно сокращает общее время пайплайна. В GitHub Actions аналогичное делается с помощью `needs:` на уровне job.

---

**Как безопасно работать с секретами в GitHub Actions?**

Секреты хранятся в Settings → Secrets репозитория или организации и никогда не отображаются в логах после первоначальной настройки. В воркфлоу доступ к ним осуществляется через `${{ secrets.MY_SECRET }}`. Правила: (1) никогда не хардкодить секреты в YAML воркфлоу; (2) никогда не делать `echo` значения секрета — GitHub автоматически маскирует известные значения секретов в логах, но это не абсолютная защита; (3) использовать environment-level secrets (`environment: production`) для production-credentials — они требуют подтверждения перед запуском задания; (4) для сторонних API-ключей использовать OIDC (OpenID Connect) федеративные токены вместо долгоживущих статических ключей, когда провайдер поддерживает это (AWS, GCP, Azure — поддерживают); (5) регулярно проверять, какие воркфлоу имеют доступ к каким секретам.

---

**Что такое Docker-in-Docker (dind) и когда он нужен в GitLab CI?**

dind (Docker-in-Docker) — конфигурация, при которой Docker-демон запускается *внутри* Docker-контейнера, чтобы команды `docker build` и `docker push` можно было выполнять внутри CI-задания, которое само работает в контейнере. Требуется в GitLab CI при использовании Docker executor (где задания работают как контейнеры) и необходимости собирать Docker-образы. Объявляется как сервис: `services: - docker:24-dind`, с переменной `DOCKER_HOST: tcp://docker:2376`. Последствия для безопасности: dind требует запуска контейнера в **привилегированном режиме** — контейнер получает полный доступ к ядру хоста, что является значительным риском в разделяемых средах. Альтернатива: Kaniko — сборка Docker-образов без демона, без привилегированного режима.

## Группа 3: Docker

**В чём разница между Docker-образом и контейнером?**

Образ — доступный только для чтения слоёный снимок файловой системы со всем необходимым для запуска приложения: базовая ОС, среда выполнения, зависимости, скомпилированный код. Образ собирается из Dockerfile и хранится в реестре. Контейнер — запущенный экземпляр образа: файловая система образа плюс перезаписываемый верхний слой, работающий как изолированный процесс на хост-ОС. Из одного образа одновременно могут работать несколько контейнеров. Аналогия: образ — это определение класса, контейнер — его экземпляр. Образы собираются один раз и продвигаются по окружениям; контейнеры эфемерны и заменяемы.

---

**Что такое слои Docker-образа и почему они важны для производительности сборки?**

Каждая инструкция Dockerfile (`FROM`, `RUN`, `COPY`, `ADD`) создаёт новый слой — дифф относительно нижнего слоя. Слои кешируются: если инструкция и её входные данные не изменились с момента последней сборки, Docker повторно использует кешированный слой и пропускает его повторное выполнение. Это делает порядок инструкций критически важным: помещайте инструкции, которые редко меняются (установка OS-пакетов, копирование `package.json`, запуск `npm install`), перед теми, что меняются часто (копирование исходного кода приложения). Изменение любого слоя инвалидирует кеш всех слоёв ниже него. Практическое правило: `COPY package*.json ./` → `RUN npm ci` → `COPY . .` — слой с `npm ci` остаётся закешированным при изменениях исходного кода.

---

**В чём разница между CMD и ENTRYPOINT, и между exec-формой и shell-формой?**

`ENTRYPOINT` задаёт фиксированный исполняемый файл, который всегда запускается. `CMD` предоставляет аргументы по умолчанию, которые можно переопределить при `docker run`. При совместном использовании контейнер запускает `ENTRYPOINT CMD-аргументы`. Shell-форма (`CMD node server.js`) вызывает `sh -c "node server.js"` — `sh` становится PID 1, Node.js — дочерним процессом, и `sh` не пробрасывает `SIGTERM` дочерним процессам → graceful shutdown сломан. Exec-форма (`CMD ["node", "server.js"]`) запускает Node.js напрямую как PID 1 — он получает `SIGTERM` и может корректно завершиться. Всегда используйте exec-форму для главного процесса. Shell-форма допустима для инструкций `RUN` на этапе сборки.

---

**Что такое PID 1 в контейнере и почему это важно?**

В Linux PID 1 — это init-процесс, первый процесс, запускаемый ядром. Он отвечает за сборку зомби-дочерних процессов и пробрасывание сигналов. В Docker-контейнере PID 1 — это процесс, заданный в `CMD`/`ENTRYPOINT`. Если `sh` является PID 1 (shell-форма CMD), он не пробрасывает `SIGTERM` дочерним процессам — когда Kubernetes останавливает pod, `SIGTERM` отправляется `sh`, который его игнорирует, Kubernetes ждёт 30 секунд (grace period), затем отправляет `SIGKILL`, уничтожая контейнер принудительно и обрывая все запросы в обработке. С exec-формой ваше приложение является PID 1, получает `SIGTERM` напрямую и может корректно завершиться. Для Node.js: `CMD ["node", "server.js"]`, а не `CMD node server.js`.

---

**Что такое multi-stage build и какую проблему он решает?**

Multi-stage build использует несколько инструкций `FROM` в одном Dockerfile, каждая из которых начинает новый этап сборки. Файлы можно копировать между этапами с помощью `COPY --from=builder`. Финальный образ содержит только то, что явно скопировано на последний этап — инструменты сборки, dev-зависимости, компилятор TypeScript и тестовые файлы остаются в предыдущих этапах. Результат: production-образ содержит только среду выполнения Node.js и скомпилированный JS, без TypeScript-компилятора, пакетов `@types/*` и тестовых фреймворков. Типичное уменьшение размера: с ~400 МБ (все инструменты сборки) до ~80 МБ (только среда выполнения). Меньший образ — быстрее скачивается, меньше поверхность атаки, меньше затраты на хранение.

---

**Почему контейнеры должны работать от непривилегированного пользователя?**

По умолчанию процесс внутри Docker-контейнера работает от root (UID 0). Если злоумышленник эксплуатирует уязвимость в приложении и выбирается из контейнера (через баг ядра или неверную конфигурацию), у него есть полный root-доступ к хост-машине. Запуск от непривилегированного пользователя ограничивает blast radius: сбежавший процесс имеет те же ограниченные права, что и обычный пользователь ОС. В Dockerfile: `USER node` (базовые образы `node:*` включают предсозданного пользователя `node` с UID 1000). Убедитесь, что файлы, в которые пишет приложение, принадлежат этому пользователю. Некоторые среды выполнения контейнеров (например, Kubernetes с Pod Security Admission) могут применять ненулевой UID как политику принудительно.

## Группа 4: Стратегии деплоя

**Что такое деплой без простоев и как он достигается?**

Деплой без простоев означает, что сервис остаётся непрерывно доступным во время обновления кода — ни один запрос не теряется, ни один пользователь не видит ошибок. Для этого нужны две вещи, работающие совместно: (1) инфраструктурная стратегия, удерживающая старые инстансы работающими, пока запускаются новые (rolling update, blue-green или canary); (2) graceful shutdown в приложении — при получении `SIGTERM` приложение прекращает принимать новые соединения, но завершает обработку текущих запросов перед выходом. Без graceful shutdown даже идеально оркестрированный rolling update теряет запросы при каждой замене pod. Миграции базы данных также должны быть обратно совместимы — старая и новая версии приложения могут работать одновременно с одной БД.

---

**В чём разница между rolling, blue-green и canary деплоями?**

**Rolling update (скользящее обновление)**: инстансы заменяются по одному (или небольшими группами). Во время выкатки одновременно работают обе версии — требуется обратная совместимость API и схемы БД. Минимальные инфраструктурные затраты. Встроено в Kubernetes и ECS. **Blue-green**: поддерживаются два идентичных окружения; новая версия полностью разворачивается в простаивающем окружении и тестируется, затем трафик переключается мгновенно через балансировщик. Нет периода одновременной работы версий, мгновенный откат (переключить обратно), но удваивает стоимость инфраструктуры. **Canary**: небольшой процент трафика (например, 5%) направляется на новую версию; остальной остаётся на старой. Новая версия наблюдается; трафик постепенно увеличивается. Минимальный blast radius, но требует инфраструктуры для разделения трафика (service mesh или умный балансировщик нагрузки).

---

**Что такое паттерн expand-contract migration и когда он нужен?**

Паттерн expand-contract решает проблему внесения ломающих изменений в схему базы данных без простоев, когда старая и новая версии приложения должны работать с одной БД во время rolling-деплоя. Одна опасная миграция разбивается на три отдельных деплоя: (1) **Expand (Расширение)** — добавляем новую колонку как nullable (нет блокировки, обратная совместимость, старый код её игнорирует); (2) **Backfill (Заполнение)** — деплоим новый код, пишущий в обе колонки; фоновым заданием заполняем новую колонку в существующих строках; (3) **Contract (Сжатие)** — как только все строки заполнены и старый код исчез, делаем колонку NOT NULL и удаляем старую. Никогда не добавляйте NOT NULL колонку с DEFAULT в одной миграции на большой таблице — PostgreSQL должен перезаписать всю таблицу, что вызывает полную блокировку.

---

**Что такое feature flag и чем он отличается от canary release?**

Feature flag (переключатель функциональности) — это `if`-условие в коде приложения, проверяющее значение конфигурации во время выполнения: `if (flags.isEnabled('new-checkout', user))`. Код уже задеплоен в production, но неактивен до включения флага. Canary release — инфраструктурная стратегия: две версии сервиса задеплоены одновременно, балансировщик нагрузки или service mesh направляет процент трафика на новую. Feature flags — уровень приложения, можно переключать для конкретного пользователя, организации или процента без деплоя. Canary release предполагает реальный запуск двух задеплоенных версий. Они дополняют друг друга: canary release для проверки нового бинарника, feature flags для контролируемого раскрытия функциональности.

---

**Что такое graceful shutdown и как его реализовать в Node.js?**

Graceful shutdown означает, что процесс перехватывает `SIGTERM`, прекращает принимать новые соединения, дожидается завершения запросов в обработке, освобождает ресурсы (соединения с БД, консьюмеры очередей сообщений) и завершается с кодом 0. В Node.js с Express:
```javascript
process.on('SIGTERM', async () => {
  server.close(async () => {        // прекращаем принимать новые HTTP-соединения
    await prisma.$disconnect();     // закрываем пул соединений с БД
    process.exit(0);
  });
});
```
Kubernetes отправляет `SIGTERM` и ждёт `terminationGracePeriodSeconds` (по умолчанию 30 с) перед отправкой `SIGKILL`. Приложение должно завершить shutdown в этом окне. Для NestJS: `app.enableShutdownHooks()` обрабатывает это через lifecycle-хуки (`OnApplicationShutdown`).

## Группа 5: Окружения, конфигурация и секреты

**В чём разница между переменной окружения, конфигурационным файлом и секретом?**

**Переменные окружения**: несекретные параметры времени выполнения (feature flags, уровень логирования, URL внешних API, имя окружения). Простые пары ключ-значение, легко переопределяются для каждого окружения, доступны через `process.env`. **Конфигурационные файлы**: структурированные несекретные настройки (имя схемы БД, значения пагинации по умолчанию, поддерживаемые локали) — хранятся в репозитории и загружаются при старте. **Секреты**: чувствительные учётные данные, которые никогда не должны появляться в исходном коде, логах или в незашифрованном хранилище (пароли БД, API-ключи, JWT signing keys, TLS private keys). Секреты хранятся в secrets manager (AWS Secrets Manager, HashiCorp Vault) и инжектируются в процесс во время выполнения через переменные окружения. Граница простая: если утечка этого значения повлечёт инцидент безопасности — это секрет.

---

**Почему `.env` файлы нельзя коммитить в систему контроля версий?**

Как только секрет закоммичен, он фактически стал постоянным — даже если удалить его в следующем коммите, он остаётся в git-истории и может быть восстановлен через `git log -p`. Любой, кто имеет доступ к репозиторию (нынешние и будущие члены команды, автоматизированные инструменты, индексирование GitHub), может прочитать его. Атаки на цепочки поставок собирают секреты из git-истории. Правильный паттерн: `.env` в `.gitignore`; вместо него коммитится `.env.example` (те же ключи, без реальных значений). Это документирует, какие переменные необходимы, не раскрывая их значений. Используйте инструменты сканирования секретов (встроенный сканер GitHub, `trufflehog`, `gitleaks`) в CI для обнаружения случайных коммитов с реальными учётными данными.

---

**Что такое configuration drift и как его предотвращает Infrastructure as Code?**

Configuration drift — постепенное расхождение окружений (staging, production) между собой и с их задуманным состоянием из-за ручных изменений, вносимых напрямую на серверы или в облачные ресурсы: «я просто быстро изменю эту настройку в консоли». Со временем staging и production отличаются в тонкостях, которые никогда не документируются, что приводит к «работает на staging, падает в production». Инструменты IaC (Terraform, AWS CDK) определяют инфраструктуру декларативно в файлах кода, хранящихся в git. IaC-инструмент вычисляет разницу между объявленным состоянием и фактическим состоянием облака и применяет только необходимые изменения. Ручные изменения через консоль перезаписываются при следующем `terraform apply` — код всегда является авторитетным источником истины.

---

**Что даёт secrets manager в отличие от простых переменных окружения?**

Хранение секретов как простых переменных окружения (в `.env` на сервере или захардкоженными в Kubernetes-манифесте) не обеспечивает аудируемости, ротации, гранулированного контроля доступа и версионирования. Secrets manager (AWS Secrets Manager, HashiCorp Vault) предоставляет: (1) **шифрование в покое** — секреты никогда не хранятся в открытом виде; (2) **журнал аудита** — каждое событие доступа (кто, что, когда) записывается; (3) **гранулированные IAM-политики** — платёжный сервис может читать `STRIPE_SECRET_KEY`, но не `SENDGRID_API_KEY`; (4) **автоматическая ротация** — пароли БД ротируются по расписанию; (5) **версионирование** — откат к предыдущему значению секрета; (6) **динамические секреты** — учётные данные, генерируемые по запросу и автоматически отзываемые после использования.

---

## Группа 6: Мониторинг и наблюдаемость

**В чём разница между мониторингом и наблюдаемостью (observability)?**

Мониторинг означает наблюдение за заранее определёнными сигналами — дашборды, алерты, проверки аптайма. Он отвечает на вопросы, которые вы уже знаете как задать: «Частота ошибок выше 1%?» «CPU выше 80%?» Наблюдаемость — это свойство системы, позволяющее понять её внутреннее состояние, изучая внешние выходные данные (логи, метрики, трейсы). Она отвечает на непредвиденные вопросы: «Почему запрос именно этого пользователя упал иначе, чем у всех остальных?» Хорошо мониторируемая система говорит, что что-то пошло не так. Высоко наблюдаемая система объясняет *почему* и *где*. Наблюдаемость строится на трёх китах: логах, метриках и распределённых трейсах.

---

**Каковы три кита наблюдаемости и на что каждый отвечает?**

**Логи**: записи дискретных событий с временными метками («пользователь 42 создал заказ 891»). Отвечают на вопрос: *что произошло*. Лучшие практики: структурированные (JSON) логи с согласованными полями (`traceId`, `userId`, `level`, `message`). **Метрики**: числовые измерения, агрегируемые во времени (запросы в секунду, p99 задержка, частота ошибок, использование CPU). Отвечают на вопрос: *сколько и как быстро*. Инструменты: Prometheus + Grafana. **Трейсы**: записи пути единственного запроса через все сервисы распределённой системы, разбитые на spans. Отвечают на вопрос: *где конкретно этот запрос потратил время и где произошёл сбой*. Инструменты: Jaeger, Zipkin, OpenTelemetry + любой совместимый бэкенд. На практике: начинаете с алерта по метрике → углубляетесь в логи для контекста → используете traceId из логов для получения полного распределённого трейса.

---

**Каковы четыре золотых сигнала и что каждый из них измеряет?**

Из книги Google SRE — четыре метрики, достаточных для понимания состояния любого сервиса с точки зрения пользователя: (1) **Latency (Задержка)**: сколько времени занимают запросы — всегда отслеживайте перцентили (p50, p95, p99), а не средние значения. p99 = 200 мс означает, что 1% пользователей ждёт ≥ 200 мс. (2) **Traffic (Трафик)**: какой спрос обрабатывает система — запросы в секунду, сообщения в секунду, активные пользователи. Устанавливает базовую линию «нормы». (3) **Errors (Ошибки)**: частота неудачных запросов — различайте 5xx (ошибки сервера), 4xx (ошибки клиента) и тихие ошибки (неверные данные с кодом 200). (4) **Saturation (Насыщенность)**: насколько «заполнена» система — CPU, память, использование пула соединений, глубина очереди. Насыщенность предсказывает сбои до их наступления.

---

**В чём разница между liveness probe и readiness probe в Kubernetes?**

**Liveness probe**: «Жив ли этот контейнер, или его нужно перезапустить?» Неудача → Kubernetes перезапускает контейнер. Должна проверять только собственное состояние процесса — простой HTTP-эндпоинт, возвращающий 200, если сервер работает. НЕ должна проверять внешние зависимости: если liveness probe проверяет БД, а БД недоступна, все поды перезапускаются в бесконечном цикле, усугубляя аутаж. **Readiness probe**: «Готов ли этот контейнер принимать трафик?» Неудача → Kubernetes удаляет pod из пула балансировщика (без перезапуска). Должна проверять, доступны ли все необходимые внешние зависимости. Именно readiness probe предотвращает направление трафика на pod, который запустился, но ещё не подключился к базе данных.

---

**Что такое distributed tracing и какую проблему он решает?**

В монолите медленный запрос легко локализовать: добавить логи с замерами, профилировать код. В микросервисной системе одно действие пользователя запускает вызовы через 5-10 сервисов. Если ответ медленный — непонятно, какой сервис виноват. Distributed tracing записывает полный путь: когда сервис A вызывает B, который вызывает C, каждый сервис добавляет span (время начала, конца, имя сервиса, ошибки) к общему трейсу. `traceId` генерируется в точке входа и передаётся через заголовок `traceparent` HTTP через все межсервисные вызовы. Результат: визуализация в виде диаграммы Ганта всего запроса, показывающая точно, какой span был медленным и почему. OpenTelemetry — стандартный SDK для инструментирования трейсов.

---

**В чём разница между SLA, SLO и SLI?**

**SLI (Service Level Indicator)**: само измерение — конкретная метрика, которую вы отслеживаете. Пример: «процент запросов, завершающихся менее чем за 300 мс за 5-минутный период». **SLO (Service Level Objective)**: внутренняя цель для SLI — порог, ниже которого вы считаете, что сервис не справляется с задачей. Пример: «99,5% запросов должны завершаться менее чем за 300 мс». Не является обязательством перед клиентами. **SLA (Service Level Agreement)**: внешнее, договорное обязательство перед клиентами с финансовыми последствиями за нарушение. Пример: «Мы гарантируем 99,9% аптайма в месяц; при нарушении клиенты получают кредиты». SLA всегда менее строгий, чем SLO — для обеспечения буфера. Нарушение SLO — внутренний сигнал тревоги; нарушение SLA влечёт финансовые и юридические последствия.

---

**Что такое error budget и как он меняет поведение команды?**

Error budget — дополнение целевого SLO: допустимый объём ненадёжности за период. Формула: `error budget = 1 − SLO`. Пример: SLO аптайма 99,9% → error budget 0,1% → 43,2 минуты допустимого простоя за 30 дней. Бюджет — общий ресурс между командой разработки (хочет деплоить фичи → каждый деплой рискует потратить бюджет) и SRE/ops-командой (хочет надёжности). Когда error budget велик, команды деплоят агрессивнее и берут больше рисков. Когда почти исчерпан — замораживают рискованные деплои и сосредотачиваются на надёжности. Это превращает надёжность в измеримый и торгуемый ресурс, а не в размытую цель. Error budgets заменяют противостояние «разработка хочет деплоить» vs «ops хочет стабильности».

---

**Что такое структурированное логирование и чем оно лучше обычного текстового?**

Неструктурированный лог: `"Пользователь 42 разместил заказ 891 в 14:30"` — читаем для человека, но враждебен к машинам. Парсинг через regex для поиска по userId среди миллионов строк медленный и хрупкий. Структурированный лог: объект JSON на каждое событие — `{ "level": "info", "msg": "order placed", "userId": 42, "orderId": 891, "traceId": "abc123", "timestamp": "2026-01-15T14:30:00Z" }`. Преимущества: (1) системы агрегации логов (Loki, Elasticsearch) индексируют поля и позволяют мгновенно делать запросы вроде `userId=42 AND level=error`; (2) поле `traceId` связывает строки логов с распределёнными трейсами; (3) поля типизированы — длительности являются числами, а не строками «заняло 120мс»; (4) дашборды и алерты можно строить по значениям полей. Используйте `pino` или `winston` в JSON-режиме для Node.js-приложений.

---

## Группа 7: Аббревиатуры и концепции

**Объясните CI, CD, IaC, SLA, SLO и SLI в одном предложении каждый.**

**CI (Continuous Integration)**: практика частого слияния кода в основную ветку, при этом каждое слияние запускает автоматический пайплайн сборки и тестирования для немедленного обнаружения проблем интеграции. **CD (Continuous Delivery/Deployment)**: Continuous Delivery означает, что каждая успешная сборка готова к деплою в production в любой момент с ручным шагом подтверждения; Continuous Deployment означает, что этот шаг автоматизирован и каждая успешная сборка уходит в production автоматически. **IaC (Infrastructure as Code)**: определение и управление инфраструктурой (серверы, сети, базы данных) в файлах кода, хранящихся в системе контроля версий, чтобы окружения были воспроизводимы, аудируемы и свободны от configuration drift. **SLA (Service Level Agreement)**: внешнее договорное обязательство перед клиентами, определяющее минимальное качество сервиса и финансовые штрафы за нарушение. **SLO (Service Level Objective)**: внутренняя цель надёжности (более строгая, чем SLA), определяющая порог, ниже которого команда считает, что сервис подводит пользователей. **SLI (Service Level Indicator)**: измеримая метрика (например, процент запросов, выполняющихся менее чем за 300 мс), используемая как входные данные для оценки соблюдения SLO.

---

**Что означает APM и какую проблему он решает?**

APM расшифровывается как Application Performance Monitoring (мониторинг производительности приложений). Он заполняет пробел между мониторингом инфраструктуры (CPU, память, диск) и пониманием поведения приложения на уровне бизнеса. APM-инструменты автоматически инструментируют приложение и захватывают: длительность и код статуса каждого запроса, медленные запросы к БД (с SQL-текстом), внешние HTTP-вызовы, ошибки с полными стектрейсами и распределённые трейсы. Они агрегируют эти данные по эндпоинту, сервису и сегменту пользователей. Проблема, которую они решают: вы видите, что сервер «здоров» (CPU 30%, память 60%), но пользователи жалуются, что чекаут медленный. APM показывает, что у `POST /checkout` p99 = 4 с и 80% этого времени занимает один неиндексированный запрос к БД. Примеры: Datadog APM, New Relic, Sentry Performance, Elastic APM.

---

**Что такое Kubernetes (K8s) и какую проблему он решает для деплоев?**

Kubernetes (8 в K8s — это 8 букв между K и s) — платформа оркестрации контейнеров. Проблема: запускать контейнеры на одном сервере просто с помощью Docker. Запускать сотни контейнеров на десятках серверов — планировать, какой контейнер работает на каком сервере, перезапускать упавшие контейнеры, масштабироваться с 2 до 20 реплик под нагрузкой, маршрутизировать сетевой трафик, выкатывать новые версии без простоев, распределять секреты — это сложная задача распределённых систем. Kubernetes решает её декларативной моделью: вы описываете желаемое состояние («хочу 5 реплик этого образа, доступных на порту 8080, с 512 МБ памяти каждая»), и Kubernetes непрерывно приводит фактическое состояние в соответствие. Он автоматически обрабатывает планирование, проверки работоспособности, масштабирование и rolling update.

---

**Что такое GitOps?**

GitOps — практика, при которой желаемое состояние как деплоев приложений, так и инфраструктуры полностью хранится в Git, а автоматизированные системы (Argo CD, Flux) непрерывно приводят живое состояние в соответствие с тем, что объявлено в Git. Все изменения — включая инфраструктурные — вносятся через pull requests. Преимущества: каждое изменение проходит ревью, аудируется и версионируется; история git является журналом деплоев; откат — это `git revert`; расхождение между объявленным и фактическим состоянием обнаруживается и исправляется автоматически. GitOps — это конвергенция IaC, CI/CD и принципа «Git — единственный источник истины». Особенно мощен в Kubernetes-средах, где манифесты могут быть полностью декларативными.

---

**Что такое DAG в контексте CI/CD-пайплайнов?**

DAG расшифровывается как Directed Acyclic Graph (направленный ациклический граф) — математическая структура, где рёбра (связи) направленные и нет циклов (ни один путь не возвращается к своей начальной точке). В CI/CD DAG описывает зависимости заданий: «задание B должно выполняться после задания A». «Directed» — A → B (не B → A). «Acyclic» — нет циклической зависимости (A → B → A). Традиционные stage-based пайплайны — это простой DAG, где все задания 2-го этапа ждут всех заданий 1-го. Современные пайплайны (ключ `needs:` в GitLab CI и GitHub Actions) допускают произвольные формы DAG: `deploy-preview` может зависеть только от `build`, выполняясь параллельно с `test-e2e`, что кардинально сокращает общее время пайплайна по сравнению с ожиданием завершения всего этапа.

---

**Что такое OpenTelemetry (OTel) и почему он стал стандартом?**

OpenTelemetry — нейтральный по отношению к вендорам, open-source фреймворк наблюдаемости, предоставляющий API, SDK и инструменты для генерации, сбора и экспорта телеметрических данных: трейсов, метрик и логов. До OTel инструментирование приложения для наблюдаемости означало выбор вендора (Datadog, New Relic, Jaeger) и установку его проприетарного агента. Смена вендора требовала повторного инструментирования всей кодовой базы. OTel решает это: вы инструментируете один раз с OTel SDK (используя стандартные API), экспортируете в OTel Collector, а затем маршрутизируете из коллектора в любой бэкенд — Jaeger, Datadog, Honeycomb, Grafana Tempo — меняя конфиг экспортёра коллектора, а не код приложения. Поддерживается CNCF и всеми крупными вендорами наблюдаемости. Для новых проектов OTel — единственный разумный выбор для инструментирования трейсов.

---

**В чём разница между uptime monitoring и APM?**

Uptime monitoring (внешний синтетический мониторинг): проба за пределами вашей инфраструктуры периодически отправляет HTTP-запросы на ваши публичные URL и сигналит, если они не возвращают успешный ответ в течение таймаута. Симулирует базовый пользовательский опыт: «Может ли пользователь вообще зайти на сайт?» Инструменты: UptimeRobot, Pingdom, Checkly. Обнаруживает: сбои DNS, аутаджи CDN, неверную конфигурацию балансировщика нагрузки, полную недоступность сервиса. APM (внутреннее инструментирование): агенты внутри приложения захватывают детальные данные о каждом запросе — SQL-запросы, внешние API-вызовы, стектрейсы ошибок, задержка по эндпоинтам. Обнаруживает: медленные участки кода, неиндексированные запросы, утечки памяти, всплески частоты ошибок для конкретных эндпоинтов. Нужны оба: uptime monitoring ловит «сайт недоступен», APM объясняет «этот конкретный эндпоинт медленный для пользователей из Европы из-за отсутствующего индекса в БД».
