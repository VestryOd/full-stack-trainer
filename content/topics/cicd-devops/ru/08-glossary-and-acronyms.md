# Глоссарий CI/CD и DevOps

Все аббревиатуры и термины, с которыми фуллстек-разработчик столкнётся в контексте CI/CD и DevOps — на собеседованиях, в описаниях вакансий и в повседневной работе. Организован по алфавиту латинских терминов (они используются в индустрии именно так).

---

## A

**Agent (Агент)**
Другое название для **runner** — машина, на которой выполняются задания (jobs) CI/CD-пайплайна. Термин «agent» используется в Jenkins, Azure Pipelines и других инструментах; «runner» — в GitHub Actions и GitLab CI. См.: *Runner*.

**ALB — Application Load Balancer (Балансировщик нагрузки приложения)**
Управляемый балансировщик нагрузки AWS (Amazon Web Services), работающий на уровне HTTP/HTTPS (7-й уровень модели OSI). В отличие от классического балансировщика, который маршрутизирует по IP/порту, ALB может маршрутизировать по пути URL, имени хоста, заголовкам или параметрам запроса. Используется в blue-green и canary-деплоях для переключения трафика между средами.

**APM — Application Performance Monitoring (Мониторинг производительности приложений)**
Категория инструментов, объединяющих распределённые трейсы, метрики и логи в единый интерфейс с фокусом на производительности приложения. APM-инструменты автоматически инструментируют ваш код, отслеживают каждый запрос, выявляют медленные участки кода и группируют ошибки. Примеры: Datadog APM, New Relic, Sentry, Elastic APM. См.: *[Мониторинг и наблюдаемость]*.

**Artifact (Артефакт)**
Любой файл или набор файлов, созданный на этапе сборки и сохранённый для использования на последующем этапе или при деплое. Примеры: скомпилированный JavaScript-бандл, Docker-образ, `.zip`-архив с кодом Lambda-функции, отчёт о покрытии тестами. Ключевой принцип: собирайте артефакт один раз и деплойте один и тот же артефакт во все окружения (staging, production) — никогда не пересобирайте из исходного кода для каждого окружения. См.: *[Основы CI/CD]*.

**Artifact Registry (Реестр артефактов)** (также: **Container Registry** применительно к Docker-образам)
Сервис хранения, где собранные артефакты загружаются, версионируются и откуда их скачивают во время деплоя. Для Docker-образов: GitHub Container Registry (ghcr.io), Docker Hub, AWS ECR (Elastic Container Registry), Google Artifact Registry. Реестр — это не то же самое, что система контроля версий (git): он хранит скомпилированный/упакованный результат, а не исходный код.

## B

**Base Image (Базовый образ)**
Отправная точка для Docker-образа, указываемая в инструкции `FROM` в Dockerfile. Каждый образ строится поверх базового. Распространённые примеры: `node:20-alpine` (Node.js на Alpine Linux), `python:3.12-slim`, `ubuntu:22.04`. `FROM scratch` начинает с полностью пустого образа — для минимальных однобинарных приложений.

**Blast Radius (Зона поражения)**
Масштаб последствий при неудачном деплое. Ключевая цель стратегий деплоя (canary, feature flags) — ограничить blast radius: если в новой версии есть баг, он затронет только небольшой процент пользователей до того, как проблема будет обнаружена и выполнен откат. «Минимизация blast radius» — стандартная фраза в инженерии надёжности.

**Blue-Green Deployment (Сине-зелёный деплой)**
Стратегия деплоя, при которой поддерживаются два идентичных production-окружения («синее» и «зелёное»). Новая версия разворачивается в простаивающем окружении, тестируется, а затем трафик мгновенно переключается через балансировщик нагрузки. Старое окружение продолжает работать для быстрого отката. Исключает период одновременной работы старой и новой версий (в отличие от rolling), но удваивает стоимость инфраструктуры. См.: *[Стратегии деплоя]*.

**Build Context (Контекст сборки)** (Docker)
Набор файлов, передаваемых Docker-демону при выполнении `docker build` — обычно содержимое текущей директории. Файлы, перечисленные в `.dockerignore`, исключаются из контекста сборки. Большой контекст сборки (например, включающий `node_modules/`) замедляет сборку, потому что все эти файлы передаются демону перед выполнением первой инструкции Dockerfile.

## C

**Canary Release (Канареечный релиз)**
Назван по аналогии с «канарейкой в угольной шахте». Стратегия деплоя, при которой небольшой процент реального пользовательского трафика (например, 5%) направляется на новую версию, пока остальные пользователи работают со стабильной версией. Новая версия наблюдается на предмет ошибок и проблем с производительностью; трафик постепенно увеличивается, если всё в порядке. Отличие от feature flag: canary — это инфраструктурное разделение трафика между двумя задеплоенными версиями. См.: *[Стратегии деплоя]*.

**CDK — Cloud Development Kit (Набор инструментов разработки облака)**
Инструмент Infrastructure as Code от AWS, позволяющий определять инфраструктуру AWS на TypeScript, Python, Java и других языках — в отличие от JSON/YAML-шаблонов. CDK-код компилируется в шаблоны CloudFormation. Ощущается как написание кода приложения, а не конфигурации. См.: *IaC*, *CloudFormation*.

**cgroups — Control Groups (Контрольные группы)**
Возможность ядра Linux, ограничивающая использование CPU, памяти, дискового ввода-вывода и пропускной способности сети группой процессов. Вместе с namespaces cgroups — технология, на которой работают Docker-контейнеры: потребление ресурсов каждого контейнера ограничено cgroups, что предотвращает «голодание» соседних контейнеров на одном хосте.

**CI — Continuous Integration (Непрерывная интеграция)**
Практика частого слияния изменений кода в общую основную ветку (несколько раз в день), при этом каждое слияние автоматически запускает пайплайн, который собирает и тестирует код. Цель: обнаруживать проблемы интеграции через минуты после их появления, а не через дни. CI — фундамент, на котором строится CD. См.: *[Основы CI/CD]*.

**CD — Continuous Delivery / Continuous Deployment (Непрерывная доставка / Непрерывный деплой)**
Два разных понятия, обозначаемых одной аббревиатурой. **Continuous Delivery (Непрерывная доставка)**: каждая успешно собранная сборка упаковывается в деплоябельный артефакт и автоматически разворачивается на staging — но деплой в production требует ручного подтверждения. **Continuous Deployment (Непрерывный деплой)**: ручной шаг подтверждения убран; каждая успешная сборка автоматически деплоится в production. Разница — в одном слове и одном человеческом клике. См.: *[Основы CI/CD]*.

**CloudFormation**
Нативный сервис Infrastructure as Code от AWS. Вы описываете ресурсы AWS (EC2-инстансы, S3-бакеты, RDS-базы данных, балансировщики нагрузки и т.д.) в JSON или YAML-шаблонах, и CloudFormation создаёт ими управляет. Зрелый сервис, глубоко интегрированный с AWS, но многословный; многие команды предпочитают CDK (который компилируется в CloudFormation) для лучшего developer experience.

**Composite Action (Составное действие)** (GitHub Actions)
Повторно используемая группа шагов, упакованная в один вызов `uses:`, встраиваемый в job. Composite actions хранятся как файлы `action.yml` (обычно в `.github/actions/`). В отличие от reusable workflows (которые являются полноценными jobs), composite action запускается как часть job наряду с другими шагами. Используется для абстрагирования последовательности шагов установки/сборки. См.: *[GitHub Actions]*.

**Configuration Drift (Дрейф конфигурации)**
Постепенное расхождение окружений (staging, production) между собой из-за ручных изменений, внесённых в одно, но не отражённых в другом — «просто на этот раз» правки в production, которые никогда не попадают в staging. Configuration drift — основная причина «работает на staging, сломано в production». Infrastructure as Code предотвращает дрейф, делая код единственным источником истины о том, что существует. См.: *IaC*.

**Container (Контейнер)**
Запущенный экземпляр Docker-образа — образ плюс перезаписываемый слой поверх него, работающий как изолированный процесс на хост-ОС. Контейнеры разделяют ядро хост-ОС (в отличие от виртуальных машин, у которых есть собственное ядро). Они запускаются и останавливаются за миллисекунды и по задумке являются эфемерными (временными, заменяемыми). См.: *[Основы Docker]*.

**Container Registry (Реестр контейнеров)**
См.: *Artifact Registry*.

**CRD — Custom Resource Definition (Определение пользовательского ресурса)** (Kubernetes)
Механизм расширения в Kubernetes, позволяющий добавлять собственные типы ресурсов в API Kubernetes. Инструменты вроде Argo Rollouts и Cert-Manager работают, устанавливая CRD в кластер и предоставляя контроллеры, которые реагируют на эти пользовательские ресурсы. Когда вы видите `apiVersion: argoproj.io/v1alpha1` в YAML-файле, это ресурс, созданный через CRD.

## D

**DAG — Directed Acyclic Graph (Направленный ациклический граф)**
Структура графа, в которой рёбра (связи между узлами) направлены в одну сторону и нет циклов — нет пути, возвращающегося к начальной точке. В CI/CD DAG описывает зависимости между заданиями пайплайна: задание B зависит от задания A (направленность), и нет циклических зависимостей (ацикличность). Ключевое слово `needs:` в GitLab CI и GitHub Actions создаёт пайплайны в форме DAG. См.: *[GitLab CI]*.

**dind — Docker-in-Docker**
Конфигурация, при которой Docker-демон запускается внутри Docker-контейнера — чтобы Docker-команды (`docker build`, `docker push`) можно было выполнять внутри CI-задания, которое само работает в контейнере. Требуется в GitLab CI при использовании Docker executor и необходимости собирать Docker-образы. Объявляется как сервис: `services: - docker:24-dind`. Имеет последствия для безопасности (требует привилегированный режим). См.: *[GitLab CI]*.

**DNS — Domain Name System (Система доменных имён)**
«Телефонная книга» интернета — распределённая база данных, переводящая человекочитаемые доменные имена (например, `api.myapp.com`) в IP-адреса. Актуально для DevOps в контексте переключения трафика: DNS-переключение в blue-green деплоях имеет задержку распространения (TTL — Time To Live) от минут до часов, что делает его непригодным для мгновенного отката. Переключение через балансировщик нагрузки предпочтительнее.

**Dockerfile**
Текстовый файл, содержащий последовательность инструкций, которые Docker выполняет для создания образа. Каждая инструкция (`FROM`, `RUN`, `COPY`, `CMD` и т.д.) создаёт слой в образе. Dockerfile хранится в репозитории и служит воспроизводимым рецептом сборки контейнерного образа приложения. См.: *[Основы Docker]*.

**Dotenv (файл `.env`)**
Обычный текстовый файл в корне проекта, хранящий переменные окружения в формате `КЛЮЧ=ЗНАЧЕНИЕ`, по одной переменной на строку. Загружается при старте приложения библиотекой `dotenv` (`import 'dotenv/config'`). Предназначен только для локальной разработки — никогда не коммитится в систему контроля версий. Вместо него коммитится файл `.env.example` (с именами переменных, но без реальных значений) — как шаблон для других разработчиков. См.: *[Окружения и конфигурация]*.

## E

**ECR — Elastic Container Registry**
Управляемый реестр Docker-контейнеров от AWS. Вы загружаете Docker-образы в ECR и скачиваете их при деплоях в ECS, Lambda и другие сервисы AWS. Образы тегируются (например, хешем коммита) и версионируются. По умолчанию приватный; доступ управляется через политики AWS IAM (Identity and Access Management — управление идентификацией и доступом).

**ECS — Elastic Container Service**
Управляемый сервис оркестрации контейнеров от AWS. Вы определяете task definitions (какой Docker-образ запускать, сколько CPU/памяти выделить, какие переменные окружения передать) и services (сколько копий запускать, к какому балансировщику нагрузки подключить). ECS управляет запуском, остановкой и проверкой работоспособности контейнеров. Альтернатива Kubernetes, более простая в эксплуатации, но менее гибкая.

**ELK Stack — Elasticsearch + Logstash + Kibana**
Популярный стек управления логами с открытым исходным кодом. **Elasticsearch**: распределённый поисковый и аналитический движок, хранящий и индексирующий данные логов. **Logstash**: пайплайн, принимающий логи из нескольких источников, преобразующий их и пересылающий в Elasticsearch. **Kibana**: веб-интерфейс для запросов и визуализации данных в Elasticsearch. Часто дополняется **Beats** (легковесными агентами сбора логов) и называется **Elastic Stack**.

**Error Budget (Бюджет ошибок)**
Допустимый объём ненадёжности (простоев, ошибок, нарушений задержек), разрешённый до нарушения SLO — выраженный в единицах времени или в количестве событий. Формула: `error budget = 1 − целевой SLO`. Пример: SLO доступности 99,9% → бюджет ошибок 0,1% → 43,2 минуты допустимого простоя за 30 дней. Когда бюджет ошибок почти исчерпан, команды замораживают рискованные изменения; когда его много — могут двигаться быстрее. См.: *SLO*, *[Мониторинг и наблюдаемость]*.

**Expand-Contract Migration (Миграция «расширить — сжать»)** (также: паттерн параллельного изменения)
Безопасная стратегия миграции базы данных для деплоя без простоев. Вместо единого деплоя с ломающими изменениями схемы, изменения разбиваются на несколько деплоев: **Expand (Расширение)** — добавляем новую колонку как nullable, обратная совместимость сохраняется; **Migrate (Миграция)** — заполняем данные в фоновом режиме; **Contract (Сжатие)** — делаем колонку NOT NULL, когда все строки заполнены; **Cleanup (Очистка)** — удаляем старую колонку, когда старый код исчез. Гарантирует, что старая и новая версии приложения могут работать одновременно с одной базой данных. См.: *[Стратегии деплоя]*.

## F

**Feature Flag** (также: Feature Toggle, Feature Switch — переключатель функциональности)
Механизм в коде приложения, позволяющий включать или отключать функцию во время выполнения без деплоя нового кода. Код функции уже задеплоен, но скрыт за условной проверкой (например, `if (flagClient.variation('new-checkout', user, false))`). Позволяет: постепенный выкат по проценту пользователей, мгновенное отключение без повторного деплоя, A/B-тестирование, деплой незавершённых функций в production (под флагом). Не является стратегией деплоя — это техника на уровне кода приложения. См.: *[Стратегии деплоя]*.

## G

**GDPR — General Data Protection Regulation (Общий регламент по защите данных)**
Регламент Европейского союза, регулирующий сбор, хранение, обработку и передачу персональных данных жителей ЕС. Актуально для DevOps: логирование PII (Personally Identifiable Information — персональные данные), таких как адреса электронной почты, телефонные номера или IP-адреса, может нарушать GDPR. В логах следует хранить идентификаторы пользователей (псевдонимизированные), а не «сырые» персональные данные.

**GitOps**
Практика, при которой желаемое состояние инфраструктуры и деплоев приложений определяется в Git, а автоматизированные системы непрерывно приводят фактическое состояние в соответствие с желаемым. Изменения инфраструктуры вносятся через pull requests (рецензируются, тестируются, мержатся), а не прямыми командами. Инструменты: Argo CD, Flux. «Если этого нет в Git — этого не существует» — этос GitOps.

**Graceful Shutdown (Плавное завершение)**
Процесс, при котором приложение корректно обрабатывает сигнал завершения (`SIGTERM`) перед выходом: прекращает принимать новые соединения, завершает обработку запросов в процессе выполнения, закрывает соединения с базой данных и завершается с кодом 0. Без graceful shutdown при каждом деплое теряются запросы, находящиеся в обработке. Необходимо для деплоя без простоев. См.: *[Стратегии деплоя]*, *SIGTERM*.

## H

**HCL — HashiCorp Configuration Language**
Предметно-ориентированный язык, используемый в Terraform и других инструментах HashiCorp для определения инфраструктуры. HCL — декларативный язык: вы описываете желаемое конечное состояние, а Terraform сам определяет, как его достичь. Человекочитаемый, поддерживает выражения, переменные, циклы и модули. Альтернатива написанию CloudFormation YAML или CDK-кода на TypeScript.

**Health Check (Проверка работоспособности)**
Периодический тест, определяющий, жив ли сервис и готов ли принимать трафик. В Kubernetes: **liveness probes** проверяют, жив ли процесс (неудача → перезапуск контейнера); **readiness probes** проверяют, готов ли сервис к трафику (неудача → удаление из пула балансировщика, без перезапуска). В Docker Compose: ключ `healthcheck` задаёт команду для проверки, используемую в `depends_on: condition: service_healthy`. См.: *[Мониторинг и наблюдаемость]*.

**Hosted Runner (Управляемый раннер)**
Раннер, предоставляемый и управляемый CI-платформой (GitHub, GitLab и т.д.). Для каждого запуска пайплайна выделяется свежая виртуальная машина с предустановленными распространёнными инструментами. Обслуживание с вашей стороны не требуется. Вы платите поминутно; железо разделяется с другими пользователями платформы; раннеры не имеют доступа к вашей приватной сети. Противопоставление: *Self-Hosted Runner*.

## I

**IaC — Infrastructure as Code (Инфраструктура как код)**
Практика определения и управления инфраструктурой (серверы, сети, базы данных, балансировщики нагрузки) в файлах кода, хранящихся в системе контроля версий — вместо кликов по веб-консоли или ручного выполнения команд. Преимущества: воспроизводимость, аудитируемость, паритет окружений, восстановление после катастроф, проверка изменений инфраструктуры через PR. Инструменты: Terraform, AWS CDK, CloudFormation, Pulumi, Ansible. См.: *[Окружения и конфигурация]*.

**Image** (Docker Image — Образ Docker)
Доступный только для чтения слоёный снимок файловой системы со всем необходимым для запуска приложения: базовая ОС, среда выполнения (Node.js, Python и т.д.), зависимости, скомпилированный код и конфигурационные файлы. Образ строится из Dockerfile. Несколько контейнеров могут одновременно запускаться из одного образа. Образ — это единица деплоя, которую собирают один раз и продвигают по окружениям. См.: *Container*, *[Основы Docker]*.

**Immutable Infrastructure (Неизменяемая инфраструктура)**
Практика, при которой серверы и контейнеры никогда не модифицируются после деплоя — вместо этого собирается новый образ, разворачивается, и старый заменяется. Если нужно обновить приложение, вы собираете новый Docker-образ и заменяете запущенные контейнеры; вы не подключаетесь по SSH к контейнеру и не патчите файлы вручную. Immutable infrastructure делает деплои воспроизводимыми и устраняет configuration drift на работающих инстансах.

---

## J

**Job (Задание)** (CI/CD)
Единица работы в пайплайне, выполняемая на одной машине раннера/агента. Job состоит из последовательности шагов (команд). Задания в одном stage пайплайна по умолчанию выполняются параллельно (в GitLab CI) или при наличии зависимостей `needs:` (в GitHub Actions). Job создаёт артефакты и сообщает результат (успех/неудача). См.: *Stage*, *Step*, *[Основы CI/CD]*.

---

## K

**K8s — Kubernetes**
Платформа оркестрации контейнеров с открытым исходным кодом, первоначально разработанная Google. «K8s» — нумероним: K + 8 букв + s. Kubernetes автоматизирует деплой, масштабирование и управление контейнеризированными приложениями в кластере машин. Ключевые концепции: pod (группа контейнеров), deployment (объявление желаемого состояния), service (сетевая точка входа), ingress (маршрутизация внешнего трафика), namespace (изоляция ресурсов внутри кластера). Де-факто стандарт для запуска контейнеров в масштабе.

---

## L

**Layer (Слой)** (Docker)
Отдельное инкрементальное изменение файловой системы Docker-образа, создаваемое инструкцией Dockerfile (`FROM`, `RUN`, `COPY`, `ADD`). Слои накапливаются — каждый записывает только разницу относительно нижнего слоя. Слои кешируются: если инструкция слоя и её входные данные не изменились с момента последней сборки, Docker использует кешированный слой. Порядок инструкций от наименее изменяемых к наиболее изменяемым максимизирует попадания в кеш и ускоряет сборку. См.: *[Основы Docker]*.

**Liveness Probe (Проба живости)**
Проверка работоспособности в Kubernetes, определяющая, жив ли процесс контейнера и должен ли он продолжать работу. При неудаче liveness probe Kubernetes перезапускает контейнер. Liveness probe должна проверять только то, отвечает ли сам процесс — а не доступны ли внешние зависимости (база данных, кеш). Если она проверяет внешние зависимости, отказ базы данных приведёт к перезапуску всех pod в бесконечном цикле. См.: *Readiness Probe*, *[Мониторинг и наблюдаемость]*.

**Loki** (Grafana Loki)
Горизонтально масштабируемая система агрегации логов, разработанная Grafana Labs по образцу Prometheus. В отличие от Elasticsearch, Loki не индексирует полное содержимое логов — только их метки (labels). Это делает его дешевле в эксплуатации в масштабе. Loki запрашивается с помощью LogQL (язык запросов, похожий на PromQL) и отображается в дашбордах Grafana. Сочетается с Prometheus (метрики) и Tempo (трейсы) для полноценного open-source стека наблюдаемости.

---

## M

**Metrics (Метрики)**
Числовые измерения свойств системы, фиксируемые во времени и агрегируемые. В отличие от логов (дискретных событий), метрики показывают «сколько» и «как быстро» — частоту запросов, частоту ошибок, использование CPU, перцентили задержек. Четыре золотых сигнала (из книги Google SRE): Latency (задержка), Traffic (трафик), Errors (ошибки), Saturation (насыщенность). Стандартные типы метрик: Counter (только растёт), Gauge (растёт и падает), Histogram (корзины). Инструмент: Prometheus. См.: *[Мониторинг и наблюдаемость]*.

**Multi-Stage Build (Многоэтапная сборка)** (Docker)
Техника Dockerfile, использующая несколько инструкций `FROM`, каждая из которых начинает новый этап сборки. Файлы можно копировать с одного этапа на другой с помощью `COPY --from=имя-этапа`. Финальный образ содержит только то, что скопировано на последний этап — инструменты сборки, dev-зависимости и исходные файлы остаются позади. Радикально уменьшает размер production-образа (например, с ~400 МБ со всеми инструментами сборки до ~80 МБ только со средой выполнения и скомпилированным кодом). См.: *[Основы Docker]*.

---

## N

**Namespace (Пространство имён)** (Kubernetes)
Механизм изоляции групп ресурсов внутри одного кластера Kubernetes. Ресурсы (pod, service, deployment) в одном namespace изолированы от ресурсов в другом. Обычно используется для разделения окружений (dev, staging, prod) внутри одного кластера или для разделения команд/приложений. Стандартные namespaces: `default`, `kube-system` (компоненты кластера), `kube-public`.

**Non-Root User (Непривилегированный пользователь)** (Docker)
Практика безопасности запуска процессов контейнера от имени непривилегированного пользователя (не root, не UID 0). По умолчанию контейнеры Docker работают от root — если злоумышленник выберется из контейнера, у него будут права root на хосте. Добавление `USER node` (или любого непривилегированного пользователя) в Dockerfile снижает этот риск. Базовые образы `node:*` включают встроенного пользователя `node` (UID 1000) для этой цели. См.: *[Основы Docker]*.

---

## O

**Observability (Наблюдаемость)**
Свойство системы, позволяющее понять её внутреннее состояние, изучая внешние выходные данные (логи, метрики, трейсы). Система «высоко наблюдаема», если вы можете диагностировать любой сценарий отказа — в том числе непредвиденный — по данным, которые система излучает. Три кита наблюдаемости: логирование, метрики и распределённое трассирование. Противопоставление: *Monitoring* (наблюдение за заранее определёнными сигналами) vs наблюдаемость (способность обнаруживать неожиданные сценарии отказа). См.: *[Мониторинг и наблюдаемость]*.

**OpenTelemetry (OTel)**
Нейтральный по отношению к вендорам, опенсорсный стандарт и SDK для инструментирования приложений с целью сбора телеметрических данных: трейсов, метрик и логов. OpenTelemetry заменяет специфичные для вендоров агенты — вы инструментируете код один раз с помощью OTel SDK и экспортируете данные в любой совместимый бэкенд (Jaeger, Datadog, Honeycomb, Grafana Tempo). Управляется CNCF (Cloud Native Computing Foundation). Текущий отраслевой стандарт инструментирования наблюдаемости.

**Orchestration (Оркестрация контейнеров)**
Автоматизированное управление контейнеризированными приложениями в кластере машин: планирование размещения контейнеров на узлах, перезапуск неработоспособных контейнеров, масштабирование количества реплик в ответ на нагрузку, управление сетевой маршрутизацией между сервисами и выкат новых версий. Kubernetes — доминирующая платформа оркестрации. AWS ECS и HashiCorp Nomad — альтернативы.

## P

**PCI DSS — Payment Card Industry Data Security Standard (Стандарт безопасности данных индустрии платёжных карт)**
Стандарт безопасности для организаций, обрабатывающих данные кредитных карт. Актуально для логирования: номера карт, CVV-коды и полные имена держателей карт не должны никогда появляться в логах или храниться в незашифрованном виде. Нарушения могут привести к штрафам и потере возможности обрабатывать карточные платежи.

**PII — Personally Identifiable Information (Персональные данные)**
Любые данные, которые можно использовать для идентификации конкретного человека: имя, адрес электронной почты, номер телефона, домашний адрес, идентификационный номер, IP-адрес (в некоторых юрисдикциях). PII требует осторожного обращения в логах, базах данных и аналитических пайплайнах в соответствии с такими нормативными актами, как GDPR, HIPAA, CCPA. В логировании: используйте идентификаторы пользователей (псевдонимизированные), а не «сырые» адреса электронной почты или имена.

**PID 1 — Process ID 1 (Идентификатор процесса 1)**
В системах Linux PID 1 — это init-процесс — первый процесс, запускаемый ядром, ответственный за запуск всех остальных процессов и сборку зомби-процессов (дочерних процессов, завершившихся, но не убранных из таблицы процессов). В Docker-контейнере PID 1 — это главный процесс. Если `CMD` задан в shell-форме (`CMD node server.js`), то `sh` становится PID 1, а Node.js запускается как дочерний процесс — `sh` не пересылает SIGTERM дочерним процессам и не собирает зомби. Всегда используйте exec-форму (`CMD ["node", "server.js"]`), чтобы ваше приложение стало PID 1. См.: *Exec form*, *SIGTERM*, *[Основы Docker]*.

**Pipeline (Пайплайн)**
Последовательность автоматизированных шагов (заданий), выполняемых в определённом порядке при срабатывании триггера (push кода, PR, расписание, ручной запуск). Пайплайн определяется как код (обычно YAML) и хранится в репозитории рядом с кодом приложения. Также называется «CI/CD-пайплайн», «build pipeline» или «workflow» (в GitHub Actions). См.: *Pipeline as Code*, *[Основы CI/CD]*.

**Pipeline as Code (Пайплайн как код)**
Практика определения CI/CD-пайплайнов в конфигурационных файлах (YAML), хранящихся в репозитории исходного кода, версионируемых вместе с кодом приложения. Преимущества: изменения пайплайна проходят тот же процесс ревью через PR, что и изменения кода; история пайплайна видна в git log; пайплайн воспроизводим в разных окружениях. Все современные CI/CD-платформы (GitHub Actions, GitLab CI, Jenkins Pipelines) реализуют эту практику.

**Prometheus**
Опенсорсная система мониторинга и база данных временных рядов, созданная в SoundCloud и теперь входящая в CNCF. Prometheus «скрейпит» (запрашивает) метрики с HTTP-эндпоинтов (`/metrics`), открытых инструментированными приложениями, с заданным интервалом (например, каждые 15 секунд). Метрики хранятся как временные ряды и запрашиваются с помощью PromQL (Prometheus Query Language). Обычно отображается в дашбордах Grafana. Часто используется в паре с Alertmanager для оповещений.

**Provisioning (Инициализация/Подготовка инфраструктуры)**
Процесс создания и настройки ресурсов инфраструктуры для приведения их в готовое к использованию состояние: создание виртуальных машин, настройка сети, установка ОС и программного обеспечения, выделение хранилища. Provisioning обычно выполняется один раз (или по расписанию при замене ресурсов). Инструменты Infrastructure as Code (Terraform, CloudFormation) автоматизируют этот процесс. Противопоставление с *Configuration Management* (текущее обслуживание уже инициализированных систем — область Ansible).

## R

**Readiness Probe (Проба готовности)**
Проверка работоспособности в Kubernetes, определяющая, готов ли контейнер принимать трафик. При неудаче readiness probe Kubernetes удаляет pod из пула маршрутизации балансировщика нагрузки, но не перезапускает контейнер. Readiness probe должна проверять доступность всех необходимых внешних зависимостей (база данных, кеш, downstream-сервисы). Противопоставление с *Liveness Probe* (которая определяет, нужно ли перезапускать). См.: *[Мониторинг и наблюдаемость]*.

**Rolling Deployment (Скользящее обновление)**
Стратегия деплоя, при которой экземпляры старой версии заменяются новой постепенно — по несколько штук за раз — пока сервис продолжает работу. Новый экземпляр проходит проверку работоспособности, после чего старый дренируется и останавливается. Во время выкатки одновременно работают обе версии, что требует обратно совместимых миграций базы данных. Встроено в Kubernetes, ECS и большинство PaaS-платформ. См.: *[Стратегии деплоя]*.

**Runner (Раннер)** (также: Agent)
Машина — виртуальная или физическая — на которой фактически выполняются задания CI/CD-пайплайна. При срабатывании пайплайна планировщик CI назначает задания доступным раннерам. **Hosted runners** предоставляются CI-платформой (GitHub-hosted, GitLab.com shared). **Self-hosted runners** — машины, которыми вы владеете и управляете, зарегистрированные в CI-платформе. См.: *[Основы CI/CD]*, *[GitHub Actions]*, *[GitLab CI]*.

---

## S

**Secrets Manager (Менеджер секретов)**
Специализированный сервис для хранения, получения, аудита и ротации чувствительных учётных данных (пароли, API-ключи, TLS-сертификаты, ключи шифрования). Предоставляет: шифрование в покое, журнал аудита (кто и когда что запрашивал), гранулированный контроль доступа, автоматическую ротацию и версионирование. Примеры: AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager, Azure Key Vault, Doppler, Infisical. Не то же самое, что менеджер паролей (для людей). См.: *[Окружения и конфигурация]*.

**Self-Hosted Runner (Самохостируемый раннер)**
Раннер, которым владеете и управляете вы, с установленным программным обеспечением раннера, зарегистрированным в вашей CI/CD-платформе. Обеспечивает доступ к приватным сетям, специфическому железу (GPU, определённая архитектура CPU) и может быть экономически выгоднее при больших объёмах пайплайнов. Проблема безопасности: самохостируемые раннеры в публичных репозиториях могут быть использованы злоумышленниками через вредоносные PR, изменяющие YAML воркфлоу для выполнения произвольного кода на вашей машине. См.: *[GitHub Actions]*, *[GitLab CI]*.

**Service Mesh (Сервисная сетка)**
Выделенный инфраструктурный уровень, управляющий коммуникацией между сервисами в микросервисной архитектуре: балансировка нагрузки, маршрутизация трафика, mTLS (взаимное TLS-шифрование между сервисами), повторные попытки, circuit breaking и наблюдаемость. Реализуется как sidecar-прокси, внедряемые в каждый pod. Примеры: Istio, Linkerd. Актуально для canary-релизов: service mesh может разделять трафик с точностью до процента без изменения кода приложения. См.: *Sidecar*.

**SIGTERM — Signal Terminate (Сигнал завершения)**
Unix/Linux-сигнал (номер сигнала 15), отправляемый операционной системой или оркестратором процессу для запроса плавного завершения. В отличие от SIGKILL (сигнал 9, немедленно убивающий процесс), SIGTERM может быть перехвачен процессом, который затем получает время на завершение (обработка текущих запросов, закрытие соединений с БД) перед выходом. Kubernetes отправляет SIGTERM перед остановкой pod, затем SIGKILL по истечении grace period (по умолчанию 30 секунд). См.: *Graceful Shutdown*, *PID 1*.

**SLA — Service Level Agreement (Соглашение об уровне сервиса)**
Договорное обязательство между поставщиком услуг и его клиентами, определяющее ожидаемый уровень сервиса (аптайм, время отклика, частота ошибок) и финансовые последствия (кредиты за сервис, возвраты, право на расторжение) при невыполнении обязательств. SLA — это внешний, юридически обязывающий документ. Он всегда менее строгий, чем внутренний SLO, для обеспечения буфера безопасности. Пример: SLA AWS S3 гарантирует 99,9% аптайма в месяц с кредитами за сервис при нарушении. См.: *SLO*, *SLI*, *[Мониторинг и наблюдаемость]*.

**SLI — Service Level Indicator (Индикатор уровня сервиса)**
Конкретная, измеримая метрика, отражающая насколько хорошо сервис работает для пользователей — само измерение. Примеры: процент запросов, завершающихся менее чем за 200 мс; процент запросов, возвращающих статус не 5xx; процент заданий, завершающихся в срок. SLI — это входные данные для SLO. См.: *SLO*, *SLA*.

**SLO — Service Level Objective (Целевой показатель уровня сервиса)**
Внутренняя цель для SLI — порог, ниже которого команда считает, что сервис не справляется с задачей. Не обязательство перед клиентами (это SLA); SLO — внутренняя инженерная цель. Разрыв между SLO и SLA даёт буфер безопасности. SLO в сочетании с концепцией error budget делает надёжность торгуемым ресурсом: если error budget велик, команды могут деплоить агрессивнее; если почти исчерпан — замораживают рискованные изменения. См.: *SLA*, *SLI*, *Error Budget*.

**SRE — Site Reliability Engineering (Инженерия надёжности сайтов)**
Дисциплина и должностная роль, основанная Google, применяющая принципы разработки программного обеспечения к проблемам инфраструктуры и эксплуатации. SRE-инженеры определяют и поддерживают SLO, управляют error budget, создают автоматизацию для замены ручной операционной работы и реагируют на инциденты. Концепция «четырёх золотых сигналов» (Latency, Traffic, Errors, Saturation) пришла из книги Google SRE. Цель SRE — сделать системы надёжными, сохраняя при этом скорость разработки.

**Stage (Этап)** (CI/CD)
Именованная группа заданий в пайплайне, которые выполняются параллельно (в пределах этапа), и все они должны завершиться успешно, прежде чем начнётся следующий этап. Ключ `stages:` в GitLab CI задаёт как имена этапов, так и порядок их выполнения. В GitHub Actions нет первоклассной концепции «stage» — порядок задаётся через ключ `needs:` между заданиями. См.: *Job*, *[Основы CI/CD]*, *[GitLab CI]*.

**State File (Файл состояния)** (Terraform)
JSON-файл, который Terraform использует для отслеживания текущего состояния всей управляемой им инфраструктуры. При выполнении `terraform apply` Terraform сравнивает желаемое состояние (из файлов `.tf`) с текущим состоянием (из файла состояния) и определяет минимальный набор изменений. В командной среде файл состояния должен храниться удалённо (например, в S3-бакете с DynamoDB для блокировки) для предотвращения одновременных изменений. Файл состояния может содержать чувствительные значения и никогда не должен коммититься в git.

**Step (Шаг)** (GitHub Actions)
Отдельная команда или действие внутри задания (job). Шаг либо выполняет команду оболочки (`run: npm test`), либо вызывает предготовленное действие (`uses: actions/checkout@v4`). Шаги внутри задания выполняются последовательно. Сравнение: *Job* (несколько шагов, одна машина) и *Stage* (несколько заданий). См.: *[GitHub Actions]*.

## T

**Terraform**
Наиболее широко используемый инструмент Infrastructure as Code, созданный HashiCorp. Написан на HCL (HashiCorp Configuration Language). Независим от провайдера: имеет провайдеры для AWS, GCP, Azure, Cloudflare, Vercel, GitHub, Kubernetes и сотен других. Рабочий процесс: `terraform init` → `terraform plan` (предпросмотр изменений) → `terraform apply` (применить изменения) → `terraform destroy` (снести). Состояние отслеживается в файле состояния. См.: *IaC*, *HCL*, *State File*.

**Trace (Трейс — распределённая трассировка)**
Запись полного пути единственного запроса через все сервисы и компоненты распределённой системы. Трейс состоит из нескольких **span** (по одному на каждый сервис/компонент). Каждый span записывает: имя операции, время начала и окончания, ошибки. Trace ID передаётся через все межсервисные вызовы (через HTTP-заголовок `traceparent` согласно стандарту W3C Trace Context). Трейсы отвечают на вопрос: «Где конкретный запрос провёл своё время и где произошёл сбой?» См.: *Span*, *OpenTelemetry*, *[Мониторинг и наблюдаемость]*.

**TTL — Time To Live (Время жизни)**
В DNS: количество секунд, в течение которых DNS-запись кешируется резолверами и клиентами перед повторным запросом к авторитетному DNS-серверу. Актуально для blue-green деплоев: если вы переключаете трафик через обновление DNS-записи и TTL равен 3600 (1 час), некоторые пользователи будут направляться в старое окружение до часа. Уменьшите TTL до 60 секунд перед плановым переключением трафика. Переключение через балансировщик нагрузки лишено этой проблемы.

## U

**Uptime Monitoring (Мониторинг аптайма)**
Практика периодической отправки запросов к вашему сервису из внешних мест и оповещения, когда сервис не отвечает корректно. «Внешний» мониторинг — симулирует то, что испытывает реальный пользователь. Отличается от внутреннего мониторинга (метрики, производимые самим приложением). Сервис может казаться здоровым внутренне, будучи недоступным извне из-за проблем с сетью/DNS/файерволом. Инструменты: UptimeRobot, Pingdom, Checkly. См.: *[Мониторинг и наблюдаемость]*.

---

## V

**Vault** (HashiCorp Vault)
Опенсорсный инструмент управления секретами от HashiCorp. Обеспечивает централизованное хранение секретов с шифрованием в покое, гранулированными политиками доступа, журналом аудита, динамическими секретами (учётные данные, генерируемые по запросу и автоматически отзываемые) и ротацией секретов. Доступен как self-hosted или HCP Vault (управляемый облачный). Сложнее в эксплуатации, чем управляемые облачные сервисы (AWS Secrets Manager), но более гибкий и не привязан к конкретному облачному провайдеру. См.: *Secrets Manager*.

**VPC — Virtual Private Cloud (Виртуальное приватное облако)**
Логически изолированный раздел сети облачного провайдера, который вы определяете и контролируете. Ваши EC2-инстансы, RDS-базы данных, Lambda-функции и другие ресурсы работают внутри вашего VPC и недоступны из публичного интернета без явной настройки маршрутизации. Актуально для CI/CD: самохостируемые раннеры внутри VPC могут обращаться к приватным базам данных и внутренним сервисам; GitHub-hosted runners (за пределами вашего VPC) — не могут.

## W

**Workflow (Воркфлоу)** (GitHub Actions)
YAML-файл в `.github/workflows/`, определяющий автоматизацию CI/CD: когда запускается (триггеры), что делает (задания и шаги) и где запускается (тип раннера). Репозиторий может иметь несколько файлов воркфлоу для разных целей (CI, деплой, релиз, запланированные задачи). См.: *[GitHub Actions]*, *Pipeline as Code*.

## Y

**YAML — YAML Ain't Markup Language (YAML — не язык разметки)**
Человекочитаемый формат сериализации данных, широко используемый для конфигурационных файлов. Название — рекурсивный акроним (определение ссылается на само себя). YAML — стандартный формат для определения CI/CD-пайплайнов (воркфлоу GitHub Actions, `.gitlab-ci.yml`, манифесты Kubernetes, файлы Docker Compose). Ключевые правила синтаксиса: отступы значимы (используйте пробелы, никогда не табуляцию); `:` разделяет ключи от значений; `-` обозначает элементы списка; `#` начинает комментарий. Чувствительность YAML к пробелам — частый источник труднодиагностируемых ошибок — используйте YAML-линтер (`yamllint`) в своём пайплайне.

## Zero-Downtime Deployment (Деплой без простоев)

Требование, при котором деплой обеспечивает непрерывную доступность сервиса для пользователей — ни секунды простоя. Достигается через стратегии: rolling updates (замена инстансов по одному после прохождения проверок работоспособности), blue-green switching (мгновенное переключение трафика после валидации нового окружения) и graceful shutdown (приложение обрабатывает текущие запросы перед завершением). Деплой без простоев на уровне инфраструктуры необходим, но недостаточен — приложение также должно корректно обрабатывать `SIGTERM`. См.: *[Стратегии деплоя]*, *Graceful Shutdown*.
