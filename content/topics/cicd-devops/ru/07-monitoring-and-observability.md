# Monitoring and Observability

## Мониторинг vs Observability — различие

Эти два термина часто используют как синонимы, но они описывают разные вещи:

**Мониторинг** (monitoring) — практика сбора заранее определённых сигналов от системы и отправки алертов, когда эти сигналы превышают пороговые значения. Отвечает на вопрос: "Что-то идёт не так?"

**Observability** (наблюдаемость) — свойство системы, позволяющее понять её внутреннее состояние, исследуя внешние выходные данные. Отвечает на вопрос: "Почему что-то идёт не так?" — даже для режимов отказа, которые вы не предвидели при проектировании системы.

```txt
Мониторинг (известные неизвестные):
  "Оповести меня, когда частота ошибок > 1%" — вы знали, что ошибки — это то, за чем следить

Observability (неизвестные неизвестные):
  "Почему поток оформления заказа работает в 3× медленнее только для пользователей
   из Германии, использующих Safari на iOS после 18:00 в будние дни?"
  — вы не предвидели этот режим отказа;
    observability позволяет обнаружить его из данных
```

Система с большим количеством мониторинга может всё равно быть плохо наблюдаемой — если вы собираете только агрегированные метрики, вы видите *что* что-то не так, но не *почему*.

## Три столпа observability

Три сигнала, которые вместе дают полную картину здоровья распределённой системы — это **логи**, **метрики** и **трейсы**. Каждый отвечает на разный вопрос.

### Столп 1: Логи (Logs)

**Лог** — это запись с временной меткой о дискретном событии, произошедшем в системе: пришёл запрос, возникла ошибка, пользователь вошёл в систему, выполнился запрос к БД.

```txt
Логи отвечают: "Что именно произошло в этот конкретный момент?"
```

Пример вывода лога (структурированный JSON — отраслевой стандарт):

```json
{
  "timestamp": "2025-03-15T14:32:01.234Z",
  "level": "error",
  "message": "Database query failed",
  "service": "user-service",
  "requestId": "req-abc123",
  "userId": "usr-456",
  "query": "SELECT * FROM users WHERE id = $1",
  "error": "connection timeout after 5000ms",
  "duration_ms": 5002
}
```

**Структурированное логирование** (JSON) vs **неструктурированное** (простой текст):

```ts
// ❌ Неструктурированное — читаемо для человека, но неудобно для машины
console.log(`User ${userId} failed login attempt from IP ${ip}`);

// ✅ Структурированное — запрашиваемое, фильтруемое, парсируемое инструментами агрегации
logger.warn('login_failed', {
  userId,
  ip,
  attempt: attemptCount,
  reason: 'invalid_password',
});
```

Со структурированными логами ваш инструмент агрегации логов (Datadog, Grafana Loki, AWS CloudWatch, ELK Stack — Elasticsearch + Logstash + Kibana) позволяет фильтровать и агрегировать:
- "Показать все ошибки для userId = usr-456 за последний час"
- "Посчитать неудачные попытки входа, сгруппированные по IP-адресу"
- "Алерт, когда количество ошибок для service = payment-service > 50 за 5 минут"

**Уровни логирования** (от наименее до наиболее серьёзного):

```txt
DEBUG   → детальная диагностическая информация; включать только при отладке;
           никогда не логировать в prod на уровне DEBUG (слишком шумно, влияние на производ.)

INFO    → нормальные, ожидаемые события (получен запрос, пользователь вошёл,
           задача завершена); дефолтный уровень логирования в production

WARN    → что-то неожиданное произошло, но система справилась
           (retry удался после первой попытки, использован устаревший API,
           приближается к лимиту rate limit); исследовать в спокойный период

ERROR   → что-то сломалось и требует немедленного внимания
           (БД недоступна, ошибка обработки платежа, необработанное исключение)

FATAL   → приложение не может продолжать работу; процесс вот-вот завершится
```

**Старший нюанс #1 — что НЕ следует логировать:**

```ts
// ❌ Никогда не логировать учётные данные или PII (Personally Identifiable Information —
//    персональные данные, по которым можно идентифицировать конкретного человека:
//    имя, email, телефон, адрес)
logger.info('user login', { email, password });        // логирует пароль!
logger.info('payment', { cardNumber, cvv });           // нарушение PCI DSS

// ✅ Логировать идентификаторы, не чувствительные данные
logger.info('user login', { userId, email: maskEmail(email) });
logger.info('payment initiated', { userId, orderId, last4: card.last4 });
```

Логирование PII или учётных данных — нарушение безопасности/compliance (GDPR — Общий регламент защиты данных ЕС, PCI DSS — стандарт безопасности платёжных карт, HIPAA — закон о медицинских данных США; все требуют защиты данных пользователей). Многие компании были оштрафованы за логирование email или телефонов.

### Столп 2: Метрики (Metrics)

**Метрика** — это числовое измерение свойства системы, собираемое во времени. Метрики агрегированы — они говорят *сколько* или *как быстро*, а не *что конкретно произошло*.

```txt
Метрики отвечают: "Как система работает в целом, с течением времени?"
```

Четыре "золотых сигнала" (придуманы книгой Google SRE — Site Reliability Engineering, инженерия надёжности сайтов):

```txt
1. Latency (задержка) — сколько времени занимает обработка запросов
                        (p50, p95, p99 — перцентили; среднее обычно вводит в заблуждение)

2. Traffic (трафик)   — сколько запросов получает система
                        (запросов в секунду, сообщений в секунду, активных соединений)

3. Errors (ошибки)    — доля неудачных запросов
                        (5xx ответы / все ответы, частота исключений, частота таймаутов)

4. Saturation         — насколько система "заполнена"; как близко к пределу мощности
(насыщенность)          (CPU %, RAM %, диск %, глубина очереди, использование пула соединений БД)
```

**Перцентили vs среднее** — критически важное различие:

```txt
Представьте 100 запросов: 99 занимают 10мс, 1 занимает 10,000мс (10 секунд):
  Средняя задержка:  ~110мс  ← выглядит нормально
  p99 задержка:   10,000мс  ← 1% пользователей ждут 10 секунд — это кризис

Среднее маскирует выбросы. Всегда мониторьте перцентили (p95, p99) для задержки.
p99 = "99% запросов завершаются в течение этого времени"
      (худший 1% исключён — это ваши самые разочарованные пользователи)
```

Распространённые типы метрик:

```txt
Counter    → монотонно возрастающее число (всего запросов, всего ошибок)
             только растёт; полезен для вычисления частот (ошибок/секунду)

Gauge      → значение, которое может расти и падать (текущее использование памяти,
             глубина очереди, количество активных соединений)

Histogram  → распределяет значения по bucket'ам (корзинам); используется для
             перцентилей задержки (сколько запросов заняли 0-10мс? 10-50мс? 50-200мс? >200мс?)

Summary    → похож на histogram, но вычисляет перцентили на стороне клиента;
             менее гибкий для агрегации по нескольким экземплярам
```

Пример — инструментирование Express-приложения Node.js с метриками Prometheus (Prometheus — open-source система сбора метрик и алертинга):

```ts
import { Registry, Counter, Histogram } from 'prom-client';

const registry = new Registry();

const httpRequestsTotal = new Counter({
  name: 'http_requests_total',
  help: 'Total number of HTTP requests',
  labelNames: ['method', 'route', 'status_code'],
  registers: [registry],
});

const httpRequestDuration = new Histogram({
  name: 'http_request_duration_seconds',
  help: 'HTTP request duration in seconds',
  labelNames: ['method', 'route'],
  buckets: [0.01, 0.05, 0.1, 0.5, 1, 5],   // границы bucket'ов в секундах
  registers: [registry],
});

// Middleware для инструментирования всех запросов
app.use((req, res, next) => {
  const end = httpRequestDuration.startTimer({ method: req.method, route: req.path });
  res.on('finish', () => {
    httpRequestsTotal.inc({
      method: req.method,
      route: req.path,
      status_code: res.statusCode,
    });
    end();   // записывает длительность
  });
  next();
});

// Эндпоинт метрик для scrape Prometheus
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', registry.contentType);
  res.end(await registry.metrics());
});
```

### Столп 3: Распределённая трассировка (Distributed Tracing)

**Трейс** (trace) — это след единственного запроса, который проходит через все сервисы и компоненты распределённой системы, фиксируя, сколько времени было потрачено на каждом шаге.

```txt
Трейсы отвечают: "Каков полный путь этого конкретного запроса,
                 и где именно он замедлился или провалился?"
```

В монолитном приложении один stack trace рассказывает всё. В микросервисной архитектуре один пользовательский запрос может затронуть 8 сервисов:

```txt
Запрос пользователя: "Загрузить дашборд"
  │
  ├── API Gateway (2мс)
  │   │
  │   ├── Auth Service — проверить JWT (5мс)
  │   │
  │   ├── User Service — получить профиль (8мс)
  │   │   └── запрос к PostgreSQL (6мс)
  │   │
  │   └── Feed Service — получить ленту активности (230мс)  ← 🐌 МЕДЛЕННО
  │       ├── проверка Redis cache (1мс) — MISS (промах)
  │       └── запрос к PostgreSQL (225мс)  ← УЗКОЕ МЕСТО
  │
  Итого: 245мс
```

Без трейсинга вы видите, что дашборд загружается за 245мс — медленно. С трейсингом вы сразу видите, что запрос к PostgreSQL в Feed Service отвечает за 225мс (92%) от общего времени.

**Как работает трейсинг:**

Каждому запросу присваивается уникальный **trace ID** в точке входа (API Gateway или первом сервисе). По мере прохождения через каждый сервис запрос несёт этот ID в заголовке (`traceparent` по стандарту W3C Trace Context). Каждый сервис создаёт **span** — именованную, хронометрированную единицу работы внутри трейса.

```txt
Trace ID: abc-123

  Span: "api-gateway"     [0мс ─────────────────────────── 245мс]
    Span: "auth-service"  [2мс ─────── 7мс]
    Span: "user-service"  [7мс ─────────── 15мс]
      Span: "db-query"    [9мс ──────── 15мс]
    Span: "feed-service"  [15мс ─────────────────────────── 245мс]
      Span: "redis-check" [15мс ── 16мс]
      Span: "db-query"    [16мс ─────────────────────────── 241мс]
```

**OpenTelemetry** (OTel) стал отраслевым стандартом инструментирования. Это vendor-neutral (не привязанный к поставщику) SDK, который однажды инструментирует код и экспортирует трейсы (а также метрики и логи) в любой бэкенд — Jaeger, Zipkin, Datadog, Honeycomb и т.д.

```ts
import { NodeSDK } from '@opentelemetry/sdk-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';
import { HttpInstrumentation } from '@opentelemetry/instrumentation-http';
import { ExpressInstrumentation } from '@opentelemetry/instrumentation-express';
import { PgInstrumentation } from '@opentelemetry/instrumentation-pg';

const sdk = new NodeSDK({
  traceExporter: new OTLPTraceExporter({
    url: 'http://otel-collector:4318/v1/traces',
  }),
  instrumentations: [
    new HttpInstrumentation(),      // авто-инструментирует все HTTP запросы/ответы
    new ExpressInstrumentation(),   // авто-инструментирует роуты и middleware Express
    new PgInstrumentation(),        // авто-инструментирует запросы PostgreSQL
  ],
});

sdk.start();
```

При такой настройке каждый HTTP-запрос, обработчик роута Express и запрос к БД автоматически трейсится — ручная инструментация для общих случаев не нужна.

## APM — Application Performance Monitoring

**APM** (Application Performance Monitoring — мониторинг производительности приложений) — категория инструментов, объединяющих метрики, трейсы и иногда логи в единое представление, специально сфокусированное на производительности на уровне приложения.

APM-инструменты обычно предоставляют:
- **Transaction tracing**: автоматически трейсить каждый запрос через приложение
- **Code-level profiling**: определять, какая строка кода / вызов функции работает медленно
- **Error tracking**: группировать похожие ошибки, показывать stack trace'ы, считать вхождения
- **Dependency mapping**: визуализировать, какие сервисы вызывают другие сервисы
- **Alerting**: уведомлять при деградации производительности ниже порога

Популярные APM-инструменты:

```txt
Datadog APM     → SaaS; отличная авто-инструментация; широкая экосистема;
                   дорого при масштабе; "Datadog APM" = их продукт для трейсинга
                   (Datadog также занимается метриками и логами на той же платформе)

New Relic       → SaaS; один из первых APM-инструментов; хорош для традиционных
                   монолитов; более новые функции для микросервисов

Sentry          → прежде всего отслеживание ошибок, но также добавляет мониторинг
                   производительности; очень популярен в frontend/fullstack мире;
                   доступна open-source self-hosted версия

Elastic APM     → часть ELK Stack (Elasticsearch, Logstash, Kibana);
                   open-source; хорош для команд, уже использующих Elasticsearch

Grafana Stack   → open-source; Grafana (дашборды) + Prometheus (метрики) +
(OSS вариант)     Loki (логи) + Tempo (трейсы); больше настройки, но полностью бесплатно
```

Разница между APM-инструментом и отдельными столпами:

```txt
Инструмент логирования (Grafana Loki, CloudWatch Logs)  → хранит и запрашивает логи
Инструмент метрик (Prometheus, Datadog Metrics)          → хранит и запрашивает метрики
Инструмент трейсинга (Jaeger, Zipkin, Tempo)             → хранит и запрашивает трейсы
APM-платформа (Datadog, New Relic, Sentry)               → объединяет всё три с UI,
                                                            сфокусированным на здоровье приложения
```

## Uptime-мониторинг

**Uptime-мониторинг** — простейшая форма мониторинга: периодически отправлять запрос к вашему сервису из внешнего местоположения и отправлять алерт, если он не отвечает корректно.

```txt
Каждую 1 минуту, с серверов в 5 регионах:
  HTTP GET https://api.myapp.com/health
    → ожидаем: HTTP 200 в течение 3 секунд
    → при 2 последовательных неудачах: алерт через PagerDuty / Slack / email
```

Это "внешний" мониторинг — он имитирует то, что испытывает реальный пользователь при попытке обратиться к сервису. Он отличается от "внутреннего" мониторинга (метрики изнутри приложения). Приложение может работать и сообщать о себе как о здоровом изнутри, оставаясь недоступным извне (изменилось правило firewall, проблема распространения DNS, неверная конфигурация load balancer).

Популярные сервисы uptime-мониторинга:

```txt
Pingdom         → SaaS; мониторинг из 100+ локаций; SMS + email алерты
UptimeRobot     → есть бесплатный тариф (интервал 5 минут); популярен для side-проектов
Checkly         → мониторинг через синтетические скрипты (не только ping — может симулировать
                   пользовательский сценарий: логин → добавить в корзину → оформить заказ)
AWS CloudWatch  → встроен, если вы на AWS; позволяет создавать health checks по URL
StatusPage.io   → не сам мониторинг, но сервис публичных страниц статуса
                   (то, что вы видите на status.stripe.com, githubstatus.com)
```

## Эндпоинты health check

**Health check endpoint** — специальный API-маршрут в приложении, который сообщает, является ли сервис здоровым и готовым принимать трафик. Это не пользовательская функция — это инфраструктурная сантехника.

```txt
GET /health  →  200 OK  {"status": "ok"}
              или
              503 Service Unavailable  {"status": "degraded", "reason": "database unreachable"}
```

Существуют два типа health check, и их путаница вызывает проблемы:

**Liveness probe** (проба живучести) — "Жив ли процесс? Нужно ли его перезапустить?"

Liveness check падает, если процесс приложения находится в невосстановимом состоянии (дедлок, нехватка памяти, заблокированный event loop). Ответ на неудачный liveness check — **перезапустить контейнер**.

```ts
// Liveness: просто убедиться, что процесс может ответить
app.get('/health/live', (_req, res) => {
  res.status(200).json({ status: 'ok' });
});
```

**Readiness probe** (проба готовности) — "Готов ли сервис принимать трафик?"

Readiness check падает, если приложение работает, но ещё не готово обслуживать запросы (ещё прогревается, соединение с БД ещё не установлено, кэш не заполнен). Ответ на неудачный readiness check — **убрать pod из load balancer** — но не перезапускать.

```ts
// Readiness: проверить, что все зависимости доступны
app.get('/health/ready', async (_req, res) => {
  const checks = await Promise.allSettled([
    db.query('SELECT 1'),        // БД доступна?
    redis.ping(),                // Redis доступен?
  ]);

  const dbOk = checks[0].status === 'fulfilled';
  const redisOk = checks[1].status === 'fulfilled';

  if (dbOk && redisOk) {
    res.status(200).json({ status: 'ok', db: 'ok', redis: 'ok' });
  } else {
    res.status(503).json({
      status: 'degraded',
      db: dbOk ? 'ok' : 'unreachable',
      redis: redisOk ? 'ok' : 'unreachable',
    });
  }
});
```

Kubernetes использует обе пробы:

```yaml
livenessProbe:
  httpGet:
    path: /health/live
    port: 3000
  initialDelaySeconds: 10     # ждать 10с перед первой проверкой (время запуска)
  periodSeconds: 10
  failureThreshold: 3         # перезапустить после 3 последовательных неудач

readinessProbe:
  httpGet:
    path: /health/ready
    port: 3000
  initialDelaySeconds: 5
  periodSeconds: 5
  failureThreshold: 2         # убрать из load balancer после 2 неудач
```

### Старший нюанс #2 — антипаттерны health check

```ts
// ❌ Антипаттерн 1: health check, который может вызвать каскадный сбой
// Если у сервиса 50 pod'ов, и каждый /health делает запрос к БД,
// это 50 запросов/минуту только для health checks.
// Под высокой нагрузкой это может перегрузить БД.
app.get('/health', async (req, res) => {
  await db.query('SELECT COUNT(*) FROM users');   // полное сканирование таблицы — плохо!
  res.json({ status: 'ok' });
});

// ✅ Используйте дешёвый, целевой запрос
app.get('/health/ready', async (req, res) => {
  await db.query('SELECT 1');   // просто проверяет соединение, не данные
  res.json({ status: 'ok' });
});

// ❌ Антипаттерн 2: liveness check, проверяющий внешние зависимости
// Если БД недоступна, падающий liveness probe перезапустит все pod'ы.
// Но перезапуск pod'ов не починит БД — вы получите бесконечный цикл перезапусков.
// Liveness должна проверять ТОЛЬКО то, что ЭТОТ ПРОЦЕСС жив.
app.get('/health/live', async (req, res) => {
  await db.query('SELECT 1');   // ← неправильно для liveness
  res.json({ status: 'ok' });
});
```

## SLA, SLO, SLI — терминология уровней сервиса

Три аббревиатуры описывают, как определяются, измеряются и согласовываются обязательства по надёжности. Это разные уровни:

```txt
SLI → что вы измеряете
SLO → чему вы обязуетесь внутри компании
SLA → что вы обещаете клиентам (с последствиями при нарушении)
```

### SLI — Service Level Indicator (Индикатор уровня сервиса)

**SLI** — конкретная, измеримая метрика, отражающая то, насколько хорошо сервис работает для пользователей. Это сырое измерение.

```txt
Примеры SLI:
  - SLI доступности:   (количество успешных запросов) / (всего запросов) × 100%
  - SLI задержки:      процент запросов, завершающихся менее чем за 200мс
  - SLI частоты ошибок:процент запросов, возвращающих статус 5xx
  - SLI пропускной сп.:количество транзакций, обработанных в секунду
```

### SLO — Service Level Objective (Цель уровня сервиса)

**SLO** — внутренний целевой показатель для SLI — порог, ниже которого вы считаете, что сервис не соответствует требованиям пользователей. Это цель, а не обещание клиентам.

```txt
Примеры SLO:
  - "Доступность ≥ 99.9% в скользящем 30-дневном окне"
  - "p99 задержка ≤ 500мс для всех API-эндпоинтов"
  - "Частота ошибок < 0.1% в любом 5-минутном окне"
```

**Error budget** (бюджет ошибок) — концепция, делающая SLO практически применимыми: количество ненадёжности, которое вы можете себе позволить до нарушения SLO.

```txt
SLO: доступность 99.9% в течение 30 дней
  30 дней = 43,200 минут
  0.1% от 43,200 минут = 43.2 минуты разрешённого простоя в месяц

Error budget = 43.2 минуты / месяц

Если вы использовали уже 40 минут → осталось только 3.2 минуты error budget
→ заморозить рискованные деплои до начала нового месяца
→ сосредоточить инженерные усилия на улучшении надёжности

Если error budget стабильно не расходуется → можно двигаться быстрее,
деплоить чаще, брать больше рисков
```

Концепция error budget из книги Google SRE устраняет разрыв между продуктовыми командами (хотят двигаться быстро) и ops-командами (хотят стабильности): она количественно выражает, сколько риска система может себе позволить, в единицах, понятных всем.

### SLA — Service Level Agreement (Соглашение об уровне сервиса)

**SLA** — контрактное обязательство между поставщиком сервиса и клиентом, определяющее ожидаемый уровень сервиса и последствия (кредиты, возврат средств, право на расторжение), если этот уровень не достигнут.

```txt
Примеры SLA:
  AWS S3 SLA:  99.9% ежемесячного uptime; при uptime < 99% клиент получает
               10% сервисного кредита; при uptime < 95% — 25% кредита

  Stripe SLA:  99.99% uptime API; простои компенсируются сервисными кредитами
               (не реальным возвратом денег — читайте мелкий шрифт)

  Enterprise SaaS SLA: 99.9% uptime; < 99.9% → клиент получает пропорциональный
                       возврат; < 95% → клиент имеет право расторгнуть контракт
```

Отношения между тремя понятиями:

```txt
SLI (измерение) → SLO (внутренняя цель) → SLA (внешнее обязательство)

SLO СТРОЖЕ, чем SLA:
  Если ваш SLA обещает 99.9%, ваш внутренний SLO может быть 99.95%.
  Разрыв между SLO и SLA — это ваш "запас безопасности":
  если вы приближаетесь к SLO, вы замечаете и исправляете это
  до нарушения SLA и финансовых последствий.
```

**"Девятки" доступности** — распространённое сокращение:

```txt
"Две девятки"   = 99%    = 3.65 дня простоя/год   (87.6 часов)
"Три девятки"   = 99.9%  = 8.76 часов простоя/год
"Четыре девятки"= 99.99% = 52.6 минуты простоя/год
"Пять девяток"  = 99.999%= 5.26 минуты простоя/год

Примечание: "пять девяток" крайне сложно достичь даже крупнейшим компаниям.
Большинство SaaS-продуктов целятся в "три девятки" или "четыре девятки".
Достижение "четырёх девяток" обычно требует: мультирегиональной избыточности,
деплоев без простоя, автоматизированного failover, подробных runbook'ов.
```

## Всё вместе: практическая настройка observability

Для Node.js + PostgreSQL приложения, развёрнутого на AWS:

```txt
Логи:
  Приложение → структурированный JSON в stdout → CloudWatch Logs (или Datadog/Loki)
  Хранение: 30 дней для INFO, 90 дней для ERROR
  Алерты: > 10 ERROR логов за 5 минут → PagerDuty alert

Метрики:
  Приложение → библиотека Prometheus client → эндпоинт /metrics
  Prometheus scrape'ит /metrics каждые 15 секунд
  Дашборды Grafana: RPS, задержка p95/p99, частота ошибок, размер пула БД
  Алерты: p99 задержка > 1с в течение 5 минут → Slack alert
          частота ошибок > 1% в течение 2 минут → PagerDuty alert

Трейсы:
  Приложение → OpenTelemetry SDK → Jaeger / Tempo / Datadog APM
  Sample rate (частота выборки): 100% на staging, 10% на prod (полная выборка дорога)
  Использовать для: отладки медленных запросов, понимания межсервисных зависимостей

Uptime-мониторинг:
  UptimeRobot / Checkly делает ping /health/live каждые 60 секунд из 3 регионов
  Алерт при 2 последовательных неудачах: SMS + Slack

Health checks:
  Kubernetes liveness: /health/live (только отзывчивость процесса)
  Kubernetes readiness: /health/ready (проверяет соединение с БД + Redis)
```

## Типичные ошибки на интервью

- **"Мониторинг и observability — одно и то же"** — мониторинг — это практика наблюдения за заранее определёнными метриками; observability — это свойство системы, позволяющее обнаружить неизвестные режимы отказа. Система может быть сильно мониторируемой, но плохо наблюдаемой.

- **"Мы всё логируем через console.log"** — сигнал незнакомства с production-логированием. Проблемы: console.log синхронный (кратко блокирует event loop); создаёт неструктурированные строки; не настраивается для фильтрации по уровню; не включает метаданные (имя сервиса, request ID, user ID). Используйте структурированный логгер (winston, pino).

- **Не знать разницу между liveness и readiness пробами** — очень распространённый вопрос на собеседованиях. Их путаница ведёт к реальным операционным проблемам: readiness probe, проверяющая внешние зависимости, вызывает перезапуски pod'ов при недоступности БД (вместо простого убирания pod из load balancer).

- **"Наш SLA — 99.99%"** — интервьюеры спросят: "Каков ваш error budget? Как вы его отслеживаете?" Если не можете ответить, это сигнал, что вы реально не работаете на таком уровне. Знайте, сколько минут допустимого простоя соответствует вашему SLA.

- **Путать SLI, SLO и SLA** — очень распространённая путаница на собеседованиях, особенно на senior/staff позиции. Запомните: SLI = измерение, SLO = внутренняя цель, SLA = контракт с клиентом. SLO строже SLA для создания буфера безопасности.

- **"Мы используем среднюю задержку как метрику"** — среднее маскирует выбросы. p99 задержка 3 секунды означает, что 1% ваших пользователей ждёт 3 секунды, но среднее может выглядеть как 200мс, потому что 99% запросов быстрые. Всегда мониторьте перцентили.

- **"У нас есть /health эндпоинт, который просто возвращает 200"** — нормально для uptime-мониторинга, но недостаточно для Kubernetes liveness/readiness. Процесс может возвращать 200, находясь в деградированном состоянии (БД недоступна, но возвращаются кэшированные данные). Readiness probe должна реально проверять доступность зависимостей.
