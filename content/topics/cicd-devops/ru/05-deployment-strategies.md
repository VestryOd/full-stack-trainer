# Deployment Strategies

## Почему стратегия деплоя важна

Деплой новой версии кода на продакшн — самый рискованный момент в жизненном цикле доставки программного обеспечения. Если что-то пойдёт не так — баг прошёл через тестирование, деградация производительности, сломанная сторонняя интеграция — пользователи затронуты немедленно. Стратегии деплоя — это техники для управления тем, *как* новая версия кода заменяет старую, с целями:

1. **Минимизация простоя** — пользователи воспринимают сервис как непрерывно доступный
2. **Ограничение blast radius** (зоны поражения) — если в новой версии есть баг, затрагивается только часть пользователей до его обнаружения
3. **Быстрый откат** — если что-то не так, можно быстро вернуться к предыдущему состоянию

Каждая стратегия делает разные компромиссы между сложностью, стоимостью ресурсов и риском.

## Zero-downtime deployment (деплой без простоя)

**Zero-downtime deployment** (развёртывание без простоя) — это общее требование: деплой не должен делать сервис недоступным для пользователей — даже на секунду. Звучит очевидно, но достичь этого нетривиально.

Наивный подход — остановить старую версию, запустить новую — создаёт разрыв:

```txt
Наивный деплой (с простоем):
  t=0с    Старая версия: работает, обслуживает трафик
  t=10с   ОСТАНОВКА старой версии   ← пользователи получают "connection refused" или 503
  t=25с   ЗАПУСК новой версии       ← 15 секунд простоя
  t=30с   Новая версия: работает, обслуживает трафик
```

Все стратегии ниже — это вариации на тему того, как устранить этот разрыв.

Предварительное условие для zero-downtime deployment: **приложение должно поддерживать graceful shutdown** (корректное завершение) — когда оно получает сигнал `SIGTERM` (который оркестратор посылает перед остановкой процесса), оно должно:
1. Перестать принимать новые соединения
2. Завершить обработку in-flight запросов (тех, что уже в процессе)
3. Корректно закрыть соединения с базой данных
4. Завершиться с кодом 0

Без graceful shutdown даже самая сложная стратегия деплоя будет дропать запросы в момент остановки контейнера.

## Rolling deployment (скользящее обновление)

**Rolling deployment** (также называемый *rolling update* — скользящее обновление) заменяет экземпляры старой версии экземплярами новой **постепенно** — по одному или по несколько — пока сервис продолжает работать.

```txt
Сервис имеет 4 экземпляра (pod/контейнера/VM) версии v1:
  [v1] [v1] [v1] [v1]   ← все обслуживают трафик

Шаг 1: заменяем один экземпляр
  [v2] [v1] [v1] [v1]   ← v2 стартует, v1 дрейнируется и останавливается

Шаг 2: заменяем ещё один
  [v2] [v2] [v1] [v1]

Шаг 3:
  [v2] [v2] [v2] [v1]

Шаг 4: завершено
  [v2] [v2] [v2] [v2]   ← все экземпляры работают на v2
```

**Как происходит замена одного экземпляра (zero-downtime на уровне экземпляра):**

```txt
1. Новый экземпляр v2 стартует и проходит health check (проверку здоровья)
2. Load balancer / прокси добавляет v2 в пул маршрутизации
3. Load balancer дрейнирует v1 — перестаёт отправлять новые запросы на него,
   ждёт завершения in-flight запросов (drain timeout: обычно 30–60с)
4. Экземпляр v1 останавливается после завершения дрейна
```

**Компромиссы:**

```txt
Преимущества:
  + Просто реализовать — встроено в Kubernetes, ECS, большинство PaaS-платформ
  + Не нужна дополнительная инфраструктура — используются существующие экземпляры
  + Низкая стоимость ресурсов (по сравнению с blue-green)

Недостатки:
  - Во время развёртывания v1 и v2 работают одновременно
    → запросы могут обслуживаться любой версией — возможна несогласованность
  - Если в v2 есть баг, одни пользователи видят его, другие нет
    (процент растёт по мере продвижения rollout)
  - Откат требует ещё одного rolling update в обратном направлении (медленно)
  - Миграции БД должны быть обратно совместимы с обеими v1 и v2
    (обе работают одновременно во время rollout)
```

Конфигурация rolling deployment в Kubernetes:

```yaml
# kubernetes deployment.yaml
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # разрешить максимум 1 дополнительный pod сверх желаемого числа
      maxUnavailable: 0  # никогда не опускаться ниже желаемого числа (zero-downtime)
  template:
    spec:
      containers:
        - name: app
          image: my-app:v2
          readinessProbe:         # pod добавляется к load balancer ТОЛЬКО после прохождения этой проверки
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 5
            periodSeconds: 5
```

`readinessProbe` — это то, что делает rolling deployment zero-downtime: pod получает трафик только после прохождения пробы, поэтому глючный v2, не прошедший health check, никогда не получит реальный пользовательский трафик — rollout останавливается и срабатывают оповещения.

## Blue-green deployment

**Blue-green deployment** поддерживает **два идентичных production-окружения** — называемых "blue" (синий) и "green" (зелёный) — и мгновенно переключает трафик между ними.

```txt
Текущее состояние: "blue" живёт (обслуживает весь трафик)
  [blue: v1, v1, v1, v1]  ←── 100% пользовательского трафика
  [green: простаивает / v1] ← в резерве (или отсутствует)

Деплой v2:
  [blue: v1, v1, v1, v1]  ← всё ещё живой, всё ещё обслуживает трафик
  [green: v2, v2, v2, v2] ← v2 задеплоен и тестируется здесь (реального трафика нет)

Переключение: обновляем правило load balancer / DNS, указывая на green
  [blue: v1, v1, v1, v1]  ← теперь простаивает (оставлен для возможного отката)
  [green: v2, v2, v2, v2] ←── 100% пользовательского трафика

Если в v2 есть баг — откат за секунды:
  [blue: v1, v1, v1, v1]  ←── переключаем обратно — 100% трафика мгновенно на v1
  [green: v2, v2, v2, v2] ← снова простаивает
```

Переключение трафика может быть изменением DNS, изменением правила load balancer или перезагрузкой конфигурации reverse proxy — и может произойти менее чем за секунду.

**Компромиссы:**

```txt
Преимущества:
  + Мгновенный откат — просто переключить load balancer обратно (секунды, не минуты)
  + Нет смешанного трафика версий — в каждый момент времени все пользователи
    видят одну и ту же версию
  + Полное окружение v2 можно протестировать изолированно перед переключением
    (smoke-тесты, нагрузочные тесты, подтверждение QA)

Недостатки:
  - Удвоенная стоимость инфраструктуры — два полных production-окружения
    должны существовать одновременно
  - Миграции БД: если в v2 есть изменения схемы, v1 должен работать с новой схемой
    в период до переключения (нужны обратно совместимые миграции)
  - Переключение через DNS имеет задержку распространения (TTL) — не по-настоящему мгновенно
    на уровне DNS; используйте переключение через load balancer, чтобы избежать этого
```

Blue-green с reverse proxy (NGINX или аналог):

```nginx
# Переключение путём обновления upstream и перезагрузки NGINX (zero-downtime reload)
upstream app {
    server green:3000;   # было: server blue:3000;
}
```

Или через облачный load balancer (AWS ALB переключение target group):

```bash
# AWS: переключить правило listener ALB на green target group
aws elbv2 modify-listener \
  --listener-arn arn:aws:elasticloadbalancing:... \
  --default-actions Type=forward,TargetGroupArn=<green-target-group-arn>
```

## Canary release

**Canary release** (канареечный релиз — назван в честь "канарейки в угольной шахте": канареек запускали в шахты для обнаружения токсичного газа до спуска людей) отправляет **небольшой процент реального пользовательского трафика** на новую версию, пока большинство по-прежнему попадает на старую. Новая версия наблюдается на предмет ошибок и деградации производительности, прежде чем трафик постепенно увеличивается до 100%.

```txt
Фаза 1: 5% канарейки
  [v2] ←── 5% трафика ("канарейка")
  [v1] [v1] [v1] [v1] ←── 95% трафика

  Мониторинг: частота ошибок, задержка, бизнес-метрики
  Если всё в порядке → увеличиваем

Фаза 2: 20% канарейки
  [v2] [v2] ←── 20% трафика
  [v1] [v1] [v1] [v1] ←── 80% трафика

Фаза 3: 50%
  ...

Фаза 4: 100% (rollout завершён)
  [v2] [v2] [v2] [v2] ←── 100% трафика
  (v1 выведен из эксплуатации)
```

Ключевое отличие от rolling deployment: при rolling update экземпляры заменяются последовательно, пока все не обновятся — нет концепции "отправить 5% на новую версию". В canary вы специально управляете *разделением трафика* (по весу в load balancer, а не по числу экземпляров).

**Где реализуется canary:**

- **Взвешенная маршрутизация load balancer** — AWS ALB weighted target groups, NGINX `weight`, Kubernetes Gateway API
- **Service mesh** (Istio, Linkerd) — детальный контроль трафика на сетевом уровне без изменения приложения
- **На уровне платформы** — Vercel (traffic splitting), AWS CodeDeploy с конфигурацией canary

```yaml
# Kubernetes Gateway API — взвешенное разделение трафика
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
spec:
  rules:
    - backendRefs:
        - name: app-v1
          port: 3000
          weight: 95
        - name: app-v2
          port: 3000
          weight: 5      # 5% канарейки
```

**Компромиссы:**

```txt
Преимущества:
  + Реальный пользовательский трафик тестирует новую версию — ловит баги,
    которые пропустил staging
  + Малый blast radius — баг в v2 изначально затрагивает только 5% пользователей
  + Постепенный rollout даёт время обнаружить деградацию производительности,
    которая появляется только под реальной нагрузкой

Недостатки:
  - Обе версии работают одновременно (те же ограничения миграций БД, что при rolling)
  - Требует load balancer с поддержкой взвешенной маршрутизации
  - Более сложная настройка observability — нужно отслеживать метрики по версиям,
    не только по сервису
  - Анализ канарейки (решение об увеличении трафика) может быть автоматизирован
    (Argo Rollouts, Flagger) или ручным — ручной подвержен ошибкам под давлением времени
```

## Feature flags

**Feature flag** (флаг фичи — также называемый *feature toggle* или *feature switch*) — это механизм в коде приложения, который включает или отключает функциональность во время выполнения — без деплоя нового кода.

**Feature flags — НЕ стратегия деплоя.** Это техника на уровне кода, которая дополняет стратегии деплоя. Различие:

```txt
Стратегия деплоя  = контролирует КАК новый код попадает в production-окружение
                    (rolling / blue-green / canary)

Feature flag      = контролирует АКТИВНА ЛИ функциональность в уже задеплоенном коде
                    для конкретного пользователя
                    (код уже в production — флаг управляет видимостью)
```

Простая реализация:

```ts
// Feature flag проверяется во время выполнения
const ENABLE_NEW_CHECKOUT = process.env.ENABLE_NEW_CHECKOUT === 'true';

app.get('/checkout', (req, res) => {
  if (ENABLE_NEW_CHECKOUT) {
    return newCheckoutHandler(req, res);
  }
  return legacyCheckoutHandler(req, res);
});
```

**Почему feature flags мощны в сочетании с CI/CD:**

```txt
Без feature flags:
  Feature-ветка → PR → мёрж когда готово → деплой → фича живая
  Проблема: долгоживущие feature-ветки → integration hell

С feature flags:
  Код фичи смёржен в main за флагом (флаг = off)
  → деплои происходят в штатном режиме, код за флагом инертен
  → QA/тестирование на продакшне с флагом = on только для внутренних пользователей
  → постепенный rollout: включить для 1% → 10% → 50% → 100%
  → мгновенный откат: выключить флаг (новый деплой не нужен)
```

**Сервисы feature flags** (мощнее, чем переменные окружения):

```ts
// Использование сервиса feature flags (LaunchDarkly, GrowthBook, Unleash и т.д.)
const variation = await flagClient.variation('new-checkout', user, false);

if (variation) {
  return newCheckoutHandler(req, res);
}
return legacyCheckoutHandler(req, res);
```

Такие сервисы позволяют:
- **Таргетинг пользователей** — включить для конкретных пользователей, групп или процента
- **Постепенный rollout** — начать с 1%, увеличивать со временем
- **Kill switch** — мгновенно отключить для всех пользователей без деплоя
- **A/B-тестирование** — измерить бизнес-влияние новой фичи до полного rollout

**Критическое различие с canary release:**

```txt
Canary release:  разделение трафика на уровне инфраструктуры (load balancer)
                 → разные *серверы* обрабатывают запросы
                 → и v1-код, и v2-код задеплоены

Feature flag:    разделение трафика на уровне приложения (внутри кода)
                 → *тот же сервер* обрабатывает запрос, но выполняет
                   разные ветки кода на основе значения флага
                 → задеплоена только одна версия кода
```

## Стратегии отката

Ни одна стратегия деплоя не устраняет возможность того, что что-то пойдёт не так. Каждый план деплоя должен включать путь отката.

### Откат при rolling deployment

Rolling откат = ещё один rolling update, но с деплоем предыдущего тега образа:

```bash
# Kubernetes: откат к предыдущему deployment
kubectl rollout undo deployment/my-app

# Или к конкретной ревизии
kubectl rollout undo deployment/my-app --to-revision=3

# Проверить статус rollout
kubectl rollout status deployment/my-app
```

Недостаток: откат занимает столько же времени, сколько и исходный rollout. Если полный rollout занимает 5 минут, откат тоже займёт 5 минут.

### Откат при blue-green deployment

```bash
# Переключить load balancer обратно на blue (предыдущая версия, которая всё ещё работает)
# Занимает секунды — blue никогда не был уничтожен
aws elbv2 modify-listener --default-actions Type=forward,TargetGroupArn=<blue-arn>
```

Вот почему двойная стоимость инфраструктуры blue-green часто оправдана: откат мгновенный.

### Откат при canary release

```yaml
# Установить вес канарейки на 0 — весь трафик обратно на стабильную версию
# (используя Argo Rollouts CRD — Custom Resource Definition)
kubectl argo rollouts set weight my-app 0
# Или просто прервать rollout
kubectl argo rollouts abort my-app
```

### Миграции БД и откат

Самая сложная часть отката — как правило, база данных. Если v2 выполнил миграцию, добавившую колонку и заполнившую её, откат к v1 означает, что v1 может не знать об этой колонке — или хуже: v2 удалил колонку, от которой зависит v1.

Безопасный паттерн — **expand-contract migration** (также называемый *parallel-change pattern* — паттерн параллельного изменения):

```txt
Фаза 1 — EXPAND (расширение, деплой v2a):
  v2a добавляет новую колонку как nullable (обратно совместимо)
  v1 и v2a работают одновременно — v1 игнорирует новую колонку, v2a пишет в неё

Фаза 2 — MIGRATE (миграция):
  Заполнить новую колонку данными для существующих строк (backfill)
  Выполняется в фоне, не как часть деплоя

Фаза 3 — CONTRACT (сжатие, деплой v2b):
  v2b делает колонку NOT NULL (теперь безопасно — во всех строках есть данные)
  Убрать ссылки на старую колонку из кода
  Старый v1-код больше не в production

Фаза 4 — CLEANUP (очистка, деплой v3):
  Удалить старую колонку, когда v2b достаточно долго работает стабильно
```

Это означает: **никогда не удаляйте колонку в той же миграции, что добавляет её замену**. Никогда не делайте колонку NOT NULL в первой миграции. Всегда деплойте в несколько фаз, чтобы откат на любой фазе был безопасным.

## Какую стратегию выбирать

```txt
Тип проекта / ограничение             → Рекомендуемая стратегия
───────────────────────────────────────────────────────────────────────
Простое приложение, небольшая         Rolling update
команда, ограниченная инфраструктура

Нетерпимость к смешанным версиям,    Blue-green
нужен мгновенный откат, можно
позволить 2x стоимость инфраструктуры

Высокий трафик, нужно валидировать    Canary release
новую версию на реальном трафике
перед полным rollout

Код готов, но фича не готова          Feature flag
для всех пользователей, или нужен
постепенный пользовательский rollout
без повторного деплоя
```

На практике большинство зрелых пайплайнов комбинируют стратегии: rolling или blue-green для деплоя на уровне инфраструктуры, feature flags для контроля на уровне приложения, и canary для высокорисковых релизов.

## Типичные ошибки на интервью

- **Путать canary release с feature flags** — это крайне распространено и специально проверяется на собеседованиях. Canary = разделение трафика на уровне инфраструктуры между двумя задеплоенными версиями. Feature flag = ветвление кода на уровне приложения внутри одной задеплоенной версии. Их можно использовать вместе, но это не одно и то же.

- **"Blue-green — это просто задеплоить дважды"** — суть blue-green в том, что старая версия продолжает работать до переключения, делая откат мгновенным. Если уничтожить blue после деплоя green, вы потеряли ключевое преимущество.

- **Забывать о миграциях БД при rolling deployments** — когда v1 и v2 работают одновременно, обе версии обращаются к одной БД. Если миграция v2 удалит колонку, которую читает v1 — v1 сломается. Все миграции при rolling update должны быть обратно совместимы (в одной волне деплоя — только аддитивные; удаления — в следующей волне, после исчезновения v1).

- **Считать `maxUnavailable: 1` и `maxSurge: 0` zero-downtime** — с этими настройками Kubernetes остановит один старый pod прежде чем запустить новый, то есть мощность временно упадёт. Для zero-downtime используйте `maxUnavailable: 0` (никогда не опускаться ниже желаемого числа) и `maxSurge: 1` (временно запустить один дополнительный pod).

- **"Откат — это просто переразвернуть старый Docker-образ"** — верно для stateless-кода, но если схема БД изменилась в "прямом" направлении и старый код не понимает новую схему, переразвёртывание старого кода сломает приложение. Откат должен быть спланирован как часть стратегии миграций.

- **Заявлять о zero-downtime без упоминания graceful shutdown** — zero-downtime deployment на уровне инфраструктуры сводится на нет, если приложение не обрабатывает `SIGTERM` корректно. Платформа ждёт завершения (drain) соединений — но если приложение немедленно выходит при получении `SIGTERM`, in-flight запросы дропаются. Код приложения должен "сотрудничать".
