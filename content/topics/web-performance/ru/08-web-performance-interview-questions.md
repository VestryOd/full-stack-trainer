## Группа 1: Core Web Vitals

**1. Что такое три Core Web Vitals и что именно измеряет каждый из них?**

**LCP (Largest Contentful Paint):** время рендера наибольшего видимого элемента во вьюпорте (обычно hero-изображение или `<h1>`). Измеряет *производительность загрузки*. Хорошо: ≤ 2.5с.

**CLS (Cumulative Layout Shift):** сумма всех неожиданных сдвигов макета за время жизни страницы, взвешенная по impact fraction × distance fraction. Измеряет *визуальную стабильность*. Хорошо: ≤ 0.1.

**INP (Interaction to Next Paint):** 75-й перцентиль задержек всех взаимодействий (клик, тап, нажатие клавиши) за время посещения страницы — от ввода пользователя до отрисовки следующего кадра. Заменил FID в марте 2024. Измеряет *отзывчивость*. Хорошо: ≤ 200мс.

---

**2. Почему Google заменил FID на INP, и что INP показывает такого, чего не показывал FID?**

FID (First Input Delay) измерял только *задержку до начала обработки* первого взаимодействия — не включая время обработки и отрисовку. Обработчик, блокирующий основной поток на 800мс после начала, показывал FID = 5мс, но ощущался полностью неотзывчивым. INP измеряет *полную длительность* от ввода до следующего кадра, для *всех* взаимодействий за время посещения (берёт 75-й перцентиль). Страница с медленными `onClick`-обработчиками на динамически загружаемом контенте получала хороший FID, но проваливала INP.

---

**3. Что вызывает CLS, и назовите три конкретных исправления.**

CLS вызывается элементами, сдвигающимися после первоначального макета.

**Изображения без размеров:**
```html
<!-- ❌ браузер не знает высоту → перерасчёт макета при загрузке -->
<img src="hero.jpg" alt="Hero">

<!-- ✅ зарезервировать место через aspect-ratio или явные размеры -->
<img src="hero.jpg" alt="Hero" width="1200" height="600">
<!-- или CSS: aspect-ratio: 2/1 -->
```

**Контент, вставляемый с задержкой:** рекламные блоки, куки-баннеры и динамические виджеты, вставляемые над существующим контентом, сдвигают его вниз. Исправление: зарезервировать место через `min-height` или skeleton-заглушки.

**Веб-шрифты, вызывающие FOUT/FOIT:** запасной шрифт имеет другие метрики, текст перерасчитывается при загрузке кастомного. Исправление: `font-display: optional` (никогда не перерасчитывает, может остаться запасным) или `font-display: swap` + `size-adjust` / `ascent-override` для совпадения метрик запасного шрифта.

---

**4. Что такое LCP и какие четыре элемента могут быть LCP-элементом?**

LCP — метка времени рендера наибольшего *содержательного* элемента, видимого во вьюпорте при первом просмотре страницы. Четыре допустимых типа элементов: `<img>`, `<image>` внутри SVG, `<video>` (постер), элементы с CSS background-image. Текстовые блоки (`<p>`, `<h1>` и т.д.) также могут быть LCP-элементом, если они наибольшие. Фоновые изображения через CSS учитываются только если загружаются по URL (не градиенты).

---

**5. Пройдите через полный чеклист оптимизации LCP.**

Время LCP = задержка сети + TTFB сервера + время загрузки ресурса + время рендера. Атакуем каждый:

**1. Снизить TTFB:** CDN, edge-кэширование, быстрый origin-сервер. TTFB > 600мс — главная причина медленного LCP.

**2. Убрать render-blocking ресурсы:** перенести некритичный CSS в `<link media="print">` или async-загрузку; defer для некритичного JS.

**3. Preload LCP-ресурс:**
```html
<link rel="preload" as="image" href="/hero.webp"
      imagesrcset="/hero-400.webp 400w, /hero-800.webp 800w"
      imagesizes="100vw">
```

**4. Современные форматы изображений:** WebP экономит ~30%, AVIF ~50% против JPEG при том же качестве.

**5. Не делать lazy-load для LCP-изображения:** `loading="lazy"` на hero-изображении задерживает LCP на задержку наблюдения за скроллом. Используйте вместо этого `fetchpriority="high"`.

**6. Избегать client-side rendering для LCP-элемента:** если LCP-элемент инжектируется JavaScript, браузер должен скачать, распарсить и выполнить JS до его отрисовки. SSR или SSG помещают элемент в начальный HTML.

## Типичные follow-up вопросы (Группа 1)

```txt
"Как измерять CWV в production, а не только в DevTools?" →
  Полевые данные: Chrome UX Report (CrUX) через PageSpeed Insights,
  отчёт CWV в Search Console, или библиотека web-vitals, отправляющая
  в вашу аналитику. Лабораторные данные (Lighthouse, DevTools) —
  синтетические и не отражают реальных условий (медленные Android,
  3G, фоновые вкладки). Полевые данные — то, что Google использует
  для ранжирования.

"Страница проходит Lighthouse, но проваливает CWV в Search Console — почему?" →
  Lighthouse — лабораторные данные на симулированном устройстве.
  CrUX — полевые данные реальных Chrome-пользователей на 75-м
  перцентиле. Реальные пользователи имеют медленные устройства,
  промахи кэша, фоновые вкладки и расширения. Lighthouse измеряет
  холодную загрузку; CWV включает soft-навигации и повторные посещения.

"Что такое TTFB и как он связан с LCP?" →
  TTFB (Time to First Byte) — время от запроса до первого байта
  ответа. Это суб-метрика LCP — медленный сервер задерживает всё
  последующее. TTFB > 600мс почти гарантированно сделает LCP > 2.5с
  даже при идеальной frontend-оптимизации.
```

## Группа 2: Загрузка ресурсов

**6. Что такое critical rendering path и какие ресурсы его блокируют?**

Critical rendering path — последовательность шагов, которые браузер должен завершить до отрисовки первого пикселя: DNS → TCP → TLS → HTTP-запрос → парсинг HTML → построение DOM → построение CSSOM → Render Tree → Layout → Paint.

**Render-blocking ресурсы:** CSS-файлы в `<head>` (браузер не рисует до построения CSSOM), синхронные теги `<script>` без `defer` или `async` (останавливают парсинг HTML), `@import` в CSS (создаёт дополнительные последовательные запросы). Изображения, шрифты и async-скрипты не блокируют первый рендер.

---

**7. `defer` vs `async` vs модульные скрипты — в чём разница?**

```html
<!-- async: загружается параллельно, выполняется сразу при готовности
     (останавливает парсинг HTML). Порядок выполнения не гарантирован. -->
<script async src="analytics.js"></script>

<!-- defer: загружается параллельно, выполняется ПОСЛЕ полного парсинга HTML,
     В ПОРЯДКЕ. Безопасен для скриптов, зависящих от DOM. -->
<script defer src="app.js"></script>

<!-- type="module": всегда defer по умолчанию. Выполняется после парсинга
     HTML, поддерживает import/export, строгий режим, собственный scope. -->
<script type="module" src="main.js"></script>
```

Правило: `defer` для скриптов, нуждающихся в DOM; `async` для независимых скриптов (аналитика, реклама), где порядок неважен; `type="module"` для ESM-бандлов — они deferred автоматически.

---

**8. Что делает `rel="preload"` и когда использовать его против `rel="prefetch"`?**

`preload`: говорит браузеру "этот ресурс скоро понадобится — начни загружать его сейчас, с высоким приоритетом." Используется для ресурсов, обнаруживаемых браузером поздно в загрузке страницы (hero-изображение в CSS, шрифт above-the-fold, критичный скрипт). Не выполняет ресурс — только загружает и кэширует.

`prefetch`: сигнализирует "пользователь может перейти на страницу, которой нужен этот ресурс." Фоновая загрузка с низким приоритетом, хранится в HTTP-кэше. Используется для ресурсов следующей навигации.

```html
<!-- preload: загрузить СЕЙЧАС, высокий приоритет, для текущей страницы -->
<link rel="preload" as="font" href="/fonts/inter.woff2" crossorigin>
<link rel="preload" as="image" href="/hero.webp">

<!-- prefetch: загрузить СКОРО, низкий приоритет, для следующей навигации -->
<link rel="prefetch" href="/checkout.js">
```

Злоупотребление `preload` для некритичных ресурсов тратит пропускную способность и конкурирует с критичными. Браузер также выдаст предупреждение: "preload был найден, но не использован в течение 3 секунд".

---

**9. Что такое connection waterfall и как диагностировать в DevTools?**

Connection waterfall — цепочка последовательных сетевых запросов, где каждый не может начаться до завершения предыдущего. Классический пример: HTML → JS-бандл → API-вызов → ещё один API-вызов. На панели Network в DevTools ищите длинные горизонтальные полосы в виде лесенки — каждый запрос начинается только после получения ответа на предыдущий.

Исправления: (1) resource hints (`preconnect`, `preload`) для раннего старта загрузок; (2) объединение API-вызовов на сервере (BFF-паттерн); (3) HTTP/2 или HTTP/3 мультиплексирование — несколько запросов по одному соединению; (4) инлайнинг критичного CSS для устранения отдельного CSS-запроса на critical path.

---

**10. Что такое HTTP/2 мультиплексирование и какую проблему производительности оно решает?**

HTTP/1.1 позволяет только один активный запрос на TCP-соединение. Браузеры компенсируют это, открывая 6 соединений на домен — но каждое имеет overhead TCP slow-start. HTTP/2 отправляет несколько потоков по одному соединению одновременно без head-of-line блокировки на уровне HTTP. Это делает: domain sharding (разбиение ресурсов на `static1.cdn.com`, `static2.cdn.com`) — анти-паттерном, а объединение всего в один огромный файл — менее необходимым: много маленьких файлов отлично работает на HTTP/2. HTTP/3 добавляет исправление HOL-блокировки на уровне соединения через QUIC (на базе UDP).

## Типичные follow-up вопросы (Группа 2)

```txt
"Когда preload шрифта УХУДШАЕТ производительность?" →
  Если шрифт не используется above-the-fold, preload конкурирует
  с LCP-ресурсами (hero-изображение, критичный CSS) за пропускную
  способность. Если файл шрифта большой (variable fonts могут быть
  500КБ+), preload полного файла для нескольких символов расточителен.
  Лучше: preload только подмножества, используемого above-the-fold,
  использовать subsetting-инструменты (pyftsubset, Fonttools).

"Какой порядок приоритетов у resource hints?" →
  preconnect > preload > prefetch > dns-prefetch.
  preconnect открывает TCP+TLS соединение (не только DNS).
  Используйте preconnect только для 2-3 критичных origins —
  каждое удерживаемое соединение потребляет ресурсы браузера.

"Lighthouse говорит 'eliminate render-blocking resources'. CSS уже
в <head> — что делать?" →
  Разбить CSS: извлечь above-the-fold стили и инлайнировать
  в <style> в <head>. Остальное загружать через <link media="print">,
  который браузер загружает без блокировки, затем переключить
  на media="all" через JS onload. Или использовать CSS-in-JS,
  который автоматически извлекает критичный CSS на компонент
  (Next.js делает это).
```

## Группа 3: Rendering Pipeline

**11. Опишите rendering pipeline браузера от HTML-байтов до пикселей.**

```txt
Байты → Символы → Токены → Узлы → DOM
                                       ↘
CSS Байты → Символы → Токены → Узлы → CSSOM
                                       ↙
                          Render Tree (только видимые узлы)
                                ↓
                            Layout (геометрия — x, y, ширина, высота)
                                ↓
                              Paint (заполнение пикселей в слои)
                                ↓
                           Composite (GPU объединяет слои → экран)
```

Ключевые моменты: (1) JS блокирует построение DOM при обнаружении (без `defer`/`async`); (2) CSS блокирует рендер (CSSOM должен быть построен до Render Tree); (3) Layout и Paint дорогие — избегайте их в циклах; (4) Composite — дешевле всего: только свойства на GPU-слоях (`transform`, `opacity`) могут анимироваться без Layout/Paint.

---

**12. Что такое layout thrashing и как это исправить?**

Layout thrashing (принудительный синхронный макет): чтение layout-свойства (`offsetHeight`, `getBoundingClientRect()`) заставляет браузер сбросить ожидающие изменения стилей и синхронно пересчитать макет. Затем запись стиля инвалидирует его при следующем чтении. В цикле это вызывает N пересчётов макета за кадр вместо одного.

```ts
// ❌ Layout thrashing — чтение форсирует макет, запись инвалидирует, цикл
elements.forEach(el => {
  const height = el.offsetHeight;       // форсирует сброс макета
  el.style.height = height * 2 + 'px'; // инвалидирует макет
});

// ✅ Сначала пакетно читать, затем пакетно писать
const heights = elements.map(el => el.offsetHeight); // один сброс макета
elements.forEach((el, i) => {
  el.style.height = heights[i] * 2 + 'px';           // пакетная запись
});
// Или используйте requestAnimationFrame для пакетирования в следующем кадре
```

---

**13. Какие CSS-свойства дёшевы для анимации, а какие — дороги?**

**Дёшевые (только compositor, без Layout или Paint):**
- `transform` (translate, rotate, scale)
- `opacity`
- `filter` (на composited-слоях)
- `will-change: transform` (переводит элемент на отдельный слой)

**Дорогие (триггерят Layout → Paint → Composite):**
- `width`, `height`, `margin`, `padding`, `top`, `left` — триггерят Layout
- `background-color`, `color`, `border-color`, `box-shadow` — триггерят Paint

Правило: анимируйте `transform: translateX()` вместо `left`, `transform: scaleX()` вместо `width`. GPU обрабатывает transform/opacity нативно; Layout и Paint работают на CPU.

---

**14. Что такое `requestAnimationFrame` и когда использовать его вместо `setTimeout`?**

`requestAnimationFrame(cb)` планирует `cb` перед следующей перерисовкой браузера, синхронизированной с частотой обновления дисплея (обычно 60fps ≈ 16.7мс). Преимущества: (1) не запускается когда вкладка скрыта — экономит CPU/батарею; (2) синхронизирован с дисплеем — нет разрывов или потраченных кадров; (3) браузер может объединить его с другой визуальной работой в одном кадре.

`setTimeout(fn, 0)` может сработать в любое время — может разделиться между кадрами (визуальные дёргания) или сработать во время фазы layout. Используйте `requestAnimationFrame` для: всех визуальных анимаций, пакетной записи DOM после чтения layout-свойств, обновлений на основе скролла. Используйте `setTimeout` для несвязанной с визуалом отложенной работы.

---

**15. Что такое свойство `will-change` и каковы риски его чрезмерного использования?**

`will-change` подсказывает браузеру, что элемент будет анимироваться, побуждая его перевести элемент на отдельный compositor-слой *до* начала анимации (избегая стоимости перевода во время анимации). Для анимаций, запускаемых по hover или через JS, это убирает дёрганье первого кадра.

Риск: каждый composited-слой потребляет GPU-память. Использование `will-change: transform` на сотнях элементов одновременно может исчерпать GPU-память, заставив браузер убрать слои — хуже, чем без него. Применяйте только к элементам, которые *реально* будут анимироваться, и убирайте после:

```ts
el.addEventListener('mouseenter', () => { el.style.willChange = 'transform'; });
el.addEventListener('animationend', () => { el.style.willChange = 'auto'; });
```

## Типичные follow-up вопросы (Группа 3)

```txt
"Что такое paint flashing в DevTools и как включить?" →
  DevTools → вкладка Rendering → чекбокс 'Paint flashing'. Области,
  перерисовываемые в каждом кадре, мигают зелёным. Полезно для поиска
  компонентов, перерисовывающихся при скролле (должно быть ноль для
  плавного скролла). Если вся страница мигает при скролле — что-то
  вызывает глобальную перерисовку, часто это fixed-элемент с box-shadow
  или некомпозированная анимация.

"Что такое stacking context и почему важен для производительности?" →
  Stacking context — 3D-пространство для z-index compositing.
  Создаётся: position + z-index, opacity < 1, transform, will-change,
  filter. Каждый stacking context отрисовывается как единица. Слишком
  много вложенных stacking context не позволяют браузеру оптимизировать
  объединение слоёв. Неожиданные stacking context объясняют "почему
  этот элемент появляется поверх всего?".

"Когда CSS-анимация превосходит JS-анимацию?" →
  CSS-анимации на compositor-only свойствах (transform, opacity)
  работают в compositor thread — блокировка JS-потока (long task)
  не роняет их кадры. JS-анимации через requestAnimationFrame работают
  в main thread — long task роняет и анимацию, и кадр. Для
  fire-and-forget анимаций на transform/opacity: CSS безопаснее.
  Для сложной физики или интерактивных анимаций: JS + requestAnimationFrame.
```

## Группа 4: Стратегии кэширования

**16. Объясните разницу между `Cache-Control: max-age`, `no-cache` и `no-store`.**

**`max-age=N`:** кэшировать ответ N секунд. Браузер отдаёт из кэша без обращения к серверу. Используется для версионированных статических ресурсов (`bundle.abc123.js`).

**`no-cache`:** *всегда* ревалидировать с сервером перед использованием кэша. Браузер отправляет условный запрос (`If-None-Match` / `If-Modified-Since`). Сервер может ответить 304 Not Modified (без тела — быстро, но всё равно round-trip). Несмотря на название, `no-cache` *сохраняет* кэш.

**`no-store`:** никогда не сохранять ответ в кэше. Каждый запрос получает свежее с сервера. Для чувствительных данных (банковские сессии, личные медданные).

```txt
Статика с content hash:     Cache-Control: max-age=31536000, immutable
HTML-страницы (ревалидация): Cache-Control: no-cache
Чувствительные данные:       Cache-Control: no-store
API (кэш 1 минута):          Cache-Control: max-age=60, s-maxage=300
```

**17. Что такое ETags и как работает ревалидация кэша?**

ETag — токен, генерируемый сервером (хэш контента). Поток: (1) первый запрос — сервер отправляет `ETag: "abc123"` в ответе; (2) браузер хранит ответ с его ETag; (3) следующий запрос (после истечения max-age) — браузер отправляет `If-None-Match: "abc123"`; (4) сервер сравнивает ETag с текущим контентом: не изменился → `304 Not Modified` (без тела, экономия трафика); изменился → `200` с новым контентом и новым ETag.

`Last-Modified` / `If-Modified-Since` — старая альтернатива с временными метками вместо хэшей. ETags надёжнее (точность временных меток, дрейф часов на мультисерверных установках).

**18. Что такое Service Worker и как он обеспечивает offline-first опыт?**

Service Worker — JavaScript-файл, работающий в фоновом потоке (отдельно от страницы), перехватывающий сетевые запросы через событие `fetch`. Может: кэшировать ответы в Cache API, отдавать кэшированный контент без сети, реализовывать стратегии stale-while-revalidate, отправлять push-уведомления.

```ts
// service-worker.ts
const CACHE_NAME = 'v1';
const PRECACHE = ['/shell.html', '/app.js', '/styles.css'];

self.addEventListener('install', (event: ExtendableEvent) => {
  event.waitUntil(
    caches.open(CACHE_NAME).then(cache => cache.addAll(PRECACHE))
  );
});

self.addEventListener('fetch', (event: FetchEvent) => {
  event.respondWith(
    caches.match(event.request).then(cached => {
      // Stale-while-revalidate: сразу отдать кэш, обновить в фоне
      const networkFetch = fetch(event.request).then(response => {
        caches.open(CACHE_NAME).then(c => c.put(event.request, response.clone()));
        return response;
      });
      return cached ?? networkFetch;
    })
  );
});
```

**19. Что такое stale-while-revalidate и когда это уместно, а когда нет?**

Stale-while-revalidate (SWR): немедленно отдать кэшированный (устаревший) ответ, одновременно загружая свежую версию в фоне для следующего запроса. Нулевая задержка для пользователя; всегда в конечном счёте актуален.

```
Cache-Control: max-age=60, stale-while-revalidate=3600
```
Означает: отдавать из кэша 60с (свежий); после 60с — отдавать устаревший и ревалидировать в фоне; после 3660с — ревалидировать синхронно.

**Уместно:** контент, меняющийся, но не критично — статьи блога, списки товаров, данные дашборда. Небольшая устарелость допустима.

**Неуместно:** пользовательские real-time данные (баланс счёта, состояние корзины), отправка форм, состояние аутентификации. Отдать устаревшие личные данные не той сессии — это баг или проблема безопасности.

```txt
"В чём разница между браузерным кэшем и кэшем Service Worker?" →
  Браузерный кэш (HTTP cache): автоматический, управляется заголовками
  Cache-Control, браузером. Нельзя программно отдать устаревшее при
  сетевой ошибке.
  Кэш Service Worker (Cache API): программный, полностью под вашим
  контролем. Вы решаете, что кэшировать, когда обновлять и что
  отдавать при отсутствии сети. SW может перехватывать запросы,
  которые HTTP-кэш никогда не увидит (cross-origin ошибки, timeout).

"Пользователь жалуется, что не видит свежих данных после деплоя.
Без хард-рефреша. Почему?" →
  Устаревшая запись кэша с долгим max-age отдаёт старый ресурс.
  Решения: (1) content-based hashing в именах файлов — URL меняется
  при деплое, инвалидируя кэш; (2) для HTML: max-age=0 или no-cache
  чтобы точка входа всегда ревалидировалась; (3) Service Worker
  с skipWaiting() + clients.claim() для немедленной активации при обновлении.

"Чем CDN-кэш отличается от браузерного кэша?" →
  Браузерный: на устройстве пользователя, для конкретного пользователя.
  Cache-Control управляет обоими.
  CDN: общий для пользователей, на edge-сервере. s-maxage переопределяет
  max-age только для CDN. Заголовок Vary говорит CDN поддерживать
  отдельные записи кэша для разных значений заголовка
  (например, Vary: Accept-Encoding).
```

## Группа 5: Производительность JavaScript

**20. Что такое long task и как он влияет на INP?**

Long task — любое выполнение JavaScript в main thread дольше 50мс. Во время long task браузер не может обрабатывать ввод пользователя — клики и нажатия клавиш ставятся в очередь. Когда long task завершается, браузер обрабатывает ввод (возможно несколько сразу) и рисует следующий кадр. INP измеряет эту задержку. Long task в 200мс после клика кнопки = INP ~200мс.

Диагностика в DevTools: панель Performance → Long Tasks помечены красными треугольниками. `PerformanceObserver` для программного мониторинга:

```ts
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.duration > 50) {
      console.warn('Long task:', entry.duration, entry.attribution);
    }
  }
});
observer.observe({ type: 'longtask', buffered: true });
```

**21. Что такое code splitting и как он сокращает начальное время загрузки?**

Code splitting откладывает загрузку JavaScript, не нужного для начального отображения. Без splitting браузер загружает, парсит и выполняет весь бандл до интерактивности страницы — включая код маршрутов, которые пользователь не посещал.

```tsx
// Без code splitting: CheckoutPage.js включён в main бандл
import CheckoutPage from './CheckoutPage';

// С React.lazy + Suspense: CheckoutPage загружается только при навигации
const CheckoutPage = React.lazy(() => import('./CheckoutPage'));

function App() {
  return (
    <Suspense fallback={<Spinner />}>
      <Routes>
        <Route path="/checkout" element={<CheckoutPage />} />
      </Routes>
    </Suspense>
  );
}
```

Next.js делает route-based code splitting автоматически. Эффект: сокращение начального JS-бандла с 500КБ до 150КБ может уменьшить Time to Interactive на 2-3 секунды на медленном устройстве.

**22. Что такое tree shaking и что мешает ему работать?**

Tree shaking — устранение неиспользуемых экспортов из ES-модулей бандлером. Основывается на статическом анализе `import`/`export` — бандлер определяет на этапе сборки, какие экспорты никогда не импортируются.

**Что мешает tree shaking:**
1. CommonJS (`require`) — динамический, не поддаётся статическому анализу
2. Импорты с побочными эффектами: `import 'polyfill'` — бандлер не знает, безопасно ли удалять
3. Отсутствие `"sideEffects": false` в `package.json` — бандлер считает все файлы с побочными эффектами
4. Реэкспорт всего: `export * from './utils'` — препятствует индивидуальному tree shaking
5. Динамические импорты с переменными путями: `import(variable)` — бандлер не знает, какие модули используются

**23. В чём разница между `debounce` и `throttle` и когда что использовать?**

**Debounce:** откладывает выполнение до N миллисекунд после *последнего* вызова. Функция срабатывает только когда пользователь *прекратил* триггерить события.

**Throttle:** гарантирует срабатывание функции не чаще одного раза в N миллисекунд, независимо от количества вызовов.

```ts
// Debounce: поле поиска — ждать пока пользователь не перестанет печатать
const debouncedSearch = debounce((query: string) => {
  fetchResults(query);
}, 300); // срабатывает через 300мс после последнего нажатия

// Throttle: обработчик скролла — ограничить обновления до 60fps
const throttledScroll = throttle(() => {
  updateScrollProgress();
}, 16); // срабатывает не чаще раза в 16мс (~60fps)
```

Правило: **debounce** для дорогих операций, триггерируемых *окончанием* активности пользователя (поиск, валидация формы, resize). **Throttle** для операций, которые должны продолжаться *во время* активности пользователя, но не перегружать браузер (прогресс скролла, отслеживание позиции мыши, рисование на canvas).

```txt
"Добавили React.lazy() везде и INP ухудшился. Почему?" →
  Ленивая загрузка триггерит сетевой запрос + парсинг + выполнение
  при первой навигации на маршрут. Если пользователь кликает кнопку,

"Что такое Scheduler API и как он улучшает INP?" →
  scheduler.postTask() позволяет планировать работу по приоритетам:
  'user-blocking' (обработка ввода), 'user-visible' (рендер),
  'background' (аналитика, некритичное). Работа может уступать
  более приоритетным задачам в процессе. Это разбивает long tasks
  на меньшие части без ручного вставления setTimeout(0).
  Доступно в Chrome 94+; полифиллируется через MessageChannel.

"Как определить, что вызывает long task?" →

  отдаёт не только длительность, но полную атрибуцию — какие скрипты
```

**24. Опишите ваш workflow в DevTools для диагностики медленного взаимодействия со страницей.**

Пошагово:

1. **Открыть DevTools → панель Performance.** Установить CPU throttle 4x или 6x (симуляция среднего Android).

2. **Нажать "Record", выполнить взаимодействие** (клик кнопки, открытие дропдауна и т.д.), остановить запись.

3. **Найти взаимодействие на таймлайне.** Искать красный прямоугольник (long task) или проверить трек "Interactions". Полоса показывает длительность от ввода до отрисовки.

4. **Развернуть стек вызовов** под long task. Самая широкая полоса снизу — реальное узкое место: React-ре-рендер, сортировка, глубокое клонирование, синхронный XHR.

5. **Проверить события "Layout" и "Paint".** Если Layout занимает > 20мс — найти что его триггерит (style recalculation → layout shift → forced synchronous layout).

6. **Использовать вкладку "Bottom-Up"** для сортировки по суммарному времени — показывает реальную горячую функцию, а не инициатор.

7. **Исправить, перезаписать, сравнить.** Измерить до и после для подтверждения эффекта.

**25. Что такое библиотека `web-vitals` и как использовать её для отправки CWV в аналитику?**

`web-vitals` — официальная Google JS-библиотека, измеряющая CWV той же логикой, что использует Chrome для CrUX. Отдаёт полевые данные (замеры реальных пользователей), а не синтетические лабораторные.

```ts
import { onCLS, onINP, onLCP, onFCP, onTTFB } from 'web-vitals';

function sendToAnalytics({ name, value, rating, id }: Metric) {
  // rating: 'good' | 'needs-improvement' | 'poor'
  navigator.sendBeacon('/analytics', JSON.stringify({
    metric: name,
    value: Math.round(name === 'CLS' ? value * 1000 : value),
    rating,
    id,   // уникален для посещения страницы, для дедупликации
    url: location.href,
    userAgent: navigator.userAgent,
  }));
}

onCLS(sendToAnalytics);
onINP(sendToAnalytics);
onLCP(sendToAnalytics);
onFCP(sendToAnalytics);
onTTFB(sendToAnalytics);
```

Ключевая деталь: `onCLS` и `onINP` отдают *финальное* значение при уходе страницы (или переходе в фон). Вызывайте один раз за посещение — они накапливают значения. Используйте `id` для дедупликации, если одно посещение отправляет несколько отчётов.
