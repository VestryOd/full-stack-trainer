# System Design Fundamentals

## Что такое System Design

System Design — проектирование архитектуры системы.

---

На интервью оценивают:

```txt
умение принимать решения

понимание компромиссов

масштабирование

надежность
```

---

# Самая важная мысль

Нет:

```txt
идеальной архитектуры
```

---

Есть:

```txt
компромиссы
```

---

# Типичный вопрос

Спроектируй:

```txt
чат

Instagram

YouTube

URL Shortener
```

---

Интервьюер не ждет идеальный ответ.

---

Он хочет понять:

```txt
как ты думаешь
```

---

# Первое правило

Никогда сразу не рисовать архитектуру.

---

Сначала уточняем требования.

---

# Functional Requirements

Что система должна делать.

---

Например чат:

```txt
отправка сообщений

получение сообщений

групповые чаты
```

---

# Non Functional Requirements

Как система должна работать.

---

Например:

```txt
100ms latency

1 млн пользователей

99.99 uptime
```

---

Очень важная тема.

---

# Scale Estimation

Очень любят спрашивать.

---

Например:

```txt
1 млн пользователей
```

---

Из них:

```txt
100k активных ежедневно
```

---

# Зачем это нужно

Чтобы понять:

```txt
размер БД

нагрузку

трафик
```

---

# Основные строительные блоки

Практически любая система состоит из:

```txt
Client

API

Database

Cache

Queue

Storage
```

---

# Базовая архитектура

```txt
Client
 ↓
API
 ↓
Database
```

---

Почти все начинается именно так.

---

# Single Point Of Failure

Очень любят спрашивать.

---

Компонент:

```txt
без резервирования
```

---

Если он падает:

```txt
падает вся система
```

---

# Пример

```txt
1 Database
```

---

Без реплик.

---

# Availability

Доступность.

---

Например:

```txt
99.9%

99.99%

99.999%
```

---

# Что означает 99.9%

Примерно:

```txt
8.7 часов простоя в год
```

---

# Latency

Время ответа.

---

Например:

```txt
50ms
100ms
300ms
```

---

# Throughput

Количество операций.

---

Например:

```txt
1000 RPS
```

---

Requests Per Second.

---

# Vertical Scaling

Очень популярный вопрос.

---

Увеличиваем сервер.

---

```txt
8 GB RAM
 ↓
32 GB RAM
```

---

# Плюсы

Просто.

---

# Минусы

Есть предел.

---

# Horizontal Scaling

Добавляем серверы.

---

```txt
Server 1

Server 2

Server 3
```

---

# Плюсы

Почти бесконечный рост.

---

# Минусы

Появляется сложность.

---

# Stateless Services

Очень любят спрашивать.

---

Сервер не хранит состояние.

---

Любой запрос может попасть:

```txt
на любой сервер
```

---

# State

Например:

```txt
Session Memory
```

---

мешает масштабированию.

---

# Stateless

Поэтому часто используют:

```txt
JWT

Redis

Database
```

---

Для хранения состояния.

---

# Read Heavy Systems

Большинство веб-приложений.

---

Например:

```txt
Instagram

YouTube

News Sites
```

---

Чтений намного больше чем записей.

---

# Write Heavy Systems

Например:

```txt
Analytics

IoT

Logs
```

---

# Почему это важно

От этого зависит выбор:

```txt
Database

Cache

Queues
```

---

# Частый вопрос

Что важнее: Availability или Consistency?

Ответ:

Зависит от бизнеса.

---

Банк:

```txt
Consistency
```

---

Соцсеть:

```txt
Availability
```

---

# Interview Answer

System Design — это процесс проектирования масштабируемых, надежных и поддерживаемых систем. На интервью важно сначала уточнить требования, оценить нагрузку и только потом выбирать архитектурные решения.
